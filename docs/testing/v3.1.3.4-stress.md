## 必过场景

### 1. self-reload bootstrapper 触发(P0 关键场景)

**场景**:dev VPS 187.77.157.220 demo 部门 git HEAD 在 v3.1.3.2 老版 update.sh,GitHub origin/main = v3.1.3.4 新版 update.sh。

**步骤**:
```bash
cd /root/tg-monitor-demo
# 确认 HEAD 是老版
git rev-parse --short HEAD     # 不是 v3.1.3.4 commit
sha256sum update.sh            # local hash X
# 跑 update.sh
bash update.sh
```

**预期输出**:
- 输出含 `🔄 检测到 update.sh 自身有新版,先拉 + exec 重启避免 bash-cache bug...`
- 输出含 `✅ update.sh 已更新,exec 重启...`
- 之后正常跑升级流程,5.5b reload 段输出 `✅ Caddy 已 reload`
- 升完后 Caddy runtime config upstream = `tg-web-demo:5001`(显式名,不是占位符)
- `web /login` HTTP 200

### 2. 已是最新版 — bootstrapper no-op

**场景**:dev VPS demo 升级完 v3.1.3.4 之后再跑一次 update.sh。

```bash
cd /root/tg-monitor-demo && bash update.sh
```

**预期**:
- 不出现 `🔄 检测到 update.sh 自身有新版...` (REMOTE_HASH == LOCAL_HASH)
- 直接进 section 0 老流程
- section 1 `git rev-parse origin/main` SHA 相等 → short-circuit exit `当前已是最新版`

### 3. fleet fanout 升级 — 仍能正常工作

**场景**:从 v3.1.3.x 起跳的客户 VPS,中央台按「全网升级」按钮。

**预期**:
- agent.upgrade container_git 路径 → 新启 bash 跑 v3.1.3.4 update.sh
- bootstrapper 检测 hash 已是最新(刚 git reset 到 main)→ no-op,不再 exec
- 5.5b reload 段直接跑(因为 bash cache 的就是 v3.1.3.4)
- update_checker.py 默认 OFF 生效

### 4. 升级通知 feature flag

**场景 4a**:`.env` 没设 `UPDATE_CHECK_NOTIFY_ENABLED`(默认 OFF)。
- 预期:tg-web 容器日志含 `new version XXX detected but UPDATE_CHECK_NOTIFY_ENABLED=false, skip TG notice`
- 客户群**不**收到「有新版」通知

**场景 4b**:`.env` 设 `UPDATE_CHECK_NOTIFY_ENABLED=true`。
- 预期:客户群收到「有新版」通知(老行为,运维需要时打开)

### 5. bootstrapper 失败兜底(防 set -e 整段挂)

**场景**:模拟 git fetch 失败 / sha256sum 失败。

- `git fetch` 失败:`set +e` + `2>/dev/null` 包住 → 不抛异常 → REMOTE_HASH 为空 → bootstrapper 跳过
- `sha256sum` 失败(罕见):同上,LOCAL_HASH 为空 → bootstrapper 跳过
- `git checkout origin/main -- update.sh` 失败:打 warning「继续按老版本跑(可能漏新增段)」,主流程继续

**预期**:任何 bootstrapper 内部失败**不阻塞升级**,fall back 到老 update.sh 跑(跟 v3.1.3.3 之前行为一致)。

### 6. exec 防无限循环

**场景**:故意制造一个 update.sh 跟 origin/main 不一致的状态,且 git checkout 也失败的情况。

**预期**:
- 第一轮:`_UPDATE_SH_REENTRY=0`,bootstrapper 检测到 hash 不一致 + git checkout 成功 → exec 设 `_UPDATE_SH_REENTRY=1`
- 第二轮:`_UPDATE_SH_REENTRY=1`,bootstrapper 整段跳过,直接进 section 0
- **不无限循环** ✅

### 7. 5.5b reload 失败时不撒谎(从 v3.1.3.3 backport 验证)

**场景**:故意改坏 Caddyfile(语法错)让 reload 失败。

**预期**:
- update.sh 输出含 `⚠ Caddy reload 失败 exit=1`
- 输出含 Caddy 错误细节(tail 5 行)
- 输出含 `⚠ Caddyfile host 端已更新但 Caddy 进程仍跑老配置,请人工排查`
- **不出现** 假的 `✅ Caddy 已 reload`

## 回滚

升级失败:
```bash
cd /root/tg-monitor-<dept>
bash rollback.sh   # 走 update.sh 内置回滚到 OLD_SHA(rollback.sh 不动 update.sh,所以恢复后 bootstrapper 又会触发)
```

`rollback.sh` 把 git reset 回升级前 SHA,update.sh 也回老版本(没 bootstrapper)。下次客户跑 update.sh 又会被 bootstrapper 拉一次新版 → exec 重启 → 升级 — 等于自愈。
