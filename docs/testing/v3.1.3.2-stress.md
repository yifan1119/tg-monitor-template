# v3.1.3.2 暴力测试清单

**版本**:v3.1.3.2 · Caddyfile upstream 显式化(防共用 Caddy DNS 撞车)
**配套 ADR**:[0044](../adr/0044-v3.1.3.2-caddyfile-explicit-upstream.md)
**改动文件**:`Caddyfile` · `install.sh` · `update.sh`

---

## 必过场景

### 1. 新部署(install.sh)— 占位符替换

**前置**:dev VPS 全新,没装过任何 tg-monitor。

```bash
cd /tmp && git clone https://github.com/<repo>/tg-monitor && cd tg-monitor
bash install.sh demo-fresh 5101
```

**预期**:
- install.sh 输出含 `✅ Caddyfile upstream 已绑定到 tg-web-demo-fresh:5001`
- 安装目录 `/root/tg-monitor-demo-fresh/Caddyfile` 第 16 行(`reverse_proxy` 那行)= `reverse_proxy {$WEB_UPSTREAM:tg-web-demo-fresh:5001} {`
- **不含** `__COMPANY_NAME__` 字面
- **不含** `web:5001` 子串
- caddy 容器起来后 `docker exec tg-caddy-demo-fresh wget -qO- http://127.0.0.1:2019/config/` 看 upstream dial = `tg-web-demo-fresh:5001`

### 2. 老客户升级(update.sh)— 模板替换 + sed 兜底

**前置**:dev VPS 已部署一台 tg-monitor 跑 v3.1.3.1,其 `Caddyfile` 第 16 行还是老的 `{$WEB_UPSTREAM:web:5001}`。

```bash
cd /root/tg-monitor-demo-old && bash update.sh
```

**预期**:
- update.sh 输出含 `🔧 Caddyfile upstream 显式化(防共用 Caddy DNS 撞车,v3.1.3.2)...`
- 输出含 `✅ upstream 已绑定到 tg-web-demo-old:5001`
- 升级后 `Caddyfile` 第 16 行 = `reverse_proxy {$WEB_UPSTREAM:tg-web-demo-old:5001} {`
- `Caddyfile.bak.upstream.<timestamp>` 备份文件存在
- Caddy bind-mount inode 跟 host 文件一致(`docker exec tg-caddy-demo-old wc -c /etc/caddy/Caddyfile` = `wc -c /root/tg-monitor-demo-old/Caddyfile`)

### 3. 共用 Caddy 场景 — 第二个部门加进来不撞车

**前置**:同一 dev VPS,场景 1/2 的部门已起来,Caddy 在跑。

```bash
# 在同 VPS 加第二个部门(模拟客户买扩容)
bash install.sh demo-second 5102
bash enable_https.sh demo-second   # 把 demo-second 站点加到现有 Caddy
```

**预期**:
- 两个部门 web 后台分别访问 `https://demo-fresh.<IP>.nip.io/login` 跟 `https://demo-second.<IP>.nip.io/login`
- 各自 redirect 到自己后端的 login 页(看 cookie 或 page title 区分)
- 两个 URL 的 `/api/v1/metrics` 返回的 accounts 数据**完全不同**(各部门独立 DB)
- Caddy 运行时 config 两个 upstream 各自显式:`tg-web-demo-fresh:5001` 跟 `tg-web-demo-second:5001`

### 4. 已 skip-worktree 部门升级 — 不被覆盖

**前置**:dev VPS 上某部门历史上手改过 Caddyfile + 加了 `git update-index --skip-worktree Caddyfile`,改后第 16 行已是显式 `tg-web-XXX:5001`。

```bash
cd /root/tg-monitor-XXX
git ls-files -v Caddyfile     # 看到 'S Caddyfile' 表示 skip-worktree 还在
bash update.sh
```

**预期**:
- `git status` 全程不报 Caddyfile 改动
- update.sh 跑到 section 5.5b 时 `NEED_UPSTREAM_REWRITE=0`(两个 grep 都不命中:无 `__COMPANY_NAME__` + 无 `web:5001`)
- 不生成新备份
- 升级后 Caddyfile 完全没动

### 5. BSD sed / GNU sed 兼容

**dev**:
```bash
# Linux GNU sed
sed -e 's/__COMPANY_NAME__/test/g' -e '/reverse_proxy/s|web:5001|tg-web-test:5001|g' Caddyfile

# Mac BSD sed
sed -e 's/__COMPANY_NAME__/test/g' -e '/reverse_proxy/s|web:5001|tg-web-test:5001|g' Caddyfile
```

**预期**:两者输出**完全一致**,均把第 16 行的 upstream 替换为 `tg-web-test:5001`,enable_https.sh 加的子部门 site block 不动。

### 6. 中央台 fanout upgrade 全网一键升级

**前置**:全网 45 个部门(43 个 agent 通)都是 v3.1.3.1。

```bash
# 中央台后台 /admin/fleet 点「⬆ 全网升级到 main」
```

**预期**:
- 43 个 agent 通的部门 fanout 跑完 60 分钟内全部升 v3.1.3.2
- 升级后逐台 ping `/api/v1/inspect` 看 version
- 抽 3 台「同 IP 多部门」VPS(运维自留清单)用同款方法验证:不同子域 web 后台显示的 accounts **互相独立**
- 已手动修的部门(运维加过 `git update-index --skip-worktree Caddyfile` 的)升级后 Caddyfile 仍是手改版,不被冲掉

## 失败/回滚演练

### 失败 1:sed 输出为空(磁盘满 / IO 错)

模拟:`mount -t tmpfs -o size=1k tmpfs /tmp`,然后跑 update.sh。

**预期**:`/tmp/Caddyfile.upstream` 大小为 0 或 sed 失败 → update.sh 跳过替换,打印 `⚠ sed 输出为空,跳过` → 老 Caddyfile **不被破坏** → 升级继续(只是 upstream 没修)。

### 失败 2:Caddyfile 已被 skip-worktree 但仍有 `web:5001`

模拟:`git update-index --skip-worktree Caddyfile`,Caddyfile 第 16 行仍是老的 `web:5001`(运维改一半)。

**预期**:section 5.5b grep 命中 `web:5001` → sed 替换 → 老内容被替换成显式名 → 升级后跑得通(此场景下覆盖客户手改是正确行为,因为客户其实没改完)。

### 回滚

升级中或升级后发现问题:

```bash
cd /root/tg-monitor-<dept>
# 回滚 Caddyfile
ls Caddyfile.bak.upstream.*
cp Caddyfile.bak.upstream.<时间戳> Caddyfile   # ⚠ 这会换 inode → docker restart caddy
docker restart tg-caddy-<dept>
# 或者整版回滚
bash rollback.sh
```
