## 必过场景

### 1. 升级后 Caddy 进程真的切到新 routes(P0-1 修复验证)

**dev VPS**:187.77.157.220 上 demo 部门跑 v3.1.3.2(目前)或更老版。

**步骤**:
```bash
# 升级前:看 Caddy runtime config 的 upstream
docker exec tg-caddy-demo wget -qO- http://127.0.0.1:2019/config/ | python3 -m json.tool | grep -A1 upstreams

# 跑 update.sh(从老版升 v3.1.3.3)
cd /root/tg-monitor-demo && bash update.sh

# 升级后:再看 Caddy runtime config
docker exec tg-caddy-demo wget -qO- http://127.0.0.1:2019/config/ | python3 -m json.tool | grep -A1 upstreams
```

**预期**:
- 升级前 upstream 显示 `web:5001`(或老的模糊别名)
- 升级中 update.sh 输出含 `✅ Caddy 已 reload (tg-caddy-demo)`
- 升级后 upstream 显示 `tg-web-demo:5001`(显式)
- **不需要任何手动 docker exec caddy reload**

### 2. reload 失败时日志不撒谎(Codex P0 fix 验证)

**dev VPS**:故意制造 Caddyfile 语法错让 reload 失败。

**步骤**:
```bash
# 故意把 Caddyfile 改坏(造成 reload 时 Caddy 拒绝)
cp Caddyfile /tmp/Caddyfile.good
echo "this is invalid caddyfile syntax" >> Caddyfile
# 跑 update.sh(走 5.5b 路径)
bash update.sh
# 还原
cat /tmp/Caddyfile.good > Caddyfile
docker restart tg-caddy-demo
```

**预期**:
- update.sh 输出**含** `⚠ Caddy reload 失败 exit=非0`
- 输出**含** `⚠ Caddyfile host 端已更新但 Caddy 进程仍跑老配置,请人工排查`
- **不出现** 假"✅ Caddy 已 reload"

### 3. 中央台 fanout 同 IP 部门真的串行(P0-2 修复验证)

**测试机**:用 `FLEET_EXTRA_DEPTS_JSON` 在中央台 .env 配 3-4 个假 dept,其中 2-3 个共享同一 IP(用 dev VPS 187.77.157.220 + 不同子域)。

**步骤**:
```bash
# 中央台后台点「全网升级」按钮(或 curl /api/admin/fleet/cmd 模拟)
# 看 docker logs tg-central-dashboard 看 fanout 顺序
```

**预期**:
- 日志显示同 IP 的部门**逐个**触发 call_agent(start 时间间隔 ≥ 单台升级耗时)
- 不同 IP 的部门**同时**触发
- 总耗时 ≈ max(各 IP 组的串行总时长),不是所有部门时长之和

### 4. 真实域名 dept fallback 退回并发(无 IP regex 命中场景)

**测试**:在 `FLEET_EXTRA_DEPTS_JSON` 加一个 base_url = `https://monitor-fake.example.com` 的假 dept。

**步骤**:
```bash
# 触发 fanout(upgrade)
# 看 docker logs 看 warning
```

**预期**:
- 日志含 `[fleet.fanout] serialize_by_ip=True 但 1 个 dept base_url 不含 IPv4(真实域名场景),仍并发处理:['fake-company']`
- 升级过程不报错(只是这个 dept 跟其他 dept 并发)

### 5. 老 caller 行为不变(向后兼容)

**测试**:
- 中央台 fleet UI 点「🔍 全网健康检查」按钮(触发 inspect action)
- 看 docker logs 看 fanout 行为

**预期**:
- inspect 走老路径(`serialize_by_ip=False` 默认)
- 所有 dept 并发,无 IP 分组 log
- `fleet_health.py:202` inspect daemon 同上,不受影响

### 6. cp 备份失败不阻塞升级(罕见盘满)

**dev VPS**:把 host 目录挂载点设为 read-only 模拟。

**步骤**:
```bash
# 不容易模拟,改用单元测试方式:手动 chmod -w 当前目录
cd /tmp/test-readonly && chmod -w .
# 模拟跑 5.5b 段(cat > /tmp/Caddyfile.upstream)
```

**预期**:
- `cp Caddyfile Caddyfile.bak.upstream.* 2>/dev/null || true` 不导致 set -e 整段挂
- 后续 sed + cat > Caddyfile 仍走完

### 7. 跨网络 reload(共用 Caddy 模式)

**测试**:dev VPS 187.77.157.220 同 IP 多部门(sim / freshtest / v225test 等)模拟。

**预期**:
- 5.5b 检测到该部门没自家 `tg-caddy-${COMPANY_NAME}` 时,通过 `PUBLIC_DOMAIN` 字符串匹配在共享 Caddy 容器内找到对应 site block
- reload 该共享 Caddy(影响全 VPS 所有部门,**但只一次,不抖动**)

## 监控指标(升级后 24h)

- **0 个** dept 升级后报「DNS 撞车」类客户投诉
- 中央台 fleet_health 异常数**不**因为升级新增
- Caddy 容器 restart 次数(`docker events --filter type=container --filter event=restart`)在升级窗口内**每台 VPS ≤ 1 次**(同 IP 多部门 VPS 总共 1 次,不是每 dept 1 次)

## 回滚预案

升级失败 / 客户报错:

```bash
cd /root/tg-monitor-<部门>
bash rollback.sh    # 走 update.sh 内置回滚到 OLD_SHA
# 或者手动:
git reset --hard $(cat .last_commit_before_v3133)
ls Caddyfile.bak.upstream.* | head -1 | xargs -I{} cp {} Caddyfile
docker restart tg-caddy-<部门>
```
