# v2.10.24.3 压力测试清单

**发布前必过**。对应 ADR-0010(预警分页整行缺失自动 writeback)。

## 核心行为验证

### T1:DB migration 跑到 user_version=2

**前提**:已有 v2.10.24.2 容器跑过的 `data/tg.db`

**步骤**:
1. `./update.sh` 升级到 v2.10.24.3
2. 容器启动后:`sqlite3 data/tg.db "PRAGMA user_version"`
3. `sqlite3 data/tg.db "PRAGMA table_info(alerts)"`

**预期**:
- `user_version = 2`
- alerts 表包含新列:`sheet_written`(INTEGER)、`keyword`(TEXT)、`last_write_error`(TEXT)
- 历史 alerts 全部 `sheet_written=1`:`SELECT COUNT(*) FROM alerts WHERE sheet_written=0` = 0

**通过**:migration 幂等跑成功,老数据不污染

### T2:第二次启动不重跑 migration

**步骤**:T1 后 `docker compose restart`,再检查 `PRAGMA user_version` 和日志

**预期**:仍是 2,不出现重复 `_migrate_to_2` log

**通过**:迁移幂等

### T3:`_alert_writeback_loop` 加入 asyncio.gather(第 11 个)

**步骤**:冷启动 → 观察日志第 1 分钟

**预期**:所有 11 个 loop 启动 — 包含 `_alert_writeback_loop`。无 exception。

**通过**:loop 启动正常

## 写入路径验证

### T4:keyword 命中正常写分页 + mark sheet_written=1

**前提**:有一个配置了 keyword 的账号,peer 发一条含该 keyword 的消息

**步骤**:
1. 触发命中
2. 立刻 `sqlite3 data/tg.db "SELECT id,type,keyword,sheet_written,message_text FROM alerts ORDER BY id DESC LIMIT 1"`
3. 查 Google Sheets「关键词监听<部门>」分页最后一行

**预期**:
- DB 有新行:`type='keyword'`, `keyword='<关键词>'`, `message_text='<纯消息内容>'`(**无** `[kw]` 前缀), `sheet_written=1`
- Sheet 分页新增一行,A-G 列齐全

**通过**:新路径正常工作

### T5:keyword 模拟 Sheet 写入失败 → sheet_written 保持 0

**模拟**:临时改 `config.SHEET_ID` 为一个假 id(或拔网络)让 Sheet API 调用都失败

**步骤**:
1. 命中 keyword → bot.py 3 retry 失败
2. 查 DB:`SELECT id, sheet_written, last_write_error FROM alerts ORDER BY id DESC LIMIT 1`

**预期**:
- 新 alert `sheet_written=0`
- `last_write_error` 字段非空,包含错误原因

**通过**:失败路径持久化状态正确

### T6:恢复后 writeback loop 补写成功 → mark sheet_written=1

**步骤**:T5 基础上恢复 SHEET_ID → 等 60-90 秒 → 查 DB + Sheet

**预期**:
- 日志出现 `writeback_pending_alerts: 完成 — 写入 1 行,失败 0 行`
- 该 alert `sheet_written=1`
- Sheet「关键词监听<部门>」分页出现那一行

**通过**:writeback loop 自愈

### T7:no_reply approved 后写入失败 → writeback 接力

**前提**:触发一条 no_reply 预警 → 审核通过 → 但此时 Sheet API 正好挂

**步骤**:
1. 模拟 Sheet API 挂 → 点审核通过(callback)
2. `SELECT sheet_written FROM alerts WHERE id=<alert_id>` → 应是 0
3. 恢复 API → 等 60s

**预期**:恢复后 loop 自动补写未回复预警分页 + mark=1

**通过**:no_reply approved 路径也走 writeback

### T8:deleted approved 同 T7 行为

**步骤**:同 T7 但用 deleted 预警

**通过**:deleted approved 路径也走 writeback

## 幂等 / 边缘 case

### T9:writeback 同 alert 跑两次不重复写

**前提**:T6 已经把某 alert mark=1

**步骤**:`docker compose restart` → 等 90 秒 → 看 Sheet 分页该行数

**预期**:该 alert 对应的 Sheet 行**只有 1 行**,不会重复 append

**通过**:`sheet_written=1` 的 alert 不会被 writeback 拾起

### T10:历史 alerts(升级前)不会被 writeback 误补

**前提**:DB 中有 v2.10.24.3 升级前的 alerts(T1 已标全部 sheet_written=1)

**步骤**:正常跑 10 分钟,观察 writeback 日志

**预期**:
- 日志只出现 `完成 — 写入 0 行`(如果新命中都成功)或不出现(无积压)
- 绝不会看到「写入 N 行」N 是历史数字(例如 500 等)

**通过**:一刀标 1 策略生效,历史不被盲补

### T11:silenced alerts 不会被 writeback 写入

**前提**:关 `ALERT_NO_REPLY_ENABLED=false` → 触发一条 no_reply → 应插入 alert 并 `status='silenced'`

**步骤**:检查 writeback 3 轮(3 分钟)日志 + Sheet「信息未回复预警」分页

**预期**:
- 该 alert `sheet_written=0` 但 `status='silenced'`
- writeback 永不拾起它(`get_unwritten_alerts` 过滤了 silenced)
- Sheet 分页没有新增行

**通过**:silenced 不受影响,客户关了告警就不该上表

### T12:pending 的 no_reply / deleted 不会被 writeback 写入

**前提**:触发 no_reply 预警但**不点审核通过**(保持 pending)

**步骤**:观察 writeback 3 轮

**预期**:
- Sheet「信息未回复预警」分页**不出现**这条
- writeback 日志不提此 alert

**通过**:pending 的 approved-only 类型不会被误写

## 配额 / 负载验证

### T13:积压 50 条时一轮写入 ~75 秒无 429

**模拟**:人为把 50 条 alert 标 sheet_written=0(`UPDATE alerts SET sheet_written=0 WHERE id IN (...)`)

**步骤**:
1. 人工造 50 条 sheet_written=0 → restart
2. Google Cloud Console → Sheets API → 监控 Writes/min
3. 等 75-90 秒看是否跑完

**预期**:
- 一轮 50 条,每秒 < 1 次 write API(_rate_limit=1.5s)
- 写完 mark 50 条为 1
- Sheet 分页增加 50 行
- 无 429

**通过**:rate limit 下积压清理安全

### T14:writeback 碰到 429 → 整轮截断

**模拟**:积压 100 条 → loop 跑到一半人为触发配额爆(例如关掉 SHEET_ID 让 API fail)

**步骤**:
1. 100 条积压
2. 人为让第 30 条之后开始 fail with 429 (或网络断)
3. 观察日志

**预期**:
- 日志出现 `写入 N 行,失败 1 行(配额爆,本轮提前结束)` (N < 30)
- 下一轮 loop 间隔(60s)后再试
- 未写入的保留 sheet_written=0

**通过**:429 触发本轮终止,等待下一轮自然退避

### T15:`ALERT_WRITEBACK_DISABLED=true` 完全停掉

**步骤**:
1. `.env` 加 `ALERT_WRITEBACK_DISABLED=true` → restart
2. 人工造 10 条 sheet_written=0 alerts
3. 观察 3 轮(3 分钟)

**预期**:日志不出现 `writeback_pending_alerts`,DB 的 sheet_written=0 不动

**通过**:开关生效

### T16:`ALERT_WRITEBACK_INTERVAL_SEC=10` 短间隔可用

**步骤**:`.env` 加 `ALERT_WRITEBACK_INTERVAL_SEC=10` restart → 观察 2 分钟

**预期**:
- 每 10 秒一轮
- 无积压时每轮不产生 log / 快速完成(`get_unwritten_alerts` 返回空)
- 总 API 消耗仍在 < 60/min

**通过**:短间隔场景配额仍安全(无积压时无 API 消耗)
