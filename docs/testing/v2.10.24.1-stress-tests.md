# v2.10.24.1 压力测试清单

**发布前必过**。对应 ADR-0008(Sheets 读 API 配额保护)。

## 核心行为验证

### T1:默认配置下 sync_headers 跑 1/10 次

**场景**:`.env` 不配 `SYNC_HEADERS_INTERVAL_SEC`(走默认 600s)、保持 `PATROL_INTERVAL=60`

**步骤**:
1. 启动容器,等 60 秒让 `_patrol_loop` 进入循环
2. `docker compose -p tg-<部门> logs tg-monitor | grep -c "sync_headers"` 持续观察 10 分钟
3. 预期:10 分钟内 `sync_headers` 只被调用 **1-2 次**(而不是 10 次)

**通过**:`sync_headers` 调用次数 ≤ 2 次 / 10 分钟

### T2:`SYNC_HEADERS_INTERVAL_SEC=60` 回退到旧行为

**场景**:客户手动降低间隔到 60s(兼容性验证)

**步骤**:
1. `.env` 加 `SYNC_HEADERS_INTERVAL_SEC=60`,restart 容器
2. 观察 10 分钟
3. 预期:`sync_headers` **每 60 秒调用一次**(每轮 patrol 都跑)

**通过**:`sync_headers` 调用次数 ≈ 10 次 / 10 分钟(允许 ±1)

### T3:`SYNC_HEADERS_DISABLED=true` 完全停掉

**步骤**:
1. `.env` 加 `SYNC_HEADERS_DISABLED=true`,restart
2. 观察 30 分钟
3. 预期:`sync_headers` **0 次调用**,但 `_patrol_loop` 其他巡检项(昵称同步、分页自愈)**正常跑**

**通过**:`sync_headers` = 0 次,`_sync_account_names` / `ensure_account_tabs` 照常

### T4:`_peer_name_consistency_loop` 默认 600s

**步骤**:
1. 不配 `PEER_NAME_CONSISTENCY_INTERVAL_SEC`(走默认 600s),restart
2. 观察 30 分钟
3. 预期:`_peer_name_consistency_loop` 主体逻辑(读 row 6)**3 次**(30 / 10)

**通过**:`广告主名同步` / `广告主名一致性巡检失败` 日志频率 ≈ 3 次 / 30 分钟

### T5:`PEER_NAME_CONSISTENCY_DISABLED=true` 完全停掉

**步骤**:
1. `.env` 加 `PEER_NAME_CONSISTENCY_DISABLED=true`,restart
2. 观察 30 分钟
3. 预期:`_peer_name_consistency_loop` **完全不进主逻辑**,但 loop 本身还在跑(隔 600s sleep 醒来继续 check)

**通过**:日志完全没有「广告主名同步」「读 row 6 失败」等,但容器没崩

## 配额压力测试(模拟客户场景)

### T6:150 账号默认配置下 10 分钟配额消耗

**场景**:在测试 GCP 项目上模拟(不要用客户生产项目打压力)

**步骤**:
1. 准备 150 个测试账号 session(或用 mock 账号)
2. 启动容器,默认配置
3. 监控 Google Cloud Console → APIs & Services → Google Sheets API → Metrics → 「Read requests per minute」
4. 观察 10 分钟

**预期**:
- 峰值 read requests ≤ **55 / min**(小于配额 60/min,留 ≥ 5 余量)
- 平均 read requests ≈ **45 / min**
- **无 429 错误**

**通过**:无 429,峰值 < 60/min

### T7:老 sed 止血客户升级兼容性(一键验证)

**场景**:客户之前跑过止血 sed(`tasks.py:165 pass` + `tasks.py:34 #loop,`),现在升级到 v2.10.24.1

**客户侧前提**:只做一个动作 — `cd /root/tg-monitor-<部门> && ./update.sh`。**不允许**要求客户跑任何 `cp` / `git` / `docker` 命令。

**步骤**:
1. 复制 `tasks.py` 为 `tasks.py.bak`,手动 sed 成 stopgap 状态(模拟客户环境)
2. 跑 `./update.sh`
3. 观察

**预期**(update.sh v2.10.24 起内建 `git stash` + `git reset --hard`):
- update.sh 发现 tracked 文件(tasks.py)有 modification → 自动 `git stash push`(日志:「检测到本地修改的 tracked 文件,自动 stash 保护」)
- `git reset --hard origin/main` 强制覆盖为 v2.10.24.1 新代码(包含 Codex P0/P1 修复)
- 重建镜像 + 健康检查 60 秒 → pass → 升级完成
- stash 保留不还原(旧 sed 止血版本已经没价值了)

**通过**:无任何客户手动操作,升级后 `docker compose -p tg-<部门> logs tg-monitor` 看不到 429

### T9:广告主名 TG 改名 → Sheets 第 6 行 10 分钟内同步

**步骤**:

2. 等 10 分钟

**通过**:10 分钟内 Sheets 显示新名字

### T10.5:`_enforce_sheet_tab_consistency` 单轮只 1 个读请求(Codex P0 修复)

**场景**:验证 `spreadsheet.worksheet(title)` 循环被改成一次 `worksheets()` 拉全量

**步骤**:
1. 启动容器,默认配置
2. `docker compose -p tg-<部门> logs --since 5m tg-monitor | grep -c "分页列表读取失败\|分页名自动修正"`
3. 可选:在 Google Cloud Console 监控 Read requests per minute,确认巡检时段无单轮集中 150 次读
4. 人为制造 rename 场景:手改某分页名,等下一轮巡检
5. 预期:日志出现「分页名自动修正: 旧 → 新」,且没有额外 fetch_sheet_metadata 风暴

**通过**:Cloud Console 峰值 read requests 没有因巡检每分钟跳到 150+,rename 场景正常生效

### T10.6:sync_headers 异常时 throttle 正确推进(Codex P1 修复)

**场景**:首版计数器 throttle 在 sync_headers 抛异常时 counter 卡住会每 60 秒重试,改成时间戳 + try/finally 后应该无论异常都等 600 秒再试

**步骤**(需模拟环境,生产不要跑):
1. 临时把 `sheets.sync_headers()` 改成 `raise RuntimeError("mock")` 触发每次必抛
2. 启动容器,观察 `_patrol_loop` 日志
3. 预期:第 1 轮(启动后 ~30s)跑一次抛异常 → 下一次**等 600 秒**才再跑(不是 60 秒)
4. 确认完成后还原 sheets.py

**通过**:异常场景下 sync_headers 每 600 秒触发一次(不是每 60 秒)

### T10:start 流程无异常

**步骤**:
1. 冷启动容器,观察 `tg-monitor` 日志第一分钟
2. 确认所有 loop 正常 gather(9 个 loop:`_sheets_flush_loop`, `_patrol_loop`, `_no_reply_loop`, `_daily_report_loop`, `_peer_name_consistency_loop`, `_media_cleanup_loop`, `_update_check_loop`, `_session_health_loop`, `_sheets_backlog_loop`)

**通过**:无 exception,全部 loop 启动
