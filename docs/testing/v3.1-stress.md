## 集成测试

### 1. 首次启动 — migration V6 + resync loop 跑首轮

```bash
# 把生产备份的 SQLite 拿到 dev
cp /tmp/customer_backup.db data/data.db

# 启动容器,看日志
docker compose up -d
docker logs -f tg-monitor-dev | grep -E "user_version|migration|sheet_resync"

# 期望日志(30s 内):
#   sqlite3 user_version 升 5 → 6
#   peers 加 next_sheet_row 列 (NULL 默认)
#   30s 后:[sheet_resync] 完成: N worksheets / M peers / 0 errors / X.XXs

# 验证 DB
sqlite3 data/data.db "SELECT COUNT(*) FROM peers WHERE next_sheet_row IS NOT NULL;"
# → 应等于有 col_group >= 0 的 peer 总数
```

### 2. Update 路径 — 命中 DB 缓存的 next_row

```bash
# 从客户 DB 找一个有 next_sheet_row 的 peer
sqlite3 data/data.db "SELECT id, account_id, col_group, next_sheet_row FROM peers WHERE col_group=0 LIMIT 1;"
# 比如 (peer_id=42, account_id=1, col_group=0, next_sheet_row=156)

# 模拟该 peer 来 5 条消息(往 messages 表插)
sqlite3 data/data.db "INSERT INTO messages (msg_id, account_id, peer_id, direction, text, timestamp, sheet_written) VALUES
  (99001, 1, 42, 'B', 'test 1', datetime('now'), 0),
  (99002, 1, 42, 'B', 'test 2', datetime('now', '+1 second'), 0),
  ..."

# 等下一轮 flush(默认 5 秒)
sleep 10

# 看 Sheet 上 row 156-160 应该有数据(列 A/B/C — col_group=0)
# 看 DB next_sheet_row 应该 +5 = 161
sqlite3 data/data.db "SELECT next_sheet_row FROM peers WHERE id=42;"
# → 161
```

### 3. Append fallback — invalidate 后

```bash
# 手动 invalidate 一个 peer 的 next_row
sqlite3 data/data.db "UPDATE peers SET next_sheet_row=NULL WHERE id=42;"

# 模拟一条消息
# 看 log 应该是 "append API" 不是 "update API"
docker logs --since 30s tg-monitor-dev | grep "写入 1 条"
# → "写入 1 条到 ... append API"

# 等下轮 resync (15 分钟,或手动触发)
curl -X POST http://localhost:5050/api/sheets/resync-now -b 'session=admin'
sleep 5
# next_sheet_row 应该被重新填值
sqlite3 data/data.db "SELECT next_sheet_row FROM peers WHERE id=42;"
# → 一个非 NULL 的值
```

### 4. 删旧消息回填(核心目标场景)

```bash
# 在测试 Sheet 上手动删 peer X 的 row 30-50 数据(选中 → Delete)
# 等下一轮 resync(或手动触发)
curl -X POST http://localhost:5050/api/sheets/resync-now -b 'session=admin'
sleep 10

# next_sheet_row 应该退到 30(或更小,看具体哪行最先空)
sqlite3 data/data.db "SELECT next_sheet_row FROM peers WHERE id=<peer_x_id>;"
# → ≤ 30

# 触发新消息 → 看 Sheet 第 30 行
# 应该填了新消息内容
```

### 7. 强保护模式(SHEET_RESYNC_VERIFY_BEFORE_WRITE)

```bash
# 改 .env 加 SHEET_RESYNC_VERIFY_BEFORE_WRITE=true
docker exec tg-monitor-dev sh -c 'echo "SHEET_RESYNC_VERIFY_BEFORE_WRITE=true" >> /app/.env'
docker restart tg-monitor-dev

# 模拟一个场景:next_sheet_row=100 但 row 100 实际有客户内容
# (DB 跟 Sheet 短暂不一致,resync 间隔内的 race)

# 在 Sheet 上 row 100 col A 手动写"客户的内容"
# 触发该 peer 一条新消息

# 看 log:
#   [write update verify] peer=X row=100 已有内容 ('客户的内容...'), 走 fallback 防覆盖
#   写入 1 条 ... append API
# DB next_sheet_row 应该已 invalidate
sqlite3 data/data.db "SELECT next_sheet_row FROM peers WHERE id=X;"
# → NULL
# 客户内容 row 100 col A 应保留
```
