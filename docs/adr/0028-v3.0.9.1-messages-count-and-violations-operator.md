# ADR-0028 · v3.0.9.1 · 中央台区间消息总数 + 违规明细加 operator 字段

## 背景

v3.0.9 给中央台暴露了 23 个 DB 业务字段 + 4 个 /api/v1/* 只读 endpoint。中央台后台
`/center/<id>` 加了日期 range picker(今天 / 昨天 / 7天 / 30天 / 本月 + 自定义)后,
所有 KPI 都跟着 range 走 — 但「今日消息」永远是 `messages_today`(参 `dashboard_api.py::messages_today`),
跟选定 range 对不上。客户选「本月」期望看到本月消息总数,实际还是今天。

同时违规明细表用户反馈要看「负责人」,目前 `/api/v1/violations` 只返 `business_tg_id` /
`owner_tg_id`(@xxx 的 TG handle),accounts.operator 是该账号的商务负责人姓名 —
更适合表格展示。

## 决策

新增 `/api/v1/messages/count?from=YYYY-MM-DD&to=YYYY-MM-DD` 只读 endpoint,
返回 `{in, out, media, deleted, total, from, to}`(跟 `messages_today` 同结构 + range 元信息),
中央台 collector 在 custom_range 时调用,fallback 到 snapshot.messages_today(部门未升级时)。

`violations()` SELECT 加 `ac.operator`,API 响应字段加 `operator`,中央台前端违规明细表
新增「负责人」列(优先 `operator` → fallback `business_tg_id` → `owner_tg_id` → `—`)。

### 实现细节

- `dashboard_api.py::messages_count_in_range(from_date, to_date)`:跟 `messages_today()` 同
  逻辑,差异在 WHERE 用 `timestamp >= ? AND timestamp < ?`(右开区间,to+1 天作为
  右边界),用索引友好的范围比较而非 LIKE 防 strftime 退化。
- `dashboard_api.py::violations()` SELECT 加 `COALESCE(ac.operator, '') AS operator`,
  返回字段加 `operator` key。
- `web.py::api_v1_messages_count`:新路由 `/api/v1/messages/count`,沿用 `_v1_check_token`
  鉴权,参数缺失 → 400,`ValueError` → 400,异常 → 500。
- 0 新表 0 新字段 0 数据迁移,纯 SELECT 扩展 + 加路由,跟 v3.0.9 同模式。

## 原因

1. **数据准确性 > 客户体验**:中央台 range picker 是 v3.0.9 暴露 violations / alerts /
   keyword 接口后自然延伸的,messages 是唯一漏的。补齐让客户看「本月」就是本月。
2. **operator 是 accounts 表已有字段**,不是新数据,纯多 SELECT 一列。
3. **沿用 metrics token 鉴权**,跟其他 v3.0.9 endpoint 一致,不引新 auth 模型。
4. **timestamp 范围比较 vs LIKE**:`timestamp >= '2026-04-01' AND timestamp < '2026-05-01'`
   能用 messages 表的 timestamp 索引 + range scan,LIKE 'YYYY-MM-DD%' 只能做 prefix 匹配
   不能跨多天范围。

## 来源

- 中央台 dashboard 客户反馈:选本月数据全跟着变,只有「今日消息」死活是今天 → 改
- 违规明细表客户反馈:要看负责人,@xxx TG handle 太抽象 → 改成 operator 名字优先

## 后果

- 客户升级到 v3.0.9.1 后,中央台「今日消息」label 在 custom range 时变「区间消息」,
  数字跟选定 range 一致
- 违规明细表多一列「负责人」,显示商务运营人员姓名而非 TG handle
- 部门未升级 v3.0.9.1 时,中央台 fallback 到 snapshot.messages_today,显示今天的数字
  (跟 v3.0.9 同行为,不破坏)
- API 完全向后兼容:v3.0.9 客户老用法不受影响,新字段是纯加

## 测试

- `dashboard_api.messages_count_in_range("2026-04-01", "2026-04-29")` 返 `total > 0`
- `dashboard_api.messages_count_in_range("2026-13-01", "2026-04-29")` raise `ValueError`
- `dashboard_api.messages_count_in_range("2026-04-29", "2026-04-01")` raise `ValueError`(to < from)
- `GET /api/v1/messages/count` 不带 token → 401
- `GET /api/v1/messages/count?from=X&to=Y` 含 token → 200 + JSON
- `GET /api/v1/violations?from=X&to=Y` 响应包含 `operator` 字段
- 中央台 detail page custom range → 「今日消息」label 变「区间消息」,数字跟范围一致
