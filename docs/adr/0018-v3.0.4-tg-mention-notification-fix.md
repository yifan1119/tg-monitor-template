# ADR-0018: v3.0.4 两段式预警 @ 通知修复 — `@username` 走 TG 原生解析

## 背景

**现象**: v3.0.0 两段式预警上线后，客户反馈「商务/负责人配了 TG ID（都是 `@username` 格式），预警群里名字显示蓝色可点，但被 @ 的人 TG 没有通知弹出 / 没有响铃 / 没有出现在「提及我」列表里」。

**已排除的可能原因**:
- 商务/负责人都已经在预警群里（用户确认）
- HTML 渲染正常（名字确实是蓝色可点，不是纯文字）
- `parse_mode="HTML"` 代码里写了（`bot.py:583`）

**实际根因**:

`bot.py:_build_tg_mention` 的老逻辑：
1. 拿到 `@an2026ni`
2. 先用 Telethon 的 `get_entity` 查这个 username → 解析成 numeric UID
3. 发送时用 `<a href="tg://user?id=UID">谢瑜（YWSC-YD）</a>` 格式（**inline mention**）

这种 inline mention 的 TG 规则是：
- **bot 发送** 的 inline mention 受反垃圾规则限制
- 被 @ 的用户如果**从未 `/start` 过这个 bot**，TG 不会触发群 @ 通知
- 只会把名字渲染成蓝色可点的链接（看起来"像 mention"但实际 silent）

客户的商务/负责人平时**不会主动 /start 预警机器人**，所以永远是 silent mention 状态。

## 决策

`bot.py:_build_tg_mention` 逻辑调整优先级：

**之前**（v3.0.0 - v3.0.3）:
```python
# 先 Telethon resolve 转 numeric UID → 再 inline mention
uid, resolved = await self._resolve_tg_entity(tg_id_or_username)
if uid:
    return f'<a href="tg://user?id={uid}">{name}</a>'
# fallback：@username text
return self._format_tg_mention(...)
```

**现在**（v3.0.4）:
```python
# username 格式 → 直接 @text（让 TG 原生 mention 解析接管）
if not val.isdigit():
    return f"@{html.escape(val)}"
# 纯数字 UID → Telethon resolve + inline mention（只能这样，没 username 可用）
uid, resolved = await self._resolve_tg_entity(val)
if uid:
    return f'<a href="tg://user?id={uid}">{name}</a>'
```

**关键点**:
- `@username` 文本交给 TG 的**原生 mention 解析器**（而不是 bot API 创建的 inline mention）
- TG 原生解析不受 bot 反垃圾规则限制：只要被 @ 的用户在群里且有 username，就会触发通知
- 客户用 `@xxx` 这种配置（多数场景）立刻就能 @ 到人，不需要对方事先 /start

**退化路径**:
- 用户配的是纯数字 UID（少数场景，或 username 已失效时）→ 仍走 Telethon + inline mention
- Telethon 都解不到 → 保底 inline 套个默认名字，起码能点开 profile

## 原因

客户反馈 — 线上真实踩坑。原设计的「Telethon 先解 numeric UID 最可靠」忽略了 TG 对 bot-generated inline mention 的反垃圾限制。实际上对大多数客户，`@username` 文本形式反而**更稳定**，因为被 @ 的对象（商务/负责人）的 TG 账号都有明确的 username（招聘时登记），不会有 username 不存在的场景。

## 来源

- 客户 demo 部署反馈（2026-04-24 上午）
- 截图证据：预警群里「谢瑜（YWSC-YD）请10分钟内及时处理」，名字蓝色可点但对应 TG 没收到通知
- 用户确认：「有显示名字但不是@」（意思是"看起来像 mention 但不是真 mention"）

## 后果

**正面**:
- 客户配 `@xxx` 立刻生效，无需对方 /start bot
- 修复了 v3.0.0 以来两段式预警的隐性通知失效 bug
- 向后兼容：numeric UID 配置路径不变

**负面/trade-off**:
- `@username` 文本显示的是 **username 字符串**（比如 `@an2026ni`），不是**显示名**（比如「谢瑜」）
  - 不影响功能（点击仍然跳 profile），但视觉上不如 inline mention 美观
  - 如果客户在意，可以改配 numeric UID → 走 Telethon 路径拿真名 → inline mention（付出「对方需 /start bot」的代价）

**未覆盖的场景**:
- 如果客户配 username 但该 username 已被废弃/更改 → `@失效名` 就是死文本，TG 不会警告
  - 建议 v3.0.5+ 加入 username 存活校验（后台设置页保存时 Telethon 试解析，不存在就 reject）
