# ADR-0004: `upsert_account` ON CONFLICT 不再覆盖业务字段

**版本**:v2.10.23
**日期**:2026-04-21

## 背景

Codex 审计发现 `database.py:94-102` + `listener.py:38-43` + `main.py:107-114` 的 bug:

**v2.10.22 的 `upsert_account`**:
```python
def upsert_account(phone, name="", username="", tg_id=None, company="", operator=""):
    conn.execute("""
        INSERT INTO accounts (phone, name, username, tg_id, company, operator)
        VALUES (?, ?, ?, ?, ?, ?)
        ON CONFLICT(phone) DO UPDATE SET
            name=excluded.name, username=excluded.username,
            tg_id=excluded.tg_id, company=excluded.company, operator=excluded.operator
    """, (phone, name, username, tg_id, company, operator))
```

**问题**:`listener.py` 和 `main.py` 启动时调 `upsert_account(phone, name=..., username=..., tg_id=...)` 没传 `company/operator`(空字符串默认),ON CONFLICT 逻辑会把已有的 `company/operator` 值**清空**。

**受影响场景**:
1. 客户在 Web 后台设置页填好「商务人员」和「所属公司」
2. 容器重启 / listener 重连 / sweep 触发 upsert
3. 业务字段被空字符串覆盖 → 客户填的数据消失
4. Sheet 分页表头同步时读到空值 → 继续污染

**v3.0.0 提前预警**:v3.0.0 要给 accounts 加 4 个业务字段(`business_tg_id` / `owner_tg_id` / `remind_30min_text` / `remind_40min_text`),如果不先修这个 bug,新字段也会被启动覆盖清空。

## 决策

**不动 `upsert_account` 签名**(向后兼容),改 ON CONFLICT 子句,**只更新 TG 身份字段**:

```python
def upsert_account(phone, name="", username="", tg_id=None, company="", operator=""):
    """v2.10.23:ON CONFLICT 只更新 TG 身份字段(name/username/tg_id),
    不再覆盖业务字段(company/operator)— 避免 listener 启动登录时空值覆盖客户填的业务配置。

    业务字段更新走 update_account_business(account_id, company=..., operator=...)。
    INSERT 时(第一次创建)仍然会写入 company/operator 初始值。"""
    conn.execute("""
        INSERT INTO accounts (phone, name, username, tg_id, company, operator)
        VALUES (?, ?, ?, ?, ?, ?)
        ON CONFLICT(phone) DO UPDATE SET
            name=excluded.name, username=excluded.username, tg_id=excluded.tg_id
    """, (phone, name, username, tg_id, company, operator))
```

**新增 `update_account_business(account_id, company=None, operator=None)`**(只在 Web 后台/配置页用)。

v3.0.0 同样模式新增 `update_account_two_stage(account_id, business_tg_id=..., owner_tg_id=..., remind_30min_text=..., remind_40min_text=...)`。

## 原因

### 为什么不改调用方传入当前值?
- 调用方(`listener.py` / `main.py`)不该关心业务字段
- 职责分离:TG 身份字段靠 Telethon 自动维护,业务字段靠人工 / Web 后台

### 为什么不拆成两个函数 `upsert_tg_identity` / `create_account`?
- 老调用方签名不变,保持向后兼容 = 回滚时老代码完全不炸
- 语义上 `upsert_account` 仍是「这个账号存在吗?没有就建,有就同步身份」— 业务字段仅首次创建时采用
- 改名涉及所有调用点要改,风险大于收益

### INSERT 时业务字段还是会写入是不是有问题?
- 第一次创建(`INSERT`):调用方如果传了 `company="某公司"`,就写入;没传就空 — 行为跟以前一样
- ON CONFLICT(`UPDATE`):跳过业务字段 — 这是修复的核心
- **旧 SQLite 老数据不受影响**,只是以后启动/sweep 不再覆盖

## 来源

Codex 审计 Major:
> 🟡 Major `database.py:94-102` + `listener.py:38-43` + `main.py:107-114`:启动登录会把账号业务字段重置为空。根因是 `upsert_account()` 每次更新 `company/operator`,而 listener/main 启动时常传空值。建议 TG 身份字段和业务配置字段分开更新;v3 新增 4 字段尤其不能走这种覆盖逻辑。

## 后果

### 立即效果
- 客户在 Web 后台填的「商务人员」「所属公司」不会再被启动清空
- 重启容器 / listener 重连 / sweep 触发 — 业务字段稳定保留

### 对 v3.0.0 的铺垫
- 新 4 字段走 `update_account_two_stage(account_id, ...)`,不走 `upsert_account`
- listener 启动不碰新字段,只负责维护 TG 身份

### 相关代码
- `database.py:upsert_account`(改 ON CONFLICT 子句)
- `database.py:update_account_business`(新增,给 Web 后台用)
- `database.py:update_account_two_stage`(v3.0.0 新增,给 Web 后台两段式设置用)
- `database.py:get_account_by_id`(v3.0.0 新增)
- `listener.py` / `main.py`:调用方签名不变,行为自然修复
