# ADR-0011: `update_checker` 版本号 regex 支持四段(修后台升级弹窗显示错版本说明)

## 背景

客户在 Web 后台「版本升级」弹窗截图反馈:明明显示 `72439e0 → d07bebb`(v2.10.24.1 → v2.10.24.3),但弹窗里的升级说明文字却是 v2.10.24 的「升级流程修补 — 升级不再撞容器冲突」— 不是真正的 v2.10.24.3 预警分页零丢失保障的说明。

查代码定位:`update_checker.py:64-78` 的 `_notes_for(short_sha, all_notes, commit_subject)` 负责从 `release_notes.json` 里挑说明:
1. 先按 short_sha 精确匹配
2. 再从 commit subject 用 regex 抓版本号去匹配

原 regex:
```python
m = re.search(r"v\d+\.\d+\.\d+", commit_subject or "")
```

只匹配**三段**版本号。commit subject `v2.10.24.3: 预警分页整行缺失自动 writeback ...` 被抓成 `v2.10.24`(三段),于是去 `release_notes.json` 拿 `v2.10.24` 的 key,那就是「容器冲突修补」。

v2.10.24.2 / v2.10.24.3 两个版本都中了同样的坑,只是之前没人对照展示内容做核对。

## 决策

两处 regex 都从 `v\d+\.\d+\.\d+` 改成 `v\d+(?:\.\d+)+`,贪婪吃完所有连续的 `.数字` 段:

```python
# update_checker.py:76 — 版本号匹配
m = re.search(r"v\d+(?:\.\d+)+", commit_subject or "")

# update_checker.py:84 — subject 前缀剥离
desc = re.sub(
    r"^(v\d+(?:\.\d+)+|docs|fix|feat|...)[:\s]*",
    "", subject, flags=re.IGNORECASE,
).strip()
```

验证矩阵(Python 真实跑过):
| subject | 匹配结果 |
|---|---|
| `v2.10.24.3: foo` | `v2.10.24.3` ✓ |
| `v2.10.24: bar` | `v2.10.24` ✓(向后兼容) |
| `v3.0.0: baz` | `v3.0.0` ✓(向后兼容) |
| `v2.10.24.3.1: x` | `v2.10.24.3.1` ✓(未来 5 段也能撑) |
| `docs: y` | `None`(不命中) |
| `v2: w` | `None`(至少两段才算版本号) |

## 原因

**为什么 greedy `+` 而不是固定 `{2,4}`?**
- `{2,4}` 假设最多 4 段 — 未来如果出 `v2.10.24.3.1` hotfix 又要改一次
- `(?:\.\d+)+` 是开放的,加几段都接得住,不留技术债

**为什么 regex 贪婪能挑最长?**
- `re.search` 从左往右找第一个匹配,但**该匹配本身**会贪婪尽量吃长
- 所以 `v2.10.24.3: ...` 会先找到 `v` 位置,然后 `\d+(?:\.\d+)+` 贪婪到 `v2.10.24.3` 才停(下一字符是 `:` 不是 `.`)

**为什么不加新字段 / 新 key?**
- release_notes.json 已经用 `"v2.10.24.3"` 这种四段 key(`v2.10.24.3` 条目在本次之前已经存在于 main 分支)
- 只是 regex 没跟上,展示层单点修复就够了
- 零风险,纯读取侧

## 来源

- **客户反馈**:2026-04-22 晚,客户 Web 后台截图显示 `72439e0 → d07bebb` 但弹窗文字是 v2.10.24 的「容器冲突修补」
- **查代码定位**:`update_checker.py:76` regex 只抓三段

## 后果

**正面**:
- 之后任何四段 / 更多段的版本号(v2.10.24.4 / v2.10.24.5 ...)升级说明都能正确展示
- 零业务影响,纯 UI 展示层修复
- 升级后下次 `update_checker` 巡检命中(每 6 小时一次)客户就看到对的说明

**需要后续观察**:
- 无。纯一行 regex 修复,升级后下一轮巡检生效

**客户侧感知**:
- 升级 v2.10.24.4 之后,后台「版本升级」弹窗会正确显示每个版本的对应说明
- 之前 v2.10.24.2 / v2.10.24.3 升级时看错说明的客户不受影响(已经升完了),未来发 v2.10.24.4+ 的时候弹窗就对了

## 教训

展示层 bug 容易漏,因为:
1. Codex 审阅只看逻辑正确性,不跑端到端
2. 开发者本地不走客户 Web 后台那条路径
3. v2.10.24.2 已经中过同样的坑(之前四段版本的),没人看就一直错

未来预防:每次发四段版本时顺手看一眼后台弹窗截图,或者加个单测。
