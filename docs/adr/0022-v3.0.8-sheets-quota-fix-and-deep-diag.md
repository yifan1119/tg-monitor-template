# ADR-0022: v3.0.8 Sheets 写入架构治本 + 卡死深度诊断 + 自助修复

## 背景

v3.0.7 之后两个不同客户案例同一周内炸出来:

**案例 A (某客户A 部门, 2026-04-25)**: 200+ 账号 × 5 peer × 2 Sheets API call (col_values 读 + update 写) ≈ 2000 calls/min, 远超 Google 60/min/user 配额。所有账号撞 429,per-account 退避升到 600s 上限,**1 天没写, 待写 3906 条**。诊断卡片说"30 分钟内自愈"实际不会自愈。

**案例 B (某客户B 部门, 同一天)**: 待写 794 条卡 1 天,**Google API 用量 ≈ 0** (Cloud Console 验证)。诊断卡片说"Sheet 不存在或权限被改"但「测试 Sheet 访问」绿勾。意味着 tg-monitor **根本没在调 Google API**, 问题在 SQL 层 (孤儿消息 / col_group=NULL / sheet_tab 缺失之类)。

两个案例都暴露:
1. **真根因不只配额**: 某客户B 是 DB 层卡住, quota 用量 0
2. **诊断不精确**: v3.0.6 的 404/403 关键词太松, false positive 把 Drive 上传 404 误判成 Sheet 不存在
3. **客户没自助路径**: 必须 SSH 跑 SQL 查 DB / docker restart / sed .env, 用户多次反馈 **不要再让客户操作终端**

## 决策

v3.0.8 4 块改动, 全部 **纯 web 自助**, 客户只需一次升级 + 浏览器点几下:

### 1. Sheets 写入架构治本 (sheets.py 核心改动)

#### 1a. write_messages 改用 `values.append` 替代 `update + col_values read`

旧路径 (v3.0.7-): `col_values` 1 次 read API + `ws.update` 1 次 write API = **每个 peer 2 次 API call**。某客户A 200 账号 × 5 peer = 2000 calls/min, 超 quota 3000%。

新路径 (v3.0.8): `ws.append_rows(rows, table_range="A:C", insert_data_option="OVERWRITE", value_input_option="USER_ENTERED")` 1 次 API call。响应里 `updates.updatedRange` 解析回 row 号给 `db.mark_written`。

**关键安全保证**: append 自动追加到 table_range 末尾, **永远不覆盖客户已有数据** (用户场景: 客户在表里手动改 / 插 / 删行, 老 update + 缓存 row counter 会覆盖客户改动, append 自动跟随当前末尾)。这是从原计划"DB 缓存 row counter"pivot 到 append API 的根本原因 — 用户审阅抓到 cache 会被客户改表破坏。

兼容: `include_values_in_response` 关键字老 gspread (<3.7) 不支持, try/except TypeError 降级。fallback: 如果响应没 `updatedRange` (极罕见), 走一次 `col_values` 反推 row 号。

#### 1b. 全局令牌桶限流 (`_rate_limit` 双层防护)

新增 `SHEETS_RATE_LIMIT_PER_MIN` (默认 50):
- 60 秒滑动窗口内最多 N 次 API call
- 满了 → sleep 到最早一个过期 + 0.1s
- 加在原 `_min_interval=1.5` 之上 (双层: per-call 间隔 + per-minute 总量)
- 用户在设置页可调 (5-60), Google 配额 60/min/user 不能超

#### 1c. SHEETS_FLUSH_INTERVAL 设置页可调

老的 `.env` 默认 5s, 大客户撞 quota 时调 30-60s 治标。设置页加输入框, 客户自助调, 不用 SSH 改 .env。

### 2. 驾驶舱深度诊断 + 自助修复

#### 2a. 新增 dashboard_api 函数

- `sheets_stuck_detail()`: 后台跑 SQL 列出 4 块明细
  - `per_account_unwritten`: 各账号未写消息数 (top 50)
  - `orphan_messages`: 有未写但 peer FK 拉不到的孤儿消息 (DB 层卡住根因 #1)
  - `peers_no_col_group`: `peer.col_group IS NULL` 的 peer (写入时 None * 3 TypeError 崩, 根因 #2)
  - `missing_worksheets`: accounts 表有但 sheet_tab 为空的账号
- `fix_orphan_messages()`: 把孤儿消息标 `sheet_written=1, last_write_error='ABANDONED_orphan_v308'` (放弃, 反正写不进去) — 解开 flush_pending 循环
- `fix_peers_no_col_group()`: 给每个 NULL peer 分配下一个空闲列组 (`MAX(col_group)+1` per account)

#### 2b. web.py 加 2 个 endpoint

- `GET /api/diag/sheets-stuck-detail` (`@login_required`): 纯读,返 stuck_detail JSON
- `POST /api/diag/sheets-fix-stuck` (`@login_required + @admin_required`): action ∈ {orphan_messages, col_group_null, all},返修复结果 + needs_restart 提示

#### 2c. dashboard.html 加「立刻深度诊断」按钮 + modal

诊断卡片 (sh-diag) 显示时旁边露出蓝色按钮 → click 弹 modal → 拉 4 块明细 → 渲染 → 检测可修复项自动显示「修复 N 条孤儿消息」/「修复 N 个 peer 列组缺失」/「一键修复全部」按钮 → confirm + POST → 显示成功 + 引导客户点「立刻重启监听器」让变动生效。

不可修复 (DB 层正常但仍卡) → 引导客户去用「立刻重启监听器」按钮 (v3.0.7.1 整合进来的能力)。

### 3. 驾驶舱「立刻重启监听器」按钮 (沿用 v3.0.7.1 设计)

诊断卡片 429 / 通用 warning 路径加按钮, 复用现有 `/api/restart` endpoint, 0 新后端代码 0 新安全模型变化。整合进 v3.0.8 同一驾驶舱。

### 4. 诊断关键词收紧 (`_diagnose_sheets_stuck` 重写)

v3.0.6 的 `404 / 403 / not found / permission denied / does not have access` 太松, 任何 Drive 上传 404 都误判成 "Sheet 不存在"。

v3.0.8 改成精确匹配:
- `worksheetnotfound` / `worksheet not found` → 「某账号 Sheet 分页被删或重命名」
- `spreadsheet not found` / `requested entity was not found` → 「整个 Spreadsheet 不存在或被删」
- `permission_denied` / `the caller does not have permission` / (`does not have access` + `sheet|range`) → 「Sheet 写权限被改或 OAuth scope 不足」
- `protected` + `range|sheet` → 「Sheet 某范围被锁定/受保护」
- 其他 → 通用 warning, 给「立刻重启监听器」按钮 + 引导深度诊断

不再误判 某客户B 那种 "Drive 上传 404 但 Sheet 端没问题"。

## 原因

1. **闭合两个客户案例**: 某客户A 走 1a/1b/1c 治本 (架构层 quota fit), 某客户B 走 2a/2b/2c 自助 (DB 层卡住可视 + 一键修)。两条独立路径不互相依赖

2. **零 SSH 自助原则**: 用户明确多次反馈"不要让客户操作终端"。所有 v3.0.8 新能力都暴露成 web UI: 设置页 input 框 + 驾驶舱按钮 + 弹窗 modal。SQL 在后端跑, 修复操作在后端执行, 客户全程浏览器

3. **append 替代 update 的安全性**: 用户在 v3.0.8 设计阶段抓出"DB 缓存 row counter 会被客户改表单破坏"的关键问题。append API 自动跟随当前末尾, 客户改/删/插行都安全, 这是从 row counter 缓存 pivot 到 append API 的根本原因 (Codex round-1 设计审阅没抓到, 用户审阅抓到)

4. **诊断关键词收紧**: v3.0.6 的关键词清单是匆忙建的 false positive 来源。v3.0.8 借机抽象 + 精确化, 跟 某客户B 案例直接相关

## 来源

- 某客户A 案例 (2026-04-25): 200+ 账号撞 Google quota 持续 1 天
- 某客户B 案例 (2026-04-25): 794 条卡 1 天, Google quota 用量 0, 测试 Sheet 绿勾, 诊断 false positive
- 用户多次反馈: "不要再让他们一直操作终端", "谷歌配额上限就是 60 没法提高"
- 用户设计阶段反馈: "写完直接 update 行号 ... 这他们表单可能会改" → pivot 到 append API
- Codex 多轮审阅 (v3.0.7 → v3.0.7.1 → v3.0.8 演进过程):
  - round-1 抓"in-process singleton 跨容器无效" → 改文件 IPC
  - round-2 抓"OAUTH_FAIL_MARKERS 两处分歧" → 抽到 oauth_helper 单一来源
  - round-3 抓"docs 提及 register_global_writer 残留 + 恢复时间各说各话" → 全部统一

## 后果

### 正面
- 某客户A 升级后 quota 用量从 ~2000/min 砍到 ~50/min, **不再撞 429**
- 某客户B 升级后驾驶舱「立刻深度诊断」按钮一点就看出 DB 层根因 + 一键修复, 不用 SSH
- 任何客户后续遇到 Sheets 卡死, 都有完整的纯 web 自助矩阵:
  - OAuth 失效 → 「去重新授权」(v3.0.6 + v3.0.7 自愈)
  - 429 配额超载 → 治本架构改动 (v3.0.8) + 「立刻重启监听器」兜底 (v3.0.7.1)
  - DB 孤儿/col_group 缺失 → 「立刻深度诊断」+ 一键修 (v3.0.8)
  - 通用 warning → 「立刻重启监听器」(v3.0.7.1)
- 设置页加 `SHEETS_FLUSH_INTERVAL` / `SHEETS_RATE_LIMIT_PER_MIN` 让客户自助调流速

### 潜在问题 / trade-off
- **append 写入比 update 慢一点点**: gspread `append_rows` 内部走 Sheets API `values:append`, latency 跟 update 同一个量级 (~200-500ms), 实测忽略不计。但响应里只返 `updatedRange`, 没返完整 cell metadata, 我们需要 regex 解析 row 号 → fallback 路径需 col_values 兜底
- **令牌桶用 list + 每次过滤**: 不如 deque popleft 快, 但 50 元素的 list filter 在 Python 是 µs 级, 不优化保持代码可读
- **修复孤儿消息标 sheet_written=1 是数据丢弃**: 这是 trade-off — 反正写不进去, 不放弃就永远卡。`last_write_error='ABANDONED_orphan_v308'` 留备忘可以追溯
- **dashboard.html 文件越来越大**: 1500+ 行 modal + JS 都堆在里面, 跟 v3.0.6 日志面板一起。下一版应该考虑抽 JS 出去

### 未涉及 (v3.0.9+)
- `tasks.py::_sheets_flush_loop` 启动健康检查 + TG 推送: 如果循环挂了 30s 没跑, 自动推预警群。考虑过, 但 某客户B 案例的"循环根本没跑"概率不高 (跟 #2 DB 层卡住相比), 留 v3.0.9
- 跨 peer batch_update: 一个账号所有 peer 合并到一次 `spreadsheets.values:batchUpdate` call (200 acc × 5 peer = 1000 → 200 calls/min), 实施复杂度高, 留 v3.0.9
- 跨 account batch_update: 整个 spreadsheet 一次 batch (1 call/min), 真终极治本, 留 v3.0.10
- 告警群推送 v3.0.8 严重事件 (孤儿消息检测 / 持续 429): 跟 v3.0.6 的「Sheet 卡」推送配套, 留 v3.0.9
- dashboard.html 抽 JS 文件: 重构, 留 v3.0.9
- 修复操作的审计日志: `fix_orphan_messages` 修了多少次, 谁修的 → 留备忘表, v3.0.9

## 升级指引

```bash
cd /root/tg-monitor-<部门> && ./update.sh
```

升级动作本身就会重启容器, 之前 某客户A/某客户B 卡住的状态自动重启清零再 + 走新代码:
- 新 append API → quota 用量自动降下来
- 新令牌桶 → 即使 quota 还是有压力也不撞 429
- 新设置页字段 → 客户能继续调
- 新驾驶舱按钮 → 客户随时点深度诊断 / 一键修复 / 重启监听器

## 验证

### 静态自查 (push 前)
- `grep -n "ws.append_rows\|ws.update.*current_row" sheets.py` — 应该有 `append_rows` 1 处, **无** `ws.update.*current_row` (除了 fallback)
- `grep -n "_call_window_max\|SHEETS_RATE_LIMIT_PER_MIN" sheets.py config.py web.py` — 三处都有定义/读取/写入
- `grep -n "sheets_stuck_detail\|fix_orphan_messages\|fix_peers_no_col_group" dashboard_api.py web.py templates/dashboard.html` — 后端 1 处定义 + endpoint 1 处 + 前端 modal 1 处
- `grep -n "/api/diag/sheets-stuck-detail\|/api/diag/sheets-fix-stuck" web.py templates/dashboard.html` — endpoint 双向引用对应
- DIV balance: dashboard.html `<div>:</div>` 必须平衡
- Python syntax + JSON syntax 全过

### 集成 (dev 容器)
1. 模拟孤儿消息: `INSERT INTO messages (account_id, peer_id, ...) VALUES (1, 99999, ...)` (peer_id=99999 不存在), `UPDATE messages SET sheet_written=0 WHERE id=last_insert_rowid()`. 驾驶舱「立刻深度诊断」应该显示 "1 条孤儿消息"
2. 点「修复 1 条孤儿消息」→ confirm → 应返 `{ok:true, fixed_count:1}`, DB 该条 sheet_written=1
3. 模拟 col_group=NULL: `UPDATE peers SET col_group=NULL WHERE id=1`, 「修复 1 个 peer 列组缺失」→ DB col_group 应被填上下一个空闲值
4. append API 真写测试: 给某 peer 写 5 条消息, log 应显示 `append API ... row N-N+4`, db.mark_written 应 5 条都填了正确 row
5. 令牌桶满测试: 配 SHEETS_RATE_LIMIT_PER_MIN=10, 触发 12 次 _rate_limit 调用, 第 11 次应 sleep 到 60s 窗口让位
6. 诊断卡片 false positive 修复: log 里只有 Drive 上传 404, **不应** 出现 "Sheet 不存在" 诊断 (走通用 warning 路径)

### 回归
- v3.0.7 OAuth 自愈路径不变
- v3.0.7.1「立刻重启监听器」按钮不变 (整合进 v3.0.8 同一驾驶舱)
- 老 `_rate_limit` 行为兼容 (仍有 1.5s per-call 间隔), 只是多了 60s 窗口的硬天花板
- `write_messages` 老 update 路径完全删除, 只保留 fallback `col_values` 兜底 (响应解析失败时)
