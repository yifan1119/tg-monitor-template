# ADR-0008: 两段式预警数据层 + UI 配置入口(批次 A)

**版本**:v2.10.25(v3.0.0 分批发布批次 A)
**日期**:2026-04-22

## 背景

客户提出两段式未回复预警需求 + accounts 加 4 字段配置 + 独立预警群。v3.0.0 整个功能拆成 4 批次发布(A/B/C/D),每批 feature flag 默认关,独立可回滚。

本 ADR 专门说批次 A — **数据层就位 + UI 入口可用,但不触发新行为**。

## 决策

### A1. DB migration_v2(已在 Day 1 完成)

```sql
-- accounts 表加 4 字段(两段式预警配置)
ALTER TABLE accounts ADD COLUMN business_tg_id TEXT DEFAULT '';
ALTER TABLE accounts ADD COLUMN owner_tg_id TEXT DEFAULT '';
ALTER TABLE accounts ADD COLUMN remind_30min_text TEXT DEFAULT '';
ALTER TABLE accounts ADD COLUMN remind_40min_text TEXT DEFAULT '';

-- alerts 表加 stage 字段(两段式阶段标识)
-- stage=0: 非两段式(老数据 / flag 关时的新数据)
-- stage=1: 两段式 stage1 已推送,等 stage2_after_min
-- stage=2: 已升级到 stage2
ALTER TABLE alerts ADD COLUMN stage INTEGER DEFAULT 0;
```

通过 `database.py:_migrate_to_2()`(幂等,基于 `PRAGMA user_version`)。保留 `type='no_reply'` 不变(见 ADR-0005)确保回滚兼容。

### A2. Config flags(已在 Day 1 完成 + 批次 A 补齐热 reload)

```python
TWO_STAGE_NO_REPLY_ENABLED = os.environ.get("TWO_STAGE_NO_REPLY_ENABLED", "false").lower() == "true"
NO_REPLY_STAGE2_AFTER_MIN = int(os.environ.get("NO_REPLY_STAGE2_AFTER_MIN", "10"))
UNREPLIED_ALERT_GROUP_ID = int(env) if env and env != "0" else 0
```

3 个 flag 默认值 = 老行为:
- 关 → 不走两段式
- 独立群 ID = 0 → fallback 到 `ALERT_GROUP_ID`(老群)
- stage2 等待 10 分钟(合计 30+10=40 分钟)

全部加入 `config.reload_if_env_changed()` 的热 reload 列表,Web 后台改完保存 → 容器里下一秒生效,不用 restart。

### A3. Account 级 4 字段配置 API

**新路由**:`GET/PATCH /api/accounts/<int:account_id>/notify-config`

```python
@app.route("/api/accounts/<int:account_id>/notify-config", methods=["GET", "PATCH"])
```

- GET:返回 4 字段当前值
- PATCH:只更新传入的字段(`db.update_account_two_stage` 内部已有「传 None 则跳过」的逻辑)
- 只接受预期字段(过滤其他 POST key),字段值 strip 空格

**为何单独 API 而不是扩 `/api/accounts` 整体 PATCH**:
- 现有 accounts 不在 Web UI 可编辑(只能 setup 时设 company/operator + sync_headers 读 B2/B3)
- 4 字段属于两段式专属,独立 endpoint 更清晰
- 后续如果 v3.0.0 要加「批量导入 4 字段」API,这个 endpoint 可以扩 POST

### A4. Settings 页加 UNREPLIED_ALERT_GROUP_ID 输入

- `templates/setup.html`:02 · TG Bot 部分「审核按钮白名单」之后加「未回复预警专用群 Chat ID」输入
- `{% if d.two_stage_no_reply_enabled %}` 条件渲染 — flag 关时不显示
- `web.py:settings_page` defaults dict 加 `unreplied_alert_group_id` + `two_stage_no_reply_enabled`
- `web.py:_save_settings` updates dict 加 `UNREPLIED_ALERT_GROUP_ID`

### A5. 账号管理页「配置两段式」按钮 + 模态框

- `templates/index.html` 账号卡片右侧加「配置两段式」按钮(紧邻「移除」按钮)
- 模态框 4 字段输入 + 保存
- `{% if two_stage_no_reply_enabled %}` 条件渲染 — flag 关时:
  - 按钮不显示
  - 模态框 HTML 不渲染
  - 模态框 JS(openNotifyConfig / saveNotifyConfig)不定义

- `web.py:get_sessions()` 在返回的 session dict 里加 `id`(account_id),方便模板调 API

### A6. 行为不变保证

**flag 关(默认)= v2.10.24 行为一字不差**:
- 设置页多出的「未回复专用群」字段不渲染 → 客户看不到
- 账号卡片「配置两段式」按钮不渲染 → 客户看不到
- API `/api/accounts/<id>/notify-config` 可访问(便于预配置)但字段值不被任何业务逻辑读取
- alerts.stage 默认 0,老扫描逻辑 `type='no_reply' status='pending'` 不带 stage 过滤也正常工作

## 原因

### 为什么不把 4 字段走 Sheet B4-B7 同步?

**方案对比**:

| 方案 | 优点 | 缺点 |
|---|---|---|
| Sheet 同步(扩展 sync_headers) | 跟现有 B2/B3 一致体验,客户在 Sheet 上统一管理 | 破坏现有 row 2-6 布局,row 5-6 已经是对话槽 header,改会冲突 |
| Web 后台配置(本方案) | 布局清爽,独立 UI,动态编辑不用每次改 Sheet | 客户 meeting 里描述偏向 Sheet 侧 |

**决定选 Web 方案**因为:
- 现有 Sheet 布局已经够复杂,再加 4 字段排版困难
- Web UI 支持热编辑、保存即生效,客户体验更好
- 如果客户强烈要求 Sheet 侧也能填,可以在批次 B 再加一个 sync_headers 扩展(读 row 2-3 之外的行),两个入口并存

### 为什么用模态框而不是 inline 展开?

- 200+ 账号页面,每行都 inline 4 字段输入 UI 太拥挤
- 模态框保持列表简洁,只有编辑时弹出
- 单账号编辑完关闭即可,不影响浏览其他账号

### 为什么 API 在 flag 关时也可访问?

- 允许 admin 用 curl 或自动化脚本预配置 4 字段(在打开 flag 之前)
- 避免鸡生蛋问题:客户要填字段之前必须先开 flag 才能看到 UI
- 字段值写了也没用(flag 关业务逻辑不读),纯数据存储

## 后果

### 立即效果
- 升级 v2.10.25 后:
  - `messages.delete_mark_pending`、`alerts.stage`、`accounts` 4 字段自动 migration(user_version 0→1→2 或 1→2)
  - .env 不改 → 行为 100% = v2.10.24
  - .env 改 `TWO_STAGE_NO_REPLY_ENABLED=true` → UI 立刻显示 4 字段配置入口(但没有 stage1/stage2 推送,因为批次 B/C 才加)
- 想用两段式的客户可以**在批次 A 阶段先填字段**,等批次 C 发布后立刻生效

### 批次 A 结束时不做的事(留给 B/C)
- ❌ bot.send_no_reply_alert_stage1 / stage2 推送函数(批次 B)
- ❌ tasks._no_reply_stage2_loop 扫描(批次 C)
- ❌ listener outbound 事件驱动(批次 C)

### 回滚
`bash rollback.sh` 回 v2.10.24。SQLite 字段保留(nullable,老代码不读),UI 自然消失(模板文件回老版本)。完全无损。
