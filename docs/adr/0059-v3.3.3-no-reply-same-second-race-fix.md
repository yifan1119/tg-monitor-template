# ADR-0059 · v3.3.3 未回复预警同秒回复误推 + stage2 兜底失效双重修

## 背景

2026-05-19 某客户线上反馈:外事号收到「未回复预警」并 @ business 负责人。但客户
实际看截图:广告主同秒发问题,商务**同秒就回了**(标准 TG Business 转介绍话术)。

SSH 上客户 VPS dump DB(脱敏后):

```
peer_id=<X> 当天全部消息:
  msg_id=N    direction=B  timestamp='2026-05-19 HH:MM:SS'  deleted=0  "<对方问题>"
  msg_id=N+1  direction=A  timestamp='2026-05-19 HH:MM:SS'  deleted=0  "<我方回复>"

peer_id=<X> 当天 alerts:
  type=no_reply  stage=2  msg_id=N  created_at='2026-05-19 HH+0:39:MM'  bot_message_id=<Y>
```

**两条消息都在 DB,都没删,timestamp 精确到秒完全相同**(15:39:09)。但系统在 30
分钟后(16:09:41)推了 stage1,又 10 分钟后升级 stage2 @ 负责人。

## 根因(双重 bug,叠加触发)

### Bug #1 — `get_unanswered_candidates` SQL 同秒 race

`database.py::get_unanswered_candidates` 找未回复对话用:

```sql
SELECT p.*, ... FROM peers p
JOIN messages m ON m.peer_id = p.id
WHERE p.account_id = ?
  AND m.id = (SELECT id FROM messages WHERE peer_id = p.id AND deleted=0
              ORDER BY timestamp DESC LIMIT 1)
  AND m.direction = 'B'
```

subquery `ORDER BY timestamp DESC LIMIT 1` **没 tiebreaker**。同 timestamp 时 SQLite
不保证选哪条。实测在本案场景下选了 159886 (B),外层 `m.direction='B'` 通过 →
peer 进 candidates → 30 分钟后推 stage1。

SQLite 文档明确:`Without an ORDER BY tiebreaker, equal-valued rows are returned in
an unspecified order`,实际行为依赖 query plan + index 选择,生产不可预测。

### Bug #2 — `_no_reply_stage2_loop` 兜底用错时间起点

`tasks.py::_no_reply_stage2_loop` 升级 stage1 → stage2 前有兜底:

```python
if db.has_outbound_since(row["peer_id"], row["created_at"]):
    db.mark_stage1_handled_by_reply(...)
    continue
```

`has_outbound_since` SQL:`WHERE direction='A' AND timestamp > since_ts`。本案传入
`since_ts = alert.created_at = 16:09:41`,但 outbound timestamp = 15:39:09(早于
alert),`timestamp > 16:09:41` 永远 false → 兜底失效 → 升级 stage2。

正确语义应该是「**alert 那条 incoming 之后**有没有 outbound」,不是「alert 创建
之后」。同秒 outbound 也应算回复(`>=` 不是 `>`)。

## 决策

### 1. `get_unanswered_candidates` subquery 加 tiebreaker

```sql
ORDER BY timestamp DESC, msg_id DESC LIMIT 1
```

- 同 timestamp 按 `msg_id DESC` 走 TG 服务器消息号(同对话内严格递增,schema
  `UNIQUE(msg_id, account_id)` 保证)
- **不偏置 direction**(Codex round 1 P1 抓出 `direction ASC` 偏置会让「同秒 A→B
  我方先回客户秒追」漏告警 — 改 `msg_id DESC` 跟 TG 客户端看到的对话顺序完全一致)
- 客户实际 case:159886(B)+159887(A)同秒,`msg_id DESC LIMIT 1` 拿到 159887 (A)
  → 外层 filter `direction='B'` 把整 peer reject → 不进 candidates → 不推 stage1
  ✓ 根治

### 2. 新增 `stage1_resolved_by_reply(peer_id, alert_msg_id)` 取代兜底用 `has_outbound_since`

```python
def stage1_resolved_by_reply(peer_id, alert_msg_id):
    in_row = conn.execute(
        "SELECT timestamp FROM messages WHERE peer_id=? AND msg_id=? LIMIT 1",
        (peer_id, alert_msg_id)
    ).fetchone()
    if not in_row:
        return False
    out = conn.execute(
        "SELECT 1 FROM messages WHERE peer_id=? AND direction='A' AND deleted=0 "
        "AND timestamp >= ? LIMIT 1",
        (peer_id, in_row["timestamp"])
    ).fetchone()
    return out is not None
```

- 时间起点改用 `alert.msg_id` 反查 incoming `timestamp`(不是 `alert.created_at`)
- 比较改 `>=`(含同秒回复)
- 精准看「**alert 那条 incoming 之后**有没有 A」,不会被「老 alert 等待时来新 B」
  误判已升级(Codex round 1 P0 关切的场景 — 老 alert 期间客户回了 A,又来新 B 追问,
  新 B 是独立对话该新创 stage1 不该把老 alert 升 stage2)

### 3. 老 `has_outbound_since` 保留不删,标 DEPRECATED

向后兼容(本版无新 caller,老 API 还在防 third-party 引用,标注 deprecated 让未来
版本知道可删)。

### 4. 调用方 `tasks.py:631` 切换到新 helper

```python
# - 老: if db.has_outbound_since(row["peer_id"], row["created_at"]):
# + 新: if db.stage1_resolved_by_reply(row["peer_id"], row["msg_id"]):
```

## 原因

- **0 schema 变 / 0 migration / 0 重登**(纯 SQL 行为修),hotfix 安全
- 加 tiebreaker 是 Codex round 1 验证后第二轮选定的「最稳」方案(对比 `direction ASC` /
  `id DESC` / `msg_id DESC`,只有 `msg_id DESC` 不偏置 direction 且跟 TG 对话原生
  顺序一致)
- 兜底从「时间起点 +`>`」改「incoming timestamp + `>=`」,直接堵了同秒 race + 老
  alert 等待场景两类失效
- 老 API `has_outbound_since` 保留,符合 Surgical Changes 原则(只动必须动的)

## 来源

- 客户线上反馈:截图「未回复预警」 + 实际对话有同秒回复
- SSH dump:`docker exec tg-monitor-<dept> python3 sqlite3 SELECT messages` 实测 A+B 同 timestamp 都进 DB
- Codex round 1 review:抓出 `peer_last_direction_is_a` 在「老 alert + 新 B」误判
  + `direction ASC` 偏置同秒方向 → 改 `stage1_resolved_by_reply` + `msg_id DESC`

## 后果

### 正向

- 客户同秒回复(TG Business 自动回 / 商务超快回 / 第三方代发同秒)不再误推 stage1
- stage2 兜底用 incoming timestamp + `>=` 同秒含,即使将来 `get_unanswered_candidates`
  漏堵也能在升 stage2 前救一次

### 风险 / 边界

- **同秒 A→B 顺序**(我方先回,客户秒追):新 SQL `msg_id DESC` 拿到 B(对话最后一条
  是客户追问)→ 进 candidates → 30 分钟后正常推 stage1。**这是想要的行为**,客户
  追问应该继续告警。
- **A 是删除消息**:`get_unanswered_candidates` subquery 用 `deleted=0` 过滤,删了的
  A 不算最后一条,可能拿到 B 触发告警 — 跟现状一致,本版不动。
- **listener 漏接 A(A 没进 DB)**:不在本版修复范围(任何 SQL 救不了不存在的数据)。
  TG Business / 第三方代发 / Telethon 离线漏接的 outbound,要靠 v3.0.0 的 polling 兜
  底拉,本版不动。

### 性能

- subquery `ORDER BY timestamp DESC, msg_id DESC` 在 `idx_msg_peer(peer_id)` 单列
  索引下需要 temp sort,但同 peer 行数通常 < 1000,sort 开销可忽略
- 若大客户(几十万消息/peer)性能 regression,可加复合索引 `idx_msg_peer_ts_msgid`
  (`peer_id`, `timestamp DESC`, `msg_id DESC`),本版不动

## 验证

### 单元测试(已在本地跑通)

5 种场景 SQL 行为:
1. ✅ A+B 同秒(A msg_id 大,客户实际 case):candidate=None,stage1_resolved=True
2. ✅ A 早 B 晚(正常未回复):candidate=B
3. ✅ 老 alert 后客户回 A,又来新 B(Codex P0):stage1_resolved=True(精准看 alert
   msg_id 对应 incoming 之后有 A → 抑制 ✓ 不会被新 B 误判)
4. ✅ 真没回复:stage1_resolved=False(该升 stage2)
5. ✅ 同秒 A→B(Codex P1,我方先回客户秒追):candidate=B(应告警 ✓)
