# ADR-0002: Session health sweep 加 `get_me()` 真 RPC 探测

**版本**:v2.10.23
**日期**:2026-04-21

## 背景

某账号案例(2026-04-21):
- 某个外事号(`+xxxxxxxxxxxx`)被 TG 官方冻结/封号(用户举报 + 人工审核确认)
- 但监控系统 UI 显示 🟢 ONLINE,预警群也没发离线通知
- 5 分钟 session 巡检通过,系统认为账号正常

查代码 `tasks.py:_check_single_session`:

```python
# v2.10.22 逻辑
authorized = await asyncio.wait_for(client.is_user_authorized(), timeout=10)
return "healthy" if authorized else "revoked"
```

**问题**:Telethon `is_user_authorized()` 只检查 session key 是否有效。TG 冻结账号的机制是:
- Session key **仍然有效**(`is_user_authorized() = True`)
- 但真正调 API(`get_me()` / `send_message()` / `iter_messages()`)时才抛 `UserDeactivatedBanError` / `UserDeactivatedError`

所以光靠 `is_user_authorized()` **永远检测不到冻结/封号**。

通过 @SpamBot 确认该账号封号原因:
> Your account was blocked for violations of the Telegram Terms of Service based on user reports confirmed by our moderators.

## 决策

`_check_single_session` 加 `get_me()` 真 RPC 探测,捕获冻结/封号相关异常:

```python
async def _check_single_session(self, phone, client):
    try:
        if not client.is_connected():
            await asyncio.wait_for(client.connect(), timeout=15)

        authorized = await asyncio.wait_for(client.is_user_authorized(), timeout=10)
        if not authorized:
            return "revoked"

        # v2.10.23: 真 RPC 探测
        try:
            await asyncio.wait_for(client.get_me(), timeout=10)
        except asyncio.TimeoutError:
            return "error"   # 网络抖动,不触发转场
        except Exception as e:
            cls = type(e).__name__
            msg = str(e)
            if "FloodWait" in cls:
                return "healthy"   # 限流 ≠ 死
            dead_kw = ("Deactivated", "AuthKey", "Revoked", "Banned", "Unauthorized", "SessionExpired")
            if any(k in cls for k in dead_kw) or any(k in msg for k in dead_kw) or \
                    any(k in msg for k in ("unauthorized", "deactivated", "banned", "revoked")):
                return "revoked"
            return "error"
        return "healthy"
    except Exception as e:
        # 兜底同老逻辑
        ...
```

## 原因

### 为什么不直接调 `get_me()` 不看 `is_user_authorized()`?
保留 `is_user_authorized()` 作为 fast path — 如果 session key 本身无效(比如 session 文件损坏),立刻 return `revoked`,不用再跑 RPC。

### 为什么不直接用异常类型判断?
不同 Telethon 版本可能抛不同异常类名(比如 1.36 vs 1.43)。用「类名命中关键字 OR 消息命中关键字」双重匹配,防御性更强。v2.10.21 已经遇到过类似问题(ImportError)。

### 为什么 FloodWait 视为 healthy?
FloodWait 只是 TG 暂时限流,账号本身健康。如果当 revoked 处理,会误报账号离线。

### 为什么 TimeoutError 视为 error 不 revoked?
Telethon 底层可能因为网络抖动 RPC 超时,不能据此判定账号死了。连续多次超时会自然累积成 `error` 状态,不触发 healthy → revoked 转场(避免 flap)。

## 来源

1. 2026-04-21 某账号案例 — 客户反馈 UI 绿色但实际冻结
2. Codex 审计 Major B5 — 建议增加真实 RPC 探测和 Telethon 异常分类
3. 本地 Bash 调试:`docker exec ... python -c "from telethon import TelegramClient; ..." ` 手动探测确认 `get_me()` 会抛 `UserDeactivatedBanError`

## 后果

### 立即效果
- 升级 v2.10.23 后,5 分钟巡检周期内,被冻结/封号的账号会被正确标红
- 预警群会收到「外事号离线预警」
- 驾驶舱 UI 从 🟢 变 🔴「会话已吊销」

### 额外 API 调用成本
- 每账号每 5 分钟多一次 `get_me()` 调用
- TG API 对 `get_me()` 限制宽松,200 账号每 5 分钟 = 40 次/分钟,远低于限流
- 可接受

### 冻结原因无法查
- TG 只给异常类名,**不给冻结原因**
- 原因只能通过 @SpamBot(手机 TG 操作)查
- 监控系统到此为止,再深入需要业务侧人工处理

### 相关代码
- `tasks.py:_check_single_session`(改动主体)
- `tasks.py:_session_health_loop`(调用方,不改)
- `bot.py:send_session_alert`(离线通知,不改)
