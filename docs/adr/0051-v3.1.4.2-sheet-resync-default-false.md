# ADR-0051 · v3.1.4.2 SHEET_RESYNC_ENABLED 默认 true → false(保 Sheet 时间顺序)

## 背景

2026-05-18 客户反馈悦达-渠道 dept Sheet「晴晴(官方收量)」分页时间错乱:

```
row 253: 2026-05-17 15:23:43
row 254: 2026-01-22 10:33:07
row 270: 2026-05-18 03:31:39
row 271: 2026-01-22 01:05:40
```

2026-05 跟 2026-01 / 2026-02 数据混杂,客户审查时无法按时间溯源。

## 根因

v3.1 ADR-0027 设计 `peers.next_sheet_row + _sheet_position_resync_loop` 用于「客户删 Sheet 中间行 → 新消息自动回填空位」。流程:

1. 客户手动删 Sheet 中间几行 → 留空位
2. `_sheet_position_resync_loop` 每 15 分钟扫 `ws.get_all_values()` → 找首个空行更新 `peers.next_sheet_row`
3. `write_messages` 用 `update` 命中 next_row(而非 append 末尾)→ **新消息塞中间空位**
4. 视觉上时间错乱(新消息 timestamp 跟旧消息混杂)

**这是 v3.1 设计就有的副作用**,跟 v3.1.4 SA 升级无关(SA 没动 sheets.py 写入逻辑)。但客户开始大量审查时才暴露(以前数据少不明显)。

## 决策

**`SHEET_RESYNC_ENABLED` 默认值改 `"true"` → `"false"`**:

```python
# config.py:455
SHEET_RESYNC_ENABLED = os.environ.get("SHEET_RESYNC_ENABLED", "false").lower() == "true"
```

升级后:
- 客户 `.env` 没设 `SHEET_RESYNC_ENABLED` → 默认 false → write_messages 永远 append 末尾(v3.0.9 行为)
- 客户 `.env` 显式 `SHEET_RESYNC_ENABLED=true` → 仍走老 next_row 回填空位逻辑(用户可选)

## 原因

1. **时间顺序优先于自动回填**:Sheet 是审查/合规工具,时间顺序错乱比偶尔有空白行严重得多。客户删 Sheet 行的频率远低于「按时间审消息」的频率
2. **append 末尾是 v3.0.9 之前的稳定行为**(数年验证):新消息永远末尾,时间单调递增,客户审查可按行号 = 时间倒序信任
3. **可选回退**:客户特殊场景仍可显式开 true(向后兼容)

## 来源

- 2026-05-18 客户截图反馈(悦达-渠道,晴晴 Sheet)
- 跟 v3.1.4 SA 一起 push(v3.1.4.2),陈家碧 9 IP 升级时一并生效

## 后果

**修改影响**:1 行(config.py:455 默认值字符串)

**升级路径**:
- 客户跑 `update.sh` → 自动 git reset 拉新 default → docker compose recreate → 下条预警按 append 模式写
- 老客户 `.env SHEET_RESYNC_ENABLED=` 没显式设值 → 自动获得新默认 false 行为
- 删 Sheet 中间行的客户(陈家碧悦达-渠道) → 升级后中间空白行不会被新消息回填(空着,客户自己看)

**回退**:`.env SHEET_RESYNC_ENABLED=true` + docker restart 即可恢复 v3.1 next_row 回填行为
