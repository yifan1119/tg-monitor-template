# ADR-0036 v3.0.17 — 账号吊销 @ 监察员 + 异地登录区分 + 实时预警全覆盖中央台

**状态**: 已实施
**版本**: v3.0.17
**日期**: 2026-05-11

## 背景

客户提的 6 项优化诉求第 3、5、6 项落到本版:

1. **账号 TG 会话失效时,预警群里没人 @,监察员看不到自己负责的号出事**
   - v2.10.4 起的 `session_revoked_alert` 文案是「外事号:xxx 状态:❌ 登录会话已失效」纯通知,
     200+ 号刷过来谁也分不清是谁负责的。

2. **VPS 多部门一台机时,预警分不清是哪台 / 哪部门**
   - 文案只带 `COMPANY_DISPLAY` 没 IP,客户多部门多 VPS,看不出物理位置。

3. **被盗号(异地登录)跟普通终止会话混一起,处理优先级分不开**
   - Telegram 的 `AuthKeyDuplicated`(别人在另一台机登录把你踢了)是高风险事件
     (对方此刻能看你聊天 + 发消息),应该跟「自己点终止其他会话」分开提示。

4. **实时预警里,session 类(吊销/恢复)+ 关键词类没走中央台路由**
   - v3.0.16 只把 stage1 未回复预警接入中央台,其他类型还是各 VPS 自己 bot 推
     → 跟「按公司归属统一管理」的目标不一致。

## 决策

**5 件事一次做:**

1. `accounts.inspector_tg_id`(v3.0.15 已加)在吊销/异地预警时被读出来 →
   `_build_tg_mention()` 解析(@username 直接 native @,纯数字 UID 走 Telethon 拿真名 inline @) →
   塞到消息里。

2. 消息标题改成 `【外事号离线预警 · 部门 (IP)】`,IP 取自 `.env::VPS_PUBLIC_IP`
   (install.sh / update.sh 已自动写,老 VPS 升级也会补)。

3. 新增 `templates.session_hijacked_alert` 文案,跟普通 revoked 区分。
   `tasks._check_single_session` 加 `AuthKeyDuplicated` 关键字优先匹配 → 返新状态 `'hijacked'`。
   `_session_health_loop` 把 healthy ↔ revoked / hijacked 三态转场都识别。

4. `bot.send_session_alert` 增加 `kind="hijacked"` 分支,共用同一推送骨架。

5. **中央台路由扩到 session + keyword**:
   - `send_session_alert` 调 `_try_central_route()`,routed → 跳本地推
   - `send_keyword_alert` 同
   - `send_no_reply_alert_stage2` / `send_delete_alert` **不走中央台** —
     这两类带「登记违规/取消」按钮,callback 必须回到原 bot,中央台另一个 bot 收不到回调
     → 强行路由会让按钮失效。这是有意保留,不是疏漏。

## 原因

- **加 @ 监察员** = 客户最常吵的痛点(「不知道谁的号挂了」),数据(`inspector_tg_id`)v3.0.15 已就位
  ,本版只缺消费侧。
- **加 IP** = 共享 VPS 多部门客户的痛点,IP 已经在 `.env::VPS_PUBLIC_IP` 里(v3.0.16 写入),
  零新基础设施。
- **细分 hijacked** = 被盗号比终止会话风险高一个数量级(对方仍能 actively 用号),
  不能用同一文案让客户掉以轻心。`AuthKeyDuplicated` 是 Telegram 官方明确语义,
  judge 准确度高,误报极低。
- **session/keyword 走中央台** = 跟 v3.0.16 的「按公司归属统一管理」目标对齐,
  减少各 VPS 单独配 bot 群的运维负担。
- **stage2/delete 不走中央台** = button callback 必须回原 bot,这是 Telegram 协议限制,
  不是设计缺陷。如要解决需中央台代理回调或 webhook bridge,留 v3.0.18+。

## 实施

| 文件 | 改动 |
|---|---|
| `templates.py` | 新增 `_vps_tag()` helper;`session_revoked_alert` / `session_restored_alert` 加 `inspector_mention` / `host_ip` / `company_display` 参数;新增 `session_hijacked_alert` 模板 |
| `tasks.py:_check_single_session` | get_me 异常分支加 `AuthKeyDuplicated` 优先匹配 → 返 `'hijacked'`;tail except 同处理 |
| `tasks.py:_session_health_loop` | 状态转场判定从 `revoked` 扩到 `(revoked, hijacked)`;恢复转场反向同样扩 |
| `bot.py:send_session_alert` | 加 `kind="hijacked"` 分支;读 `account.inspector_tg_id` 解析 mention;读 `VPS_PUBLIC_IP`;调 `_try_central_route` 优先路由;`parse_mode="HTML"` 渲染 inline mention |
| `bot.py:send_keyword_alert` | 加 `_try_central_route` 调用,routed → 跳本地 |

## 来源

- 客户提的 6 项优化诉求(plan: hazy-floating-tiger.md 第 3、5、6 项)
- 用户拍板(2026-05-08):「内事号 = 监察员/商务自己的某个标识」「VPS 位置显示部门名 + IP 都显示」「实时预警全覆盖,stage2/删除/关键词/吊销 全走中央台」
- 设计妥协:stage2/delete 因 callback button 协议限制保持本地,跟用户口头「全走」打折扣 —
  在文档显式说明 trade-off,留 v3.0.18+ 探讨 webhook bridge 方案。

## 后果

**正面:**
- 监察员一眼看到自己负责的号出事(@ 直接通知到人)
- 多 VPS 客户分得清是哪台机出事(IP 标签)
- 被盗号场景有专门高优先级文案(三步处理:终止会话 → 改两步密码 → 重登录)
- session/keyword 类预警按公司归属统一推送,减少各 VPS 配 bot 负担

**负面 / 注意:**
- 老账号没设 `inspector_tg_id` → 模板兼容,无 @ 行,文案其余照旧(向后兼容)
- 老 VPS 没设 `VPS_PUBLIC_IP` → IP 括号不显示,标题退化为 `【... · 部门】`(v3.0.16 install.sh / update.sh 已自动补,本版不需要客户额外配)
- stage2 / delete 仍走本地推,跟「全走中央台」的口头预期有 gap(已在本 ADR 显式说明)
- 中央台未配置(`CENTRAL_PUSH_URL` 空)→ session/keyword 自动 fallback 本地推,老客户升级零感知

## 验证

1. **吊销文案**:`./update.sh` 后 dev VPS 手动 logout 一个号 → 5 分钟内 `_session_health_loop` 触发 → 预警群收消息含 `【外事号离线预警 · 悦达-运营 (76.13.x.x)】` + `监察员:@xxxx`
2. **异地文案**:用第二台机器 telethon 登录同一个号 → 第一台 `get_me()` 抛 `AuthKeyDuplicated` → 预警群收 `【⚠ 外事号异地登录 · ...】` + 三步处理指引
3. **恢复文案**:重新登录该号 → `_session_health_loop` 检测 healthy 转场 → 预警群收 `【外事号恢复通知 · 部门 (IP)】`
4. **中央台路由**:配好 `CENTRAL_PUSH_URL/TOKEN` + 中央台对应公司路由 → session/keyword 预警走中央台对应 bot 群,本地 ALERT_GROUP_ID 不收
5. **fallback**:中央台关掉 → session/keyword 自动 fallback 本地 ALERT_GROUP_ID
