# ADR-0027: v3.1 Sheet 后台扫描 + 自动回填空位 — 客户删旧消息后新消息自动补上

## 背景

线上某客户(悦达渠道,72.62.252.175)反馈:`Tommie Scoles` 这个 peer 在 Sheet 上看着只有 4-21 的一条数据,后面大片空白,但驾驶舱显示"Sheet 写入健康",bot 也推了 22:32 的关键词告警 — 客户认为系统漏写。

**深查 v3.0.8 `sheets.py:1063` `write_messages` 发现**:

```python
# table_range = "{col_a}:{col_c}" 限定本列组的 3 列, 不会跨列组打架  ← 错误注释
ws.append_rows(rows, table_range=f"{col_a}:{col_c}", insert_data_option="OVERWRITE", ...)
```

**Google Sheets API 实际行为**:`table_range` 参数只是**提示**,Google 自动检测整张 worksheet 的"table boundary"(连接的数据区),`values.append` 实际写到**整张表的最底部行 + 1**,**不限于 3 列范围**。

**实测验证**(API 拉 38 个 peer 的 sheet_row 范围):
```
col_g  peer            msgs  min_row  max_row
3      百合花 绿        38    7        421
8      Tommie Scoles  72    7        420
20     若风          110    7        309    ← 110 条消息但只占 7~309 行 (有空)
28     VL             28    7        349
37     大卫           6     328      333    ← 后期加的 peer 一开始就在 328
```

**所有 peer 的 max_row 都被全表"齐头并进"推到 280-421**,无论自身消息数。`Tommie Scoles` 4-21 写第 7 行 → 4-22~4-25 期间其它 peer 写到 row 350 → 4-26 Tommie 又来消息时 Google 把"全表底部"算成 row 350+ → Tommie 自己的 Y/Z/AA 列在 row 8~349 大片空白 → 客户截图就是这个。

更关键痛点:**客户手动删 Sheet 旧消息**时,append 永远跟全表底部走,**不会回填空位**。这是 v3.0.8 设计 trade-off 的副作用(为了砍 quota 从 update+col_values 换成 append),客户期望"删旧消息系统自动接续"完全不满足。

## 决策

v3.1 双路径写入 + 后台定时扫描更新 DB 缓存的 peer 行号:

### 1. `database.py` migration V6
`peers` 加两列:
- `next_sheet_row INTEGER DEFAULT NULL` — 下次 update 应写第几行(NULL = 未扫过 → fallback append)
- `next_sheet_row_resynced_at TEXT DEFAULT NULL` — 上次 resync loop 计算时间(driver 显示 + 跨容器读)

新增 helpers:`get_peer_next_sheet_row` / `set_peer_next_sheet_row` / `bump_peer_next_sheet_row` / `invalidate_peer_next_sheet_row` / `get_all_peers_with_col_group` / `get_max_resynced_at` / `get_all_accounts`。

### 2. `config.py` 3 新 config
```python
SHEET_RESYNC_INTERVAL_MINUTES = 15        # 后台扫描周期
SHEET_RESYNC_ENABLED          = True      # 关掉退回 v3.0.9 纯 append
SHEET_RESYNC_VERIFY_BEFORE_WRITE = False  # 强保护:写前 acell 验证那行真空(quota×2)
```

### 3. `sheets.py` 写入路径双轨

**`write_messages` 决策**:
1. `SHEET_RESYNC_ENABLED=false` → 永远 append(等价 v3.0.9)
2. `peers.next_sheet_row IS NULL` → 没扫过 → fallback append(等下轮 resync)
3. 否则走新 update 路径,命中 DB 缓存的行,写完 `bump_peer_next_sheet_row(+N)`

**`_write_messages_via_update`**:`ws.update(f"{col_a}{next_row}:{col_c}{end_row}", rows)`。响应 `updatedRange` 校验 `actual_start == next_row`,不一致 invalidate(防御 gspread 行为变化)。强保护开关下写前 `ws.acell` 验空,quota 翻倍但 0 覆盖。

**`_write_messages_via_append`**:抽出 v3.0.8 老逻辑(零行为变化),作 fallback 兜底。

**`_post_write_finalize`**:两路径共用收尾 — `mark_written` + 删除标红 backfill。

### 4. `sheets.py` 新增 `resync_peer_positions()`
- 持 `_write_lock` 跟 `flush_pending` 串行(不撞 update 路径的 bump)
- 每个 worksheet **1 次** `ws.get_all_values()`(API 节俭关键 — 跟 peer 数无关)
- 本地 `_scan_first_empty(values, col_group * 3)` 找首空行 → `db.set_peer_next_sheet_row`
- 跑完更新 `self._last_resync_ts` / `self._last_resync_stats` 给 dashboard 看

**`_scan_first_empty(values, col_start, header_rows=6)`** 是模块级纯函数(可单测,不依赖 SheetsWriter 实例):跳过前 6 行 header,从 row 7 开始找第一个 (a/b/c) 三列都空的 row,1-based 返回。整列满 → `len(values) + 1`。

### 5. `tasks.py` 新增 `_sheet_position_resync_loop`
- 启动后等 30s(避冷启动)
- 走 `asyncio.to_thread(self.sheets.resync_peer_positions)`(`_rate_limit` 内部 `time.sleep` 同步,放进线程不阻 event loop)
- `SHEET_RESYNC_ENABLED=false` 时 5s 轻量心跳(跑不动事但 stop 能响应)
- **on-demand flag**:每 5s 检查 `data/.sheet_resync_request` 文件,存在就立即跑一轮 + 删文件(给「立刻重扫」按钮用,跨容器 IPC,无需 RPC)

### 6. `dashboard_api.sheets_health()` 加 4 字段
`resync_enabled / resync_interval_min / last_resync / last_resync_human`。跨容器(tg-web 拿不到 SheetsWriter)走 `db.get_max_resynced_at()` — 读 `MAX(peers.next_sheet_row_resynced_at)`。

### 7. `web.py` `/api/sheets/resync-now`(@admin_required)
写文件标志 `data/.sheet_resync_request`,由 tasks loop 每 5s 检查触发。**跨容器 IPC 模式**跟 ADR-0021 OAuth token 共享文件一致。

### 8. `setup.html` / `dashboard.html` UI
- 设置页加扫描间隔 + 启用开关 input
- 驾驶舱 Sheet 健康卡新增「行号扫描:N 分钟前 (每 15 分钟) [立刻重扫]」(admin-only 按钮)

## 原因

1. **客户痛点真实存在且我之前误诊** — 我最初告诉客户"38 peer 共用 16 槽导致混乱",Plan agent 实测发现 col_group 单调递增不复用,**真实是 Google append 跨列推高行号**,不是 slot 问题。这次 ADR 拨乱反正。

2. **跟 v3.0.8 治本目标互补不冲突** — v3.0.8 砍 quota(`append` 替代 `update + col_values`),F 治"客户感知行号错位 + 删除回填"。新 update 路径**仍是 1 次 API/peer**(跟 append 同 quota 占用),后台扫描走读 quota(独立预算 300/min,跟写 60/min 不抢)。

3. **batchGet 优化关键** — 1 worksheet 只调 1 次 `get_all_values`,无视 peer 数。100 worksheet × 4 次/小时 = 6.7 reads/min,远低于 300/min 上限。

4. **0 数据迁移 0 重登 0 强制行为变化** — `peers.next_sheet_row=NULL` 老数据天然走 append fallback,首次 resync 后切 update 路径。客户 < 16 peer 体感零变化。

5. **跨容器 IPC 用文件标志,不引入新 RPC** — 跟 v3.0.7 OAuth token reload 共享 docker volume 同模式,代码风险面小。

## 来源

- 客户截图(悦达渠道 Tommie Scoles 列大片空白)+ 客户反复质疑 "为什么会跑到 200 行去"
- API 实测拉 38 peer sheet_row 分布(`/api/v1/messages?account_id=130&peer_id=...&limit=1000`,v3.0.9 新 endpoint)
- `sheets.py:1063` 错误注释发现:`# table_range 限定本列组的 3 列, 不会跨列组打架` ← 跟 Google API 实际行为不符
- Plan agent 深查代码 + Google Sheets API 文档(`spreadsheets.values.append` 的 table 自动检测语义)
- Codex round 待补 — 等用户审 plan 后再走

## 后果

### 正面
- ✅ **客户手动删 Sheet 旧消息 → 15 分钟内自动回填** — 解决 ADR 背景里的核心痛点
- ✅ **Tommie Scoles 一类的 peer 在自己的 Y/Z/AA 列紧凑写入** — 行号不再被全表推高
- ✅ **0 quota 撞顶**:小客户 read 0.33/min,大客户 6.7/min,Google quota 300/min 余量充足
- ✅ **写 quota 不变**:仍 1 次 API/peer,跟 v3.0.8 持平
- ✅ **完全 feature flag**:`SHEET_RESYNC_ENABLED=false` 退回 v3.0.9 行为,跑老逻辑
- ✅ **0 数据迁移**:DDL 加列 NULL 默认,老 peer 自动走 append fallback
- ✅ **0 账号重登**:不动 listener / sessions
- ✅ **跨容器 IPC**:文件标志机制,跟 v3.0.7 同架构,无新 RPC

### 潜在问题 / trade-off
- ~~**race window**:resync 间隔内客户在 next_row 处贴内容 → 我们覆盖。罕见场景。默认接受。~~ **(Codex P0-1 改)** 默认 `SHEET_RESYNC_VERIFY_BEFORE_WRITE=true` 写前必读 1 cell 验空,只 +1 read API/peer/写批次,quota 影响极小(~50/min vs 300/min limit)。客户写到 next_row 时 verify 失败 → invalidate + 走 append fallback,**0 覆盖风险**。设 false 才是"省 quota 但接受 race"。
- **大客户 200 worksheet × 15min 扫**:API 13.3/min,接近写 quota 边缘。UI 设置页应自动提示 worksheet > 100 改 30+ 分钟(暂未实现 Auto-suggest,留 v3.1.1)。
- **gspread `ws.update` 行为变化风险**:某些版本可能在指定 range 写入时跨行偏移。`updatedRange` 校验 + invalidate 兜底,fallback append 路径保证零覆盖。
- **跨进程 SheetsWriter 状态读不到**:tg-web 跨容器拿不到 `_last_resync_ts`,走 DB MAX 兜底(微小延迟 ms 级,可接受)。

### Codex P0/P1 审阅修复(post-design)

| 等级 | 问题 | 修法 |
|---|---|---|
| **P0-1** | 默认 update 路径无写前校验,客户改 Sheet 中段时被覆盖 | `SHEET_RESYNC_VERIFY_BEFORE_WRITE` 默认改 ON,每条 update 前读 1 cell 验空,失败 → invalidate + raise 让上层 append fallback |
| **P0-2** | feature flag 关→开 时陈旧 next_row 覆盖老行 | flag 关时 write_messages 顺手 `invalidate_peer_next_sheet_row()` 清缓存,下次开 flag 时走 NULL fallback append 等下轮 resync 重算 |
| **P1-1** | `resync_peer_positions` 持 `_write_lock` 跑 100 worksheets × N 秒 → 卡死 flush_pending | 把 worksheets() + get_all_values 拉出锁外,SQLite UPDATE 本身原子无需 _write_lock 保护;接受短暂 staleness(下轮 resync 修) |
| **P1-2** | 测试覆盖不到 update 路径 / verify / flag 切换 / lock | 加【4】测试块:5 case 含 stub FakeWorksheet 模拟 update + acell 路径 + 静态 grep 验证 resync 不持锁 |

### 未解决留 v3.2+
- **多 spreadsheet 自动分裂**:200 sheets/spreadsheet 上限场景,等大客户出现再做
- **每 peer 一个 worksheet**:Plan agent 探讨过但取消(38 worksheet/account × 多 account 容易撞 200 上限)
- **老 peer col_group 整理工具**:38/16 视觉异常的存量客户,提供一键重排(留 v3.1.1)
- **`MEDIA_STORAGE_MODE=tg_archive` 跟 update 路径回归测试**:`=HYPERLINK()` 公式渲染应该兼容(`USER_ENTERED` 不变),但需要实际 tg_archive 客户验证

## 升级指引

```bash
cd /root/tg-monitor-<部门> && ./update.sh
```

升级后:
- DB 自动跑 V6 加 2 列(NULL 默认),毫秒完成
- 30 秒后后台 loop 启动,首次扫整张 Sheet 填 `next_sheet_row`
- **首次扫描完成前**(几秒到几十秒,看 worksheet 数)write_messages 走 append fallback
- **首次扫描完成后** write_messages 切 update 路径,新消息精确命中行号
- 200+ TG 账号 0 重登(不动 listener / sessions)
- 客户体感:Sheet 上同 peer 的新消息开始紧凑接续,不再全表底部齐头并进

### 集成
1. **首次启动**:V6 升级 → 30s 后 resync loop 跑首轮 → peers.next_sheet_row 全部填值
2. **Update 路径**:模拟 5 条消息 / peer → ws.update 命中 next_row → DB bump +5
3. **Append fallback**:invalidate next_row → 下条消息走 append → 不覆盖
4. **删旧消息回填**:客户 Sheet 删 row 30~50 → resync 后 next_sheet_row 退到 30 → 新消息从 30 写
5. **on-demand 触发**:POST /api/sheets/resync-now → 文件标志 → loop 5s 内触发 + 删文件

- `SHEET_RESYNC_ENABLED=false` 全走 append 等价 v3.0.9
- v3.0.7 OAuth 自愈 / v3.0.8 深度诊断 / v3.0.8.3 重启 fallback 行为不变
- `/api/v1/metrics` 返 4 个新 resync 字段,老中央台不读不影响
