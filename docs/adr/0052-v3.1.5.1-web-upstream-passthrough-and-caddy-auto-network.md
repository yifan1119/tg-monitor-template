# ADR-0052 · v3.1.5.1 WEB_UPSTREAM 透传 caddy + caddy 自动接所有 dept network 治根

## 背景

v3.1.5 治根了「site block 被 git reset 冲掉」(ADR-0049),但 2026-05-18 修陈家碧 187.127.114.68 / 82.112.236.78 时暴露**还有 2 个未治根 gap**:

### Gap 1:`WEB_UPSTREAM` 没透传到 caddy 容器

陈家碧 dept 目录名含 `--https` 后缀(例 `tg-monitor-yunyingzhongxin-wanyouyinli--https`),docker compose project name sanitize 砍掉非法后缀 → **实际容器名** `tg-web-yunyingzhongxin-wanyouyinli`(不带 --https)。

但 `Caddyfile` 默认上游 `{$WEB_UPSTREAM:tg-web-${COMPANY_NAME}:5001}` 里 `${COMPANY_NAME}` = `.env COMPANY_NAME` = 带 `--https` → 默认值 `tg-web-yunyingzhongxin-wanyouyinli--https:5001` → DNS 解析失败 → **502**。

客户在 `.env` 加 `WEB_UPSTREAM=tg-web-yunyingzhongxin-wanyouyinli:5001` 应该能覆盖,但 `docker-compose.yml caddy environment` **没透传** `WEB_UPSTREAM` → caddy 容器内拿不到这个 env → Caddyfile 仍读默认值 → 配置白做。

### Gap 2:caddy 接子 dept network 是手术(没固化到 enable_https.sh / update.sh)

共用 Caddy 模式下,主 dept caddy 反代其他 dept 时需要 caddy 容器接进所有 dept 的 docker network(才能 DNS 解析对应 `tg-web-X` 容器)。

之前(v3.0.13)update.sh 5b 已加「web 跟 caddy 同 network」自愈,但**只检测主 dept 自己的 web**(`tg-web-${COMPANY_NAME}`),**不查 conf.d/*.caddy 里 reverse_proxy 引用的其他 dept 容器**。

每次客户跑 update.sh `docker compose up --force-recreate` caddy 容器会重建 → 之前手术 `docker network connect` 加的子 dept network 都丢 → 别 dept 502。

## 决策

**3 处治根**:

### A. `docker-compose.yml caddy environment` 加 `WEB_UPSTREAM` 透传

```yaml
environment:
  - WEB_UPSTREAM=${WEB_UPSTREAM:-}
```

`.env WEB_UPSTREAM=tg-web-X:5001` 才能真覆盖 Caddyfile 默认上游名(Gap 1 治根)。

### B. `enable_https.sh` 加 sanitize 容器名检测

`docker ps` 查实际容器名,跟 `${COMPANY}` 比对,不一致自动写 `.env WEB_UPSTREAM=<真实容器名>:${WEB_PORT}`,客户不用手工加。

### C. `update.sh` 加 5c:扫 conf.d/*.caddy + 主 Caddyfile 自动接 network

```bash
# 收集所有 reverse_proxy upstream 容器名
UPSTREAMS=$( ... grep ... Caddyfile ... conf.d/*.caddy ... )
# 对每个 upstream 容器,inspect 拿 network,接进本 caddy(幂等)
for up in $UPSTREAMS; do
    UP_NETS=$(docker inspect "$up" --format '{{range $k,$v := .NetworkSettings.Networks}}{{$k}} {{end}}')
    for net in $UP_NETS; do
        docker network connect "$net" "$MY_CADDY" 2>/dev/null || true
    done
done
docker exec "$MY_CADDY" caddy reload  # 接完 reload
```

跳过 localhost / IP / host.docker.internal 等非容器名。幂等(已接的 network 跳过)。

## 原因

1. **Gap 1**:WEB_UPSTREAM 是 docker-compose 模板配置,但漏给 caddy 容器透传,等于半个 feature(只有 web/tg-monitor 容器认 env,caddy 不认)→ 修。
2. **Gap 2**:update.sh 5b 只看主 dept 不够,共用 Caddy 模式得查 conf.d/*.caddy 所有 reverse_proxy 容器名 → 扩展。

## 来源

- 2026-05-18 陈家碧 187.127.114.68(dingfenggs4 反代 wanyouyinli/fangyouyinli/hudiexiaoying1)+ 82.112.236.78(dingfenggs01--https 反代 dingfenggs-yunyingzx1)反复撞 502
- 现场手术验通后(`.env WEB_UPSTREAM` + `docker network connect` 手动加),固化进 enable_https.sh / update.sh / docker-compose.yml

## 后果

**改动 3 文件**:
- `docker-compose.yml`:caddy environment 加 `WEB_UPSTREAM` 透传(2 行)
- `enable_https.sh`:加 sanitize 容器名检测 + 自动写 `.env WEB_UPSTREAM`(15 行)
- `update.sh`:加 5c 自动扫 reverse_proxy 接 network(40 行)

**升级路径**:
- 客户跑 `update.sh` 升 v3.1.5.1 → git reset 拉新 docker-compose.yml + update.sh → docker compose up --build recreate caddy → 5c 自动接所有 dept network → 502 不再复发

**未来行为**:
- 新部署 enable_https.sh 自动检测 sanitize 容器名,写 .env WEB_UPSTREAM(根治)
- 客户每次升级 update.sh 5c 自动重接 network(防 force-recreate 丢失)

**回滚**:不需要;新机制是 strict superset(老客户没 .env WEB_UPSTREAM 行为不变)
