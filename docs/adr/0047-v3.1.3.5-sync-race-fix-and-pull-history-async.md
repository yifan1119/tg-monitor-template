# ADR-0047 · v3.1.3.5 · 数据同步双向 race 修 + pull_history 异步化

## 背景

2026-05-14 全网 35 IP × 62 dept 巡检发现 2 个 P0:

### P0-1:`sync_headers` 双向同步互相覆盖,清空客户填的归属

v3.0.15(ADR-0034)引入 `_sync_account_business_to_sheet` 做 DB→Sheet 单向覆盖(用户拍板 Web 优先),但 v2.10.23 起就有的 `sheets._sync_one_account_headers`(Sheet→DB)路径**没改**,两条同步方向互相打架。

时序:
1. **t=0**:客户 web 后台「配置」modal 填 `operator='张三' / inspector_tg_id='@admin'` → DB 保存
2. **t=600s**:`_patrol_loop` 跑 `sync_headers`(Sheet→DB):读 Sheet B2='' → 跟 DB 不一致 → **UPDATE DB.operator=''**(清空!)
3. **t=600s 之后**:`_sync_account_business_to_sheet`(DB→Sheet):DB 已空,Sheet 也保持空
4. **t=600s+**:客户回 web 看,字段又空了

**实测影响**:
- hengfeng-hf:DB operator 0/19 + inspector_tg_id 0/19(全清,company 19/19 是 `恒丰-恒丰` 默认值不是客户填的)
- 全网 18 dept v3.0.15+ 已升 + DB 业务字段疑似被清空
- 影响所有升 v3.0.15+ 的部门(随版本扩散越来越广)

**根因**:`_sync_one_account_headers` 老逻辑「Sheet 跟 DB 不一致就直接覆盖 DB」,**没有「Sheet 空 + DB 有值就跳过」的保护**。

### P0-2:`pull_history` 阻塞 TaskScheduler 启动

`main.py:185` 同步 `await listener.pull_history(phone, days=2)` 给每个号补拉历史。telegram 反扒虫限速频繁触发 `Sleeping for 29s GetHistoryRequest flood wait`,几十个号累加 → 几小时拉不完 → **TaskScheduler.run() 永远不被 spawn → patrol_loop / listener / bot 全没起**。

**实测影响**:
- 全网 42/62 dept(67%)容器 Up healthy 但「全部服务启动」=0
- 关键词预警 / 未回复预警 / 删除消息预警 / 离线预警**全部不会触发**
- 客户感觉:web 后台能开,预警从来不响

**根因**:升级后反复重启(OAuth 失效修复 / dept hotfix / sync 调试)24h 内每 dept 重启 8+ 次,触发 telegram 累积惩罚,补拉历史成本指数上升。pull_history 阻塞设计 → patrol_loop 永远等不到。

### P0-3:patrol_loop 静默失败无人察觉

42 dept patrol_loop 没起的状态被忽略两周以上,直到客户报「填的归属消失」才系统排查发现。

**根因**:tg-monitor 没有「启动后业务循环 N 分钟内必须跑过一轮」的健康监控,容器 healthcheck 只看 web 端口能 listen,不验证业务循环。

## 决策

### 改动 1:`sheets._sync_one_account_headers` 加「Sheet 空 + DB 有值跳过」保护

```python
# v3.1.3.5: Sheet 空 + DB 有值 → 跳过(防双向 race 清 DB);Sheet 真有内容才回写 DB
db_op = account["operator"] or ""
db_co = account["company"] or ""
new_op = operator if operator else db_op
new_co = company if company else db_co
if new_op != db_op or new_co != db_co:
    db.get_conn().execute(
        "UPDATE accounts SET operator=?, company=? WHERE id=?",
        (new_op, new_co, account["id"])
    )
```

5 个场景对比:

| 场景 | 老行为 | 新行为 | 是否 regression |
|---|---|---|---|
| Sheet 有值 + DB 不一致 | Sheet 覆盖 DB | 同 | 否 |
| Sheet 空 + DB 也空 | 无操作 | 同 | 否 |
| Sheet 空 + DB 有值 | **覆盖成空(P0)** | **跳过保留 DB** | 🆕 修 P0 |
| Sheet 有值 + DB 一致 | 无操作 | 同 | 否 |
| Sheet 有值 + DB 不一致 | Sheet 覆盖 DB | 同 | 否 |

**唯一变的场景** = 客户在 Sheet 上手填后又删空想清 DB → 现在不会同步。但实际客户**通过 web 后台清字段**(`web.PATCH /api/accounts/<id>` 直接写 DB),不会去 Sheet 删,所以不算 regression。

### 改动 2:`main.py` `pull_history` 改异步 fire-and-forget

```python
async def _bg_pull_all_history():
    logger.info("[bg_pull_history] 开始背景补拉最近 %d 天历史消息(不阻塞业务循环)", config.HISTORY_DAYS)
    for phone in listener.clients:
        try:
            await listener.pull_history(phone, days=config.HISTORY_DAYS)
        except Exception as e:
            logger.warning("[bg_pull_history] phone=%s 失败 (跳过,继续下一个): %s", phone, e)
    logger.info("[bg_pull_history] 全部账号背景补拉完成")

_pull_history_task = asyncio.create_task(_bg_pull_all_history())
def _on_pull_done(t):
    if t.cancelled(): return
    exc = t.exception()
    if exc is not None:
        logger.error("[bg_pull_history] 背景任务异常退出: %s", exc, exc_info=exc)
_pull_history_task.add_done_callback(_on_pull_done)
```

**关键设计**:
- `asyncio.create_task` fire-and-forget,不 await
- 后续代码立刻 spawn TaskScheduler / listener / bot
- 单号 pull_history 失败不阻塞其他号(try/except per phone)
- `add_done_callback` 防异常被 asyncio 静默吞(rule 19:静默失败必须告警)
- 历史消息回填会跟实时新消息并发写 DB — sqlite WAL 模式 + 单进程安全

### 改动 3:patrol_loop 加启动 INFO log + `_first_patrol_done` 标志

```python
logger.info("[patrol_loop] 启动,interval=%ds(每轮:sync_headers/sync_account_names/bus_sync/...)", config.PATROL_INTERVAL)
# ... 每轮成功执行后:
self._first_patrol_done = True
```

让 docker logs grep `patrol_loop` 能直接看到启动证据 + 让健康监控有钩子。

### 改动 4:新增 `_startup_health_loop` 15 分钟自检告警

```python
async def _startup_health_loop(self):
    await asyncio.sleep(900)  # 15 分钟
    if self._first_patrol_done:
        logger.info("[startup_health] patrol_loop 15 分钟内已正常跑过,健康")
        return
    msg = "⚠ tg-monitor 启动 15 分钟内 patrol_loop 未运行 — 业务循环可能卡住,请检查 docker logs"
    logger.error("[startup_health] %s", msg)
    if self.bot and self.bot.bot and config.ALERT_GROUP_ID:
        await self.bot.bot.send_message(config.ALERT_GROUP_ID, msg)
```

只检查一次(不循环),最后一道防线。

**2026-05-14 阈值修正(5min → 15min)**:Wave 1 麦小麦 linghang-qd@103(23 号)+ Wave 2A 苏木 dingfengsw(11 号)实测,sheets API 撞 429 拖慢 patrol_loop 第一轮跑完时间到 8-10 分钟,5 分钟阈值频繁 false positive。改成 15 分钟覆盖大 dept 第一轮极端情况,仍能在真死(代码 bug / asyncio task 没 spawn)时报警。

## 原因

### 为什么不直接关掉 sync_headers(Sheet→DB)

理论上 v3.0.15 ADR-0034 拍板「Web 优先单向覆盖」,sync_headers 老路径可以全删。但保留这条路径有价值:
- 老客户在 Sheet B2/B3 手填的习惯(v2.x 时期有)还能用
- 加保护逻辑只 8 行,删整个老路径牵连 A2 label 同步 / row 6 PEER_ROLE_LABEL 同步等无关功能

「最小改动」 + 「向后兼容」原则,加保护比删更稳。

### 为什么 pull_history 不直接 `HISTORY_DAYS=0` 跳过

客户的实时新消息走 listener 事件驱动不受影响,但**升级后头几小时漏接收的消息**没了 — 客户依赖这个补拉。设成 0 = 永久关掉补拉功能,这不是「修 bug」是「砍功能」。

异步化让 pull_history 仍然在背景跑,只是不阻塞 patrol_loop 启动。两全其美。

### 为什么不加 listener pull_history 重启 backoff

可以加(避免短期重启反复触发 telegram 限速)但这是更大的改动 + 引入新状态(上次启动时间记录)。本版只解决「启动卡死」,重启 backoff 留 v3.2.x。

### 为什么 `_first_patrol_done` 是 bool 不是 int / counter

只关心「跑过没」不关心跑了几次。bool 简单 + 正确。`_startup_health_loop` 一次性检查,之后这个 flag 没人读。

## 后果

### 正面

- ✅ 客户填的 operator/company/inspector_tg_id **不再被周期性清空**
- ✅ tg-monitor 启动后 30s 内 patrol_loop 一定起来,不被 telegram 限速卡死
- ✅ 业务循环卡死(任何原因)5 分钟内推预警群,运维秒级感知
- ✅ 老 Sheet → DB 同步功能保留(向后兼容,客户在 Sheet 手填仍生效)
- ✅ 0 数据迁移、0 schema 变更、0 配置变更
- ✅ rule 6:Feature flag 不需要(纯 bug 修复,无新功能)

### 中性 / 负面

- ⚠ pull_history 改异步后,**历史消息回填顺序不再确定**(老的:严格按 phone iteration 顺序;新的:并发跟新消息交错)— 但 Sheet 写入按 ts 排序,客户视角无差
- ⚠ pull_history 失败(单号 / 全部)只 WARNING/ERROR log,不再让 main 退出 — 客户看不到「拉历史失败」红字。需要靠 docker logs 排查。但这是正确权衡:阻塞 main 退出导致 docker restart loop 比静默失败更糟
- ⚠ 升级后 `_startup_health_loop` 第一次跑会发一条「patrol_loop 已正常」INFO log + 万一卡住会发一条 ALERT 到客户群 — 客户群多一条噪音,但是必要的「我活着」信号
- ⚠ 麦小麦 7 dept 升级后需要手动撤掉 `.env SYNC_HEADERS_DISABLED=true`(本来就是临时 workaround)
