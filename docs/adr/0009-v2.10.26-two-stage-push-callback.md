# ADR-0009 — v2.10.26 两段式未回复预警:推送 / 模板 / callback(批次 B)

## 背景

v3.0.0 两段式未回复预警拆 4 批次发布(见 ADR-0005)。批次 A(v2.10.25)完成了数据层 + UI
入口:`accounts` 表 +4 字段(`business_tg_id` / `owner_tg_id` / `remind_30min_text` /
`remind_40min_text`)、`alerts.stage`、3 个 feature flag(`TWO_STAGE_NO_REPLY_ENABLED` /
`NO_REPLY_STAGE2_AFTER_MIN` / `UNREPLIED_ALERT_GROUP_ID`)、后台「配置两段式」模态框 +
设置页「未回复预警专用群」入口 + API `PATCH /api/accounts/<id>/notify-config`。

批次 B 要做的:**把数据 + 配置真的用起来**,让 bot 能走两段式推送 / 响应新按钮。不做 loop
扫描(批次 C 的活)。

## 决策

### 1. `templates.py` 新增两个模板函数

- `no_reply_alert_stage1(..., business_mention, custom_text)` — 30 分钟提醒版
- `no_reply_alert_stage2(..., owner_mention, custom_text)`    — 40 分钟升级版

都支持 `*_mention` 为空时**退化为纯提醒文案**(即账号没配 `business_tg_id` / `owner_tg_id`
时推送不带 @,不报错),`custom_text` 为空时用内置默认文案(「请尽快回复」/「已超过 40 分钟未回复,
请处理」)。

模板用 plain text 拼接 + HTML `<a>` tag 嵌入 @mention,因此发送时必须 `parse_mode="HTML"`。

### 2. `bot.py` 改动

#### 2.1 工具函数 `_format_tg_mention(tg_id_or_username, display_name)`

**分支判断**:
- 输入纯数字 → HTML inline mention `<a href="tg://user?id=N">{name}</a>`
  - 优势:对方**没设 username 也能触发群 @ 通知**
  - `display_name` 会 `html.escape`,防注入
- 输入其他(字母 / 下划线 / 混合) → `@xxx` 字符串拼接
  - TG username 天然合法字符集,无需 escape
- 空值 / 空白 → 返回 `""`,模板据此退化不带 @

#### 2.2 `_make_keyboard_stage2(alert_id)` 新按钮工厂

按钮:
- `violation:{alert_id}` — 登记违规
- `cancel:{alert_id}`   — 取消

动作名跟 alert 状态映射:`violation_logged` / `cancelled`,不跟老 `approved` / `rejected` 混。

#### 2.3 老 `send_no_reply_alert` 入口加 flag 分流

```python
async def send_no_reply_alert(self, account_id, peer, message_text, msg_id):
    config.reload_if_env_changed()
    if config.TWO_STAGE_NO_REPLY_ENABLED:
        return await self.send_no_reply_alert_stage1(account_id, peer, message_text, msg_id)
    # ↓ 以下保留 v2.10.24 原逻辑 100% 不变
```

**关键**:flag=false(默认)时**一行没改老路径**,保证向后兼容。

#### 2.4 `send_no_reply_alert_stage1(...)` 新函数

- 推送群:`UNREPLIED_ALERT_GROUP_ID or ALERT_GROUP_ID`(fallback 老群)
- 沿用 `has_alert_today("no_reply")` 天级去重(跟老单段版一致)
- 沿用 `ALERT_NO_REPLY_ENABLED` 子开关,关时插 `insert_stage1_alert` 并标 silenced
- 正常路径:`insert_stage1_alert` → `send_message(parse_mode="HTML")` → `update_alert_bot_msg`
- 失败语义:`bot_message_id` 保持 null,`has_alert_today` 不认作已推,下轮重试
  (跟 ADR-0003 v2.10.23 立的语义一致)

#### 2.5 `send_no_reply_alert_stage2(alert_id)` 新函数(批次 C 才会有 loop 调用)

- 入口:批次 C 的 `_no_reply_stage2_loop` 会扫 `get_pending_stage1_alerts(after_minutes=NO_REPLY_STAGE2_AFTER_MIN)` 然后逐条调本函数
- 并发保护:`db.upgrade_to_stage2(alert_id)` 原子 UPDATE WHERE stage=1 AND status='pending',
  rowcount=1 才继续推
- 推送群同 stage1(用 `UNREPLIED_ALERT_GROUP_ID`)
- 推送后 `update_alert_bot_msg` 用 stage2 新 message_id 覆盖(callback edit_text 要用)
- 失败 trade-off:已经 UPDATE `stage=2` 无法回退;但 status 仍 `pending`,下轮可重推;极少数推送失败
  的 stage2 会在群里少一条消息,可接受(不会打错 / 漏登记)

#### 2.6 新 callback handler `on_stage2_action`

```python
@self.dp.callback_query(F.data.startswith("violation:") | F.data.startswith("cancel:"))
```

逻辑跟老 `on_audit` 对齐:
1. 权限校验(沿用 `CALLBACK_AUTH_USER_IDS`)
2. `claim_alert_for_review(alert_id, new_status)` 原子抢占(WHERE status='pending',不限 stage)
3. `violation` → `_write_alert_to_sheet(alert)` 写违规登记;`cancel` → 仅 edit_text
4. 异常兜底 `callback.answer` 提示,不抛

### 3. `_write_alert_to_sheet` 扩展

`alert["type"] == "no_reply"` 分支里加 stage 末列标记:
- `stage == 2` → 末列写「违规登记」
- 其他(stage=0 老路径 / stage=1 理论上不会走到,因为 stage=1 无按钮) → 末列**不追加**
  保持老 6 列结构,客户表格完全无感

决策理由:跟你确认的方案 A(Q2)— 写到同一张`信息未回复预警{部门}`分表,不新开表,结构向后兼容。

## 原因

### Q1 确认:stage2 按钮权限沿用老白名单

**选项**:
- A. 跟 approve/reject 一样,用 `CALLBACK_AUTH_USER_IDS`(任何管理员都能点)
- B. 只有 `owner_tg_id` 本人能点

**选 A**:客户审查员已经在老白名单里管理,沿用同一套免去训练;管理员兜底防负责人失踪。

### Q2 确认:违规记录写同一张预警表

**选项**:
- A. 同一张`信息未回复预警{部门}`,末列加 stage 标记
- B. 新开一张`信息违规登记{部门}`

**选 A**:表格结构不变 → 回滚干净(flag 关掉后老写入路径 6 列跟以前一模一样,不会出现跨版本混乱的列数)。

### @mention 策略:numeric → HTML inline / username → `@xxx`

TG username 规则允许字母 / 数字 / 下划线,5-32 字符。**TEXT 类型字段同时支持两种填法**:

- 客户填纯数字 → `@userinfobot` 查到的 numeric user_id,最可靠
- 客户填 `@username` 或 `username` → 兼容用户名场景,但要求对方设了 username

`_format_tg_mention` 统一分支处理,前端 UI 不做校验(交给客户灵活填)。

### feature flag 默认关 + 单入口分流

遵循全局规则「所有新功能 flag 默认关,升级后行为 = 老版本」。
`send_no_reply_alert` 单点分流,后续批次 C 增加 `_no_reply_stage2_loop` 也只在 flag=true 时跑。

## 后果

### 正面

- 向后兼容:`TWO_STAGE_NO_REPLY_ENABLED=false`(默认)时,`send_no_reply_alert`、`_write_alert_to_sheet`、
  预警分表结构跟 v2.10.24 一模一样,老客户零感知
- 推送失败可重试(继承 ADR-0003 的语义)
- stage1 → stage2 升级原子,并发安全
- @mention 两种格式都支持,客户填法自由

### 负面 / 遗留

- 批次 B **不做 loop**,单独部署看不到 stage2 自动升级效果。必须等批次 C 把
  `_no_reply_stage2_loop` 接上才完整可用。批次 B 可单独测 stage1 推送 + 手动触发 stage2。
- 老单段 `send_no_reply_alert` 里 `send_message` 没加 `parse_mode`,保持原默认行为不动;两段式
  路径独立用 HTML。两条路径 parse_mode 不同是有意选择,不作改动。
- stage2 推送失败目前会「吃掉一次升级机会」(stage=2 无法回退到 1)。
  trade-off 已写在 `send_no_reply_alert_stage2` docstring,视后续生产数据决定是否加 retry。

### 回滚

- 设 `TWO_STAGE_NO_REPLY_ENABLED=false` 重启 → 瞬间回 v2.10.24 行为
- DB 层 stage 列保留(无害),UI 按钮条件渲染自动隐藏
- Sheets 已有违规登记记录不动(客户历史数据保留)
