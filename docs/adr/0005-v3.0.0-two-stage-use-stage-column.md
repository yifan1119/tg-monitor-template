# ADR-0005: 两段式未回复预警用 `alerts.stage` 字段,不改 `type`

**版本**:v3.0.0(开发中)
**日期**:2026-04-21

## 背景

v3.0.0 新需求(客户苏总):未回复预警改两段式
- **Stage 1(30 分钟触发)**:@商务人员 + 可选文案,无按钮
- **Stage 2(Stage 1 后 10 分钟仍未处理)**:@负责人 + 可选文案 + 按钮「登记违规 / 取消」

**初始方案(错误的)**:
- 把 `alerts.type` 从 `'no_reply'` 改成 `'no_reply_stage2'` 表示升级
- stage2 扫描 WHERE `type='no_reply' status='pending' created_at < now-10min`

## 决策

**保持 `alerts.type='no_reply'` 不动,新增 `stage INTEGER DEFAULT 0` 字段**:
- `stage=0`:v2.x 老数据默认 / 未激活两段式
- `stage=1`:两段式 stage1 已推送,等 stage2_after_min
- `stage=2`:已升级到 stage2

所有扫描 / 去重逻辑显式用 `WHERE stage=N` 过滤。

### Schema migration(user_version=2)
```sql
ALTER TABLE alerts ADD COLUMN stage INTEGER DEFAULT 0;
```

### stage2 扫描 SQL
```sql
SELECT * FROM alerts
WHERE type='no_reply' AND stage=1 AND status='pending'
  AND bot_message_id IS NOT NULL
  AND created_at <= now - NO_REPLY_STAGE2_AFTER_MIN
```

## 原因

Codex 审计指出 3 个 Critical 问题,都是「改 type」方案导致的:

### Critical B1:stage2 loop 误触 v2.x 老 pending
如果把 type 改成 `no_reply_stage2`,那 stage2 扫描用 `type='no_reply' status='pending'` 会扫到 v2.x 遗留的老按钮预警(那些 alerts.type='no_reply' 还在等人审核)→ 错发 stage2 → 把老按钮消息变成 stage2 带违规按钮,很混乱。

**显式 stage=1 过滤后**:老数据 stage=0,不会被扫到。

### Critical B2:升级到 stage2 后重复发 stage1
```python
# 错误方案
db.execute("UPDATE alerts SET type='no_reply_stage2' WHERE id=?", ...)

# 下次 _no_reply_loop 扫 has_alert_today('no_reply', peer_id)
# → 原记录已经变成 no_reply_stage2,has_alert_today 返回 False(因为 WHERE type='no_reply')
# → 错误地再推一次 stage1
```

**保持 type='no_reply'**:去重逻辑永远生效,同一对话同一天只有一条 type='no_reply' 记录,不管 stage 多少。

### Critical B3:回滚到 v2.10.23 后老代码不认识新 type
```python
# 如果 v3.0.0 推出后回滚到 v2.10.23,DB 里有 type='no_reply_stage2' 记录
# v2.10.23 老代码只认 ('no_reply', 'keyword', 'deleted'):
#   - 统计漏掉 stage2 记录
#   - 审核按钮 callback 可能找不到对应 alert
```

**保持 type='no_reply'**:老代码完全能识别,stage 字段老代码看不到但不影响查询(SELECT * 多一列 nullable,dict 访问 .get 也 OK)。

## 原因(设计权衡)

### 为什么不新建 `two_stage_alerts` 表?
- 新表意味着 callback 路由逻辑要判断「这个 alert_id 是 alerts 还是 two_stage_alerts」,复杂度 +1
- 审核记录、日报统计、预警分表写入等 15+ 处要改
- 保守方案:一个 stage INTEGER,改动最小

### 为什么用 stage 而不用 sub_type?
- stage 是 INTEGER,检索性能好
- 明确表达「阶段」语义,如果未来要加 stage3(罕见)也自然扩展
- ORDER BY / WHERE stage < N 这类查询直接能用

### 为什么默认 0 不是 NULL?
- 老数据(v2.x 的 pending 按钮预警)默认值 0
- stage=0 意味着「跟新两段式无关」— 明确语义
- NULL 会带入三值逻辑复杂度

## 来源

Codex (GPT-5.4 高推理) 外部审计,v3.0.0 方案评审:
- **Critical B1**:会误处理 v2 遗留 pending 未回复告警
- **Critical B2**:升级到 stage2 后可能重复发 Stage 1
- **Critical B3**:回滚 v2.10.22 后,`type='no_reply_stage2'` 老代码不认识

Codex 原话:
> 保留 `alerts.type='no_reply'`,新增 `stage INTEGER DEFAULT 0`、`stage1_sent_at`、`stage2_sent_at`、`stage2_bot_message_id`、`target_group_id`、`reviewer_tg_id`。根因是复用 `type/status` 表达阶段会破坏去重和回滚。若不想扩 alerts,可新建 `no_reply_workflow`,但我更建议扩 alerts,兼容性更好。

## 后果

### 立即效果(v3.0.0 激活后)
- 两段式扫描 loop 用 WHERE stage=1 显式过滤,不误触老数据
- has_alert_today 和所有 WHERE type='no_reply' 的查询逻辑不变
- 回滚安全:老代码能正常处理,stage 字段保留不影响

### migration 风险
- `ALTER TABLE alerts ADD COLUMN stage INTEGER DEFAULT 0` 是幂等的(`_safe_add_column`)
- 没动原有行为:老 loop 不看 stage 字段,新 loop 显式看

### 待 v3.0.0 继续跟进
- `stage1_sent_at` / `stage2_sent_at` 是否需要单独列?— 目前用 `alerts.created_at` 和 stage2 时的 `bot_message_id` 更新时间足够,暂不加
- Codex 建议的 `stage2_bot_message_id` — 如果 stage2 推送成功,需要一个单独字段记录 stage2 消息 id(不同于 stage1 的 bot_message_id)→ **v3.0.0 Day 2 实现时确认,可能要加 `stage2_bot_message_id` 字段**
