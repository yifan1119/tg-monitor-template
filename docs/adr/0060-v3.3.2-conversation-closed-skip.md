# ADR-0060 · v3.3.2 对话告一段落跳过 — 扩 SKIP 词单 + 双层匹配 + 新增 CLOSE_PHRASE outbound 检测

## 背景

客户线上反馈:外事号已经主动说了结束语(「OK 等通知」「再联系」「拜拜」「商务合作请
联系」),对方应答一句「好的辛苦了」后,30 分钟系统照样推「未回复预警」,业务上误推。

两类场景:

| 场景 A | 对方最后说结束语 | 例:对方发「好的辛苦了。」 |
| 场景 B | 我方主动结束 + 对方应答 | 例:我方「OK 等通知」→ 对方「好嘞」 |

**v3.0.10 现状**:
- `is_trivial_no_reply` **精确整句匹配**(`s in SKIP_NO_REPLY_TEXTS` set)
- 词单含「辛苦了」「好的」但不含「好的辛苦了」整句
- 「好的辛苦了。」(末尾标点)精确匹配 miss → 不跳 → 误推
- 只看对方那条消息,看不到我方说过结束语

## 决策

### 1. SKIP_NO_REPLY_TEXTS 双层匹配(场景 A)

`SKIP_CONTAIN_MIN_LEN = 5` 阈值:
- **< 5 字短词**精确匹配(`s == phrase`)— 防短业务词(「上架」「打款」)吃业务问句(「什么时候上架?」)
- **≥ 5 字长结束语**子串匹配(`phrase in s`)— 吃标点变体(「好的辛苦了。」/「好的辛苦了!」匹配「好的辛苦了」)

阈值 5 经验值:
- 业务问句一般 ≥ 5 字,但极少整段包含 5+ 字告一段落整句(「好的辛苦了」不会出现在业务问句中间)
- 4 字以下的「上架/打款/支付」等业务术语必须精确匹配避免误吞

### 2. 扩 SKIP_NO_REPLY_TEXTS 默认词单(场景 A)

新加 40+ 整句:
- 告一段落短词(3-4 字精确):再联系 / 拜拜 / OK收到 / 了解 / 没问题 / 知道了 / 搞定了 ...
- 告一段落整句(5+ 字子串):好的辛苦了 / 好的没问题 / 就这样吧 / 等通知就好 / 等消息就好 ...

**清理 v3.0.10 残留**:
- 删「业务短词」段:支付 / 支付通道 / 通道 / 费率 / 进群了
- 跟 KEYWORDS 默认列表(到期/续费/暂停/下架/上架/地址/打款/欠费/返点/返利/回扣)语义重叠
- 单字消息「打款」业务上应该回(可能是问支付),不该跳未回复预警

### 3. 新增 CLOSE_PHRASE_TEXTS 列表(场景 B)

跟 SKIP_NO_REPLY_TEXTS 不同维度:
- SKIP 看**对方**最后一条说啥
- CLOSE 看**我方**最近 N 条 outbound 里有没有结束语

默认 40+ 词,全子串匹配:
- 主动结束:等通知 / 等消息 / 再联系 / 再聊 / 改天聊 / 下次聊
- 推开:就这样 / 先这样吧 / 行就这样
- 转介绍(常见 TG Business 自动回话术):商务合作请联系 / 私聊 / 找他 / 对接
- 推迟:晚点回复 / 稍后联系
- 完结:搞定了 / 完事了 / 处理好了 / 已经发了

### 4. tasks.py:_no_reply_loop 加 outbound 检测

```python
if config.CLOSE_PHRASE_TEXTS:
    recent_a = db.recent_outbound_texts(row["id"], limit=config.CLOSE_PHRASE_LOOKBACK)
    if any(config._contains_any(t, config.CLOSE_PHRASE_TEXTS) for t in recent_a):
        continue
```

- 在 `is_trivial_no_reply` 检查之后(语义嵌套:先看对方说啥跳过明显闲聊,再看我方有没有结束意图)
- `CLOSE_PHRASE_LOOKBACK = 3` 默认看最近 3 条 outbound,客户可在 web 改
- 复用 `_contains_any(text, phrases)` 子串包含 helper(跟 ≥ 5 字 SKIP 同语义)

### 5. database.py:recent_outbound_texts(peer_id, limit) helper

```sql
SELECT text FROM messages WHERE peer_id=? AND direction='A' AND deleted=0
ORDER BY timestamp DESC, msg_id DESC LIMIT ?
```

- 复用 v3.3.1 的 `ORDER BY timestamp DESC, msg_id DESC` tiebreaker(同秒按 TG msg_id 倒序)
- 走 `idx_msg_peer(peer_id)` 索引 + 同 peer 内 sort,limit 3 性能极轻

### 6. listener.py + tasks.py:GIF 加入媒体跳过列表(承 sticker)

`listener.py` 3 处媒体分类(`on_incoming` / pull_history 两路)在 `msg.video` **之前**
加 `msg.gif` 分支(Telethon GIF 是 MP4,同时带 Animated+Video attribute,顺序错会
落到 video 分支):

```python
elif msg.gif:
    media_type, text = "gif", "[GIF]"
elif msg.video:
    media_type, text = "video", "[视频]"
```

`tasks.py:_no_reply_loop` 把 `"gif"` 跟 `"sticker"` 并到同一跳过集:

```python
if last_media in ("sticker", "gif"):
    continue
```

理由:TG 里 GIF 99% 是「滑稽脸 / 拜拜挥手 / 鼓掌」表情包性质,业务消息几乎不用 GIF。

⚠ 暂不处理 `video_note`(圆形短视频)— 客户用嘴说事可能是业务也可能是闲聊,保守
不跳。voice / photo / video / file 同保守(原设计)。

### 7. Web 后台 setup.html 加可配字段

- `close_phrase_texts` textarea(逗号/换行分隔,留空用默认 40+ 词)
- `close_phrase_lookback` 数字(默认 3,1-20 范围)
- 路由 web.py:save_settings + reload_if_env_changed 已支持 hot reload

## 原因

- 客户 200+ 账号每天产生大量「对话告一段落」对话,误推预警 = 监察员误判 = 商务被错怪
- 双层匹配 + 扩词单是最小改动覆盖场景 A
- CLOSE_PHRASE outbound 检测是新维度,覆盖场景 B,但默认 ON(向后兼容:客户没设也不会破老行为,因为 default 词单的语义就是「我方明确说了结束语」,业务上几乎不会出现在非结束语境下)
- 清理 v3.0.10「业务短词」残留 — 借机修正历史设计错误

- 客户线上反馈 + 截图(2026-05-19 多次案例)

## 后果

### 正向

- 客户应答(辛苦了 / 好的辛苦了 / OK收到 等)不再误推 ✓
- 我方主动结束对话(等通知 / 拜拜 / 商务合作请联系)+ 对方应答不再误推 ✓
- 客户可在 web 后台覆盖默认词单(灵活性保留)
- 0 schema 变 / 0 migration / 0 重登

### 风险 / 边界

- **子串匹配长结束语漏判**(语义边界):「就这样」5 字子串吃到「不能就这样」业务问句 → 误跳。
  接受:5+ 字短句完整出现在业务问句几率极低,跟「不告警」相比「漏告警」是更小代价(商务还是看
  到了对方消息,只是少推一次提醒)。
- **CLOSE_PHRASE_TEXTS 维度激进**:「等通知」「再联系」「就这样」是常见日常用语,可能误抑制
  本来该升级 stage2 的对话。`CLOSE_PHRASE_LOOKBACK=3` 默认看最近 3 条 outbound,客户嫌激进可调
  小到 1 或 web 改空 CLOSE_PHRASE_TEXTS 关闭该检测。
- **历史数据兼容**:升级前已经在 DB 的对话, _no_reply_loop 下一轮跑就按新规则。无回溯影响。
- **「业务短词」删除**:原 v3.0.10 客户没改过 .env 的 dept(全网默认)单字消息「支付」「通道」
  「费率」会从「跳过」变成「触发预警」 — 业务上是更正(单字业务术语该回)。可能监察员预警量
  短期上升,但属修正不算 regression。
