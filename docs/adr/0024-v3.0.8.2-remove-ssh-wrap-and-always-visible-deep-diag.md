# ADR-0024: v3.0.8.2 升级命令去掉 SSH wrap + 复制按钮 fallback + 深度诊断永远可见入口

## 背景

v3.0.8.1 升级后客户连续 2 个反馈:

1. **「然后复制点了也没复制」** — 升级 modal 里的「复制命令」按钮点了无反应
2. **「为啥前面要给 SSH」** — 升级 modal 显示 `ssh root@187.77.157.220 "cd /root/tg-monitor-demo && bash update.sh"`,客户疑惑为什么要给个 SSH 包装(且没有客户的 root 凭据可用)
3. **「我更新了还是没看到你说的按钮啊」** — 客户 Sheet 状态正常 (0 待写, 17 秒前刚写入),v3.0.8 设计是只在卡死时才显示「立刻深度诊断」按钮,客户**没法主动测试这个能力**

按规矩查代码定根因:

**问题 1 (复制按钮没反应)**:
`templates/dashboard.html:1324` (+ index.html / setup.html 同样位置) 的「⚠ 此版本涉及镜像重建」modal 内嵌 inline `onclick="navigator.clipboard.writeText(...)..."`, **没有 fallback**。HTTP 场景 (客户用 IP:5001 直连不走 HTTPS) 下 `navigator.clipboard` API 不可用 → silently fail → 客户点没反应。
其他位置 (line 1308) 的 try/catch + prompt fallback 是正确的, 但 modal 这个独立 inline onclick 漏了。

**问题 2 (SSH wrap 误导)**:
`upgrader.py:248-250` `build_upgrade_cmd`:
```python
ip = env.get("VPS_PUBLIC_IP", "").strip()
if ip:
    return f'ssh root@{ip} "cd {install_dir} && bash update.sh"'
return f"cd {install_dir} && bash update.sh"
```
.env 有 `VPS_PUBLIC_IP` 时会 wrap 成 `ssh root@<IP>` 包装。问题:
- 客户没有从其他主机登 root 的凭据 (这条命令需要从某个跳板机执行)
- 实际客户已经 SSH 到 VPS 上了, 不需要再 ssh
- 按规矩:命令本身应该是「在 VPS 跑」的,wrap 只增加困惑

**问题 3 (深度诊断按钮藏起来)**:
`templates/dashboard.html::renderSheets`:
```js
if (status === 'ok' || s.status_msg === '正常') {
  diag.style.display = 'none';
  if (deepDiagWrap) deepDiagWrap.style.display = 'none';
  return;
}
```
按 v3.0.8 设计 — 「立刻深度诊断」按钮挂在诊断卡片 `sh-diag` 旁边, 只在异常 (sh-diag.display=block) 时显示。Sheet 状态正常时按设计完全藏起来不打扰客户。
但客户**主动想看**(测试新功能 / 主动巡检 / 想确认健康状态) → 没入口。

## 决策

v3.0.8.2 三件事一次发, 全 hotfix 范围, 不动 v3.0.8 / v3.0.8.1 核心逻辑:

### 1. `upgrader.build_upgrade_cmd` 去掉 SSH wrap

```python
# 旧 (v3.0.8.1-)
if ip:
    return f'ssh root@{ip} "cd {install_dir} && bash update.sh"'
return f"cd {install_dir} && bash update.sh"

# 新 (v3.0.8.2)
return f"cd {install_dir} && bash update.sh"
```

只返本地命令。客户已经 SSH 到 VPS 上了, 直接贴这条跑。文案改 "📋 复制 SSH 命令" → "📋 复制升级命令", 底部说明改 "涉及镜像重建的版本要在服务器跑命令(给你复制好,SSH 上去贴一下就行)"。

### 2. 复制按钮 `copyTextFallback` 三层兜底

3 个 templates (dashboard / index / setup) 都加 helper:

```javascript
window.copyTextFallback = function(text, btn) {
  if (navigator.clipboard && navigator.clipboard.writeText) {
    navigator.clipboard.writeText(text).then(() => { if (btn) btn.textContent = '✓ 已复制'; })
      .catch(() => _legacyCopy(text, btn));
  } else {
    _legacyCopy(text, btn);
  }
};
function _legacyCopy(text, btn) {
  // 创建临时 textarea + execCommand('copy') — HTTP 场景有效
  // 失败再 prompt() 让客户手动复制
}
```

3 处 inline `onclick="navigator.clipboard.writeText(...)"` 全改成 `onclick="copyTextFallback(..., this)"`。

三层防御:
- HTTPS 现代浏览器 → `navigator.clipboard` (主路径)
- HTTP 现代浏览器 → `document.execCommand('copy')` (兜底, 老 API 但 100% 兼容)
- 极端兼容性问题 → `prompt()` 弹窗给客户手动 Ctrl+C (终极兜底)

### 3. Sheet 深度诊断永远可见入口 (admin only)

`templates/dashboard.html` 在日志查看面板上方加固定区块:

```html
{% if is_admin %}
<div style="margin-top:18px;">
  <div class="section-title">
    <span>Sheet 写入诊断</span>
    <button onclick="openSheetsDeepDiag()">▸ 立刻深度诊断</button>
    <span>任何时候都能查 — 看各账号未写数 / 孤儿消息 / col_group 缺失明细 + 一键修复</span>
  </div>
</div>
{% endif %}
```

复用 v3.0.8 的 `openSheetsDeepDiag()` 函数 (modal + 后台 SQL + 一键修复 — 完全沿用,无新代码)。客户随时点能看健康明细。

`{% if is_admin %}` Jinja 条件 — 跟 v3.0.8.1 设计一致,普通成员看不到这个 admin-only 入口。

## 原因

1. **客户反馈优先**: 复制按钮没反应是 P0 直接体验破坏,SSH wrap 误导是 P1 但容易让客户错点出问题, 一起修

2. **SSH wrap 反模式**: 命令包装应该是 "怎么执行的描述" 不是 "包多一层执行壳"。客户已在 VPS 上, ssh root@self 没意义

3. **深度诊断永远可见**: v3.0.8 设计是 "状态 OK 不打扰", 但客户想主动巡检时找不到入口违背 "纯 web 自助" 原则。永远可见 + admin only 兼顾两边 — 普通成员不打扰, 管理员随时查

4. **三层 fallback**: HTTP 场景客户没用 HTTPS 部署 (没绑域名 / 用 IP 直连 / install.sh `--no-https`) 仍是常见, navigator.clipboard 在 HTTP 直接 throw, 必须 fallback

## 来源

- 客户反馈 1: 截图显示 ssh root@187.77.157.220 包装命令 + 「然后复制点了也没复制」
- 客户反馈 2: 「单纯给更新代码就好别多个SSH误导」
- 客户反馈 3: 「我更新了还是没看到你说的按钮啊」 — 状态截图 0 待写 17s 前写入 = 正常 → 按 v3.0.8 设计本来就藏

## 后果

### 正面
- 升级 modal 复制按钮在 HTTP / HTTPS 任何场景都能工作 (3 个页面都修)
- 客户看到的升级命令直接是 `cd /root/tg-monitor-xxx && bash update.sh`, 无 SSH 包装无误导
- 管理员在驾驶舱日志面板上方有「立刻深度诊断」按钮永远可见, 主动巡检 + 测试新功能都能用
- 普通成员仍看不到深度诊断入口 (跟 v3.0.8.1 admin button gating 一致)

### 潜在问题 / trade-off
- **新「Sheet 写入诊断」section 占垂直空间**: 一行 (~40px) 在日志面板上方。可接受, 比放卡片旁边更显眼
- **`copyTextFallback` 在 3 个 templates 各 inline 一份**: 没抽公共 .js 文件, 维护时要 3 处同步。但 v3.0.9 应该考虑抽 dashboard.js / setup.js / index.js 出去 (本版不动)
- **execCommand('copy') 已被 W3C 标记 deprecated**: 但浏览器 (Chrome / Firefox / Safari) 都还支持且短期内不会移除, 作为 HTTP 场景兜底完全够用

### 未涉及 (v3.0.9+)
- 抽 templates 公共 JS 到独立 .js 文件 → v3.0.9
- `users.json` atomic write (Codex v3.0.8.1 P1 deferred) → v3.0.9
- 升级流程进度条 polling 改 SSE → v3.0.9
- 删除 `VPS_PUBLIC_IP` env (现在没用了) → 留 v3.0.9 一并清理

## 升级指引

```bash
cd /root/tg-monitor-<部门> && ./update.sh
```

升级后:
- 「版本升级」modal 复制命令按钮在 HTTP/HTTPS 任何场景都能用
- 升级命令是 `cd /root/tg-monitor-xxx && bash update.sh` (无 SSH wrap)
- 管理员账号在驾驶舱看到「Sheet 写入诊断 ▸ 立刻深度诊断」永远可见入口 (日志面板上方)
- 普通成员看不到深度诊断入口 (admin-only)

## 验证

### 静态自查
- `grep -nE "ssh root@" upgrader.py` — 只剩注释提及, 无 code 生成
- `grep -nE "navigator.clipboard.writeText" templates/*.html` — 仍存在但都包在 try/catch + prompt fallback 或 copyTextFallback 内, 无裸 inline
- `grep -nE "copyTextFallback" templates/*.html` — 3 个 templates 各 1 处定义 + 1 处 button 调用
- `grep -nE "openSheetsDeepDiag" templates/dashboard.html` — 至少 2 处 (老 sh-deep-diag-wrap 1 处 + 新 always-visible 1 处)
- `grep -nE "is_admin" templates/dashboard.html` — Jinja `{% if is_admin %}` 至少 1 处 (新 always-visible section)
- DIV balance: 3 个 templates `<div>:</div>` 平衡

### 集成 (dev / 客户测试)
1. **HTTP 场景复制**: 用 IP:5001 直连驾驶舱, 触发升级 modal (有 docker-compose.yml 改动) 的「复制命令」按钮 → 应该弹 prompt 或显示 ✓ 已复制
2. **HTTPS 场景复制**: 用 https://xxx.nip.io 驾驶舱同操作 → 应该 ✓ 已复制 (走 navigator.clipboard 主路径)
3. **升级命令文本**: modal 显示的命令应该是 `cd /root/tg-monitor-xxx && bash update.sh`, 不带 ssh 前缀
4. **深度诊断永远可见**: admin 账号登录, 驾驶舱日志面板上方应有「Sheet 写入诊断」section + 「立刻深度诊断」按钮
5. **普通成员隐藏**: 普通成员账号登录, 同位置 **不应** 有「Sheet 写入诊断」section

### 回归
- v3.0.8 + v3.0.8.1 全部功能不变 (append API / 令牌桶 / 诊断卡片 / 立刻重启 / docker cp 列表)
- 卡片下方 sh-deep-diag-wrap 按钮(诊断卡片显示时跟着出)行为不变 — 跟新永远可见入口共存, 客户哪个先看到点哪个
- /api/restart / /api/diag/sheets-stuck-detail / /api/diag/sheets-fix-stuck endpoints 后端不动
