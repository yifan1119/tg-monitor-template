# ADR-0009: 预警分页历史空白自动回填(承接 v2.10.24.1)

**版本**:v2.10.24.2(hotfix 续)
**日期**:2026-04-22
**状态**:Accepted

## 背景

v2.10.24.1 修好了 `sync_headers` 的 429 配额爆问题,但留下一个历史脏数据问题:

**时间线**(某客户 150+ 账号事故):
- 04-22 15:07 客户 Google Sheets 读配额爆(450 reads/min),`sync_headers` 每调必 429
- 04-22 16:25 运维在 VPS 手动 `sed` 把 `tasks.py:165` 改成 `pass` 止血 —— 彻底关停 `sync_headers`
- 04-22 16:25 后 客户在 TG 新登录若干外事号,系统自动建对应 Sheets 分页
- 客户在新分页 B2(商务人员)/ B3(所属公司)填好 —— 但 `sync_headers` 已被关,值**永远不会同步到 DB**
- 期间关键词命中 / 未回复 / 删除预警触发 —— `bot.py:142/147/190` 写三个预警分页时从 DB 读 `account.operator` / `account.company` 全是空 —— 写入的 A/B 栏全是空白

**客户视角**:升级 v2.10.24.1 后打开表格,关键词监听分页 A 栏(所属公司)+ B 栏(商务人员)还是一大片空白。三个预警分页同样症状。

**代码层面**:全代码没有任何 loop 会扫回「关键词监听/未回复预警/删除预警」分页的历史行然后重写 A/B 栏(查过 `Grep backfill|回填|reprocess`,只有 listener 的 `_backfill_peer` 和 sheets 的 `[delete_backfill]`,都不管 operator/company)。`append_row` 是命中瞬间一次性写入,当时 DB 什么值就写什么,之后 `sync_headers` 只更新 `accounts` 表不会回去改 Sheets 历史。

**客户需求**:一键 `./update.sh` 升级就自动补好,不能要求运维跑任何 `docker` / `python` / `sql` 命令(「用户他们根本不会用」)。

## 决策

v2.10.24.2 加入**启动立刻补 + 定期巡检补漏**的自动化回填机制:

### 1. 新方法 `SheetsWriter.backfill_alert_history()`(`sheets.py`)

```python
def backfill_alert_history(self):
    """扫三个预警分页,空 A/B 栏用 DB accounts.company/operator 回填。

    幂等 — 只填空栏,已有值的行不动。
    配额 — ~7 次 API(1 worksheets + 3 get_all_values + 3 batch_update)。
    返回 (filled_rows, skipped_rows_db_empty)。
    """
    alert_tabs = [f"信息未回复预警{suffix}", f"关键词监听{suffix}", f"信息删除预警{suffix}"]
    # 1. 读 DB 所有 accounts 的 name → (operator, company) 映射
    # 2. 一次 worksheets() 拿分页索引
    # 3. 对每个分页:
    #    a. get_all_values() 拿全部行
    #    b. 遍历数据行(跳过表头):
    #       - C 栏(外事号)取出
    #       - 如果 A 或 B 空 且 DB 对应账号有值 → 加进 batch_updates
    #       - 如果 A B 都空 且 DB 也空 → 计入 missing_accounts,下次再来
    #    c. batch_update 一次性写回(`USER_ENTERED`)
    # 4. 日志输出 filled / skipped / missing 清单
```

**统一结构**(已验证):三个预警分页 A 所属公司, B 商务人员, C 外事号, D 广告主, ... — 见 `sheets.py:367-371` ALERT_HEADERS。

### 2. 启动时立刻跑一次(`sheets.py:__init__`)

```python
self.ensure_alert_tabs()
self.ensure_account_tabs()
if config.BACKFILL_ALERT_HISTORY:
    try:
        self.backfill_alert_history()
    except Exception as e:
        logger.warning("backfill_alert_history 启动时失败(不阻塞启动): %s", e)
```

- 补 90%+ 历史(客户能立刻看到表格变干净)
- 异常隔离 —— 回填失败不阻塞启动

### 3. 独立 loop `_alert_backfill_loop`(`tasks.py`)

```python
async def _alert_backfill_loop(self):
    await asyncio.sleep(600)   # 启动后等 10 分钟,让 sync_headers 先同步
    while self._running:
        try:
            if config.BACKFILL_ALERT_HISTORY and self.sheets:
                self.sheets.backfill_alert_history()
        except Exception as e:
            logger.error("alert_backfill_loop 异常: %s", e)
        await asyncio.sleep(config.BACKFILL_ALERT_INTERVAL_SEC)  # 默认 3600s
```

- 补新命中但当时 DB 还没 operator 的行(新登录账号 <10 分钟内命中场景)
- 加进 `start()` 的 `asyncio.gather` 列表

### 4. 新配置(`config.py`)

```python
BACKFILL_ALERT_HISTORY = os.environ.get("BACKFILL_ALERT_HISTORY", "true").lower() == "true"
BACKFILL_ALERT_INTERVAL_SEC = int(os.environ.get("BACKFILL_ALERT_INTERVAL_SEC", "3600"))
```

- 默认**开**(客户零操作升级生效)
- 可关 —— 不想用的客户 `.env` 加 `BACKFILL_ALERT_HISTORY=false`

## 原因

### 为什么不提供独立脚本(`scripts/backfill.py`)
- 客户不会跑终端命令(ADR-0008 改 update.sh 的同一个原因)
- 单独脚本要 `docker compose exec python scripts/backfill.py`,客户看都看不懂
- 做进主代码 + 开关默认开 → `./update.sh` 一键升级自动生效

### 为什么启动时 + 定期巡检,不只做启动
- 客户 **未来还会**新登录外事号
- 新账号登录后的头几分钟,`sync_headers` 还没同步 operator/company 到 DB
- 这几分钟内如果有关键词命中 → 预警行 A/B 栏又会是空
- 启动只跑一次无法覆盖这种情况 → 定期巡检补漏

### 为什么默认 1 小时一次
- 配额:每轮 ~7 次 API,1 小时 7 次 = 0.12/min,绝对安全
- 时效:客户能接受历史数据 1 小时内补齐
- 可调 —— 客户想更快设 `BACKFILL_ALERT_INTERVAL_SEC=600`(10 分钟),想更省设 `=86400`(1 天)

### 为什么幂等(只填空栏)不覆盖已有值
- 客户可能**手动**在预警分页改过某行的 operator(比如发现原归属错了手动改掉)
- 如果回填时「强制用 DB 值覆盖」,客户的手改会被抹掉 → 客户投诉
- 只填空栏 —— 客户从来不填空表示「保持空」,有值就不动 —— 幂等安全

### 为什么先读全部分页数据再 `batch_update`(不是逐行 `update`)
- `get_all_values` 一次 API 拿全部行,比逐个 row 读快
- `batch_update` 一次 API 写多行,比逐个 cell 写省
- 三个分页合起来 ~7 次 API,远低于 Google 60/min 配额

### 为什么 `_alert_backfill_loop` 启动后等 10 分钟才开始第一轮
- 启动时 `sheets.__init__` 已经跑过一次
- 独立 loop 第一轮 10 分钟后才跑 —— 让 `sync_headers` (600s 间隔)先把新账号的 operator 同步到 DB,再来补
- 避免「启动补一次 → 10 秒后 loop 又补一次」重复开销

## Codex review 修复(medium reasoning)

### Major 1:同名外事号错误归属

**问题**:首版用 `accounts.name` 做 key 构建 `name_to_info` 映射,但 DB schema 没保证 `name` 唯一。若两个外事号昵称相同(客户复制昵称 / TG 允许重名),dict 只保留最后一个,导致同名所有历史行被错误归属到同一人。

**修复**:检测重复 name → 标 `ambiguous_names` → 从映射里踢出 → 这些外事号的历史行统一跳过回填 + log 提醒客户哪些同名。

### Major 2:`batch_update` 范围粒度过粗,会重写客户手填值

**问题**:首版对「A 空 B 有值」的行执行 `batch_update([{"range": f"A{i}:B{i}", "values": [[new_company, new_operator]]}])`,虽然 `new_operator == operator_cell`(即回写客户手填值),但:
1. 如果 B 栏是公式(客户可能用 `=VLOOKUP(...)`),`USER_ENTERED` 会把公式替换成当前显示值 → 客户公式丢失
2. 读后写前如果客户改了 B 栏,回填会把旧值写回去 → 覆盖并发改动

**修复**:按单元格粒度更新 —— 只在 `company_cell` 空且 `db_company` 有值时写 `A{i}`,只在 `operator_cell` 空且 `db_operator` 有值时写 `B{i}`。完全不动已有值的单元格,严格满足「只填空栏」语义。

## 后果

### 正面
- **客户零操作体验**:`./update.sh` 升级 → 重启自动补 90%+ 历史 → 每小时巡检补漏
- **配额安全**:每轮 7 次 API,不会搞出新的 429
- **幂等安全**:跑多少次都不会破坏客户已有数据
- **透明可观测**:找不到 DB 值的外事号会 log 清单,客户按清单补 B2/B3 → 下轮自动补

### 中性
- 启动日志多一行 `backfill_alert_history: 完成 — 补填 N 行, 跳过 M 行`
- `.env` 多两个可选配置(默认值就是最优解,客户不用看)

### 需要客户做的(只在 log 告诉他们)
- 某些外事号 DB 里 operator/company 也空(客户至今没在该外事号分页 B2/B3 填过) → log 会输出外事号清单
- 客户去对应外事号分页 B2/B3 填完 → 等下一轮(1 小时内)`sync_headers` 同步 + 回填自动接力

### 后续版本可能的改进(不在本版做)
- `SheetsWriter._fetch_bulk_alert_values` 把三个 `get_all_values` 合成一次 `batch_get` —— 对未来 400+ 账号分页万行级别时有意义
- Web UI 加「立即补填」按钮 —— 客户不想等 1 小时可以手动触发
- 回填统计推送到 TG 预警群 —— 有 N 行待补时通知运维关注
