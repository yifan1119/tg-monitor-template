# ADR-0016: v3.0.0 两段式未回复预警 — 推送 / 模板 / callback / Telethon 真名解析 / 自动升级 loop

**版本**:v3.0.0
**日期**:2026-04-23
**状态**:accepted(集成到 `integration/v3.0.0-on-main`,等测试 + Codex round2 + merge main + tag v3.0.0)

## 背景

ADR-0015 把数据层(migration V5 + accounts 4 字段 + alerts.stage + 5 个 flag + 3 个装置伪装字段)做好,
本 ADR 处理 **bot / templates / tasks / listener 四个模块的行为实现**,让配置真的能跑起来。

目标:`TWO_STAGE_NO_REPLY_ENABLED=true` 时,完整两段式链路可用 —
1. Stage 1(30 分钟)推 @ 商务,无按钮
2. 员工在外事号侧 outbound 回复 → 自动结案
3. 未回复满 stage2_after_min → 自动升级 Stage 2 @ 负责人 + 违规/取消按钮
4. 审核员点按钮 → 登记违规到 Sheet / 取消

## 决策

### 1. `templates.py` — stage1 / stage2 HTML 模板

新增两个函数:

```python
def no_reply_alert_stage1(*, company, operator, account_name, peer_name, message_text,
                          minutes, business_mention, custom_text) -> str: ...

def no_reply_alert_stage2(*, company, operator, account_name, peer_name, message_text,
                          minutes, owner_mention, custom_text) -> str: ...
```

**关键规则**:
- 纯文本字段(company / operator / account_name / peer_name / message_text)全部过 `html.escape`
  — 防客户填入含 `<>&` 的值把 HTML parser 整崩
- `business_mention` / `owner_mention` **已经是 HTML**(由 `bot._build_tg_mention` 输出),不 escape
- `custom_text` 也 escape(客户自定义的纯文本)
- `*_mention` 为空 / `custom_text` 为空时模板退化不显示对应段(不报错)
- 所有推送必须 `parse_mode="HTML"`

### 2. `bot.py` — TG mention 三层解析

#### 2.1 `_format_tg_mention(tg_id_or_username, display_name)` — 最底层兜底

分支判断:

| 输入 | 输出 | 备注 |
|---|---|---|
| 纯数字(例如 `123456789`) | `<a href="tg://user?id=N">{escaped_name}</a>` | **HTML inline mention**,对方无 username 也能触发群 @ 通知 |
| 含字母 / `@xxx` / 混合 | `@xxx` | TG username 字符集天然合法,无需 escape |
| 空 / 空白 | `""` | 模板据此退化不带 @ |

`display_name` 一律 `html.escape`。

#### 2.2 `_resolve_tg_entity(tg_id_or_username)` — Telethon 查 entity

v2.10.26 测试期客户反馈:`@username` 在群里显示蓝字 username,不够醒目。希望显示**真名**。

方案:用 listener 的 Telethon client(main.py 注入 `bot.listener = listener`),调
`await client.get_entity(...)` 拿 numeric user_id + first_name / last_name,拼成 HTML inline mention。

```python
async def _resolve_tg_entity(raw):
    if not raw or not self.listener:
        return None
    try:
        entity = await self.listener.client.get_entity(raw)
        display = " ".join(x for x in (entity.first_name, entity.last_name) if x).strip() or str(entity.id)
        return entity.id, display
    except Exception:
        return None    # 失败兜底走 _format_tg_mention
```

**优先解析 numeric 输入**:客户填 `123456789` 也走 Telethon 查真名,显示「王小明」而不是一串数字。
**解析失败兜底**:网络异常 / 对方没 username 且 bot 没见过 → 返回 None,上层 fallback 到 `_format_tg_mention`。

#### 2.3 `_build_tg_mention(raw)` — 上层入口

```python
async def _build_tg_mention(raw):
    if not raw: return ""
    resolved = await self._resolve_tg_entity(raw)
    if resolved:
        uid, display = resolved
        return f'<a href="tg://user?id={uid}">{html.escape(display)}</a>'
    # fallback
    return self._format_tg_mention(raw, display_name=raw)
```

### 3. `bot.py` — 推送函数

#### 3.1 `send_no_reply_alert` 老入口 flag 分流

```python
async def send_no_reply_alert(self, account_id, peer, message_text, msg_id):
    config.reload_if_env_changed()
    if config.TWO_STAGE_NO_REPLY_ENABLED:
        return await self.send_no_reply_alert_stage1(account_id, peer, message_text, msg_id)
    # ↓ 以下保留 v2.10.25 原逻辑 100% 不变
```

**flag=False 时一行没改老路径**,保证向后兼容。

#### 3.2 `send_no_reply_alert_stage1`

- 推送群:`UNREPLIED_ALERT_GROUP_ID or ALERT_GROUP_ID`(fallback 老群)
- 沿用 `has_alert_today("no_reply", peer_id)` 天级去重
- 沿用 `ALERT_NO_REPLY_ENABLED` 子开关,关时插 `insert_stage1_alert` 并标 silenced
- 文案优先级:`account.remind_30min_text` → `config.REMIND_30MIN_TEXT` → 模板默认
- `business_mention = await self._build_tg_mention(account.business_tg_id)`
- `insert_stage1_alert(stage=1)` → `send_message(..., parse_mode="HTML")` → `update_alert_bot_msg(msg_id)`
- 失败语义:`bot_message_id` 保持 null → `has_alert_today` 不认作已推 → 下轮重试
  (沿用 ADR-0003 的 v2.10.23 语义)

#### 3.3 `send_no_reply_alert_stage2(alert_id)`

入口由 tasks 的 `_no_reply_stage2_loop` 调用。

- **并发保护**:`db.upgrade_to_stage2(alert_id)` 是原子 `UPDATE alerts SET stage=2
  WHERE id=? AND stage=1 AND status='pending'`,`rowcount=1` 才继续推
- 推送群同 stage1(优先 `UNREPLIED_ALERT_GROUP_ID`)
- 文案优先级:`account.remind_40min_text` → `config.REMIND_40MIN_TEXT` → 模板默认
- `owner_mention = await self._build_tg_mention(account.owner_tg_id)`
- 附 `_make_keyboard_stage2(alert_id)` 按钮(violation / cancel)
- 推送后 `update_alert_bot_msg(stage2_message_id)` 覆盖 message_id,让 callback edit_text 用新 id
- 失败 trade-off:stage=2 已写无法回退;但 status 仍 pending → 下轮可重推
  (极少数 stage2 推送失败会在群里少一条消息,可接受)

#### 3.4 `_make_keyboard_stage2(alert_id)` 按钮工厂

两个 inline button:
- callback_data: `violation:{alert_id}` — 登记违规
- callback_data: `cancel:{alert_id}` — 取消

动作名独立于老 `approved` / `rejected`,不跟单段审核按钮混。

#### 3.5 `on_stage2_action` callback handler

```python
@self.dp.callback_query(F.data.startswith("violation:") | F.data.startswith("cancel:"))
async def on_stage2_action(callback): ...
```

流程:
1. **权限校验**:`callback.from_user.id in config.CALLBACK_AUTH_USER_IDS`(沿用老白名单)
2. **原子抢占**:`db.claim_alert_for_review(alert_id, new_status)` — WHERE status='pending',不限 stage
   (防两个审核员同时点)
3. `violation` → 写 Sheet(见第 4 节);`cancel` → 仅 edit_text 删按钮显示「已取消」
4. 异常兜底 `callback.answer("操作失败,请看日志")`,不抛 bot 崩

### 4. `_write_alert_to_sheet` 扩展(Sheet 写入格式)

**v2.10.26 测试期客户反馈调整**:不要在预警分表末尾加「违规登记」第 7 列,跟老 6 列结构保持一致。

最终写入:
- 走原有 6 列结构(时间 / 账号 / 广告主 / 消息 / @ / 商务 操作人)
- `violation_logged` 状态直接写进「处理状态」列,不额外加末列
- 老客户回滚到 v2.10.25 看这张表不会多出一列 → 完全向后兼容

Q2 决策:**同一张 `信息未回复预警{部门}` 分表,结构不变,回滚干净**(不新开表)。

### 5. `tasks.py` — `_no_reply_stage2_loop` 自动升级

```python
async def _no_reply_stage2_loop(bot, db):
    while True:
        config.reload_if_env_changed()
        if not config.TWO_STAGE_NO_REPLY_ENABLED:
            await asyncio.sleep(60)
            continue
        after_min = config.NO_REPLY_STAGE2_AFTER_MIN
        pending = db.get_pending_stage1_alerts(after_minutes=after_min)
        for alert in pending:
            try:
                await bot.send_no_reply_alert_stage2(alert["id"])
            except Exception:
                logger.exception("stage2 升级失败")
        await asyncio.sleep(60)
```

**事件驱动 + poll 兜底的双路径**:
- **事件驱动(主路径)**:listener outbound 钩子(见第 6 节)收到员工回复立刻 `mark_stage1_handled_by_reply`,
  把 alert 的 status 改掉 → loop 下次扫不到
- **poll 兜底(次路径)**:每 60s 扫一遍 `stage=1 status='pending' created_at < now - after_min`,
  推 stage2

flag 关时 loop 空转 sleep,不浪费 CPU。

### 6. `listener.py` — outbound 钩子 + TG 装置伪装

#### 6.1 outbound `_handle_message` 钩子

员工在外事号 App 回复消息时,listener 的 `on_outgoing` 事件会触发。
增补一段:

```python
async def _handle_message(self, event, ...):
    # ... 原有落盘逻辑
    if not event.out:
        return   # 只对 outbound 生效
    if config.TWO_STAGE_NO_REPLY_ENABLED:
        db.mark_stage1_handled_by_reply(account_id, peer_id)
```

`mark_stage1_handled_by_reply` 把该账号 × 该对话当天所有 `stage=1 status='pending'` 的 alert
改成 `status='resolved_by_reply'`,不会再被 stage2 loop 扫到。

#### 6.2 TG 装置伪装(长期防风控)

Telethon 默认 session 会带 `Telethon/1.x` 装置签名,TG 风控容易识别为 userbot。外事号是客户核心资产,
冻结损失大。

在创建 `TelegramClient` 时显式传:

```python
client = TelegramClient(
    session_path,
    api_id, api_hash,
    device_model=config.TG_DEVICE_MODEL,        # 默认 "shencha"
    system_version=config.TG_SYSTEM_VERSION,    # 默认 "1.0"
    app_version=config.TG_APP_VERSION,          # 默认 "tglistener 1.0"
)
```

默认值够中性。风险敏感客户可自定义成 `TG Desktop` / `Windows 10` / `5.1.1` 模拟 TG Desktop。

## 原因

### Q1 — stage2 按钮权限沿用 `CALLBACK_AUTH_USER_IDS`(不是只限 owner_tg_id 本人)

**选项**:
- A. 沿用老审核按钮白名单 `CALLBACK_AUTH_USER_IDS`
- B. 只有 `owner_tg_id` 本人能点

**选 A**:客户审查员已经在老白名单里管理,沿用同一套免去训练;管理员兜底防负责人失踪。

### Q2 — 违规登记写同一张预警表,不加末列

**选项**:
- A. 同一张 `信息未回复预警{部门}`,末列加「违规登记」第 7 列
- B. 同一张 `信息未回复预警{部门}`,不加列,`violation_logged` 写进「处理状态」列
- C. 新开一张 `信息违规登记{部门}`

**选 B**(**v2.10.26 测试期从 A 改 B**):客户反馈「多一列看着乱」。改成 B 后回滚 v2.10.25 老代码看同一张表结构 100% 一致。

### Q3 — @mention 显示真名而不是 `@username`(v2.10.26 测试期客户反馈)

客户:「@xxx 群里显示蓝字 username 没存在感,看不出是哪位王经理。」

解决:`_build_tg_mention` 走 Telethon `get_entity` 查真名 → HTML inline mention 显示「王小明」而不是 `@wang123`。
解析失败降级到 `_format_tg_mention`(纯 `@xxx`),不会崩。

### Q4 — 文案全域统一,不要每个号单独填(v2.10.26 测试期客户反馈)

客户 200+ 账号,每个都单独填 30min/40min 文案不现实。
最终方案:账号级 4 字段仍在(给 VIP 特殊需求用),空值 fallback 到 `.env` 全域 `REMIND_30MIN_TEXT` / `REMIND_40MIN_TEXT`。
客户一次改 `.env` 全部账号生效。

### Q5 — TG 装置伪装 3 字段为什么默认 `shencha` / `1.0` / `tglistener 1.0`?

- 不能用 `Telethon` 关键词 — TG 风控 ML 模型学过
- 不能用 `TG Desktop` / `TG Android` 等真实产品名 — 冒用官方产品名风险
- `shencha`(审查)+ `tglistener` 是中性内部命名,不撞真实产品
- 客户觉得风险高可以自改成 `TG Desktop` / `Windows 10` / `5.1.1`(这些是官方 App 签名,不做任何风控上的保证)

### Q6 — 事件驱动 vs 纯 poll?

**双路径**:
- 事件驱动(listener outbound 钩子)— 员工回复后即时结案,客户体验好(stage2 loop 下一轮就不扫)
- poll 兜底(60s)— 事件漏掉 / listener 短暂不可达时兜底升级

只做事件驱动风险:listener crash / 消息漏触发 → 永远升不到 stage2。加 poll 是廉价的兜底保险。

## 后果

### 正面

- **完整两段式链路可用**:30min @ 商务 → 40min @ 负责人 + 违规/取消按钮
- **向后兼容**:flag=False 时 `send_no_reply_alert` / `_write_alert_to_sheet` / 预警分表结构跟 v2.10.25 一模一样
- **推送失败可重试**:继承 ADR-0003 语义
- **stage1 → stage2 升级原子**:`upgrade_to_stage2` rowcount=1 才推,并发安全
- **真名 @ 显示**:Telethon 查 entity 拿 numeric id + first_name + last_name → HTML inline mention
- **员工回复自动结案**:事件驱动(outbound 钩子)+ poll 兜底,客户体验好
- **装置伪装降低风控风险**:session 签名不再是 Telethon 默认

### 副作用 / 需要后续跟进

- **stage2 推送失败吃掉一次升级机会**:`stage=2` 无法回退到 1。视生产数据决定是否加 retry
- **Telethon get_entity 缓存**:首次解析 username 要访问 TG API(慢 ~1s),后续走 cache 快。大并发场景可能打 TG rate limit → 未来考虑本地 entity cache
- **装置伪装无法保证不被 TG 风控识别**:只降低概率,不是 0 风险
- **全域文案优先级需代码清晰**:`account.text if account.text else config.GLOBAL_TEXT`(不能反)

### 代码改动点

- `templates.py` — `no_reply_alert_stage1` / `stage2` + `html.escape` 统一
- `bot.py` — `_format_tg_mention` / `_resolve_tg_entity` / `_build_tg_mention` / `_make_keyboard_stage2` / `send_no_reply_alert_stage1` / `send_no_reply_alert_stage2` / `on_stage2_action` / `_write_alert_to_sheet` 扩展
- `tasks.py` — `_no_reply_stage2_loop`
- `listener.py` — `_handle_message` outbound 钩子 + `TelegramClient` 三字段伪装
- `main.py` — `bot.listener = listener` 注入(给 `_resolve_tg_entity` 用)
