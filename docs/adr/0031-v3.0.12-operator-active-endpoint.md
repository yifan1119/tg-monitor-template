# ADR-0031 · v3.0.12 · 商务人员每日活跃广告主数 endpoint

## 背景

2026-04-30 客户反馈:中央台想看每个商务人员的「日活跃广告主数」绩效统计 —
即该商务负责的所有外事号(`accounts.operator`)当天跟多少个不同广告主
(`peer_id`)有过任意方向消息。

## 决策

新增 `/api/v1/operator_active?from=X&to=Y` 只读 endpoint:

```sql
SELECT a.operator, substr(m.timestamp, 1, 10) AS day,
       COUNT(DISTINCT m.peer_id) AS active_peers
FROM messages m JOIN accounts a ON m.account_id = a.id
WHERE m.timestamp >= ? AND m.timestamp < ?
  AND a.operator != ''
GROUP BY a.operator, day
```

返回:
```json
{
  "ok": true,
  "items": [
    {"operator": "徐郎", "day": "2026-04-29", "active_peers": 12, "account_count": 3},
    ...
  ]
}
```

`account_count` 是该 operator 名下外事号总数(给中央台算「平均/号」用)。

## 实现细节

- `dashboard_api.py::operator_active(from, to)` — SQL + 验参 + 抛 ValueError
- `web.py::api_v1_operator_active` — `/api/v1/operator_active` 路由,
  沿用 `_v1_check_token` 鉴权 + 错误码契约(401/400/500)
- 跟 v3.0.9.1 / v3.0.10 / v3.0.11 同模式:0 新表 0 新字段 0 数据迁移

## 原因

1. **SQL 只读** — 不动 listener / sheets / bot,200+ TG 账号不重登
2. **JOIN 性能** — messages 表已有 timestamp + account_id 索引,GROUP BY
   常规客户量(100+ 号 × 1000 msg/天)毫秒级完成
3. **`account_count` 顺手算一个** — 中央台前端不用反查 accounts 表
4. **timestamp 范围比较 vs LIKE** — `timestamp >= ? AND timestamp < ?` 走索引,
   to+1 天作右开区间

## 来源

2026-04-30 客户反馈:「可以把每个商务人员每天的活跃数统计出来吗 — 这个商务的
所有外事号每天与多少广告主进行沟通过」

## 后果

- 中央台升级后能拉这个统计 + 看板加「商务活跃榜」表格
- 任意方向消息都算(广告主→外事号 OR 外事号→广告主),最直观
- 副作用:无(纯只读)
- 想要「只算回复过的」「只算今日新增」等更细维度 → 留下版本

## 测试

- `operator_active("2026-04-01", "2026-04-30")` 返回非空
- 错参(`from=garbage` / `to < from`)→ ValueError → endpoint 返 400
- 不带 token → 401
- 该 operator 当天对多个 peer 收发消息 → active_peers 是去重后的 peer 数
