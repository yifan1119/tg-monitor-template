# ADR-0058 · v3.3.0 peers.first_seen_at 拓客量统计字段

## 背景

中央台「商务活跃榜」原本只有 5 个指标:外事号数 / 总活跃广告主×天 / 平均/天 / 活跃天数。客户反馈需要看「拓客量」 — 该商务名下外事号在区间内**第一次出现**的广告主数,区分「老客户日常聊」和「拉到新客」。

之前没记录 peer 第一次出现时间,只能从 `messages.timestamp` 推回(MIN per peer_id),百万级消息表每次报表查 2-5s。

## 决策

`peers` 表加 `first_seen_at TEXT DEFAULT NULL`:
- 新 peer 在 `upsert_peer` INSERT 时写入 NOW(BJ ISO)
- 老 peer 一次性 migration backfill `MIN(messages.timestamp)`
- 加索引 `idx_peers_first_seen_at` 给报表查询用

中央台「商务活跃榜」加一列「新增活跃对话」(operator + day GROUP BY → 中央台 (operator, dept) 复合 key 累加),前端 8 列展示。

## 原因

1. **性能稳定**:报表查询从「翻全量 messages MIN」→ 「peers 字段直读 + 索引」,毫秒级
2. **零数据迁移成本**:migration 一次跑完,后续不再扫消息表
3. **向后兼容**:老 peer 没 first_seen_at(NULL)→ COUNT(*) 自动跳过(WHERE first_seen_at IS NOT NULL 隐含 GROUP BY substr 跳 NULL),不影响老查询语义
4. **跟旧逻辑配合**:operator_active 主 SQL(按 messages 算 active_peers)不动,新增 SQL 单独算 new_peers,merge 到 (operator, day) key,语义清晰

## 来源

2026-05-19 用户截图反馈「这个区块需要新增一个 新增活跃对话」→ 用户解释「就是该外事号新增的对话」 → 决定加 first_seen_at 字段(方案 B,vs 每次查询 MIN 算的方案 A)。

## 后果

**代码改动**:
- `database.py`:_migrate_to_9 + upsert_peer 加 first_seen_at + `sync_peer_first_seen_at_from_messages` helper + `idx_msg_peer_ts` 复合索引(~30 行)
- `dashboard_api.py`:operator_active 加第二段 SQL + union (operator,day) keyset 合并(~20 行)
- `listener.py`:pull_history 单 peer 历史拉完后调 sync 校正(3 行)
- `web.py`:docstring 补 new_peers(1 行)

**Codex review P0/P1 修正**(round 1):
- **P0**(`listener.py:434`):pull_history 给老 peer 调 upsert_peer 写 NOW,但实际消息是历史时间 → 升级首日新增对话虚高。修法:`db.sync_peer_first_seen_at_from_messages(peer["id"])` 在该 peer inner iter_messages 跑完后调一次,UPDATE first_seen_at = MIN(messages.timestamp);COALESCE 兜底 messages 表空时不破坏。
- **P1.1**(`database.py:_migrate_to_9`):backfill 用相关子查询 MIN,只有 idx_msg_peer 单列索引 → 大表慢。加 `idx_msg_peer_ts ON messages(peer_id, timestamp)` 复合索引让 backfill 走 index-only,从 30s 缩到 < 5s
- **P1.2**(`dashboard_api.py:operator_active`):原 merge 只迭代 messages 维度的 rows,(op, day) 只有 new_peers 但无 messages 的极端 case 漏。改 union 两边 keyset 后再 merge
- **P1.3**(`center_detail.html`):colgroup 加起来不到 100% → 改 50px + 7 列 100% 平衡

**升级路径**:
- 客户跑 `hotfix_v3.1.8.sh`(TARGET_TAG=v3.3.0)→ recreate tg-monitor → migration 自动跑
- backfill 耗时:200 账号 × 50 万 messages dept 约 5-15 秒(EXPLAIN QUERY PLAN 走索引 idx_msg_peer)
- 跑完中央台报表 5 分钟内自动反映新增活跃对话数

**风险**:
- migration backfill 失败(SQLite write lock 长持)→ 容器启动卡住:风险低,SQLite 单连接 + 200 账号规模常见 < 15s,容器 health check 60s 默认 timeout 够
- 配套中央台 v0.42 没升 → 老中央台收到 new_peers 字段但 UI 不展示,无伤害(JSON 多一字段忽略)
- 单升中央台 v0.42 没升 dept v3.3.0 → operator_active 返回 dict 无 new_peers,中央台 `r.get("new_peers") or 0` 兜底为 0,UI 显示全 0 列(无报错,只是没数据)

**回滚**:
- 代码层:git revert
- DB 层:`PRAGMA user_version=8`(列保留,无副作用)
