# ADR-0060: v3.3.8 — stage2 升级 / 删除预警按钮恢复

## 背景

2026-05-19 用户截图反馈:多个客户群里 stage2 未回复升级预警(40 分钟超时 @ 负责人)只显示纯文本,没有「登记违规 / 取消」按钮。监察员无法在群里一键处理,必须回 web 后台,效率低。

**范围澄清**(Codex P1):
- ✅ 受影响的:`send_no_reply_alert_stage2`(stage2 升级)、`send_delete_alert`(删除预警 owner_tg_id 路径 + 老 approve/reject 路径)、`send_no_reply_alert`(老单段未配 business_tg_id 路径)
- ❌ 不受影响的:`send_keyword_alert`(关键词预警从设计起就无按钮,`bot.py:488` 直接 `send_message` 不附 `reply_markup`)、`send_no_reply_alert_stage1`(30 分钟 stage1 设计上无按钮,docstring 明示「@ 商务人员,无按钮」)

排查发现:**v3.1.9 [ADR-0055] 的 `BOT_POLLING_DISABLED` 短路逻辑跟 `_callback_meta_for` 的配置要求互相打架**,fallback 路径下 44/47 dept 按钮被删。

### 现场数据(2026-05-19 21:30 全网 fleet inspect)

| 状态 | dept 数 |
|---|---|
| 路由配齐(VPS_PUBLIC_IP 已设,中央台路由正常生效) | **2** |
| 缺 VPS_PUBLIC_IP / DEPT_PUBLIC_URL / PUBLIC_HOSTNAME | **44** |
| inspect 失败 | 1 |

过去 24h 中央台 `pending_callbacks` 表只有 2 个 route 产生过条目(领航商务 + 悦达商务),其他 **34 路由的预警全部 fallback 到本地** → 因 BOT_POLLING_DISABLED 短路 → 按钮被删。

### 调用链(stage2 预警,有缺 VPS_PUBLIC_IP 的 dept)

```
tasks._no_reply_stage2_loop
  → bot.send_no_reply_alert_stage2
  → _callback_meta_for(account_id, alert_id, buttons)
    → metrics_token=ok, dept_url=""  # ← VPS_PUBLIC_IP/PUBLIC_HOSTNAME 都缺
    → return None
  → cb_meta is None,跳过 _try_central_route
  → self.bot.send_message(reply_markup=_make_keyboard_stage2(alert_id))
    → _make_keyboard_stage2:
       getattr(config, "BOT_POLLING_DISABLED", False) == True
       → return None  ← 按钮被删
  → 消息发出,纯文本无按钮
```

## 决策

**revert v3.1.9 的 `BOT_POLLING_DISABLED → None` 短路。永远附按钮。**

具体改动:`bot.py` 两个方法删 `if getattr(config, "BOT_POLLING_DISABLED", False): return None`:
- `_make_keyboard(self, alert_id)` — keyword / delete / 历史 stage1 通过/拒绝按钮
- `_make_keyboard_stage2(self, alert_id)` — stage2 升级 登记违规/取消按钮

老的 callback_data 格式保留(`approve:N` / `reject:N` / `violation:N` / `cancel:N`)。

## 原因

### 1. v3.1.9 的预设条件没满足

v3.1.9 的逻辑前提:**共用 BOT_TOKEN 的客户必须通过中央台路由 + 中央台 callback_listener 处理按钮**。

但实际部署:
- 44/47 dept `.env` 没配齐 `VPS_PUBLIC_IP` / `DEPT_PUBLIC_URL`
- → `_callback_meta_for` 返 None
- → 跳过中央台路由,fallback 本地
- → 然后被 v3.1.9 短路删按钮
- → **比 v3.1.9 之前更糟**(之前至少按钮可见,现在彻底纯文本)

### 2. 「看得到点不动」≠「彻底没按钮」

v3.1.9 顾虑:按钮 click 没人接 → 监察员体验差。

但实际:
- **没按钮 → 完全无视监察员**(他在群里看到「请处理」,但没办法处理)
- **有按钮 click 失败 → 至少监察员意识到「这是个动作项」**(去 web 后台或催问技术)

按钮一直在 = 信息密度高,监察员有目标。比纯文本好。

### 3. 大部分客户本来就是独立 bot

独立 bot 客户(每个 dept 有自己 `BOT_TOKEN` 且本地 polling):
- 按钮 click 走 dept 本地 bot.py callback handler → 工作
- 这些 dept 应该是 BOT_POLLING_DISABLED=false

共用 bot 客户(多 dept 用同 `BOT_TOKEN`,如商务/渠道/运营 3 中心共一 bot):
- 必须 dept 端 BOT_POLLING_DISABLED=true(避免 409 Conflict)
- 按钮 click 由中央台 polling 处理(`cb:<>:<>` 格式 OR dept 老格式 — 详见后续)

revert 短路后:
- 独立 bot:本地接 → 完全工作 ✅
- 共用 bot + 路由配齐:中央台接 → 工作 ✅
- 共用 bot + 路由缺失:按钮 dead(跟之前同一难堪),但**至少不比之前更糟**

## 来源

- 2026-05-19 用户群截图 + 「全部都要检查下不是只有恒丰 必须让按钮能正常使用」反馈
- 全网 fleet inspect:44/47 缺 VPS_PUBLIC_IP
- 中央台 DB `pending_callbacks` 表:24h 只有 route_id 1/19 产生条目(34/36 路由 fallback 失败静默)

## 后果

### 正面

1. **44 dept 按钮立刻恢复**(stage2 / 删除 / 关键词预警)
2. 没有 schema 变 / 没有 .env 变 / 0 重登
3. 老用户升级零感知(按钮多出来,旧文案完整保留)

### 负面 / 风险

1. **「点了没人接」按钮**(Codex P1):共用 bot 客户 + 中央台路由没配齐的 dept,fallback 路径发出按钮但 click 不会有反应:
   - `send_no_reply_alert`(老单段)走的是 `routed = await _try_central_route(account, "no_reply", msg)` **不带 callback_meta**(line 765),中央台 push_alert 路径不会创建 `pending_callback` → 即使路由配齐 click 也无人接
   - `send_no_reply_alert_stage2` / `send_delete_alert` 在 fallback 本地推时直接发原始 `violation:<id>` / `cancel:<id>` callback_data,中央台 polling 看到不认识的 callback_data 也无法 dispatch 回 dept
   
   修法路径(单独追,不在本 ADR 范围):
   - 配置层:给 44 dept 补 `VPS_PUBLIC_IP` + 中央台白名单加全部 36 bot,让 stage2/delete 走 cb_meta 路径
   - 代码层:`send_no_reply_alert` 也加 `callback_meta` 参数(老单段路径也走中央台 bridge),或者中央台 polling 加 `violation:<id>` / `cancel:<id>` 反查 dispatch 回原 dept
   - 本版**只保证按钮可见**(比纯文本好,监察员知道是个动作项),click 闭环留给配置 + 后续代码改造

2. 不解决「中央台路由全 fanout 可靠性」根本问题(部分 dept 的 `DEPT_PUBLIC_URL` 可能用 IPv6 内网地址 → 中央台 POST callback 不通)。需要后续巡检 + 标准化。

### 验证

1. fleet fanout 全网升 v3.3.8
2. 等任一 dept 触发 stage2 预警 → 群里看消息**带按钮** ✅
3. 中央台路由配齐的 dept(领航商务 / 悦达商务 + 后续补)→ click 按钮 → 中央台 polling 接 → POST 回 dept `/api/v1/callback` → DB 状态变 → 群里消息 edit 成「✅ 已处理」
4. 路由没配齐的 dept → click 按钮无响应(预期内,等配置补全)
