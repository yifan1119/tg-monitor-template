## 背景

客户/团队 2026-05-08 提出 6 项优化需求(plan: hazy-floating-tiger.md),v3.0.15 实现「外事号信息全自动管理」第 1+2 项:

> **登录即录入 + 改一处全同步**:监控登录时自动写入外事号信息和所属 VPS 位置,无需人工填表。外事号信息变动?只改 web 后台,每日数据表格自动同步最新信息。

### 现状痛点

外事号登录时只录入 phone/name/username/tg_id,**业务字段(operator / company)必须客户去 Sheet B2/B3 手填**等 60s `sync_headers` 反向同步到 DB。改字段两条路:
- Sheet 改 → DB 等 60s
- DB 改 → 没 web UI 暴露,只能 SSH 改

监察员调岗 / 外事号转给别人 / 改部门归属,**这事儿没法 web 一键搞定**,客户痛。

### 用户拍板的 5 个 v3.0.15 相关 clarify(plan 阶段问完)

1. 「内事号」= **监察员/商务自己的某个标识**(新加 `inspector_tg_id` 字段)
2. DB↔Sheet 冲突 = **Web 改了立刻覆盖 Sheet**(单向 web 优先)
3. 监察员 TG 字段 = **新增 inspector_tg_id**(独立,不复用 owner_tg_id)
4. 内事号填 Sheet 哪 = **加到 row 2/3 后面 = row 4**
5. 5 个字段:operator / company / inspector_tg_id / business_tg_id (已有) / owner_tg_id (已有)

## 决策

### 1. accounts 表加 `inspector_tg_id` 字段(migration V7)

```sql
ALTER TABLE accounts ADD COLUMN inspector_tg_id TEXT DEFAULT '';
```

跟 `business_tg_id` (v3.0.5) / `owner_tg_id` (v3.0.5) 同维度独立字段。理由:监察员 ≠ 业务负责人,角色独立(监察员对接「会话失效」运维,owner 对接「业务怠工」预警审核)。

`update_account_business(account_id, ...)` 扩 `inspector_tg_id` 参数,跟 operator/company 同套半结构化 update。

### 2. `/api/accounts/<id>/notify-config` PATCH 扩 3 字段

老接口已有 `business_tg_id` / `owner_tg_id` / `remind_*_text`,扩加:
- `operator` (商务人员名)
- `company` (中心/部门名)
- `inspector_tg_id` (监察员 TG)

`inspector_tg_id` 走同套 `TG_MENTION_RE` 校验(纯数字 / @username 5-32 字符)。

### 3. Web 编辑 modal 扩字段(`templates/index.html`)

现有「配置两段式预警」modal **顶部新加「业务归属」section**(operator / company / inspector_tg_id),老的「两段式 @ 通知」section 不动。

**Codex round1 P1 fix**:`openNotifyConfig` 进入时 `ncAccountId=null` + 保存按钮 `disabled=true`,GET 成功后才启用。防加载未完成时点保存,空字段 PATCH 误清掉客户原配置。

### 4. DB → Sheet 业务字段反向同步 loop(`tasks.py:_sync_account_business_to_sheet`)

跟现有 `_sync_account_names`(改 name 同步分页标题)同模式,放 `_patrol_loop` 60s 一轮。

**单向覆盖语义**(用户拍板 web 优先):
- 拉所有 accounts(从 DB)
- **Codex round1 P0 fix**: 用 `spreadsheet.values_batch_get(ranges=...)` 一次拉所有 ws 的 B2:B4(单引号 escape sheet 名)
  - 老实现 200 账号 = 200 次 batch_get + 200 次 _rate_limit ≈ 阻塞 5 分钟
  - 新实现 200 账号无差异 = 2 次 API call ~3 秒
- **Codex round1 P1 fix**: B2/B3/B4 三字段全用 `db != sheet` 比较(允许空串清空覆盖,跟 web 优先单向覆盖语义一致)
  - 老实现 `if db_operator:` 跳过空串,违反单向覆盖

**feature flag** `BUSINESS_FIELD_SYNC_ENABLED=true` 默认开,客户出问题可在 .env 关回老行为。

### 5. Sheet 模板 row 4 = 「监察员」label + value

**保持 `frozenRowCount=6` 不变**(零数据迁移):
- Row 1:CYAN 横条
- Row 2:`A2=商务人员 B2=<operator>`
- Row 3:`A3=中心/部门 B3=<company>`
- **Row 4:`A4=监察员 B4=<inspector_tg_id>`(新)**(老模板原本是「白底白字 spacer」placeholder)
- Row 5-6:对话槽表头
- Row 7+:消息数据(不动)

`create_account_tab_full(name, operator, company, inspector_tg_id="")` 加参数。
`upgrade_minimal_tab(ws)` 老分页升级:把 row 4 格式从「白底白字 spacer」改成「白底正常字」+ 写 A4 = "监察员"(B4 让 60s patrol 自动从 DB inspector_tg_id 填)。

### 6. Migration 跨进程 race 容错(Codex round1 P0 fix)

`_safe_add_column` ALTER 抛 `duplicate column` 时 catch + recheck schema:
- tg-monitor / tg-web 是独立 docker 容器,启动时各自跑 init_db()
- 两进程同时从 user_version<7 升级 → PRAGMA check 都没看到列 → 都跑 ALTER → 第二个撞 duplicate
- 修法:catch duplicate-column → recheck 列已在 → 视为成功;真错原样 raise

## 原因

### 为何「Web 改立刻覆盖 Sheet」单向?

用户明确拍板。Sheet 上手改的会被 DB 值压回去 — **预期行为**(防止双向同步竞态导致用户困惑)。Release notes 已明确提示。

### 为何 inspector_tg_id 独立字段不复用 owner_tg_id?

3 个理由:
1. **语义不同**:owner = 业务负责人(stage2 预警 @),inspector = 运维监察员(账号被吊销 @ — v3.0.17 用)
2. **未来扩展**:监察员可能多人轮值,后续可能加 `inspector_tg_ids` 字段
3. **零回归**:不动 owner_tg_id 老语义,v3.0.5 老逻辑完全保留

### 为何 row 4 占用 + 不改 frozen=6?

**零数据迁移** = 客户老分页 row 7+ 数据完全不动:
- 模板上 row 4 老 placeholder 是「白底白字 spacer」 — 视觉上空白
- 改成「白底正常字」(B4 显示监察员 ID)— 多了一行可见文字,**老 row 7+ 完全不影响**
- `frozenRowCount=6` 保持 — 客户滚 sheet 顶部固定还是 6 行(包含 row 4)

如果改 frozen=7,row 7 现有数据要往下挪一行,**这是大数据迁移风险**,不值。

### 为何收紧版只做核心字段不含 audit?

按 CLAUDE.md「分批发布」原则,**用户明确要求别一次推全部**。原 v3.0.15 包含 audit + 中央台同步 ~600 行,拆成:
- v3.0.15(本版) = 外事号 5 字段 + Sheet 同步 ~250 行
- v3.0.16 = 中央台同步 ~100 行
- v3.0.18 = 操作审计 ~250 行

每版独立 ADR + Codex + 灰度 + 回滚。

## 后果

### 收益

- ✅ 监察员 web 自助改 5 字段(operator / company / inspector_tg_id / business_tg_id / owner_tg_id),不再依赖 Sheet 手填
- ✅ 60s 内 DB → Sheet 单向同步,客户改完立刻生效
- ✅ inspector_tg_id 字段为 v3.0.17「会话失效 @ 监察员」铺路
- ✅ Sheet row 4 显示监察员,业务一眼看清账号归属

### 已知限制

- ⚠ 业务字段 DB → Sheet **单向覆盖**(用户拍板),客户在 Sheet 上手改的会被压回。Release notes 明确提示
- ⚠ 中央台报表**仍按 VPS 归类**(改 account.company 不影响中央台 dept/center 归属)— v3.0.16 解决
- ⚠ 老分页升级时,row 4 格式从 spacer 改成可见 — **客户视觉上会感知到「多了一行监察员」**(release notes 预告)

### 兼容性

- 老客户不升级:行为完全不变 (`accounts.inspector_tg_id` 字段不存在)
- 老 .env 不动:`BUSINESS_FIELD_SYNC_ENABLED` 默认 true 自动开,客户改 false 关回 v3.0.14 行为
- 老分页:升级时 `upgrade_minimal_tab` 把 row 4 改可见 + 写 A4 label,B4 让 60s patrol 自动填
- 升级失败回滚 v3.0.14:DB 字段 `inspector_tg_id` 保留(SQLite ADD COLUMN 不破坏)

### 性能

- migration V7:< 5ms(添加 1 列)
- `_sync_account_business_to_sheet` 60s 一轮:1 次 worksheets() + 1 次 batch_get + 0-N 次 update_acell(只对差异写)。200 账号无差异 = **2 次 API ~3 秒**(对比老实现 200 次 API ~5 分钟,Codex P0 fix 优化 100 倍)
