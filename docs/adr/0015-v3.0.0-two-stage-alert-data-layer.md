# ADR-0015: v3.0.0 两段式未回复预警 — 数据层 + config flag + demo 错位 DB 兼容修复

**版本**:v3.0.0
**日期**:2026-04-23
**状态**:accepted(集成到 `integration/v3.0.0-on-main`,等测试 + Codex round2 + merge main + tag v3.0.0)

## 背景

v2.x 单段未回复预警(30 分钟触发 @ 商务,有「已处理 / 不必处理」审核按钮)在部分客户场景不够用。
2026-04-21 客户对接提出两段式需求:

- **Stage 1(30 分钟触发)**:@ 商务人员 + 可选自定义文案,无按钮(温和提醒)
- **Stage 2(Stage 1 后默认 10 分钟仍未处理,即累计 40 分钟)**:@ 部门负责人 + 自定义文案 + 按钮「登记违规 / 取消」
- 员工在本账号 outbound 回复了消息,Stage 1 要自动结案,不能再升 Stage 2

这是 v3.0.0 的核心功能,本 ADR 处理**数据层 + feature flag + migration**,不碰推送 / callback(见 ADR-0016)。

### 关键约束

1. **回滚兼容**:v3.0.0 推出后,DB 写入的新结构必须让 v2.x 老代码读得懂(rollback.sh 后老代码不报错)
2. **向后兼容**:老 `.env` 不改,升级后行为 = v2.10.25 一字不差(flag 默认关)
3. **防 demo DB 错位**:之前 v3.0.0 开发期间,demo 部门 DB 被改坏过 — 某些 migration 半途失败导致 `PRAGMA user_version` 跟实际 schema 不一致(例如 `user_version=5` 但 `alerts.stage` 列缺失 / `accounts.business_tg_id` 列缺失)。升级后必须能自愈

## 决策

### 1. Schema 变动(migration V5)

在 main 的 V4(`media_seq` / `archive_msg_id` / `account_seq`)之后,V5 做两件事:

```sql
-- accounts 表加 4 字段(两段式预警配置)
ALTER TABLE accounts ADD COLUMN business_tg_id        TEXT DEFAULT '';
ALTER TABLE accounts ADD COLUMN owner_tg_id           TEXT DEFAULT '';
ALTER TABLE accounts ADD COLUMN remind_30min_text     TEXT DEFAULT '';
ALTER TABLE accounts ADD COLUMN remind_40min_text     TEXT DEFAULT '';

-- alerts 表加 stage 列
ALTER TABLE alerts ADD COLUMN stage INTEGER DEFAULT 0;
```

字段语义:

| 字段 | 用途 |
|---|---|
| `accounts.business_tg_id` | 30 分钟 stage1 推送时 @ 谁(商务人员;可填 numeric ID / username / 空) |
| `accounts.owner_tg_id` | 40 分钟 stage2 推送时 @ 谁(部门负责人;同上) |
| `accounts.remind_30min_text` | stage1 自定义文案(空 → 用全域 `REMIND_30MIN_TEXT`) |
| `accounts.remind_40min_text` | stage2 自定义文案(空 → 用全域 `REMIND_40MIN_TEXT`) |
| `alerts.stage` | 0 = v2.x 老路径 / flag 关的新记录;1 = 已推 stage1 等升级;2 = 已升 stage2 |

所有 migration 都走 `database.py:_safe_add_column`(检测存在性幂等 `ALTER TABLE ... ADD COLUMN`),
跑多少次都安全。

### 2. 用 `stage` 列而不是改 `alerts.type`(沿用 ADR-0005 决策)

保持 `alerts.type='no_reply'` 不变,**不新增** `type='no_reply_stage2'`。原因见 ADR-0005,这里只复述要点:

- **回滚安全**:v2.x 老代码只认 `('no_reply', 'keyword', 'deleted')` 三个 type,看到 `stage` 多余列会直接忽略
  (`dict.get()` 拿不到不报错)。若改 type,回滚后统计 / 审核按钮 callback 全乱
- **去重不炸**:`has_alert_today('no_reply', peer_id)` 逻辑不变。同一对话同一天只有一条 `type='no_reply'` 记录,
  不管 stage 升到几
- **扫描不误触**:stage2 扫描显式 `WHERE stage=1 status='pending'`,v2.x 遗留 `stage=0` 老按钮预警不会被误扫

### 3. demo 错位 DB 兼容修复(Codex C 方案)

开发期 demo 部门遇到过:`PRAGMA user_version=5` 但实际 `alerts.stage` 列缺失(migration 半途崩 / 手工改 user_version)。
若 `_run_migrations` 只看 version 就跳过,会让后续 INSERT / SELECT `stage` 列直接 SQL 错。

**C 方案**:`_run_migrations` 每次启动都对关键新列做存在性检查,若 `user_version` 声称已到但列不存在,
强制补跑 `_safe_add_column`。具体做法:

```python
def _run_migrations(conn):
    cur = conn.execute("PRAGMA user_version")
    v = cur.fetchone()[0]

    if v < 5:
        _migrate_to_5(conn)

    # 兼容补丁:即便 user_version >= 5,也检查 V5 关键列是否真的存在
    # (防 demo 开发期 user_version 错位导致的 migration 半崩)
    _safe_add_column(conn, "accounts", "business_tg_id",    "TEXT DEFAULT ''")
    _safe_add_column(conn, "accounts", "owner_tg_id",       "TEXT DEFAULT ''")
    _safe_add_column(conn, "accounts", "remind_30min_text", "TEXT DEFAULT ''")
    _safe_add_column(conn, "accounts", "remind_40min_text", "TEXT DEFAULT ''")
    _safe_add_column(conn, "alerts",   "stage",             "INTEGER DEFAULT 0")
```

`_safe_add_column` 内部判重(`PRAGMA table_info` 检查),已存在就 no-op,幂等安全。
效果:无论 user_version 是对是错,升级后列一定在。

### 4. Config feature flags

`config.py` 新增:

```python
TWO_STAGE_NO_REPLY_ENABLED = env_bool("TWO_STAGE_NO_REPLY_ENABLED", False)
NO_REPLY_STAGE2_AFTER_MIN  = env_int ("NO_REPLY_STAGE2_AFTER_MIN",  10)
UNREPLIED_ALERT_GROUP_ID   = env_int ("UNREPLIED_ALERT_GROUP_ID",   0)
REMIND_30MIN_TEXT          = env_str ("REMIND_30MIN_TEXT",          "请尽快回复")
REMIND_40MIN_TEXT          = env_str ("REMIND_40MIN_TEXT",          "已超过 40 分钟未回复,请处理")

# TG 装置伪装(外事号 session 防风控,见 ADR-0016 附带决策)
TG_DEVICE_MODEL = env_str("TG_DEVICE_MODEL", "shencha")
TG_SYSTEM_VERSION = env_str("TG_SYSTEM_VERSION", "1.0")
TG_APP_VERSION = env_str("TG_APP_VERSION", "tglistener 1.0")
```

全部默认值 = **老行为**:
- `TWO_STAGE_NO_REPLY_ENABLED=False` → 推送走 v2.x 单段路径(stage=0)
- `UNREPLIED_ALERT_GROUP_ID=0` → fallback 到 `ALERT_GROUP_ID`(老预警群)
- `NO_REPLY_STAGE2_AFTER_MIN=10` → stage1 后 10 分钟升级(累计 40 分钟)

所有 flag 加入 `config.reload_if_env_changed()` 热 reload 列表,Web 后台改 `.env` 保存后下一秒生效,
不必重启容器。

### 5. 全域文案 vs 账号级文案(客户反馈调整)

v2.10.26 测试期客户反馈:**希望文案全域统一改**,而不是每个外事号都单独填一次。
最终保留两种填法:

- `accounts.remind_30min_text` / `remind_40min_text` — 账号级,空值 fallback 到全域
- `REMIND_30MIN_TEXT` / `REMIND_40MIN_TEXT` — 全域 .env 默认值

**读取优先级**:账号级非空 → 账号级;否则 → 全域。
客户 200+ 账号一次性用全域即可,特殊账号(例如 VIP 组)单独填账号级覆盖。

## 原因

### 为什么不新开 `two_stage_alerts` 表?

见 ADR-0005 的详细权衡。简述:新表让 callback 路由要判断「这个 alert_id 是哪张表」,15+ 处 SELECT/UPDATE 要改。
扩 `alerts` 只加一列,改动最小、回滚最安全。

### 为什么 `stage` 默认 0 不是 NULL?

- 老数据(v2.x 遗留的 `type='no_reply'` pending 按钮预警)默认 0,明确语义「跟两段式无关」
- NULL 会带入 SQL 三值逻辑(`WHERE stage=0` 不匹配 NULL),增加心智负担
- 0 就是「老路径 / 非两段式」,跟 1 / 2 形成清晰梯度

### 为什么 config flag 加 TG 装置伪装 3 字段?

外事号被 TG 官方风控冻结是客户长期痛点(v2.10.23 加 `get_me()` 真探测也是这背景)。
Telethon `TelegramClient(..., device_model=..., system_version=..., app_version=...)` 三字段让 session
看起来像 TG Desktop 而不是 Telethon 默认签名(默认带 `Telethon/...` 字样,容易被识别)。

默认值 `shencha` / `1.0` / `tglistener 1.0` 够中性,不会撞真实产品名。客户觉得风险高可自定义成 `TG Desktop` / `Windows 10` / `5.1.1`。

这 3 字段放 ADR-0015 是因为属于 config.py 的 .env 新字段,跟两段式 flag 一起合并到 V5 发布。
实际装置伪装在 session 层生效在 listener.py(见 ADR-0016)。

### 为什么 demo DB 兼容修复用 C 方案不是「强制 rebuild」?

- 强制 rebuild(drop + recreate)会丢客户历史数据(messages / alerts / account_seq),不可接受
- C 方案是幂等补列,对正常 DB(列已存在)no-op 零副作用
- 客户侧实际遇到错位 DB 的概率极低(只有 demo 开发期手改过 user_version),但防御成本很低 → 加上

## 来源

- **客户对接需求**(2026-04-21):两段式预警 + @ 商务 / @ 负责人 + 独立预警群
- **Codex 审计 v3.0.0 方案评审**:保留 `type='no_reply'` 新增 `stage` 列(见 ADR-0005 详细对话)
- **v2.10.26 测试期客户反馈**:文案全域统一,不要每个号单独填
- **demo 错位 DB 事故**:开发期手改 `user_version=5` 但某次 migration 缺一列,启动崩 → 加 C 方案防御

## 后果

### 正面

- **回滚绝对安全**:rollback.sh 到 v2.10.25 后老代码照常跑,V5 新列保留无害
- **升级零感知**:所有 flag 默认关 → 老客户 `./update.sh` 后行为 = v2.10.25
- **错位 DB 自愈**:即便客户手改过 user_version 或历史 migration 半崩,启动时自动补列
- **全域文案一次改**:200+ 账号客户不必逐个填,改 `.env` 的 `REMIND_30MIN_TEXT` 即可

### 副作用 / 需要后续跟进

- **stage 列回滚后浪费 1 个 INTEGER 列**:可接受,不追求严格 clean schema
- **device_model 装置伪装**:若 TG 未来更严风控,可能仍被识别为 Telethon。保持观察
- **全域文案覆盖账号级**的优先级:代码必须显式 if account.text else global_text,不能反过来
  (反了客户明明填了账号级却被全域覆盖,体验差)

### 代码改动点

- `database.py:_migrate_to_5` — V5 migration
- `database.py:_run_migrations` — 加 C 方案兼容补丁
- `database.py:update_account_two_stage` / `get_account_by_id` — 读写 4 字段
- `database.py:insert_stage1_alert` / `get_pending_stage1_alerts` / `upgrade_to_stage2` / `mark_stage1_handled_by_reply` — stage 列查询 helpers
- `config.py` — 5 个两段式 flag + 3 个装置伪装字段 + `reload_if_env_changed` 加入

### 相关 ADR

- 前置:**ADR-0005**(为什么用 stage 列而不改 type)
- 后续:**ADR-0016**(数据层基础上的 push / callback / Telethon 真名解析 / 自动升级 loop)
