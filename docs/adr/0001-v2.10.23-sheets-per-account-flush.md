# ADR-0001: Sheets flush 改成按账号分桶 + 429 per-account 退避

**版本**:v2.10.23
**日期**:2026-04-21

## 背景

客户苏总反馈「Google Sheets 表格有消息空白没写进去」,但 SQLite DB 里查 `messages` 表,这些消息**是存在的**,只是 `sheet_written=0`(未写入标记)。

Codex(GPT-5.4 高推理)外部审计指出 `sheets.py:759-815` + `database.py:183-189` 的根因:

```python
# v2.10.22 的 flush_pending
def flush_pending(self):
    unwritten = db.get_unwritten_messages()   # 全局 LIMIT 500
    if not unwritten: return 0
    with self._write_lock:
        return self._do_flush(unwritten)

def _do_flush(self, unwritten):
    by_account = {...}
    for account_id, msgs in by_account.items():
        # 某个账号 write_messages raise → 整个 for 循环中断
        # 下一轮 flush 又从同一批老消息开始,永远过不去
```

**具体场景**:
- 200+ 账号,每个账号 N 条 pending
- 其中一个账号触发 Google Sheets 429(quota exceeded)
- `gspread.exceptions.APIError` 抛出 → `_do_flush` 整个 for loop 中断
- 下次 flush,又取同一批老消息(LIMIT 500 按 timestamp),继续被同一个账号卡住
- 结果:**所有其他账号的新消息也写不进去**(表格空白),DB 持续积压

## 决策

改造 `flush_pending` / `_do_flush` 为**按账号分桶**模式:

1. **新增 DB helper**(`database.py`):
   - `get_accounts_with_unwritten()` — 返回有未写消息的所有 account_id
   - `get_unwritten_messages_by_account(account_id, limit=100)` — 按账号取
   - `count_unwritten_older_than(minutes=10)` — 积压告警用

2. **`flush_pending` 改造**(`sheets.py`):
   - 外层 loop 遍历每个有积压的账号
   - 单账号 try/except 隔离:`gspread.exceptions.APIError` / `Exception` 都捕获,不中断 for 循环
   - 每账号每轮最多 100 条(分桶限额)

3. **429 per-account 指数退避**:
   - `_flush_backoff = {account_id: next_retry_ts}`
   - `_flush_backoff_level = {account_id: 退避级别}`
   - 触 429 → `wait = min(5 * (2 ** level), 600)` — 5s → 10s → 20s → ... → 600s
   - 退避期内该账号跳过,**其他账号继续写**
   - 成功一次 → 重置退避

4. **积压告警**(`tasks.py` 新增 `_sheets_backlog_loop`):
   - 每 5 分钟扫 `count_unwritten_older_than(10)`
   - 超 `SHEETS_BACKLOG_ALERT_THRESHOLD`(默认 500)→ 推预警群
   - 1 小时冷却不刷屏

## 原因

### 为什么不直接 catch 异常让 for 继续?
可以,但不解决根本问题 — 老消息会一直在队列前面,下次取还是它们。必须**按账号隔离**才能让不受影响的账号正常写。

### 为什么不用真正的队列(Celery / Redis)?
- 引入新依赖(Redis),Docker 镜像变大,运维复杂
- 当前 SQLite `messages.sheet_written=0` 已经是天然的持久化队列
- 改动集中在 `sheets.py`,其他模块零感知

### 为什么不改成事件驱动(listener 直接推送写)?
- 现有架构 listener → DB → 定时 flush 的解耦很稳
- 事件驱动会让 listener 协程被 Sheets API 慢调用阻塞
- 定时 flush 天然解耦消息接收和 Sheet 写入

## 来源

1. 客户苏总 2026-04-20 反馈「表格有空白」
2. Codex (GPT-5.4 高推理) 审计 Critical A1 — 明确指出全局 LIMIT 500 + 单账号失败中断整批 是根因
3. 本地 Bash 调试确认 SQLite `SELECT COUNT(*) FROM messages WHERE sheet_written=0` 数字持续增长

## 后果

### 立即效果
- 升级 v2.10.23 后,积压的 `sheet_written=0` 消息会被**按账号并行**补写进 Sheet
- 预期 30 分钟 - 2 小时内追平(取决于积压数量)
- 客户无感知,不用操作

### 长期影响
- 单账号撞限额不再影响全局,系统整体吞吐更稳
- 积压告警提供可观测性,配额爆了第一时间知道
- 退避级别保留在内存(容器重启丢),重启后可能再撞一次 429 — 可接受

### 待关注
- `_flush_backoff_level` 重启丢 → 如果成为痛点,后续可持久化到 DB(P2)
- 积压告警阈值 500 是默认,某些超大客户可能要调整(通过 `SHEETS_BACKLOG_ALERT_THRESHOLD` env)
