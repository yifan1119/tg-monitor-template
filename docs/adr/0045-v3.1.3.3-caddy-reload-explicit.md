# ADR-0045 · v3.1.3.3 · 升级时显式 reload Caddy + 中央台 fanout 按 IP 串行

## 背景

v3.1.3.2(ADR-0044)修客户 DNS 撞车 bug,把 Caddyfile 模板 `web:5001` 模糊别名改为 `tg-web-__COMPANY_NAME__:5001` 显式占位符,`update.sh` 加 section 5.5b 自愈替换占位符 / 老模糊别名。

发版后 Plan agent 复审 + Codex 独立 review 抓出 2 个 P0,实质让 v3.1.3.2 的修复**不能完全生效**:

### P0-1:5.5b 改完 Caddyfile 没显式 reload Caddy

`update.sh:5.5b` 用 `cat /tmp/Caddyfile.upstream > Caddyfile` 保 bind-mount inode 写入。我之前的假设是「靠后面 section 5.6 inode 自愈 restart Caddy 兜底」。但实际:

- 5.6 比较的是 **host 文件 size vs 容器内 size**(line 581)
- 保 inode 写入后,host 和容器内 size 始终一致(读的是同一个 inode 的数据)
- **5.6 永不触发 restart**
- Caddy 进程内存里的 routes **仍是老的 `web:5001`**(Caddy 不自动 watch 文件)
- → DNS 撞车 bug 没修

dev VPS 端到端验证 v3.1.3.2 时,我**手动**跑了 `docker exec caddy reload`,所以看着对;真实客户升级流程**没有这一步**。

### P0-2:同 IP 多部门 VPS 并发 fanout → 共用 Caddy 反复 restart

中央台 `fleet.py:fanout` 用 `ThreadPoolExecutor(max_workers=10)` 并发调所有 dept。全网 9 台 VPS 是「同 IP 多部门」(部分共用 Caddy):

- fanout 同时让该 VPS 上 2-3 个部门跑 update.sh
- 每个 update.sh 跑到 section 5.5(异地 IP 清理)/ 5.5b(upstream)/ 5.6(inode 自愈)都可能 restart 或 reload 共用 Caddy
- → 30-60 秒内 Caddy 被反复 restart/reload → HTTPS 全停
- 同时 2-3 个 `docker compose up -d --build --force-recreate` 撞 docker daemon 锁

### Codex 复审我的修法本身又抓出 2 个细节 P0

最初我打算这样改 reload 段:

```bash
docker exec MY_CADDY caddy reload 2>&1 | tail -3 | sed 's/^/    /' || true
echo "  ✅ Caddy 已 reload"
```

Codex 抓出:`|| true` 在管道末尾吞掉 reload 失败,后面无条件 echo 撒谎日志。如果 Caddy reload 真失败(配置语法错 / 内部异常),客户日志看着"修好了",实际 routes 还是老的。

IP 分组用 `re.search(r"(\d+\.\d+\.\d+\.\d+)", base_url)`:fleet.py 设计上**允许** dept base_url 是真实域名(`https://monitor.example.com`),IP regex 漏覆盖 → 真实域名 dept 仍并发,P0-2 没修干净。

## 决策

### 改动 1:`update.sh` 5.5b 末尾加显式 reload(if/else 判定不吞错)

sed 替换成功后,inline 加 MY_CADDY 检测(沿用 5.6 的检测模式,但提前到 5.5b 段内) + caddy reload。

reload 命令的 stdout/stderr 跟 exit code 分开捕获:

```bash
set +e
reload_out=$(docker exec MY_CADDY caddy reload --config /etc/caddy/Caddyfile 2>&1)
reload_rc=$?
set -e
if [ "$reload_rc" = "0" ]; then
    echo "  ✅ Caddy 已 reload"
else
    echo "  ⚠ Caddy reload 失败 exit=${reload_rc}"
    echo "$reload_out" | tail -5
    echo "  ⚠ Caddyfile host 端已更新但 Caddy 进程仍跑老配置,请人工排查"
fi
```

**关键不吞错原则**(Codex P0 fix):
- 不用 `|| true` 包裹 reload 命令
- exit code 显式存到变量再判定
- 失败时 echo 实情 + tail 5 行错误输出
- 成功时才打印 "已 reload"

**MY_CADDY 找不到时不阻塞升级**:罕见场景(部门没用 HTTPS / 没 Caddy),5.5b sed 已经改 Caddyfile host 文件,只是不 reload。打 warning 让客户人工补 reload,继续走 5.6 / 5b。

### 改动 2:`update.sh:524` cp 加 `|| true`

`cp Caddyfile Caddyfile.bak.upstream.* 2>/dev/null` → 加 `|| true` 防 set -e 在罕见盘满场景挂掉整段升级。

### 改动 3:中央台 `fleet.py:fanout` 加 `serialize_by_ip` 参数

新增 `serialize_by_ip=False` 可选参数:
- 默认 False → 老路径,所有 caller 行为不变
- True → 按 dept base_url 的 IPv4 分组,同 IP 部门**串行**,跨 IP 并发

IP 抓取用 `urllib.parse.urlparse().hostname` 拿主机名 + regex 抽 IPv4(比直接 regex base_url 严谨,避开 query string 假阳性)。

**真实域名 fallback**(Codex P0 fix):base_url 不含 IPv4 的 dept(自定义域名场景)无法 IP 去重 → 退回并发 + log warning。理由:
- 目前 sheet 上 100% 是 nip.io 含 IP,无影响
- 未来扩自定义域名时 ops 看 log 知道这些不享受串行保护,手动错峰
- 加 reverse DNS 解析复杂度大,违反 rule 13 Simplicity First,留 v0.23+

### 改动 4:`app.py:/api/admin/fleet/cmd` upgrade/restart_svc 启用串行

只在动 Caddyfile / 重启容器的 write action 启用 serialize_by_ip。只读 action(inspect / tail_logs / reload_oauth / fix_sheets)仍走老并发路径,性能不变。

### 改动 5:`templates/fleet.html` 「全网升级」confirm 文案

加「同 IP 多部门 VPS 自动串行(防共用 Caddy 抖动),整体约 1-2 小时,慢时最久 ~72 分钟」让 ops 心里有数。

## 原因

**为什么不抽 MY_CADDY 检测成函数共享给 5.5b / 5.6**:rule 14 surgical changes,5.5b / 5.6 / 5b 三段历史上各自有自己的 MY_CADDY 风格(略有差别),共享函数要 refactor 三处,scope drift。inline 加在 5.5b 内是最小动。

**为什么 IP 分组不 reverse DNS 解析**:rule 13 Simplicity First。目前真实 sheet 上不存在自定义域名 dept,加复杂逻辑是投机性设计。退回并发 + warning 已经足够 safe-by-default。

**为什么 fanout 默认 serialize_by_ip=False**:rule 11 Feature flag 默认安全。所有现有 caller(`fleet_health.py:202` inspect daemon、app.py 其他 endpoint)行为不变。

## 后果

### 正面

- v3.1.3.2 的修复**真正生效**:升级后 Caddy 进程内存的 routes 立刻切到显式 `tg-web-<dept>:5001`,DNS 撞车 bug 根治。
- 同 IP 多部门 VPS 升级不抖:同 IP 串行后 Caddy 一次只被 reload 一次,HTTPS 不停。
- reload 失败可见:日志不撒谎,客户能在 fleet UI 看到具体错误。
- 真实域名 dept(目前 0 台)有 warning 提示,运维有迹可循。

### 中性 / 负面

- 同 IP 9 台部门最坏 72 分钟(超时 8 分钟 × 9 串行);现实中 update.sh 平均 3-5 分钟,真实约 30-45 分钟。
- MY_CADDY 检测在 5.5b / 5.6 各一次,代码重复 15 行(rule 14 守关,不抽函数)。
- fanout 在 serialize_by_ip 模式下,组内某 dept 卡超时会阻塞整组后续部门(P1,**预期行为**,记入 release notes 让 ops 知道)。

### 升级路径

| 状态 | v3.1.3.2 跑 update.sh | v3.1.3.3 跑 update.sh |
|---|---|---|
| 单 dept 部署 | Caddyfile 改了但 Caddy 没 reload(P0-1)| 改 + reload 立即生效 ✅ |
| 同 IP 多部门并发 fanout | Caddy 反复 restart 30-60s 抖动(P0-2) | 同 IP 串行,Caddy 一次 reload ✅ |
| 真实域名 dept(目前 0 台)| 无 | 退回并发 + log warning |
| reload 失败 | 静默撒谎"已 reload" | 显式打 warning + tail 错误 |
| Caddyfile.bak.upstream.* 备份失败 | set -e 整段挂 | `|| true` 继续升级 |
