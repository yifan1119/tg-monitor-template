# ADR-0046 · v3.1.3.4 · update.sh self-reload bootstrapper + 升级通知默认关

## 背景

### Bug 1:bash-cache 让升级新增段静默不执行(P0)

2026-05-14 dev VPS 完整跑 v3.1.3.3 update.sh 实测,发现:

- bash 启动 update.sh 时把整个文件读进内存(cache)
- update.sh 跑 `git fetch + reset --hard origin/main` 把文件更新成 v3.1.3.3 新版
- **但 bash 仍按内存里的老 v3.1.3.x 版本继续跑**
- 老版没有 v3.1.3.3 的 5.5b reload 段
- → 跳过 caddy reload → Caddyfile 占位符 `__COMPANY_NAME__` 不替换 / 替换了但 Caddy 不 reload → web /login 502

**影响**:43/43 全部 agent 可达客户走 SSH 手动跑 `update.sh` 升级路径都会撞(全网没人是 v3.1.3.2 起跳)。

**不影响**:中央台 fanout(走 `agent.upgrade` container_git 路径)新启 bash + git 已 reset → bash cache 是新版 update.sh,不撞 bug。

**Plan agent + Codex review 都没抓出来**,靠 dev VPS 端到端实测捞到。又一次 CLAUDE.md rule 9 验证铁律救命。

### 问题 2:升级通知打扰客户群

`update_checker` 每 6 小时查 GitHub 新版,有就推 ALERT_GROUP_ID(客户预警群)。但实际升级 100% 我们运维触发(中央台 fleet fanout 或 SSH),客户从来不主动跑 update.sh。客户群被「有新版」打扰 → 客户问「为啥不升」造成困扰。

## 决策

### 改动 1:`update.sh` 顶部加 self-reload bootstrapper

```bash
if [ "${_UPDATE_SH_REENTRY:-0}" != "1" ]; then
    set +e
    git fetch origin --tags --force --quiet 2>/dev/null
    REMOTE_HASH=$(git show "origin/main:update.sh" 2>/dev/null | sha256sum | awk '{print $1}')
    LOCAL_HASH=$(sha256sum "$0" 2>/dev/null | awk '{print $1}')
    set -e
    if [ -n "$REMOTE_HASH" ] && [ -n "$LOCAL_HASH" ] && [ "$REMOTE_HASH" != "$LOCAL_HASH" ]; then
        echo "🔄 检测到 update.sh 自身有新版,先拉 + exec 重启避免 bash-cache bug..."
        if git checkout origin/main -- update.sh 2>/dev/null; then
            export _UPDATE_SH_REENTRY=1
            exec bash "$0" "$@"
        fi
    fi
fi
```

**关键设计**:
- `_UPDATE_SH_REENTRY` 标志位防 exec 无限循环 — 第二轮跳过 bootstrapper 直接进流程
- 只 `git checkout` update.sh 一个文件(不动整个 repo,主流程后面会做 git reset)
- `set +e/-e` 包住 git 命令,任何失败不阻塞主流程(老版本兜底跟 v3.1.3.3 之前行为一致)
- 整段失败不阻塞升级(走老版兜底)

### 改动 2:`update.sh` 5.5b 段加显式 caddy reload(从 v3.1.3.3 backport)

修 v3.1.3.3 同款 P0:5.5b sed 替换 Caddyfile 后必须显式 `docker exec caddy reload`,5.6 inode 自愈不会触发(`cat > redirect` 保 inode 让 size 始终一致)。

`docker exec ... caddy reload` 的 exit code 显式存变量再 if/else 判定,**不用 `|| true` 吞错** — reload 失败打 warning + tail 错误细节,不撒谎日志。

`cp Caddyfile.bak.upstream.*` 加 `|| true` 防罕见盘满 set -e 整段挂(P1 fix)。

### 改动 3:`update_checker.py` 加 feature flag,默认 OFF

```python
notify_enabled = os.environ.get("UPDATE_CHECK_NOTIFY_ENABLED", "false").lower() == "true"
if not notify_enabled:
    logger.info(f"new version {latest_sha[:7]} detected but UPDATE_CHECK_NOTIFY_ENABLED=false, skip TG notice")
    return
```

`check_once` 仍然跑(状态写到 state 文件给 web 后台显示「有新版」UI 用),只是不主动推 TG 客户群。运维想恢复 → `.env UPDATE_CHECK_NOTIFY_ENABLED=true`。

## 原因

**为什么不把 5.5b 段移到 git fetch 之前**(rule 14 surgical 候选最简方案):

理论上:5.5b 提到 section 0,所有路径(SHA 相等 short-circuit / 升级路径)都跑。但实际 bash cache 老版 update.sh 在 section 0 位置**根本没 5.5b 段**,提不提到前面无意义。

**根本问题不是位置**,是 bash cache 的 update.sh 是老版。唯一彻底解的办法 = self-reload exec 让 bash 重新载入新文件。

**为什么 self-reload 在顶部不在末尾**:

末尾 self-reload(升级成功后再 exec)走完全流程,但 5.5b / 5.6 / 5b 段已经按老版跑过,如果新版有改动这些段不会按新版跑。

顶部 bootstrapper exec 让新 bash 从头跑,所有段都按新版执行。

**为什么 update_checker 默认 OFF 而不是删整段代码**:

rule 11 Feature flag 默认安全。`check_once` 仍跑用于 web 后台 UI 显示版本,只关 TG 推送。可逆,如果未来要恢复客户群通知 → `.env` 一行就行。

**为什么 v3.1.3.3 直接 revert 不是叠加修**:

v3.1.3.3 已经 push main,客户/运维 SSH 跑 update.sh 会撞 bash cache bug 立即 502。立刻 revert 防止任何客户中招(see PR #37)。然后用 v3.1.3.4 一次性把 5.5b reload + bash cache 修 + 通知关全部装齐。

## 后果

### 正面

- **bash-cache bug 根治**:任何后续版本升级,bash cache 老 update.sh 时,bootstrapper 第一时间检测 hash + exec 新版本,新 bash cache 新文件,所有段按新版跑。
- **升级通知不再打扰客户群**:客户安静,不再问「为啥不升」。
- **5.5b reload 真生效**:Caddyfile 改完 Caddy 立刻切到新 routes,DNS 撞车修复完整生效。
- **向后兼容**:老 .env 不改 → `UPDATE_CHECK_NOTIFY_ENABLED` 默认 false → 不推 TG。运维想恢复一行就行。

### 中性 / 负面

- v3.1.3.4 **第一次升级仍受 bash cache 影响**(老 update.sh 没 bootstrapper),所以**必须走 fanout 升级**(中央台 fleet 按按钮),不要让运维 SSH 手动跑。
- 升完后所有客户在 v3.1.3.4 含 bootstrapper,**未来 SSH 升级 v3.1.3.5+ 时安全**。
- bootstrapper 每次跑都做一次 git fetch(几百毫秒),不影响实际功能。
- 老版本 update_checker 仍然推(还没升上 v3.1.3.4 的客户),升完后默默拔掉。

## Codex review 抓出的 P1(本版处理 1 个,2 个留 v3.1.3.5)

| P1 | 状态 |
|---|---|
| **P1-1** `curl ... \| bash` 入口 `$0=bash` 命令名,`sha256sum "$0"` 失败 → bootstrapper 失效 | **本版修**:加 `[ -f "$0" ]` 检测,非文件场景跳过比对(失败兜底 = 老版本行为,跟 v3.1.3.3 之前一致,无新风险) |
| **P1-2** bootstrapper 同步 `git fetch` 无 timeout,慢网卡升级 | **本版修**:加 `command -v timeout` 检测 + `timeout 15 git fetch ...`,timeout 不存在裸跑(行为退回老版可接受) |
| **P1-3** `UPDATE_CHECK_NOTIFY_ENABLED` 不是热 reload(改 `.env` 后要重启进程) | **不修**:升级本来就重启容器,默认 false 升完自动生效;运维想恢复推送时改 `.env` + `docker compose restart` 一行即可。留 v3.1.3.5+ 加 `config.reload_if_env_changed` 兼容 |
| **P1-1 兜底** git checkout 成功但 exec 失败的罕见路径 | **不修**:bash exec 失败属操作系统级异常(资源耗尽/权限改变),罕见且无优雅恢复路径,fail loud 即可 |
