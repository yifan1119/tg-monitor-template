# ADR-0042: v3.0.26 — Remote Agent(中央台远程运维 dept VPS)

## 背景

30+ 客户 VPS 分散运维,每次出现问题:
- 客户问:中央台没接通?Sheets 没更新?预警没推?
- 我们答:你 SSH 上去跑 `cat .env`、`docker logs`、`./update.sh`...
- 一台一台问 + 教,小问题 5 分钟,大问题 1 小时

最近 v3.0.21-v3.0.24 上线后,扫描 sheet 上 44 个 VPS 部署清单,发现 18 台还在 v2.10.22(28 个版本之前),14 台在 v3.0.13,5 台在 v3.0.14,只有 2 台跑到 v3.0.15。**「客户跑过 update.sh 但没生效」是高频痛**,根因往往是 git 卡在 feature 分支 / Docker compose up 没真 rebuild / update.sh 自身老。

需要一个机制:**让我们能从中央台一键看 / 改 / 升级所有 dept VPS,无须客户配合**。

## 决策

dept 端加 `agent.py` 模块 + Flask `/api/v1/admin/cmd` endpoint。

### 鉴权(4 层)

1. `Authorization: Bearer <METRICS_TOKEN>`(已存在,跟 `/api/v1/metrics` 同 token)
2. `X-Agent-Ts: <unix_ts>` ±5 分钟时钟差
3. `X-Agent-Nonce: <16+ hex>` 5 分钟内去重(防重放)
4. `X-Agent-Sig: <hex>` = `HMAC_SHA256(token, f"{ts}.{nonce}.{body_hex}")`

加 rate limit(同 token 5 次/分钟)+ audit_log(每次调用写表,actor=fleet_admin / IP=中央台 IP)。

### 白名单 3 action(MVP)

| action | 干啥 | 实现 |
|---|---|---|
| `inspect` | 健康检查(只读)— 版本 / 容器 / sessions / disk / .env 摘要 / accounts count / alerts 24h | `_read_*` helpers + docker SDK |
| `upgrade` | `git fetch + checkout <target> + pull + ./update.sh` 同步等完成 | subprocess.run(timeout 8min)|
| `restart_svc` | `docker restart tg-monitor-<dept> / tg-web-<dept>` | docker SDK `.restart()` |

**永禁**:任意 shell exec / 任意 SQL / 删 sessions / 删 DB / 改 BOT_TOKEN / 改 OAuth secret。任何「加紧急 patch 想塞 shell」的诱惑必须忍住 → 走「改 endpoint 代码 → 升级 → 触发」正路。

### 配套:中央台 v0.19 fleet UI

`/admin/fleet` 页面从 Google Sheet `1W_L8DY...` 拉 dept 清单,fanout 并发调每台 agent,渲染表格 + 单台/批量按钮(inspect / upgrade / restart)。

## 原因

为啥不直接搞 SSH key + Ansible 这种业界做法:
- **客户信任**:服务方在客户 VPS 留 root SSH key = 未告知 backdoor,客户 IT 发现就解约
- **私钥泄露半径**:私钥丢 = 30+ VPS 任意 RCE,比 token 严重一个数量级
- **白名单 cmd 模式安全可控**:任何 action 都是我们写好的代码,不可能写出「删客户数据」

为啥 HMAC 多此一举,token 不够吗:
- token 是长寿命密钥,可能在 access log / nginx log / 客户 IT 工单 / 截图泄露
- HMAC + nonce + timestamp 把每个请求绑死到一次性签名 + 5 分钟时间窗,token 泄露也无法重放老请求
- 业界标准做法(AWS S3 / GitHub webhook 都这样)

为啥不做 push 模式(dept 主动 phone home 拉指令):
- HTTP pull 模型简单可靠,失败立刻知道(同步 response)
- pull 模型需要 dept 端起后台 daemon,增加 listener 复杂度
- 现有 web.py 已经在 Flask listen,加一个 endpoint 0 额外开销

## 来源

用户反馈:「30+ 台 VPS 一台一台 SSH 太麻烦,有没有办法直接帮他们处理」

## 后果

✅ 利:
- **90% 运维场景**(升级 / 改配置 / 重启 / 看 log / 健康检查)远程一键
- **客户感受 0**(就是 web 后台多了个 endpoint,没 UI 改变)
- **可扩**:以后加 `set_env` / `tail_logs` / `diag_sheets` / `reload_oauth` 等 action 只是写新函数

⚠ 风险:
- **首次必须客户跑一次 `./update.sh`** 把 agent 部上(无解,代码不存在就调不到)
- METRICS_TOKEN 权限提升:之前只读,现在能升级 + 重启。**因此 HMAC 防重放是强制要求**,不开 IP allowlist 也接受(token 已经分散在客户 sheet 上不好控)
- audit_log 表会增长(已有 90 天自动清,见 v3.0.18)
- subprocess.run upgrade timeout 8 min,慢网 git pull 可能不够 — 监控调

## 验证

本地单测(`agent.py` 6 个关键路径):
- ✓ HMAC 正确签名通过 / 错签拒绝
- ✓ timestamp 窗口校验
- ✓ nonce 5 分钟去重(第 2 次同 nonce 拒)
- ✓ rate limit(同 token 5 次后第 6 次返 False)
- ✓ `action_inspect` 返完整字段(version / env / sessions / disk / containers)
- ✓ `action_upgrade(target_tag="main; rm -rf /")` 被 regex 拒(防 shell injection)

demo VPS 端到端(部 agent.py + 通过中央台 fleet.py 调):
- ✓ `inspect` 返 200,完整 JSON,含 version + accounts + sessions
- ✓ `upgrade dry_run` 返 200,确认 cmd 但不执行

待客户配合一次升级到 v3.0.26 后,可全网远程操控。

## 后续(v3.0.27+)

- `tail_logs` action(docker logs 最近 N 行)
- `diag_sheets` / `fix_sheets` action(包装现有 `_diagnose_sheets_stuck` / `/api/diag/sheets-fix-stuck`)
- `reload_oauth` action(包装现有 `SheetsWriter.reload_credentials`)
- `list_alerts` / `audit_tail` 只读 action(分页查告警 / 审计)
- 异步升级 job:upgrade 返 job_id,后续 inspect 看进度,免 8min HTTP timeout
