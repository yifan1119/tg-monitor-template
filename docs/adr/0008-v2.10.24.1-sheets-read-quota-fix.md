# ADR-0008: Sheets 读 API 配额保护 — sync_headers + peer_name_consistency 间隔独立化

**版本**:v2.10.24.1(hotfix)
**日期**:2026-04-22

## 背景

### 客户反馈(2026-04-22 15:07)

某客户(150+ 账号)主机日志大量刷:

```
sheets WARNING: [sync_headers] 账号 id=717 phone=+xxx 同步失败,跳过:
APIError: [429]: Quota exceeded for quota metric 'Read requests'
and limit 'Read requests per minute per user'
of service 'sheets.googleapis.com' for consumer 'project_number:<project-id>'.

sheets WARNING: [sheets_flush] account=<id> 触 429/quota,退避 600s (level=8)

__main__ WARNING: 实时推送 Sheet 标记失败 peer=<peer-id>: APIError: [429]
```

**症状放大路径**:
1. `sync_headers` 被 429 拒绝 → 客户 B2/B3「操作人员/中心」推送显示空白
2. `_peer_name_consistency_loop` 被 429 拒绝 → 广告主角色标签停更
3. **读配额打爆连锁影响写路径**:`sheets_flush` 也开始 429(gspread `batch_update` 内部会先 read 做元数据校验),退避已到 level=8(600s)
4. **实时推送 `mark_message_pushed` 也 429** → **消息写入停了 → 数据丢失**

事件严重程度:**Critical**(不是只有显示问题,已经在丢数据)。

### 根因(查代码确认)

**配额消耗源 1:`sync_headers`([sheets.py:564](../../sheets.py#L564))**

- 每账号 2 个读请求:
  - `ws.get("A2:B3")` — 读商务人员 / 中心部门([sheets.py:594](../../sheets.py#L594))
  - `ws.row_values(6)` — 读第 6 行角色标签([sheets.py:621](../../sheets.py#L621))
- 由 `_patrol_loop`([tasks.py:165](../../tasks.py#L165))每 `PATROL_INTERVAL`(60s)调一次
- **150 账号 × 2 reads / 60s = 300 reads/min**

**配额消耗源 2:`_peer_name_consistency_loop`([tasks.py:260](../../tasks.py#L260))**

- 每账号 1 个读请求:`ws.row_values(6)`([tasks.py:279](../../tasks.py#L279))
- Loop docstring 写「每 10 分钟」但代码 `await asyncio.sleep(config.PATROL_INTERVAL)` 实际是 60 秒
- **文档-代码不一致 bug**(顺手修)
- **150 账号 × 1 read / 60s = 150 reads/min**

**配额消耗源 3:`_enforce_sheet_tab_consistency`([tasks.py:97](../../tasks.py#L97))**

- 每账号 1 个读请求:`spreadsheet.worksheet(target)` → gspread 6.x 每次都 `fetch_sheet_metadata()`(真 API 读)
- 由 `_patrol_loop` 每 `PATROL_INTERVAL`(60s)调一次
- 实测受 sheets._rate_limit(1.5s 间隔)串行化,效率约 32-40 reads/min 持续
- **这条在首版 hotfix 里被漏掉,Codex review(medium reasoning)发现**

**Google Sheets API 配额**:
- Read requests per minute per user = **60**
- 三源合计持续 ~80 reads/min + sync_headers 爆发期 300/轮 ≈ **超配额**
- 写路径同理受影响(共享部分内部 read)

## 决策

### 1. `sync_headers` 间隔独立化(默认 600s)

**配置**(`config.py`):

```python
SYNC_HEADERS_INTERVAL_SEC = int(os.environ.get("SYNC_HEADERS_INTERVAL_SEC", "600"))
SYNC_HEADERS_DISABLED = os.environ.get("SYNC_HEADERS_DISABLED", "false").lower() == "true"
```

**实现**(`tasks.py` `_patrol_loop`):时间戳式 throttle(Codex P1 修复),异常也会正确推进:

```python
_last_sync_headers_at = 0.0
while self._running:
    try:
        if not config.SYNC_HEADERS_DISABLED:
            _now = time.monotonic()
            if (_now - _last_sync_headers_at) >= config.SYNC_HEADERS_INTERVAL_SEC:
                try:
                    self.sheets.sync_headers()
                finally:
                    _last_sync_headers_at = _now  # 异常也要更新,避免 429 自我循环
        ...  # 其他巡检项不变,跟随 PATROL_INTERVAL
```

**初版方案对比**:首版用 `_patrol_count % _every_n == 0` 计数器,但 sync_headers 抛异常时 `_patrol_count += 1` 不会执行 → counter 卡住 → 下一轮 60s 再试 → 429 loop。时间戳式用 `finally` 兜底。

**配额变化**:
- 默认 600s:`300 / 600 * 60 = 30 reads/min`(均摊)
- 如果客户手动设成 `SYNC_HEADERS_INTERVAL_SEC=60`,回到旧行为
- `SYNC_HEADERS_DISABLED=true` 完全关掉(紧急开关)

### 2. `_peer_name_consistency_loop` 间隔独立化(默认 600s,对齐原 docstring)

**配置**:

```python
PEER_NAME_CONSISTENCY_INTERVAL_SEC = int(os.environ.get("PEER_NAME_CONSISTENCY_INTERVAL_SEC", "600"))
PEER_NAME_CONSISTENCY_DISABLED = os.environ.get("PEER_NAME_CONSISTENCY_DISABLED", "false").lower() == "true"
```

**实现**(`tasks.py`):

```python
async def _peer_name_consistency_loop(self):
    """每 PEER_NAME_CONSISTENCY_INTERVAL_SEC 秒(默认 600 = 10 分钟)..."""
    await asyncio.sleep(120)
    while self._running:
        if config.PEER_NAME_CONSISTENCY_DISABLED:
            await asyncio.sleep(config.PEER_NAME_CONSISTENCY_INTERVAL_SEC)
            continue
        try:
            ...
        except Exception as e:
            logger.error("广告主名一致性巡检失败: %s", e)
        await asyncio.sleep(config.PEER_NAME_CONSISTENCY_INTERVAL_SEC)  # 从 PATROL_INTERVAL 独立
```

**配额变化**:
- 默认 600s:`150 / 600 * 60 = 15 reads/min`
- `PEER_NAME_CONSISTENCY_DISABLED=true` 完全关

### 3. `_enforce_sheet_tab_consistency` 改成一次性拉全量分页标题(Codex P0 修复)

**问题**:原实现对每个账号调 `self.sheets.spreadsheet.worksheet(target)`,gspread 6.x 每次都走 `fetch_sheet_metadata()` → 150 账号 = 150 reads/轮。

**修复**(`tasks.py` `_enforce_sheet_tab_consistency`):一次 `worksheets()` 拿全部标题到内存,for 循环做 set lookup:

```python
async def _enforce_sheet_tab_consistency(self):
    try:
        self.sheets._rate_limit()
        all_worksheets = {ws.title: ws for ws in self.sheets.spreadsheet.worksheets()}
    except Exception as e:
        logger.warning("分页列表读取失败,跳过本轮巡检: %s", e)
        return
    for account in db.get_all_accounts():
        target = account["name"]
        if target in all_worksheets:  # 内存 lookup,0 API 读
            # 对齐 sheet_tab
            ...
            continue
        old = account["sheet_tab"]
        ws = all_worksheets.get(old)  # 改名场景也不需要额外读
        ...
```

**配额变化**:150 reads/轮 → **1 read/轮**(降 99.3%)。

### 4. 总合配额

| 项 | v2.10.24 | v2.10.24.1 首版 | v2.10.24.1 最终 |
|---|---|---|---|
| sync_headers | 300 reads/min | 30 reads/min | 30 reads/min |
| peer_name_consistency | 150 reads/min | 15 reads/min | 15 reads/min |
| enforce_sheet_tab_consistency | ~32 reads/min(持续) | ~32 reads/min(漏改) | **~1 read/min** |
| **合计** | ~480 | ~77 | **~46** |

| 利用率 | **800%**(爆) | **128%**(仍可能爆) | **77%**(安全线下) |

## 原因

### 为什么选间隔调整而不是 `batch_get`

`batch_get`(一次合并 150 账号的读到 1-3 个 request)是更根本的方案,但:
1. 需要改 `sync_headers` 整个函数结构(读完 batch 后再分发处理)
2. gspread 的 `spreadsheet.values_batch_get` 对 ranges 格式有要求,每个 ranges 要包含 worksheet 标题
3. 单账号 worksheet 不存在时要 fallback 处理
4. 测试量大,v3.0.0 再合入更合适

**间隔调整**属于最小改动,**立即见效**,副作用极小(B2/B3 手改同步延迟从 60s → 最多 600s,业务影响可忽略)。

### 为什么默认 600s

- Google 读配额 60/min,两源合计消耗 45/min < 60/min,留 25% 余量给其他 API 调用(`_sheets_flush_loop` 的偶发 read / `mark_message_pushed` / `get_or_create_sheet`)
- 客户改 B2/B3 后 10 分钟内同步是可接受延迟(非交互式场景,后台运维操作)

### 为什么 sync_headers throttle 用时间戳而不是计数器

- **首版用计数器** `_patrol_count % _every_n == 0`,最小改动 + 配置化清晰
- **Codex review 发现 P1**:sync_headers 抛异常(如 429)时 `_patrol_count += 1` 不会执行 → counter 卡住 → 下一轮 60s 再试 → 429 自我循环
- **最终方案**:改 `time.monotonic()` 时间戳 + `try/finally` 更新 → 异常也正确推进 throttle
- 向后兼容:`SYNC_HEADERS_INTERVAL_SEC=60` 回到近似旧行为(每 patrol 都跑)

### 为什么 `_enforce_sheet_tab_consistency` 要一次性拉全量标题

- **首版只节流 sync_headers + peer_name_consistency**,漏了这条:`spreadsheet.worksheet(title)` 在 gspread 6.x 每次都 `fetch_sheet_metadata()`(真 API 读)
- **Codex review 发现 P0**:150 账号 × 1 read / 60s ≈ 32-40 reads/min 持续,跟 sync_headers burst 叠加仍超配额
- **方案**:一次 `spreadsheet.worksheets()` 拿所有 Worksheet 对象到 dict,改名场景复用同一个 dict,避免 2× fetch_sheet_metadata

### 为什么顺手修 `_peer_name_consistency_loop` docstring

- Docstring 写「每 10 分钟」但代码用 `PATROL_INTERVAL`(60s)→ 文档-代码不一致
- 既然要配置化,顺手对齐 docstring 到实际默认行为(都是 600s)
- 符合 CLAUDE.md 规则「文档就是代码,代码就是文档,没有第三种状态」

## 后果

### 正面
- ✅ 大账号数客户(>100)配额不再爆(三源合计 ~46 reads/min < 60/min)
- ✅ 连锁影响消失:`sheets_flush` + 实时推送 Sheet 标记 恢复正常
- ✅ 加了两个紧急开关(`_DISABLED`),未来遇到类似问题可一键关而不用再改代码
- ✅ 可配置化:客户可根据自己账号数调低间隔(账号少的客户可以 `SYNC_HEADERS_INTERVAL_SEC=120` 更快同步)
- ✅ 修掉 `_peer_name_consistency_loop` docstring-代码不一致 bug
- ✅ throttle 用时间戳式,异常也正确推进(首版计数器实现有 429 自我循环风险,Codex review 抓出)
- ✅ `_enforce_sheet_tab_consistency` 150 reads/轮 → 1 read/轮(Codex P0 修复)

### 负面
- ⚠ B2/B3 手改后同步延迟最多 10 分钟(原 60 秒)— 业务影响极小(参见「原因」)
- ⚠ 广告主名 TG 改名后 Sheets 第 6 行同步延迟最多 10 分钟(原 60 秒)— 业务影响极小

### 风险
- **仅降低频率不是根治**:如果客户账号数飙到 300+,默认 600s 也可能再爆(理论临界 ~400 账号)
  - **缓解**:加了 `_DISABLED` 紧急开关,可临时停到 0 reads/min
  - **根治**:`batch_get` 重构排到 v3.0.0(1 request/轮覆盖所有账号)
- **env 未配置情况下自动走默认值**:老 client 升级后不用改 .env,行为自动从「60s 爆配额」切到「600s 不爆」(feature flag 默认 = 更安全行为)

### 客户升级影响(向后兼容)
- 老 `.env` 不动,`SYNC_HEADERS_INTERVAL_SEC` / `PEER_NAME_CONSISTENCY_INTERVAL_SEC` 自动取默认 600
- **客户一键 `./update.sh` 即可**,无需任何手动操作:
  - `update.sh` v2.10.24 起(ADR-0007)内建 `git stash` + `git reset --hard origin/main` 流程
  - sed 止血改过的 `tasks.py` 会被自动 stash 保护,然后强制覆盖为 v2.10.24.1 新代码
  - 健康检查 60 秒失败 → 自动回滚 + stash 还原
  - 运维不用懂 `cp` / `git` / `docker` 任何命令
