# ADR-0017: Caddyfile 热更新的 Docker file bind mount inode 断裂问题

**版本**: v3.0.2
**日期**: 2026-04-23
**来源**: 线上客户 VPS 反馈(一台机部署多部门 HTTPS 失败)

---

## 背景

客户场景: 一台 VPS 同时部署多个 TG 监控部门,共用一个 Caddy 容器(shared 模式),每个部门对应一个 `<dept>.<ip>.nip.io` 子域。

`enable_https.sh` 的 shared 模式逻辑:
1. 发现 80/443 被已有 Caddy 容器占用
2. 把新部门的 web 容器接到该 Caddy 所在的 docker network
3. 在共享 Caddyfile 末尾 `echo >> Caddyfile` 追加新部门的 site block
4. `docker exec caddy reload --config /etc/caddy/Caddyfile` 热加载

第 3/4 步在线上部分 VPS 一直不工作 —— 追加明明写进 host 文件里,Caddy 却始终看不到,reload 说 `config is unchanged`,新部门的 HTTPS 永远上不去。

## 根因

`docker-compose.yml` 里给 Caddy 的 Caddyfile 是 **file bind mount**(不是目录):

```yaml
volumes:
  - ./Caddyfile:/etc/caddy/Caddyfile:ro
```

Docker file bind mount 按 **inode** 绑定,不是按路径。容器启动时记住 host 上 Caddyfile 的 inode,之后:

- `echo >> Caddyfile` 追加 → 同 inode → 容器能看到新内容 ✓
- `sed -i` / `cp` / `vim :wq` / 编辑器原子替换 → 新 inode → 容器的 mount 仍指旧 inode(文件已被 unlink 但句柄存活)→ 容器永远看到旧内容 ✗

线上 VPS 的 Caddyfile 历史上被 `sed -i` / 手动编辑过,inode 早就断了。enable_https.sh 的追加写进了 host 文件(新 inode),但容器内 `/etc/caddy/Caddyfile` 还是老快照,新 site block 永远不生效。

Caddy 容器侧的症状:

```
{"msg":"config is unchanged"}   # reload 读的是容器内老 Caddyfile
```

Host 侧的症状 (诊断命令):

```
$ stat -c "inode=%i" /root/tg-monitor-demo/Caddyfile    # host
inode=522331
$ docker exec tg-caddy-demo stat -c "inode=%i" /etc/caddy/Caddyfile   # container
inode=512384
```

**两个 inode 不一样 = bind mount 断了**。

## 决策

### 1. `enable_https.sh` 加 inode 同步校验 + 自愈

追加 site block 后不直接 reload,先对比 host 文件大小 vs 容器内文件大小:

- 一致 → 正常 `caddy reload` (不中断服务)
- 不一致 → `docker restart caddy` 兜底(5-10s HTTPS 短暂中断,但保证修好),再校验

### 2. 所有 silent fail 改成明确报错

- 找不到 Caddyfile bind mount → `exit 1` (不再 fallback 到 warning 后返回 0 让 install.sh 以为成功)
- 追加后 host 文件没 grep 到刚追加的内容 → `exit 1`
- 证书 90 秒没签下来 → 明确告诉用户 3 种最常见原因(云防火墙 / 限流 / 死站堵队列),而不是悄悄继续

### 3. 新增 `scripts/caddy-doctor.sh` 自查工具

线上维护用。6 项检查:
1. 容器状态
2. Caddyfile host 和容器 hash/size 对比(这次问题核心)
3. Caddyfile 语法校验
4. 列所有 site block,对比运行中容器,标出死站
5. 证书目录 vs site block,标出没签下来的域
6. 最近 5 分钟 ACME/TLS 错误摘要

任何问题退出码 1,并打印修法提示。

### 4. 生成的 site block 清除冗余 header_up

Caddy 2 的 `reverse_proxy` 默认会加 `X-Forwarded-For` / `X-Forwarded-Proto` / `X-Forwarded-Host`,模板里写这些会被 Caddy 打 warning(`Unnecessary header_up`)。保留 `Host` 和 `X-Real-IP`(后者 Caddy 不自动加),其他删掉。

## 原因

**为什么不直接改成目录 bind mount?** 比如 `./caddy:/etc/caddy`,把 Caddyfile 放进子目录。目录 mount 绑定的是目录 inode,文件替换不影响。

理论上更干净,但:
- 破坏性改动,现有客户部署路径都要迁移
- 需要 enable_https.sh 兼容"老路径 Caddyfile"和"新路径 caddy/Caddyfile"
- `docker-compose.yml` / `install.sh` / `update.sh` / 所有文档路径都要改

本次只是修 bug,不是大重构。选 inode 自愈 + restart 兜底 = **对现有客户零感知**,老部署照跑,新部署用修过的脚本直接正常。未来若决定做目录迁移(v3.1+?),另外开 ADR。

## 来源

- 2026-04-23 某 VPS 部署 freshtest 部门 HTTPS 失败
- 排查历时约 40 分钟,过程中几度误以为是 Let's Encrypt 限流、Caddyfile 语法错、docker network 问题
- 真正定位到 inode 断裂是在对比 `docker exec caddy wc -c Caddyfile` 和 host `wc -c Caddyfile` 发现 size 不一致之后

## 后果

### 正面
- **shared caddy 模式从此稳定**: 一台 VPS 部任意多部门 HTTPS 互不干扰,inode 断了会自愈
- **故障可诊断**: 以后客户反馈"HTTPS 打不开",第一步跑 `caddy-doctor.sh`,结果说明一切
- **fail loud**: 安装脚本不会再骗客户"✅ HTTPS 已启用"但实际没启用

### 负面
- `docker restart caddy` 兜底会带来 5-10s HTTPS 中断,但仅在 inode 断的时候触发。正常情况走 reload(无中断)。
- `caddy-doctor.sh` 依赖 `docker exec` 和 `sha256sum`,极少数精简镜像可能没 sha256sum,降级到纯 size 对比(已做 fallback)。

### 中性
- 给客户升级的 `release_notes.json` 说明: v3.0.2 是技术修复,功能行为不变。

## 复盘教训

1. **Docker file bind mount 是按 inode 绑的** — 这个冷知识过去没写进团队记忆,这次吃了大亏。CLAUDE.md 必须记下来。
2. **沉默失败 = 最坏的 bug** — enable_https.sh 几个分支都是出问题后返回 0 打印"HTTPS 已启用",掩盖了真实错误 40 分钟。以后所有脚本 = fail loud,不允许 silent fallback。
3. **客户自查工具比客户 SSH 贴 log 快 10 倍** — caddy-doctor.sh 三十秒给全景诊断,比客户贴一堆零散 docker 命令快多了。

## v3.0.3 后续 (2026-04-23,用户反馈 "不要让用户一直操作终端")

v3.0.2 落地后用户提出:caddy-doctor.sh 是客户要主动跑的,那些历史上 inode 已经断掉但 HTTPS 当下仍暂时工作的客户,不会主动去跑体检。等他们下次 `install.sh <新部门>` 才触发 enable_https.sh 的自愈,中间的时间窗口他们的 HTTPS 其实是半死不活的。

**决策**: `update.sh` 升级末尾加一段 "5.6 Caddy inode 自愈",客户跑 update.sh 就自动顺手检查修复。

**关键安全约束** (用户强调 "但不能搞坏他们上面其他项目"):
- 只找跟当前部门直接相关的那一个 Caddy:
  1. 本部门有自己的 Caddy `tg-caddy-<COMPANY>` → 就是它
  2. 本部门用 shared Caddy → 扫 `^tg-caddy-*` 容器,找 Caddyfile 里包含本部门 `PUBLIC_DOMAIN` 的
- 其他 Caddy 容器一概不动
- 命名不匹配 `^tg-caddy-` 的容器(比如客户自建 Caddy `caddy-my-blog`)一律跳过

这条约束保证客户一台 VPS 可以安全混跑 TG 监控和其他 Caddy 项目,我们的脚本不越界。

```bash
# update.sh 新增段(简化版)
MY_CADDY=""
if docker ps --format '{{.Names}}' | grep -qx "tg-caddy-${COMPANY_NAME}"; then
    MY_CADDY="tg-caddy-${COMPANY_NAME}"
else
    MY_DOMAIN=$(grep "^PUBLIC_DOMAIN=" .env | cut -d= -f2)
    if [ -n "$MY_DOMAIN" ]; then
        for caddy in $(docker ps --format '{{.Names}}' | grep -E '^tg-caddy-'); do
            if docker exec "$caddy" grep -qF "$MY_DOMAIN" /etc/caddy/Caddyfile 2>/dev/null; then
                MY_CADDY="$caddy"; break
            fi
        done
    fi
fi
# 只对 MY_CADDY 做 size 对比 + restart 自愈
```

VPS 实测验证(freshtest 部署场景):
- freshtest 识别出自己用的是 `tg-caddy-demo`(shared mode),只会动它
- `tg-monitor-multi`(用户另一个无关项目)不在识别范围,不被触碰
