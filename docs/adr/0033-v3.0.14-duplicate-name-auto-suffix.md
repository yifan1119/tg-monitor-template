# ADR-0033: v3.0.14 同名外事号自动加 phone 后缀,根治 Sheets 唯一性导致的分页冲突

**日期**: 2026-05-08
**状态**: Accepted (Codex round1 P0/P1 修完)
**相关 ADR**: ADR-0021(OAuth 文件 IPC 模式)

## Codex round1 修法记录(2026-05-08)

**P0**: 后 4 位 phone 撞 / 真实 name 撞 / 已有 sheet_tab 撞 → 静默生成两个相同 sheet_tab。
- 修法:`_gen_unique_tab(base_name, phone, used_set)` 候选链(后 4 位 → 后 6 位 → 全 phone → 加序号 -2 ~ -99)循环找全局唯一,`used_set` 包含已有 sheet_tab + 所有 raw name + 同批次刚分的。
- 单测验证:3 个 Alice phone 后 4 位都 6384 → tabs=[None, Alice-6384, Alice-006384] ✓;真实 name=「张三-6045」占位 → 第二号自动跳 `张三-816045` ✓

**P1**: dedupe / 回查失败时 fallback 裸 name → race / 锁冲突下静默回退老 bug。
- 修法:`_create_sheet_tab` 在 `phone` 非空时 dedupe 失败 **fail closed**,不调 `create_account_tab_full`,延迟到 60s patrol `ensure_account_tabs` 兜底自愈。

---

## 背景

Google Sheets 的 worksheet 标题是**全表唯一**的(API 约束)。tg-monitor 现有架构里:

- `web.py:_create_sheet_tab` 登录时直接用 `tg_name`(TG 昵称)调 `add_worksheet(title=tg_name)`
- `sheets.py:ensure_account_tabs` 启动 + 60s patrol 用 `tab_name = sheet_tab or name or phone` 决定建哪个分页
- `sheets.py:flush_pending` 写消息时用同样的 `tab_name` 找分页

**默认假设**: TG 昵称(`accounts.name`)在一个部门内唯一。

**线上实际**:
- 一个部门 200+ 账号
- 业务员 TG 昵称重名很常见(「张三」「老王」「小明」「Caer」之类)
- 重名第二个登录时,`add_worksheet('张三')` 抛异常或被「幂等返回老分页」吞掉,**不会**自动建第二个独立分页
- 但 `flush_pending` 写消息照样用 `tab_name = '张三'` → **第二号的所有消息混入第一号的「张三」分页**(数据混乱已发生,无法回溯分离)

**踩坑案例**(2026-05-08):
- 客户 某客户部门 部门,两个号 `+85500000045` 和 `+85500000084` 都叫「张三」
- web 后台显示 29 个账号,Sheets 只有 27 个分页(差 2 个全是同名撞掉的)
- 客户反复反馈「为啥少分页」,排查才知道是设计层面的限制

**已知半解方案** (sheet_tab 字段):
- `accounts.sheet_tab` 字段优先级最高(老就有),允许人工 SQL 覆盖
- 但**没有 UI 让客户改**,**没有自动加后缀**,纯靠运维 SSH 干预

---

## 决策

新增 module-level helper `sheets.dedupe_assign_sheet_tabs(conn) -> int`,在两个入口被调:

1. **`web.py:_create_sheet_tab`(新登录路径)**:
   - 加 `phone` 参数,登录前先调 dedupe + 重新查 DB 拿可能已被加后缀的 `sheet_tab`
   - 用最终名字调 `create_account_tab_full` 建分页
   - `phone` 默认 `""` 保持向后兼容(老调用方不传 phone 行为不变)

2. **`sheets.py:ensure_account_tabs`(启动 sweep + 60s patrol 路径)**:
   - 第一行调 dedupe,然后照常 SELECT(会拿到新设的 sheet_tab)走原 `tab_name = sheet_tab or name or phone` 分支

**dedupe 算法**(`dedupe_assign_sheet_tabs`):
- 扫 `accounts` 表,按 `name` 分组
- 同 name ≥ 2 个 → 按 `phone` 字典序排
- **第一个保留 name 不变**(老分页不动,**零数据迁移**)
- 第二个起 `sheet_tab = "<name>-<phone后4位>"`(例:`张三-6384`)
- 已有 `sheet_tab` 的账号 → **不动**(尊重客户/老逻辑设的值,幂等)
- name 为空 / 同名只 1 个 / phone 为空 → 跳过

---

## 原因

### 为何「第一个保留 name」而不是全部加后缀?

- **零数据迁移**:第一号已有的「张三」分页历史消息全留,无需重命名
- **零客户感知**:已建好的 Sheets 分页不动,客户截图/链接/书签全部还能用
- 第二号反正之前也没分页(消息混入第一号),从升级时刻起新建独立分页接收新消息
- 升级前已混入第一号分页的第二号历史消息**无法回溯分离**(Sheets 行没有「来源 phone」标记),patch **不承诺解决历史混乱**,只承诺**从升级时刻起干净隔离**

### 为何 `phone` 后 4 位作后缀?

- 短(4 字符)+ 易识别(同业务员看自己的 phone 后 4 位一眼认出)
- 200 号里 N 个同名,N=2-3 时 phone 后 4 位再撞概率 < 1/3000(数学忽略)
- 不依赖 username(@xxx),很多账号没设 username
- 不依赖 tg_id(数字一长串,可读性差)

### 为何 dedupe 放 `sheets.py` module level 而不是 class method?

- `_create_sheet_tab`(web.py 进程)和 `ensure_account_tabs`(sheets.py 进程)**两个不同 docker 容器都要调**
- 跨进程共享一份**纯函数式**算法(只读 conn,DB UPDATE 用 WHERE id),避免重复实现
- 跟 ADR-0021 OAuth 的 module-level `is_oauth_failure()` 同样模式

### 为何按 `phone` 字典序排定「第一个」?

- **幂等性**:每次调用排序结果不变,「第一个」永远是同一个 phone
- 即使 dedupe 被并发调用 N 次,UPDATE 的目标 id 永远一致
- WAL 模式下两个 conn 同时 UPDATE 同一 id 后写覆盖前写,**值相同**(确定性算法),不出错

## 来源

- 用户(2026-05-08)反馈线上案例:某客户部门 部门两个「张三」分页只建了 1 个
- 之前 v2.x 也偶发,客户用 SQL UPDATE sheet_tab 临时救火,无系统性 fix
- 改名方案被 evaluation 后排除:`tasks.py:_sync_account_names` 没有 ownership 校验,改第二号 TG 昵称会把第一号的分页错改名 → 数据错位风险

## 后果

### 收益

- ✅ 升级后**自动 fix 所有现存重名账号**,无需 SSH 干预
- ✅ 新登录的同名账号自动加后缀,**前端无感**
- ✅ 不止单部门,所有客户所有同名 case 一次性根治
- ✅ 零数据迁移、零重登、零 client 改动 (`sheet_tab` 字段早就有)

### 已知限制

- ⚠️ 升级前已混入第一号分页的第二号历史消息**无法回溯分离**(Sheets 行没标记)
- ⚠️ 依赖 `phone` 字段非空(99.9% case 都有,极端无 phone 场景跳过不 fix)

### 兼容性

- `_create_sheet_tab` 旧签名 `(name, operator, company)` 仍然可调(`phone` 默认空)
- 已有 `sheet_tab` 字段的账号(客户手动设过的)**不动**
- 同名只 1 个的账号 `sheet_tab` 仍空,继续用 `name` → **跟老行为一致**
- 老客户不升级:行为完全不变,跟 v3.0.13 等价
- 升级失败回滚 v3.0.13:DB 里被 patch 加的 sheet_tab 仍然有效(老代码也读 sheet_tab 优先),不破坏

### 性能

- dedupe 一次 SELECT + N 次 UPDATE(只针对重名账号,N 通常 0-5),开销 < 10ms
- 60s patrol 调用 dedupe 也是同样开销
- web 登录路径加一次 SELECT + 可能一次 UPDATE,开销 < 5ms

## 验证

- 单元逻辑:模拟 [张三/+xx45, 张三/+xx84, 老王/+xx00] 的 accounts 表 → dedupe 后 [张三/+xx45 sheet_tab=NULL, 张三/+xx84 sheet_tab='张三-xx84', 老王/+xx00 sheet_tab=NULL]
- 幂等性:连续调 dedupe 3 次,DB 状态不变
- web 登录新同名号:phone=+xx99 → 调 dedupe → DB 出现 sheet_tab='张三-xx99' → 建分页「张三-xx99」✓
- 升级实战:dev 部门跑两个同名号 → ./update.sh → 启动 log 出现 `[dedupe_sheet_tab]` → Sheets 出现「张三-后4位」分页 ✓
