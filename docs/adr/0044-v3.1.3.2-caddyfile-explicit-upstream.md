# ADR-0044 · v3.1.3.2 · Caddyfile upstream 显式化(防共用 Caddy DNS 撞车)

## 背景

某客户某 VPS(同一 IP 上跑了两个独立部门,分别 dept-A 老大 / dept-B 后接入)在 2026-05-13 反馈:两个不同部门的 web 后台 URL 不同,但访问后看到的**已登录账号列表完全一字不差**(同样 29 个号、同电话号、同 TG username)。客户怀疑是数据混淆。

排查路径:

1. 比对两台容器内 `data.db` accounts 表 → 两边数据**完全不同**(dept-A 29 个号 vs dept-B 15 个号),证明 DB 没串。
2. `docker inspect tg-caddy-<dept-A>` → 发现 Caddy 容器同时接入 `tg-<dept-A>_default` 和 `tg-<dept-B>_default` 两个 network(运维 v3.0.13 共用 Caddy 模式下,后加入的 dept-B 跑 enable_https.sh 时 `docker network connect` 把 Caddy 容器拉进 dept-B network)。
3. `docker exec tg-caddy-<dept-A> getent hosts web` → 解析到 **dept-B 的 web 容器 IP**(172.19.x.x),不是 dept-A 自己的 web。
4. 查 Caddyfile 运行时 config(`/config/apps/http/servers/srv0/routes`)→ dept-A 主子域 upstream 字段是 **`web:5001`**(模糊别名),dept-B 子域 upstream 是 **`tg-web-<dept-B>:5001`**(显式名,enable_https.sh 加的)。

**根因确认**:主仓 `Caddyfile` 模板第 16 行用 `{$WEB_UPSTREAM:web:5001}` 模糊别名。Docker network 内 service 默认有 `web` 这个 alias(docker-compose service 名)。一台 VPS 单部门时,Caddy 容器只在一个 network,`web` 唯一指向自己 → 正常。一旦运维把 Caddy 拉进第二个 network(给同 VPS 第二个部门反代),两个 network 各自有名为 `web` 的容器 → Docker DNS 撞车 → 第一个部门的子域被路由到第二个部门的后端。

中央台 fleet inspect 显示:全网 45 个部门里同 IP 多部门有 4 处(具体清单见本地 `.claude/private-notes.md`)。今天客户那台是触发的第 1 例,**剩 3 台 IP 已埋雷**(同款 DNS 撞车风险但尚未触发或未被发现)。

更严重的是:`update.sh` 升级时跑 `git reset --hard origin/main`,会把手动修好的客户 Caddyfile **覆盖回主仓有 bug 的版本** → 中央台「⬆ 全网升级到 main」按钮按下去会让 4 台同时炸。

## 决策

主仓 Caddyfile 模板改用显式容器名占位符,install/update 时替换成实际部门名。

### 改动 1:`Caddyfile` 模板

```diff
 {$PUBLIC_DOMAIN} {
-    reverse_proxy {$WEB_UPSTREAM:web:5001} {
+    reverse_proxy {$WEB_UPSTREAM:tg-web-__COMPANY_NAME__:5001} {
```

占位符用 `__COMPANY_NAME__`(双下划线包围)而不是 `${COMPANY_NAME}`,因为 Caddyfile 内 `$` 已被 Caddy 用作环境变量语法,避开冲突。

### 改动 2:`install.sh` 新装时替换

```bash
# 此时 Caddy 容器还没启,Caddyfile 没 bind mount,sed -i 换 inode 安全
if [ -f Caddyfile ] && grep -q "__COMPANY_NAME__" Caddyfile; then
    sed -i.bak "s/__COMPANY_NAME__/${COMPANY_NAME}/g" Caddyfile && rm -f Caddyfile.bak
fi
```

### 改动 3:`update.sh` 升级时自愈(新 section 5.5b)

升级时 `git reset --hard` 后 Caddyfile 变成主仓新模板含占位符,且老部署可能残留裸 `web:5001`,两种情况都要处理。位于现有 section 5.5(异地 IP 清理)之后,5.6(inode 自愈)之前。

```bash
if [ -f Caddyfile ]; then
    NEED_UPSTREAM_REWRITE=0
    if grep -q "__COMPANY_NAME__" Caddyfile 2>/dev/null; then
        NEED_UPSTREAM_REWRITE=1
    fi
    if grep -q "web:5001" Caddyfile 2>/dev/null; then
        NEED_UPSTREAM_REWRITE=1
    fi
    if [ "$NEED_UPSTREAM_REWRITE" = "1" ]; then
        cp Caddyfile Caddyfile.bak.upstream.$(date +%s) 2>/dev/null
        sed -e "s/__COMPANY_NAME__/${COMPANY_NAME}/g" \
            -e "/reverse_proxy/s|web:5001|tg-web-${COMPANY_NAME}:5001|g" \
            Caddyfile > /tmp/Caddyfile.upstream
        if [ -s /tmp/Caddyfile.upstream ]; then
            cat /tmp/Caddyfile.upstream > Caddyfile   # > redirect 保 inode
        fi
    fi
fi
```

**关键 sed 安全性论证**:

- 第一行 `s/__COMPANY_NAME__/${COMPANY_NAME}/g`:占位符特殊串,不会误伤其他内容。
- 第二行 `/reverse_proxy/s|web:5001|tg-web-${COMPANY_NAME}:5001|g`:**限定在 `reverse_proxy` 行内**替换。已显式的 `tg-web-XXX:5001` 行内含子串 `web-XXX:5001`(部门名隔开),**不含子串 `web:5001`**,sed 不替换 → 已显式行不被误伤。
- enable_https.sh 加的子部门 site block 用显式 `tg-web-<dept>:5001` → 不动 ✅
- 注释行不含 `reverse_proxy` 关键字 → sed 不替换 ✅
- BSD sed / GNU sed 都兼容(无 `\b` 等扩展)。

**inode 保护**:`cat /tmp/Caddyfile.upstream > Caddyfile` 而非 `mv`,保持 docker bind mount 不断裂(CLAUDE.md 硬规定 #9)。

## 原因

**为什么不用 envsubst 全替换**:envsubst 替换所有 `$VAR`,会撞 Caddyfile 自己用的 `$PUBLIC_DOMAIN` / `$WEB_UPSTREAM`。手动选定占位符 `__COMPANY_NAME__` 更精确。

**为什么不在 docker-compose 层把 `web` alias 改成 `tg-web-${COMPANY_NAME}`**:docker-compose service 名 `web` 是历史约定,改 alias 会破坏其他代码(`enable_https.sh` 也用 `web` 作 fallback;`docker compose -p tg-XXX restart web` 等命令)。

**为什么不让运维加部门时手动改 Caddyfile**:CLAUDE.md rule 6 — Feature flag 默认安全,「出厂模板就防呆」比「靠运维记住手改」可靠。

**为什么 update.sh 加自愈而不是只改模板**:存量 45 台客户 VPS 上 Caddyfile 已经是老 `web:5001` 模板(git reset 之后变成 placeholder 但还没替换),没自愈段就要逐台 SSH 改。

## 来源

- 客户截图(2026-05-13)— 两个 URL 不同子域显示同一份账号列表
- 排查过程见上文「背景」段
- CLAUDE.md rule #9(Caddyfile bind mount inode 不能换)沿用 `cat > redirect` 模式
- CLAUDE.md rule #20(Docker bind-mount 用目录不用单文件)— Caddyfile 是历史单文件 mount,不在本次改造范围,长期改造留 v3.2.x

## 后果

### 正面

- 同 IP 多部门共用 Caddy 场景下,DNS 撞车风险**根治** — 出厂模板就是显式容器名。
- 老客户跑一次 `update.sh` 升 v3.1.3.2 → section 5.5b 自动把老 `web:5001` 替换成 `tg-web-<dept>:5001` → 升级一次永久修复。
- 中央台「⬆ 全网升级到 main」按钮可以安全按下去 — 不会再因为 git reset 覆盖回老 buggy Caddyfile。
- 已手动修过的部门(用 `git update-index --skip-worktree Caddyfile` 保护)升级时 git reset 不动 Caddyfile,sed 跑下来也是 no-op(已是显式名) ✅

### 中性 / 负面

- 每次升级 update.sh 都会跑一次 sed + 生成一份 `Caddyfile.bak.upstream.*` 备份(无害,但累积下来要靠后续手动清理)。
- 占位符方案依赖 `install.sh` 替换 — 如果客户绕过 `install.sh` 手动起 Caddy(罕见),Caddy 会拿到字面 `tg-web-__COMPANY_NAME__:5001` 解析失败。但出现这种场景的客户本来就需要懂 Caddyfile,可以自己改;另外 `update.sh` 5.5b 自愈也会兜底替换。

### 升级路径

| 客户当前状态 | 升 v3.1.3.2 后 |
|---|---|
| 老 `web:5001` 模板,Caddyfile 未手改 | git reset 拉到主仓新 placeholder 模板 → 5.5b sed 替换占位符 → 显式 ✅ |
| 老 `web:5001` 模板,本地有手改但未 commit | git stash → reset → 5.5b sed 替换占位符 → stash pop 可能 conflict(同 dept-A 触发路径) |
| 已 `skip-worktree`(运维手动加固过的部门) | git reset 不动 Caddyfile → 5.5b sed 检测 `web:5001` 不在显式行 → no-op ✅ |
| 全新 v3.1.3.2 部署 | install.sh sed 替换占位符 → 显式 ✅ |
