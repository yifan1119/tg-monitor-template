# ADR-0013: `release_notes.json` 文案必须白话 — 客户看的是业务不是技术

## 背景

用户(2026-04-22 晚)反馈后台「版本升级」弹窗显示的 v2.10.24 累计说明(ADR-0012 写的内容)太技术化 — 客户看不懂,用户自己也看不懂。

原文案例(ADR-0012 里的 v2.10.24 累计说明):
```
v2.10.24.1 — Sheets 读配额保护 🚨(账号多客户必升)
- sync_headers 节流:原 60 秒读一次每账号 2 读 → 150 账号 × 2 = 300 reads/min 爆配额
- _peer_name_consistency_loop 间隔独立化:原跟 PATROL_INTERVAL=60s 每账号 1 读,独立到 PEER_NAME_CONSISTENCY_INTERVAL_SEC=600

v2.10.24.3 — 预警分页零丢失保障 🛡️
- 修预警分页写入只重试 3 次 × 2 秒(共 6 秒容错)老 bug
- DB alerts 表加 sheet_written 标记 + claim_alert_for_sheet_write 三态语义
- 三轮 Codex 审阅修正:事件循环阻塞 / append-mark 非原子 / crash window P0
```

**问题**:
- 出现 `sync_headers` / `_peer_name_consistency_loop` / `PATROL_INTERVAL` / `DB` / `Codex` / `crash window P0` 等内部词
- 客户不知道也不关心这些 — 他只想知道「这版解决了什么问题,对我有什么好处」

## 决策

**`release_notes.json` 的 `title` 和 `body` 必须遵守白话原则**:

**✅ 白话可以用的词**:
- 具象业务名词:「表格」「预警群」「账号」「预警分页」「后台」「升级」「弹窗」
- 问题描述:「卡住」「写不进去」「丢了」「看不到」「冲突」「空白」「报错」
- 效果描述:「自动补」「不再卡」「一次升级搞定」「不用动任何东西」「零丢失」

**❌ 禁止出现的词**(这些进不了 release_notes,放 ADR 里):
- 文件名 / 函数名:`sync_headers`、`_peer_name_consistency_loop`、`_alert_writeback_loop`、`bot.py`、`sheets.py` ...
- 技术缩写 / 术语:`regex`、`config`、`SQL`、`API`、`DB`、`claim-first`、`三态语义`、`event loop`、`crash window`、`append-mark`、`idempotent`
- 代码细节:`sheet_written=0` / `PATROL_INTERVAL` / `SYNC_HEADERS_INTERVAL_SEC=600` / regex 表达式
- 工具名:`Codex`、`gspread`、`Telethon`、`aiogram`
- 配额数字:「60/min 配额」「150 × 2 = 300 reads/min」

**结构模板**:
```
[1. 问题标题 — 用白话]
以前 <具象现象> — <影响>。现在 <修好之后的效果>。

[2. ...]

**升级一条命令,已登录账号不用重登,新配置默认都是开的:**
`cd /root/tg-monitor-<部门> && ./update.sh`
```

**长度**:每条 1-2 句,整个 body 不超过 10 行。

## 原因

**为什么分成两套文档(release_notes 白话 / ADR 技术)?**

- 客户看 `release_notes.json` 的后台弹窗 — 他想知道「对我有什么好处」,不想看技术细节
- 开发者看 ADR / commit message — 需要技术细节才能审阅、回滚、追 bug
- 一条 release_note 同时满足两类读者 = 都不爽

**客户看不懂的真实代价**:
- 看不懂 → 不升级 → 旧 bug 继续遇到
- 看不懂 → 觉得系统很复杂 / 不稳定 → 信任度下降
- 看不懂 → 来问开发 → 浪费双方时间

**白话不是弱化 — 是精准的沟通层次匹配**:
- ADR-0010 写「claim-first 三态语义 + stale 回收」→ 对开发
- release_notes 写「预警绝不丢行,写失败系统每 60 秒自动补写到成功为止」→ 对客户
- 两者讲同一件事,针对不同听众

## 来源

- **用户反馈**(2026-04-22):截图显示原 v2.10.24 累计说明,原话:「这种更新说明不要写的那么复杂白话简短说明优化什么 解决什么 不然他们哪看得懂 我都看不懂了」
- 用户自己都看不懂是关键信号 — 客户更看不懂

## 后果

**正面**:
- 客户后台弹窗看了就懂,「升级解决什么」一眼看清
- 建立 release_notes 的永久规则,未来每次发版文案都白话

**需要后续遵守**:
- 以后每发 release,写文案先自检:有没有技术词?有就放 ADR / commit message,release_notes 只留白话
- Review 时看 release_notes 读一遍,有内部词 / 内部结构 → 打回去改
- 写 release_notes 时想象「发给一个不懂技术的客户看,他关不关心这句?」

**本次修复范围**:
- v2.10.24 key(累计说明)改白话 — 已改(ADR-0012 那次)
- v2.10.24.5 / v2.10.24.6 key 改白话 — 本次
- 历史 key(v2.10.24.1 ~ v2.10.24.4 独立 key)保留原技术版 — 理由:这些 key 只有已升到 v2.10.24.4+ 的客户精确匹配才会读到,而这些客户马上会升 v2.10.24.5 / .6 之后看新 key,历史独立 key 其实已经读不到了,改不改影响不大
- 下次如果再发四段版本,一定用白话写

## 教训

写 release_notes 时开发者容易「把 commit message 复制过去」— 这是错的。
- Commit message:对 git reader(开发、CI、未来自己)
- ADR:对 reviewer / 回滚者 / 新人 onboarding
- Release notes:对客户 — 唯一一个不读代码的读者

三个文档服务三个听众,内容不能混用。
