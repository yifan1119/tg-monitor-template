# ADR-0021: v3.0.7 OAuth 重新授权后 Sheets 自愈 (闭合 v3.0.6 的诊断—修复链路)

## 背景

v3.0.6 把驾驶舱「Sheet 写入健康」做成自动诊断 + 修复按钮:OAuth 失效红条 + 「去设置页重新授权 →」按钮。**诊断这一半做对了**,但**修复那一半留了 UX bug**:

客户点了「去重新授权」按钮 → 走完 Google OAuth → `oauth_helper.save_token()` 成功把新 refresh_token 写到 `data/google_oauth_token.json` ✓ — 但 **`SheetsWriter.gc` 还是握着启动时缓存的旧 Credentials**(`SheetsWriter.__init__` 只在容器启动时跑一次),`_sheets_flush_loop` 继续每 5 秒撞 `invalid_grant`,**永远不写入**。

客户体感:「我授权完了,为什么还是积压?授权再多次都没用」。
唯一解法:SSH 上服务器 `docker restart tg-monitor-<部门>`。

某客户 2026-04-25 现网命中,2991 条积压 5 小时,反复授权无效后才反馈。

**根因诊断**:
- `sheets.py:31-36` `SheetsWriter.__init__` 创建 `gspread.authorize(creds)`,绑定旧 Credentials 对象进 `self.gc`
- `gspread` 内部 google-auth `AuthorizedSession` 缓存这个 `Credentials`,后续每次调用 API 都用它的 refresh_token 换 access_token → token revoked 后永远 `invalid_grant`
- `sheets.py::flush_pending` 现有 `except gspread.exceptions.APIError` 只识别 429/quota,**不识别 `invalid_grant` / `RefreshError`** → 自身没有自愈兜底

## 决策

**目标**: 客户**只通过 web 重新授权**就能恢复 Sheets 写入,零 SSH。

实施 5 项改动:

### 1. 共享 OAuth 失效词汇表 + helper (`oauth_helper.py`)

`oauth_helper.py` 新增 module-level `OAUTH_FAIL_MARKERS` tuple + `is_oauth_failure(text)` helper。

**为什么**:
- v3.0.6 在 `dashboard_api.py:680-684` 维护了一份 OAuth 失效关键词清单
- v3.0.7 自愈逻辑也要识别 OAuth 失效
- 两份独立维护必然会漂移 — Codex 审阅指出原来的 `dashboard_api.py` 清单已经有 bug:`refreshError`(camelCase)在 `.lower()` 过的 `log_text` 里永远不匹配,`oauth.*revoked` 是正则语法但用在 substring `in` 检查里,`401` 太宽松误吞非 OAuth 401

**关键词最终清单**(根据 google-auth 源码 + OAuth 2.0 RFC 6749 误码定义):
```
invalid_grant | token has been expired | refresh_error | credentials do not contain
revoked | unauthorized_client | invalid_client
```

`is_oauth_failure(text)` 内部 `.lower()` + 任一关键词 substring 命中即 True。

`sheets.py` 的 flush_pending 兜底 + `dashboard_api.py` 的诊断卡片**都从 `oauth_helper` 引入**,单一来源。

### 2. `SheetsWriter._write_lock` 改 RLock

`sheets.py:40` `threading.Lock()` → `threading.RLock()`。

**为什么**: `flush_pending` 在 `with self._write_lock:` 内 catch `RefreshError` → 调 `reload_credentials` → `reload_credentials` 也要拿 `_write_lock`(原子重建 gc + 清退避) → 非可重入 Lock 立即死锁。

确认无副作用:全文只 `flush_pending` 和 `flush_alert_records` 两处用 `_write_lock`,都在 SheetsWriter 实例方法内同线程,不依赖非可重入语义。

### 3. `SheetsWriter.reload_credentials()` 方法

新方法,做四件事:
1. 调 `oauth_helper.get_credentials()` 重读 token 文件
2. `gspread.authorize()` 拿到新 gc
3. `gc.open_by_key(SHEET_ID)` 拿到新 spreadsheet handle
4. **清空所有账号的退避状态** (`_flush_backoff` + `_flush_backoff_level`),否则新 token 拿到了但还卡在 600s 退避里

**原子替换**:`gspread.authorize` + `open_by_key` 都成功后才赋值给 `self.gc` / `self.spreadsheet`,中途失败保留旧值不破坏现状。
全程在 `_write_lock`(RLock) 内。
新增 `self._oauth_reload_count` 计数器,日志 `[oauth_reload] gc 已重建,backoff 已清空 (累计 N 次)` 可观察自愈次数。

任何步骤失败 → 不抛异常,返 False,下轮 flush 自动再试。

### 4. `flush_pending` 加三层 OAuth 失效 catch (顺序硬性规定)

```python
except google.auth.exceptions.RefreshError as e:    # 主路径 — google-auth 透传
    self.reload_credentials(); continue
except gspread.exceptions.APIError as e:
    if oauth_helper.is_oauth_failure(str(e)):       # 罕见兜底:gspread 包了 401
        self.reload_credentials(); continue
    if 429/quota path: ...                          # 现有路径不变
except Exception as e:                              # bare 兜底也匹配
    if oauth_helper.is_oauth_failure(str(e)):
        self.reload_credentials(); continue
    ...
```

**关键决策**: OAuth 检查必须在 429 检查**之前**。`google-auth` changelog 出现过 `"invalid_grant — quota project context lost"` 字样,会被宽松的 `"quota" in msg` 误吞到 600s 退避路径。

### 5. 不在 web.py callback 里直接通知 SheetsWriter (跨进程不可行)

**Codex 审阅抓到的 P0**: 我最初的设计是 module-level `_GLOBAL_WRITER` + `register_global_writer()` from `main.py`,然后 `web.py` OAuth callback 调 `get_global_writer().reload_credentials()` 主动通知。

**这个方案是错的** — `docker-compose.yml` 把 `tg-monitor` 和 `tg-web` 拆成**两个独立容器**(分别独立的 Python 进程):
- `tg-monitor` 进程跑 `main.py` → 创建 `SheetsWriter` → 注册到本进程 `_GLOBAL_WRITER`
- `tg-web` 进程跑 Flask → OAuth callback fire → `get_global_writer()` 在 tg-web 进程拿到的永远是 `None`(没注册)
- 跨容器没有共享内存,singleton 在两个进程里完全独立

**最终设计 (干净简单)**: 只用 `flush_pending` self-heal 路径。两个进程**通过共享 docker volume** `data/google_oauth_token.json` 文件交换状态:
1. `tg-web` callback 调 `oauth_helper.save_token()` 写新 token 到 `data/...json`
2. `tg-monitor` 下一轮 `_sheets_flush_loop`(默认 5 秒)用旧 gc 撞 `RefreshError`
3. self-heal catch 调 `reload_credentials()` → 重读**已经是新 token** 的文件 → 重建 gc
4. 再下一轮 flush 直接成功,积压自动追完

**实测恢复时间**: 5-30 秒(视 SHEETS_FLUSH_INTERVAL 配置 + 队列里有没有消息触发首次写)。
**对比 v3.0.6**: 之前要 SSH `docker restart` 才行,现在零 SSH。
**对比 Drive 端**: `media_uploader.reset_drive_cache()` 之所以能在 web.py callback 里直接调,是因为 `media_uploader` 跟 web.py 都在 tg-web 进程内(Drive 上传走 web 路径)。Sheets 写入只在 tg-monitor 进程内,所以走文件 + 自愈而非 in-process reset。

### 6. Banner 文案

设置页 OAuth 完成 banner 加 `(Google Sheets 写入将在 5-30 秒内自动恢复)` 提示客户在等什么。

## 原因

1. **闭合 v3.0.6 的诊断—修复链路**: 上一版做了一半(诊断到位、修复按钮挂了 UI 但点了不灵),客户体感"工具骗我"。这版把承诺的"点按钮就修好"做实。

2. **跨容器的根本约束 → 选最简方案**: tg-monitor 和 tg-web 分开是合理的(Telethon 长连接 vs Flask 短连接,生命周期/资源/重启策略都不同)。在这个约束下,最干净的 IPC 是文件系统 — 反正 token 本来就要落盘 `data/google_oauth_token.json`,自愈逻辑直接消费这个文件就行,不引入新 IPC 机制。

3. **多层防御**:
   - 客户走 web reauth → token 文件更新 → flush self-heal 在下次写入时识别并重建
   - 客户在 Google 控制台直接撤销+重新授权(没经过本系统) → 同样路径触发自愈
   - 罕见 gspread 包装异常 → APIError + bare Exception 双层兜底匹配关键词

4. **失败兜底**: reload 失败保留旧 `self.gc`(不破坏现状),下轮 flush 自动再试。最坏情况退化到 v3.0.6 行为(SSH restart 仍可救急)。

5. **跟 v3.0.6 词汇表统一**: 共享 `oauth_helper.OAUTH_FAIL_MARKERS` 之后,诊断卡片说什么、自愈识别什么完全一致 — 不会出现"驾驶舱说 OAuth 失效但日志看自愈没动"的 ghost bug。

## 来源

- 客户反馈: 「重新授权过很多次了都没有写入」(2026-04-25 某客户部门)
- 现场诊断: 驾驶舱日志面板看到 `invalid_grant` 重复刷屏 + 客户确认授权 UI 显示成功
- 代码审计 (Plan agent, Sonnet 4.5): 抓出 3 个原始设计的 P0:
  - `__init__` 自注册会被 web.py 临时实例覆盖 → 改显式 `register_global_writer`(后续被 Codex 推翻整个 in-process 路径)
  - `threading.Lock()` 非可重入会跟 RLock 路径死锁 → 改 RLock(保留)
  - 漏 `google.auth.exceptions.RefreshError`(gspread 不包) → 单独加 catch 在 APIError 之前(保留)
- 代码审计 (Codex CLI, GPT-5.4 high reasoning): 抓出 2 个 P0+P1:
  - **P0**: in-process singleton 跨容器无效 — `register_global_writer` 在 tg-monitor,`get_global_writer` 在 tg-web,永远拿不到 → 重新设计为 self-heal only(本 ADR 第 5 节)
  - **P1**: `OAUTH_FAIL_MARKERS` 跟 `dashboard_api.py:680-684` 已经分歧(`refreshError` 在 lowercased text 里 dead-match,`oauth.*revoked` 是正则用作 substring,`401` 太宽松) → 抽到 `oauth_helper.py` 单一来源 + helper 函数

## 后果

### 正面
- 客户在驾驶舱点「去重新授权」按钮 → 走完 OAuth → **不用 SSH**, Sheets 写入 5-30 秒内自愈
- 即使没经过本系统的 reauth(客户在 Google 控制台重新授权) → flush_pending 自己也能识别并自愈
- 日志 `[oauth_reload] gc 已重建` 可观察自愈次数 + 触发路径
- 跟 v3.0.6 诊断卡片配套,客户体感「工具说怎么修就能修」
- `OAUTH_FAIL_MARKERS` 单一来源,以后维护只动一处

### 潜在问题 / trade-off
- **空队列时不会立即自愈**: 如果客户重新授权时队列里没消息(`_flush_backoff` 也没 active 退避),`flush_pending` 不会主动 raise → reload_credentials 不会触发。但这不是问题:没消息要写就没问题。下一条 inbound 消息触发的写入会撞 RefreshError → 自愈 → 写入成功
- **`_write_lock` 改 RLock 性能差异**: 可忽略 (CPython RLock 仅多一次 owner check),但留个记号 — 如果未来发现锁竞争热点,要重新评估
- **共享 `OAUTH_FAIL_MARKERS` 引入循环依赖风险**: `oauth_helper` 不依赖 sheets/dashboard_api,只是被它们 import → 单向依赖,无循环风险
- **dashboard_api 简化掉了 `401`**: 原来的 `401` 太宽松(任何 401 — 包括 Google Sheets 接口本身的 401 而非 OAuth refresh 失败),新 list 把 `unauthorized_client` / `invalid_client` 从 RFC 6749 加回来,更精确

### 未涉及(刻意不做)
- **OAuth refresh token 健康主动巡检**: 比如每小时探一次 token,提前预警过期。考虑过,会增加 Google API 调用频率(占配额),且 token 失效低频事件,被动检测足够
- **OAuth 失效推 TG 群预警**: v3.0.6 的诊断卡片已经在 web 端可见,推 TG 群是另一个产品决策(老板/运维谁该收),留给客户提需求再做
- **跨进程 IPC 通知机制**(file flag 让 tg-monitor 主动 reload 而不等下次 flush): 当前 5-30 秒延迟可接受,引入 IPC 增加复杂度不划算
- **`tasks.py:_sheets_flush_loop` bare except 优化**: 那个吞所有异常的 except 是 pre-existing 行为,本版本不动 (out of scope)。本 ADR 通过 `flush_pending` 内的精细化 catch 已经能 cover OAuth 路径

## 升级指引

```bash
cd /root/tg-monitor-<部门> && ./update.sh
```

零感知升级。功能升级后立刻可用:
- 之前 OAuth 失效已经在用 v3.0.6 诊断卡片的客户:这次 update 会重启容器(必然 recreate,因为镜像 hash 变了)→ SheetsWriter 重新初始化读到新 token → 历史积压自动追完
- 之后再次遇到 OAuth 失效:点驾驶舱「去重新授权」按钮一气呵成,5-30 秒自愈,不用 SSH

## 验证

### 静态自查 (push 前)
- `grep -n "OAUTH_FAIL_MARKERS" oauth_helper.py sheets.py dashboard_api.py` — `oauth_helper.py` 1 处定义,sheets/dashboard 各从 helper 引入
- `grep -n "is_oauth_failure" oauth_helper.py sheets.py dashboard_api.py` — helper 在两处使用
- `grep -n "_write_lock = threading" sheets.py` — 必须是 `RLock()`,无 `Lock()`
- `grep -n "RefreshError" sheets.py` — 必在 APIError catch 之前
- `grep -n "register_global_writer\|get_global_writer\|_GLOBAL_WRITER" .` — **必须 0 命中**(已删干净)
- `grep -n "reload_credentials" sheets.py web.py main.py` — sheets.py 1 处定义 + flush_pending 多处自愈调用,web.py / main.py 0 命中

### Smoke test (Python REPL)
```python
import oauth_helper
assert oauth_helper.is_oauth_failure("invalid_grant: Token has been expired or revoked.") is True
assert oauth_helper.is_oauth_failure("429 quota exceeded") is False
assert oauth_helper.is_oauth_failure("") is False
```

### 集成 (dev 容器)
1. **死锁测试**: 同线程持 `_write_lock` 调 `reload_credentials`,必须返回不挂(RLock 修复确认)
2. **invalid_grant 路由**: monkeypatch `_flush_account` raise `RefreshError("invalid_grant: ...")`,spy `reload_credentials` 必被调,`_flush_backoff_level[account_id]` 保持 0(没误进退避)
3. **E2E 模拟**: dev 容器 sed 改坏 token → 看 RefreshError + `[oauth_reload] gc 已重建` log → 走 callback 恢复真 token → 等 5-30 秒 → 看 `[oauth_reload] gc 已重建` 再次出现 + 下轮 `批量写入 N 条` 出现

### 回归 (不能破坏现有)
- 429 路径仍走原指数退避 5s→600s,不被 OAuth catch 误吞
- `web.py` 三处临时 `SheetsWriter()` 行为不变(原本就跟自愈无关)
- `tasks.py::_sheets_flush_loop` bare except 行为不变(pre-existing)
- v3.0.6 驾驶舱诊断卡片继续工作 — 共享词汇表后甚至更精确
