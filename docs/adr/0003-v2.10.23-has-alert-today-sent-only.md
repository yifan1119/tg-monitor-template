# ADR-0003: `has_alert_today` 只认真送达的,失败/静默不参与去重

**版本**:v2.10.23
**日期**:2026-04-21

## 背景

Codex 审计发现 `bot.py:196-225` + `database.py:306-313` 的 bug:

**v2.10.22 的 `has_alert_today`**:
```python
def has_alert_today(alert_type, peer_id):
    today = datetime.now(TZ_BJ).strftime("%Y-%m-%d")
    row = get_conn().execute(
        "SELECT COUNT(*) FROM alerts WHERE type=? AND peer_id=? AND created_at LIKE ?",
        (alert_type, peer_id, f"{today}%")
    ).fetchone()
    return row[0] > 0
```

**v2.10.22 的 `send_no_reply_alert`**:
```python
# 先 insert_alert,再 send_message
alert_id = db.insert_alert(...)   # DB 立刻有记录
msg = templates.no_reply_alert(...)
if not config.ALERT_NO_REPLY_ENABLED:
    return    # 静默分支 — alerts 表已有记录
try:
    sent = await self.bot.send_message(...)
    db.update_alert_bot_msg(alert_id, sent.message_id)
except Exception as e:
    logger.error("发送失败: %s", e)   # 失败分支 — alerts 表仍有记录,bot_message_id 为 null
```

**两个 bug**:
1. **推送失败当天不再重试**:`send_message` 失败后 `alerts` 表已有 `type='no_reply'` 记录,下一次 `_no_reply_loop` 扫到同一对话 `has_alert_today('no_reply', peer_id)=True` → 跳过 → 整天不再重试
2. **静默模式占去重名额**:`ALERT_NO_REPLY_ENABLED=False` 的情况下也插入记录,如果后续开回 `true`,同一对话下次 `has_alert_today` 还是 True,当天不推 — 静默期的对话整天收不到预警

## 决策

1. **`has_alert_today` 改成只认「真送达 OR 明确静默」**:
   ```python
   # v2.10.23
   row = get_conn().execute(
       "SELECT COUNT(*) FROM alerts WHERE type=? AND peer_id=? AND created_at LIKE ? "
       "AND (bot_message_id IS NOT NULL OR status='silenced')",
       (alert_type, peer_id, f"{today}%")
   ).fetchone()
   ```

2. **静默路径明确标 `status='silenced'`**(`bot.py`):
   ```python
   if not config.ALERT_NO_REPLY_ENABLED:
       alert_id = db.insert_alert(...)
       db.update_alert_status(alert_id, "silenced")   # v2.10.23: 显式静默
       logger.info("[ALERT_NO_REPLY_DISABLED] 跳过 peer=%s (静默记录)", peer["id"])
       return
   ```

3. **发送失败路径不做额外处理**:`bot_message_id` 保持 null,`has_alert_today` 不认作已推 → 下次扫描自动重试

同样逻辑应用到 `send_delete_alert`。

## 原因

### 为什么不在失败时删掉 alert 记录?
- 保留失败记录便于事后排查(故障时间、对象、文本)
- 删掉 vs 保留 bot_message_id=null 功能等价,但保留更有可观测性

### 为什么不用新字段 `sent_at` 或 `sent_status`?
可以,但目前靠 `bot_message_id IS NOT NULL` 语义足够明确 — 只有 `send_message` 成功 `update_alert_bot_msg` 才会被设置。加新字段增加 schema 复杂度。

### 为什么静默用 `status='silenced'` 而不是删掉?
保留静默期的 alerts 记录,如果将来要回溯「静默期哪些对话被跳过了」有据可查。

## 来源

Codex 审计 Critical A2:
> 🔴 Critical `bot.py:196-225` + `database.py:306-313`:未回复告警发送失败后当天不再重试。根因是先 `insert_alert()` 再 `send_message()`,`has_alert_today()` 只看是否有记录,不看是否成功推送;关闭 `ALERT_NO_REPLY_ENABLED` 时也会写入去重记录。建议区分 `created/sent/send_failed/suppressed`,只有 `sent` 或明确静音才参与"已推送"去重。

## 后果

### 立即效果
- 推送失败的 alert 下次扫描(1 分钟后)会自动重试
- 静默模式切回 true 时,当天还能收到新的预警

### DB 影响
- `status` 字段新增合法取值 `'silenced'`
- 老数据(v2.10.22 的 pending 且无 bot_message_id)在新逻辑下**不会**被认作已推送
  - 影响:升级瞬间,老的失败 alert 会被重试一次 — **可接受**,最多多推一条
  - 若不希望这种重发,可以在 migration_v1 里把老 pending 且老于 1 天的标为 `silenced`,但嫌复杂暂不做
