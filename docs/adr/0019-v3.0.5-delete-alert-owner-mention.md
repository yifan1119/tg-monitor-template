# ADR-0019: v3.0.5 删除消息预警对齐 stage2 审批体验

## 背景

**现象**: 客户反馈看到两种预警在预警群里"体验不一致":

- **两段式未回复 stage2 预警**(v3.0.0 起): 尾行 `@负责人 请核实并做审批` + 按钮「登记违规 / 取消」
- **删除消息预警**(v2.x 到 v3.0.4): 尾行无 @ + 按钮「通过 / 拒绝」

客服需要在同一个预警群里同时处理两种预警。老的「通过/拒绝」两颗按钮语义跟「登记违规/取消」不同，客服习惯切不过来。业务方希望两种预警的审批体验统一，而且删除消息也要能直接 @ 到对应账号的负责人让他立即关注。

## 决策

`send_delete_alert` 改成**数据驱动**:
- 账号配了 `owner_tg_id` → 新格式
  - 模板追加 `{owner_mention} {custom_text or "请核实并做审批"}`
  - `parse_mode="HTML"` + `_make_keyboard_stage2(alert_id)`
  - callback 复用 v3.0.0 已有的 `on_stage2_action` handler (`violation:` / `cancel:`)
- 账号没配 `owner_tg_id` → 保留老格式
  - `_make_keyboard(alert_id)` (通过/拒绝)
  - 无 `parse_mode`
  - 完全不变

新增可选配置 `REMIND_DELETE_TEXT`(全域)，空则用内置默认「请核实并做审批」。

`templates.delete_alert` 加 `owner_mention=""` / `custom_text=""` 参数。有 owner_mention 时字段全 `html.escape`，无则走老路径完全不变。

## 原因

**采用「数据驱动」而非 feature flag (参考 v3.0.1 经验)**:
- 一个 feature flag (比如 `DELETE_ALERT_OWNER_MENTION_ENABLED`) 会让客户升级后还要额外去系统设置里打开开关才生效 — 反了「零感知升级」原则
- 已经配过两段式的客户(有 `owner_tg_id`) → 自动升级到新格式，不用再配置
- 没配两段式的客户 → 行为完全不变，零感知
- 没有 flag 管理成本，代码路径也更干净

**callback 复用 `on_stage2_action`**:
- `violation` / `cancel` 的语义对两种预警都适用 — 删除消息也常见违规行为
- 复用写 Sheet 路径 `_write_alert_to_sheet`，它已经识别 `alert.type == "deleted"` 写到「信息删除预警」分页
- 不新建 handler 减少并发/race 隐患

**保留老按钮路径**:
- 有客户还没配过两段式(老部署)，删除预警一直用「通过/拒绝」运行稳定
- 突然切到「登记违规/取消」会让没准备的客服困惑
- 数据驱动 = 只有主动配过 `owner_tg_id` 的账号才切，是客户自己的决策

## 后果

### 正面
- 两种预警审批体验统一，客服学习成本降低
- 删除消息也能 @ 触发负责人的即时通知(v3.0.4 修的 TG 原生 @ 解析一起生效)
- 可选 `REMIND_DELETE_TEXT` 让客户自定义提示文案(跟 stage1/stage2 对齐)
- `_write_alert_to_sheet` 的「登记违规」写入已经覆盖 `deleted` 类型，无需额外 schema 改动

### 潜在问题
- 已配 `owner_tg_id` 的客户升级后**按钮从「通过/拒绝」变「登记违规/取消」**
  - 是按设计的行为，也是客户反馈要的
  - 但客户 IT 对接人要提前告诉一线客服，避免 UX 突变
  - release notes 已白话说明

### 未覆盖
- 删除预警**推送路由**仍走主 `ALERT_GROUP_ID`，不走 `UNREPLIED_ALERT_GROUP_ID`
  - stage2 是因为「升级后才 @ 负责人所以推独立群」的设计
  - 删除预警不是升级事件，直接推主群符合原有语义
  - 如果客户要求也推独立群，单独开 feature 请求

## 验证

模板单测（`python3 -c "..."` 已跑过）:
- 老路径(无 owner_mention): 纯文本，无 HTML escape，无 @
- 新路径(有 owner_mention): HTML escape，追加 `@xxx 请核实并做审批`
- 自定义文案: `custom_text` 覆盖默认
- HTML escape: `<company>` / `张三&李四` / `<tag>` 都正确 `&lt;` `&amp;` 转义

线上验证（客户升级后）:
让配过 `owner_tg_id` 的账号对应广告主删一条消息，预警群里出现:
```
【信息删除预警XXX】

<结构化字段>
已删除信息：<内容>

@owner_username 请核实并做审批
[登记违规] [取消]
```
点「登记违规」→ 写入「信息删除预警XXX」分页 + 消息编辑 `✅ 已登记违规 — <审核人名>`

```bash
cd /root/tg-monitor-<部门> && ./update.sh
```
