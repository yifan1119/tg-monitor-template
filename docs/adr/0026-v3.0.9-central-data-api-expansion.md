# ADR-0026: v3.0.9 中央台数据接口扩展 — 把 DB 业务字段全量暴露给 metrics token

## 背景

v2.8.0 给中央台留了 `/api/v1/metrics` Bearer token 鉴权接口(web.py:2426),返回 `dashboard_api.snapshot()`,字段够给前端驾驶舱用,但**给中央台聚合不够**:

线上某客户案例:中央台想出"商务中心日/周/月违规登记报表"(7 个部门聚合),发现:

1. **`alerts.status` 字段不返**:`alerts_recent` 的 SQL `SELECT a.id, a.type, a.message_text, a.created_at, a.bot_message_id, p.name, ac.name`(dashboard_api.py:526)漏 select status,**violation_logged 终态拿不到**,中央台没法做"违规登记明细表"。
2. **`alerts.stage` 不返**:stage1 / stage2 升级状态看不到,无法分析两段式预警的实际升级率。
3. **`alerts.keyword` 不返**:关键词监听明细看不到具体哪个关键词触发的。
4. **`alerts.reviewed_at` / `claimed_at` 不返**:审核响应时间 / 谁先抢到 review 都看不到。
5. **`alerts.sheet_written` / `last_write_error` 不返**:Sheet 写入失败的具体原因(per-row)拿不到,只能看聚合的 `sheets.status_msg`。
6. **`accounts.tg_id` / `business_tg_id` / `owner_tg_id` 不返**:`accounts_matrix` SQL `SELECT id, phone, name, username, sheet_tab, company, operator FROM accounts`(dashboard_api.py:332)漏这 3 列,中央台无法显示"谁是商务/谁是负责人"。
7. **`accounts.remind_30min_text` / `remind_40min_text` 不返**:每个号自定义的提醒模板没暴露。
8. **`peers` 表完全没专门 endpoint**:只有 `top_peers_active(5)` / `top_peers_silent(5)`,**1420 个监听聊天的明细拿不到**,中央台想做分组聚合(广告主 / 业务线)无米下锅。
9. **`messages` 表完全没暴露**:只有 `messages_today` 聚合统计(deleted/in/media/out/total),消息内容 / `deleted_at` / `media_seq` / `archive_msg_id` 全拿不到。中央台想做删除消息溯源(反查 archive 群备份)做不到。

7 台 VPS 实测确认:中央台必需的 23 个 DB 字段里,**有 15 个完全没暴露,4 个间接暴露(JOIN name 但没 ID)**。

## 决策

v3.0.9 一次性把 DB 业务字段全量暴露,**0 新表 / 0 新字段 / 0 数据迁移**,只改 SQL SELECT + 加 endpoint。

### 1. `dashboard_api.py` 现有 2 个函数扩展 SELECT(纯加字段,向后兼容)

**`accounts_matrix()`**(line 326)新增:`tg_id` / `business_tg_id` / `owner_tg_id` / `remind_30min_text` / `remind_40min_text` / `sheet_tab`(后者已在 SELECT,但旧代码没放 dict)。

**`alerts_recent()`**(line 523)新增:`account_id` / `peer_id` / `msg_id` / `status` / `stage` / `keyword` / `reviewed_at` / `sheet_written` / `claimed_at` / `last_write_error`。

老前端 `dashboard.html` 读老字段不受影响,新字段它不读就不读,**100% 向后兼容**。

### 2. `dashboard_api.py` 新增 4 个查询函数

```python
violations(from_date, to_date, alert_type)          # status='violation_logged' 明细
alerts_filtered(from, to, status, stage, type, limit, offset)  # 通用 alerts + 分页
peers_all(account_id, group)                        # peers 全表
messages_filtered(account_id, peer_id, from, to, deleted_only, limit, offset)
```

关键设计点(**v3.0.9 Codex 审阅 6 条 P1 全修后版**):
- **硬上限**:`_MAX_ALERTS_LIMIT=1000` / `_MAX_PEERS_LIMIT=5000` / `_MAX_MESSAGES_LIMIT=1000`。`_clamp_int()` helper 防客户端传非法 limit 值(限于值范围语义合理的参数,不用于必填字段)。
- **`messages_filtered` 强制 `account_id + peer_id` 必填走 `_require_int()` 硬失败**:`messages` 表是项目里最大的(几十万行级),没复合 key 不让查,防整表扫拖死 SQLite。**Codex P1 修**:之前是函数内部静默返 `{"items": [], "error": "..."}` 200 OK,违反"强制必填"安全语义 → 改成 `ValueError`,endpoint 转 400,客户端能区分成功 vs 失败。
- **日期参数走严格 regex + `datetime.strptime` 真实解析**:`_parse_date()` 不仅 `^\d{4}-\d{2}-\d{2}$` 正则匹配还要 `strptime` 验证(防 `2026-99-99` / `2026-04-01junk` 之类伪日期)。**Codex P1 修**:之前 `_date_like_pattern()` 只查前 10 字符 `[4]=='-' [7]=='-'` 就放行,坏请求会伪装成"正常但范围错"。
- **status / type / stage 白名单校验集中维护**:`_VALID_ALERT_STATUS` / `_VALID_ALERT_TYPE` / `_VALID_ALERT_STAGE` module 级常量,`_validate_choice()` helper 统一过非法值 `ValueError`。即使走参数化 SQL 多一层防御,且**非法值不静默忽略**(防中央台脚本误传 `status=DROP_TABLE` 拿到全部 alerts)。
- **`peers_all` 非法 `account_id` / `group` 不再静默忽略**(Codex P1 修):之前 `try: int(); except: pass` 会让 `account_id=abc` 静默退化成"返回全部门 peers",中央台拼错参数会拉错数据。改成 `ValueError` → 400 显式失败。
- **`_safe()` 兜底废弃**(Codex P1 修):4 个新查询函数原本走 `_safe(_q, [])` 静默吞 SQL 异常返空数组,导致客户端看到 `count=0` 误判"今天 0 条违规"。**改成不包 `_safe()`,DB 异常往上抛,endpoint try/except 显式分流**:`ValueError` → 400(用户错),其它 `Exception` → 500(服务端错),客户端能正确响应。
- **`dept` 字段对齐 ADR 契约**(Codex P1 修):`violations()` 返回字段加 `"dept": company`(同时保留 `"account_company"` 老字段名向后兼容),`alerts_filtered` 也加,跟 ADR 文档约定一致。

### 3. `web.py` 新增 4 个 `/api/v1/*` endpoint

```python
GET /api/v1/violations?from=&to=&type=
GET /api/v1/alerts?from=&to=&status=&stage=&type=&limit=&offset=
GET /api/v1/peers?account_id=&group=
GET /api/v1/messages?account_id=&peer_id=&from=&to=&deleted_only=&limit=&offset=
```

鉴权:抽出 `_v1_check_token()` helper 共用 `_ensure_metrics_token()`(跟现有 `/api/v1/metrics` 同一套 Bearer / `?token=` 双路径)。

错误码:401 unauthorized / 400 bad_request(缺必填字段)/ 500 internal_error。

每次调用都进 `_metrics_log_access()` 跟现有 metrics 一致,**设置页"metrics 调用记录"模块自动能看到 4 个新 endpoint 的命中**。

### 4. `/api/v1/metrics` snapshot 自动跟着扩展

`snapshot()` 函数本身没改,但因为它内部调 `accounts_matrix()` 和 `alerts_recent()`,**那两个函数加字段后,snapshot 输出的 JSON 自然多 13 个新字段**,中央台读这条接口也能拿到大部分新数据(messages/peers 全表除外,那两个数据量大用专门 endpoint)。

## 原因

1. **客户当下需要**:7 台 VPS 商务中心准备做日/周/月违规登记报表,中央台拉 metrics 发现 violation_logged 拿不到 → 没法做。本版本之前**只有 SSH 直查 SQLite 一条路**(客户没 SSH 凭据 + 安全合规问题)。

2. **0 新表 0 新字段 0 数据迁移 = 0 风险**:数据库 schema 完全不动,升级 = `git pull && docker compose up -d --build`,sessions / data 卷不动,**200+ 已登录 TG 账号不重登**。
   - 跟客户约束(CLAUDE.md「绝不能让 200+ 已登录账号重新登录」)对齐。

3. **跟主业务路径完全解耦**:不动 `listener.py` / `bot.py` / `sheets.py` / `tasks.py` / `database.py`,只动 `dashboard_api.py` + `web.py` 的只读路径。**4 个新 endpoint 任意一个挂掉,主业务监控/推送/写表完全不受影响**。

4. **历史数据 100% 保留**:`alerts.status` 字段从 v1 就存在(database.py:67),`bot.py:198-199` 每次客户点"登记违规"按钮就把 status 改成 `'violation_logged'`,**历史每一条都老老实实写在数据库里**。`web.py:2244` 的"清空今日去重"按钮只删今天的且要管理员手动按,**alerts 表实际是只增不删**。升级后立刻能查到从客户首次部署到现在的所有违规记录。

5. **回滚成本 0**:出问题 `./rollback.sh` 回 v3.0.8.3,因为没动 DB 也没动 sessions,**回滚后跟从没升级过一样**,比 v3.0.8 那次 Sheets 写入架构改动(改了 token bucket + values.append)安全得多。

## 来源

- 客户反馈:**线上某商务中心 7 台 VPS 部署完中央台聚合脚本,跑 `/api/v1/metrics` 发现 alerts_recent 没 status 字段** → 无法识别哪些是登记违规。
- 7 台 VPS 实测 metrics 接口:`r1-r7.json` JSON 顶层字段全跑了一遍,确认 23 个 DB 字段缺口(详见 ADR 背景表)。
- DB schema 审计:`grep -nE "CREATE TABLE|ADD COLUMN" database.py` 列出 4 张表 48 个字段,逐一对照 `dashboard_api.py` 的 SELECT 子句,确认 9 类字段没暴露(accounts.tg_id 系列 / peers 全表 / messages 全表 / alerts.status 系列)。
- CLAUDE.md「客户约束」:本版严格遵循"绝不能让 200+ 账号重新登录" + "向后兼容老 .env 不改也能跑"。

## 后果

### 正面
- **中央台一次性满足**:23 个业务字段全暴露,日/周/月违规登记报表 / 删除消息溯源 / 关键词命中分析 / stage2 升级率分析 全部可做,不再需要 SSH 直查 DB。
- **中央台脚本不用每次需求加字段都升一次版**:这次一次性把 DB 业务面全暴露,后续中央台报表迭代只在客户端做,服务端零改动。
- **隐式给老 metrics 也加了字段**:`/api/v1/metrics` snapshot 自动多 13 个新字段(因为 accounts_matrix + alerts_recent 扩了),旧客户读老字段不受影响,新客户(中央台)能直接用扩展数据。
- **`messages_filtered` 强制必填 `account_id + peer_id` 防整表扫**:即使中央台脚本写错 query 也不会把 SQLite 拖死。
- **`_metrics_log_access` 自动覆盖 4 个新 endpoint**:设置页 metrics 调用记录模块直接能看到中央台 IP / 时间 / 命中接口,审计完整。

### 潜在问题 / trade-off
- **`/api/v1/messages` 单次拉 1000 条 + 每条带 text 字段** → JSON 体积可能 1-2 MB。Flask 默认配置可接受,但中央台如果跑全量数据迁移要分页。**已加 limit/offset 标准分页**。
- **`alerts.status` 透出后**,如果中央台把这数据拿去聚合"已通过 vs 已拒"率,老 v3.0.0 之前的 alerts 没经过两段式审核流程(全是默认 pending 没 reviewed_at),**统计需要排除 created_at < v3.0.0 部署日期**。这是中央台脚本端的事,不在本版本 scope。
- **`accounts.business_tg_id` / `owner_tg_id` 暴露**给中央台 = TG 用户名 / ID 出 VPS,虽然走 Bearer 鉴权,**算敏感运维数据**。考虑到这俩字段本来就是设置页让客户填的不是机密,且 metrics token 本身就握在中央台运维手上,可接受。**比传 Sheet ID 还低一档敏感**。

### 未涉及 (留 v3.1+)
- **`/api/v1/metrics/access_log` Bearer 化**(目前是 `@login_required`):访问日志查询接口当前必须从 web 后台登入才能看,中央台想监控自己接口调用记录得绕路。但这是**审计内部用**不是中央台数据,本版本不动。
- **`messages.text` 大字段 default 全文返回**:大文本消息(>1000 字符)会让 JSON 膨胀。本版本不截断(中央台需要原文),如果实测体积问题再加 `?truncate=N` query param。
- **WebSocket 推送实时变更**:中央台如果要"违规即推",目前只能轮询。WebSocket 留 v3.1 看真实需求决定。

## 升级指引

```bash
cd /root/tg-monitor-<部门> && ./update.sh
```

升级后:
- 7 台 VPS 立刻支持 4 个新 endpoint,中央台脚本可以并发拉 violations / alerts / peers / messages
- 现有 `/api/v1/metrics` 返回的 JSON 多了 13 个字段(账号 5 个 + alerts 8 个),旧中央台脚本不读不影响
- 鉴权 token 不变(沿用现有 METRICS_TOKEN),新 endpoint 直接复用

- `python3 -m py_compile dashboard_api.py web.py` — 语法 OK
- `grep -n "def violations\|def alerts_filtered\|def peers_all\|def messages_filtered" dashboard_api.py` — 4 个新函数都在
- `grep -n "@app.route.*/api/v1/" web.py` — 6 个 v1 endpoint(原 2 个 + 新 4 个)
- `grep -n "_v1_check_token" web.py` — 4 处调用(每个新 endpoint 1 次)

### 集成(本地 dev 容器)

```bash
TOKEN=$(grep METRICS_TOKEN .env | cut -d= -f2)
BASE=http://localhost:5050   # 视 dev 配置而定

# 1. /api/v1/violations
curl -H "Authorization: Bearer $TOKEN" "$BASE/api/v1/violations" | jq
curl -H "Authorization: Bearer $TOKEN" "$BASE/api/v1/violations?from=2026-04-21&to=2026-04-28&type=no_reply" | jq

# 2. /api/v1/alerts
curl -H "Authorization: Bearer $TOKEN" "$BASE/api/v1/alerts?status=violation_logged&limit=50" | jq
curl -H "Authorization: Bearer $TOKEN" "$BASE/api/v1/alerts?stage=2&from=2026-04-25" | jq

# 3. /api/v1/peers
curl -H "Authorization: Bearer $TOKEN" "$BASE/api/v1/peers" | jq '.returned'
curl -H "Authorization: Bearer $TOKEN" "$BASE/api/v1/peers?account_id=1" | jq

# 4. /api/v1/messages(必填 account_id + peer_id)
curl -H "Authorization: Bearer $TOKEN" "$BASE/api/v1/messages" | jq    # → 400 bad_request
curl -H "Authorization: Bearer $TOKEN" "$BASE/api/v1/messages?account_id=1&peer_id=1&limit=10" | jq

# 5. 401 校验
curl "$BASE/api/v1/violations" | jq    # → 401 unauthorized
curl -H "Authorization: Bearer wrongtoken" "$BASE/api/v1/violations" | jq

# 6. limit clamp 校验
curl -H "Authorization: Bearer $TOKEN" "$BASE/api/v1/alerts?limit=99999" | jq '.limit'    # → 1000(被 clamp)
```

- `dashboard.html` 前端驾驶舱(读 `accounts_matrix` + `alerts_recent`):新字段不读不影响,UI 显示完全一致
- `tasks.py / sheets.py / bot.py / listener.py` 没动 → 监听 / 推送 / 写表 / Session 全部 0 改动
- `_start_tg_monitor()` / `/api/restart` / `_diagnose_sheets_stuck` v3.0.8.3 fallback 行为不变
- 设置页 metrics token 显示 / 重置 / access_log 行为不变
