# ADR-0055 · v3.1.9 dept 加 BOT_POLLING_DISABLED flag(完成 v3.0.22 callback bridge 链路)

## 背景

v3.0.22(ADR-0037 webhook bridge)设计了:
- 中央台 `callback_listener.py` polling 所有 alert_routes.bot_token
- 收按钮点击 → POST 回 dept `web.py:/api/v1/callback`(METRICS_TOKEN 鉴权)
- dept 本地 polling 关掉,避免双 polling 撞 409

但**这套链路从未真正启用**:
- 中央台 v0.17.1 紧急 fix:`CALLBACK_LISTENER_ENABLED` 默认 OFF,原因 — 客户 alert_routes 里 bot_token 跟 dept BOT_TOKEN 同款时,中央台 polling + dept polling 双重 polling 撞 409 Conflict spam log
- dept `bot.py:1185` `dp.start_polling()` 没条件判断,只要 BOT_TOKEN 非空就 polling

2026-05-19 升级陈家碧 13 dept 时发现:
- 4 个 dingfenggs* dept(`dingfenggs-yunyingzx` / `dingfenggs2` / `dingfenggs3` / `dingfenggs4`)共用同 1 个 BOT_TOKEN
- Telegram 只把 callback 发给当前持有 long polling 的 1 个 dept → 另 3 个 dept 收不到按钮点击
- 业务影响:3/4 dept 的「通过/拒绝」「登记违规/取消」按钮**点了没反应**

客户反馈"不要再申请一堆 BOT_TOKEN",必须完成 v3.0.22 链路治根。

## 决策

**dept `bot.py` 加 `BOT_POLLING_DISABLED` flag**(默认 false,向后兼容):

### 代码改动

1. **`config.py`** 加字段:
```python
BOT_POLLING_DISABLED = os.environ.get("BOT_POLLING_DISABLED", "false").lower() == "true"
```

2. **`bot.py:1180 async def start()`** 加分支:
```python
if getattr(config, "BOT_POLLING_DISABLED", False):
    logger.info("Bot polling 已禁用 — callback 由中央台接管")
    return
await self.dp.start_polling(self.bot)
```

### 配套(运维侧,非代码)

3. **中央台 AWS `.env`** 加:
```
CALLBACK_LISTENER_ENABLED=true
```

4. **中央台 `alert_routes` 表**确保每行 `bot_token` 配的是 dept BOT_TOKEN(共用就共用,中央台 polling 接管)

5. **dept `.env` 加**(共用 BOT_TOKEN 的 dept):
```
BOT_POLLING_DISABLED=true
```

6. 全网 dept 升 v3.1.9 hotfix(走 `hotfix_v3.1.8.sh` 同款模式,只 recreate tg-monitor)

## 原因

1. **bot 实例仍存活**:start() 提前 return 只跳过 polling,`Bot(token=...)` 实例仍可调 sendMessage / editMessage(预警推送 fallback 本地路径仍可用)
2. **默认 false 向后兼容**:老 dept 升级零感知,只有运维主动配 `BOT_POLLING_DISABLED=true` 才切换
3. **callback 走 webhook 鉴权**:中央台 → dept `/api/v1/callback` POST 用 METRICS_TOKEN(Bearer),已存在 endpoint(`web.py:2958`)
4. **claim-first 防并发**:中央台 `_handle_callback` v0.17 Codex P1 已加 `mark_pending_callback_handled` 原子抢占,两人同时点不会重复调 dept

## 来源

- 2026-05-19 升级陈家碧 13 dept 时发现 4 dingfenggs* 共用 BOT_TOKEN
- 客户拍板"不要再申请一堆 BOT,治根"
- 复用 v3.0.22 / ADR-0037 既有设计

## 后果

**代码改动**(超小):
- `config.py`:+7 行(加字段 + 注释)
- `bot.py`:+5 行(start 加 if 分支 + 注释)

**升级路径**:
- 客户跑 `hotfix_v3.1.8.sh`(脚本通用,改 TARGET_TAG=v3.1.9)→ 只 recreate tg-monitor,HTTPS 零变动
- 共用 token 的 dept 在 `.env` 加 `BOT_POLLING_DISABLED=true`
- 中央台 AWS 改 `.env CALLBACK_LISTENER_ENABLED=true` + 重启

**风险与边界**(Codex round1 P0/P1 抓出):

1. **按钮失效场景**(BOT_POLLING_DISABLED=true 时):
   - 中央台路由通(_try_central_route 返 True)→ 按钮由中央台 cb:<pending_id>:<action> 格式发出,中央台 callback_listener polling 接到 → POST 回 dept /api/v1/callback ✅
   - 中央台路由不通(网络挂 / CENTRAL_PUSH_URL 没配 / 路由 not found)→ fallback 本地预警,**`_make_keyboard*` 守护返 None,纯文本无按钮发出**。运维看到无按钮预警就知道要人工处理 + 修中央台
   - **send_no_reply_alert 单段路径**(`bot.py:575`,只在账号没配 business_tg_id 时触发):没接入 _try_central_route,直接本地。BOT_POLLING_DISABLED=true 时同样纯文本无按钮。两段式 stage1+stage2 是主路径,单段实际很少触发(配 business_tg_id 的账号会进两段式)

2. **`/start` `/bind` `/chatid` 命令失效**:
   - `/chatid`:客户在 web 后台设置精灵手填 chat_id(已有 UI)
   - `/bind` 暂无 web UI 替代,启用 flag 前确认无新管理员需要 TG 绑定(后续版本补 web 端绑定流程)

3. **中央台 callback_listener 异常停** → 共用 token 的 dept 按钮全失效(独立 token dept 不受影响)
   - 缓解:中央台 fleet_health daemon(v0.21)已有,挂会推警

**回滚**:`.env` 删 `BOT_POLLING_DISABLED=true` → dept 恢复 polling(老行为)

## Codex round1 处理记录

- P0 send_no_reply_alert 单段路径按钮失效 → 通过 `_make_keyboard` 守护返 None 改成纯文本预警避免「看得到点不动」(运维看无按钮 = 中央台异常 / 单段路径,要人工处理)
- P1 fallback 本地按钮失效 → 同上,`_make_keyboard*` 守护统一处理
- P1 `/bind` 无 web 替代 → 文档化警告,留后续版本补
