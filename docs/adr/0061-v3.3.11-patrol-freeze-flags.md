# ADR-0061: v3.3.11 — Patrol freeze flags for safe backlog flush

## 背景

2026-05-19 全网巡检发现 24/47 dept 有 Sheets 写入积压(`pending` 50-3634 条),最严重的恒丰二部积压 3634 条 4 天没写。

根因排查:
- SA(Service Account)已部署在 42 dept(分配到 4 个 SA)
- Google Sheets API per-user 配额硬上限 60 read/min(付费才能调,但 per-user 上限就是 60 不让升)
- 4 个 SA 服务 42 dept → 每 SA 60/min 被多 dept 共享 → 任何读 patrol 都 429
- 结果:**写入 backlog 被读 patrol 挤压,持续无法消化**

但如果直接关掉 patrol 大量消化 backlog,会撞另一个风险:**串号 / 顺序错**。

## 串号 / 顺序错三大风险源

| # | 风险 | 根因 |
|---|---|---|
| 1 | **广告主串号** | `messages.col_group=NULL` 孤儿消息 flush 时跨 peer 列边界 |
| 2 | **外事号串号** | `dedupe_assign_sheet_tabs` 在 backlog 期间改 `accounts.sheet_tab` → 老 pending messages 写到新 tab(可能是别号占用) |
| 3 | **顺序乱** | `_sheet_position_resync_loop` 跟 flush 撞期 → `next_sheet_row` 算错 |

## 决策

v3.3.11 加 **patrol freeze flag** 让运维能临时冻结会改 Sheet 布局的 3 个 patrol,安全消化 backlog 后再开回。

| Flag | 默认 | 控制 patrol |
|---|---|---|
| `SHEET_RESYNC_ENABLED` | ON(已有,v3.1)| `_sheet_position_resync_loop` |
| `PEER_NAME_CONSISTENCY_DISABLED` | OFF(已有,v2.10.24.1)| `_peer_name_consistency_loop` |
| **`SHEET_DEDUPE_PATROL_ENABLED`** | **ON**(本版新增) | `sheets.ensure_account_tabs` 启动 + 60s patrol 调 dedupe |

## 实施

`config.py`:
```python
SHEET_DEDUPE_PATROL_ENABLED = os.environ.get("SHEET_DEDUPE_PATROL_ENABLED", "true").lower() == "true"
```

`sheets.py:ensure_account_tabs`:
```python
if getattr(config, "SHEET_DEDUPE_PATROL_ENABLED", True):
    dedupe_assign_sheet_tabs(db.get_conn())
else:
    logger.info("[sheet_dedupe] SHEET_DEDUPE_PATROL_ENABLED=false, 跳过 dedupe 保护 backlog")
```

默认 ON(向后兼容,正常生产场景需要 dedupe)。

## 临时冻结流程(运维)

backlog 消化场景:
1. `.env` 加:
   ```
   SHEET_DEDUPE_PATROL_ENABLED=false
   PEER_NAME_CONSISTENCY_DISABLED=true
   SHEET_RESYNC_ENABLED=false
   ```
2. 重启 `tg-monitor` 容器
3. 跑 SQL 修 `messages.col_group=NULL` 孤儿:
   ```sql
   UPDATE messages SET col_group=(SELECT p.col_group FROM peers p WHERE p.id=messages.peer_id)
   WHERE sheet_written=0 AND col_group IS NULL;
   -- 修不上的(peer 也没 col_group)→ 标永不写避免串号
   UPDATE messages SET sheet_written=2 WHERE sheet_written=0 AND col_group IS NULL;
   ```
4. 让 backlog 消化(`_alert_writeback_loop` + `_sheets_flush_loop` 走原 path)
5. backlog 清空后,3 个 flag 改回默认 + 重启

## 原因

- **数据安全 > 实时性**:积压期间冻结布局变动比串号事故代价小得多
- **Feature flag 默认安全**:默认 ON 保持现有行为
- **无 schema 变化 / 0 数据迁移**:纯加 flag,客户升级零感知

## 来源

- 2026-05-19 用户反馈 Sheets 积压报警 + 串号 / 顺序错担忧
- 全网 audit 显示 4 SA 共享 42 dept,quota 撞顶,关 patrol 是唯一短期可行路径

## 后果

- ✅ 运维有冻结开关可临时止血
- ✅ 不撞 dedupe / resync / peer_name patrol 触发的布局变动
- ⚠ 冻结期间客户改归属 / peer 改名不实时同步,但 backlog 消化优先
- 长期:仍需 GCP 升 quota(付费)或多 SA(per-dept SA)永久解
