# ADR-0025: v3.0.8.3 重启 / 诊断容器名 fallback — 解决 .env COMPANY_NAME 跟实际容器名对不齐

## 背景

某客户部门 (URL: `dingfengg s2 -yunyingzx.187.127.113.127.nip.io`) 升级 v3.0.8.2 后:

1. 账号管理页显示**「会话已吊销」**(实际 TG 设备列表里 session 还活着 → false positive,session_health get_me() 一次失败误判)
2. 客户点「立刻重启监听器」按钮 → **404 Client Error No such container: tg-monitor-dingfenggs1-yunyingzx**

注意 URL 里是 `gs2`,容器名找的是 `gs1`。差一个字符。

按规矩查代码定根因:

`web.py:2229 /api/restart`:
```python
container_name = "tg-monitor-" + config.COMPANY_NAME
container = client.containers.get(container_name)  # NotFound 直接 throw
```

硬依赖 `.env COMPANY_NAME` 拼容器名。但客户部门的状态:
- `.env COMPANY_NAME = dingfenggs1-yunyingzx` (老,设置页锁的不能改)
- 实际 docker compose project name = `dingfenggs2-yunyingzx` (新)
- Caddy URL/site block = `dingfenggs2-yunyingzx.187...nip.io`

**部署遗留 inconsistency**: 客户在某个时点改了 docker compose project (or 重新部署用了新名)但 `.env COMPANY_NAME` 仍是老值。设置页 UI 把 COMPANY_NAME 设计成**安装时定死锁定**(因为改了会破坏 docker compose 引用),客户没法从 web 修。

更糟: 同一个 web.py 文件里 `_start_tg_monitor()` (line 450, 账号增删 auto-restart 用) **早就有 fallback**:
```python
try: client.containers.get(f"tg-monitor-{company}")
except NotFound:
    for c in client.containers.list(all=True):
        if c.name.startswith("tg-monitor-"): c.restart(); ...
```

但 `/api/restart` (line 2229) **没复用** — 重复实现 + 没 fallback。两个重启路径不一致。

`dashboard_api.py:671 _diagnose_sheets_stuck` 同样硬拼,COMPANY_NAME 不对就**整个诊断失败**(扫不到 log → 走 except → "无法自动诊断")。

## 决策

v3.0.8.3 hotfix, 2 处改动:

### 0. `_start_tg_monitor()` fallback 升级到 compose label-based (Codex P1 修)

Codex 审阅指出原 prefix `tg-monitor-*` fallback 在**多部门 VPS** 不安全 — 会误重启别人的容器。

新 fallback 链 (3 级):
1. **`tg-monitor-{COMPANY_NAME}`** (老路径, 大多数客户单部门 VPS 一发即中)
2. **compose project label fallback** — `socket.gethostname()` 拿当前 web 容器, 读 `com.docker.compose.project` label, 查同 project 内 `service=tg-monitor` 的兄弟容器。**多部门 VPS 安全 — 永远只 restart 自己 compose project 的兄弟**
3. **prefix fallback** — 最后兜底 `tg-monitor-*` 第一个 (compose label 都没有的极端异常部署)

`dashboard_api._diagnose_sheets_stuck` 应用同策略。

### 1. `web.py /api/restart` 改用 `_start_tg_monitor()`

```python
@app.route("/api/restart", methods=["POST"])
@login_required
def restart_monitor():
    ok, msg = _start_tg_monitor()
    if ok:
        return jsonify({"ok": True, "msg": msg})
    return jsonify({"ok": False, "error": msg})
```

**0 新代码** — 直接复用现有 fallback 逻辑(line 450-471)。

### 2. `dashboard_api.py _diagnose_sheets_stuck` 加 fallback

```python
primary = f"tg-monitor-{config.COMPANY_NAME or 'default'}"
try:
    c = client.containers.get(primary)
except docker_sdk.errors.NotFound:
    cands = [x for x in client.containers.list(all=True) if x.name.startswith("tg-monitor-")]
    if not cands:
        return "warning", f"⚠ 积压 {pending} 条,无法自动诊断(找不到 tg-monitor 容器)", None
    c = cands[0]
```

跟 `_start_tg_monitor` fallback 一致策略。

### 3. **不动** 设置页 COMPANY_NAME 锁定逻辑

设置页 UI 仍写"已锁定(安装时定死,不可修改)"。理由:
- 改 COMPANY_NAME 会影响 docker compose project name / Caddy site block / install dir / sessions 路径等多处,没 web 端能改全的安全方案
- 这次 hotfix 是让系统**容忍 inconsistency**,不是支持客户改

但 ADR 留个记号 — 真正完美的方案是 **v3.1+ 重新设计部署生命周期**: install → rename / migrate flow → unlock / re-lock。当前 hotfix 是 pragmatic 治标。

## 原因

1. **客户当下卡死立刻不可用**: `/api/restart` 404 = 会话已吊销 false positive 没法清,客户**没任何纯 web 路径救自己**(设置页 COMPANY_NAME 锁的)
2. **0 风险 0 新代码**: 复用 `_start_tg_monitor()` 现成 fallback 逻辑,只是换调用入口
3. **跟 v3.0.8 治本路径互补**: v3.0.8 治 Sheets 写入卡死(append + 令牌桶 + 深度诊断 + 一键修),v3.0.8.3 治"想救 Sheets 但容器名都拼不对"的更深层运维 inconsistency

- 用户问: 「这里是不能改的啊」 (设置页 COMPANY_NAME 锁定字段截图)

## 后果

### 正面
- 客户立刻能在驾驶舱「立刻重启监听器」按钮救自己 — 不再卡 COMPANY_NAME 拼错
- 「会话已吊销」false positive 可以通过重启清掉 (重启后 session_health 重新探测,真活的 session 会自动转绿)
- 诊断卡片在 COMPANY_NAME 错的部门也能正常诊断 Sheet 状态
- 任何后续 部署遗留 inconsistency (compose project rename / vps 迁移) 都不再卡

### 潜在问题 / trade-off
- **多 dept VPS 共享 web 容器场景**: 极罕见 (每个 dept 应该独立 docker compose project / 独立 web 容器), 但如果发生 fallback 会 restart 第一个找到的 tg-monitor-*, 可能不是预期那个。当前 _start_tg_monitor 的 fallback 已是这种行为, v3.0.8.3 没引入新风险
- **隐藏了真正的部署 inconsistency**: 客户的 .env COMPANY_NAME 跟实际容器对不齐, 是更深层的部署历史问题。这次 fallback 是 workaround, 不是修。**v3.1+ 的部署 lifecycle 重新设计才能根治**

### 未涉及 (v3.1+)
- 部署生命周期重新设计: install → rename → migrate → 解锁 COMPANY_NAME 修改路径
- `_start_tg_monitor` fallback 改用 docker compose project label (当前是 prefix match, 跟 v3.1 lifecycle 设计一起做)
- session_health get_me() false positive: 加重试 (1 次失败再试 1 次, 都失败才标吊销) — 留 v3.0.9
- /api/restart 跟 _start_tg_monitor 真正合并到一个函数 (现在是后者复用前者, 但后者还有别处 caller) — 留 v3.0.9 重构

## 升级指引

```bash
cd /root/tg-monitor-<部门> && ./update.sh
```

升级后:
- 任何 `.env COMPANY_NAME` 跟实际容器名对不齐的部门, 「立刻重启监听器」按钮自动 fallback 到本机任意 tg-monitor-* 容器
- 「Sheet 写入诊断」也不再因为找不到容器而 fail
- 设置页 COMPANY_NAME 锁定行为 **不变** (本版不改这个)

## 验证

### 静态自查
- `grep -nE "client.containers.get.*tg-monitor" web.py dashboard_api.py` — restart_monitor 应该不再直接调, 改 _start_tg_monitor; dashboard_api 有 try/except + fallback
- `grep -A3 "@app.route.*/api/restart" web.py` — 函数体应该是 `ok, msg = _start_tg_monitor()` 简洁版
- Python syntax OK + JSON OK

### 集成 (dev 容器)
1. **正确容器名场景**: COMPANY_NAME 跟容器名一致 → /api/restart 正常 restart 原容器, msg 包含正确容器名
2. **错配场景** (mock):
   - 部署一个 tg-monitor-foo 容器, 把 .env COMPANY_NAME 改成 bar
   - 点「立刻重启监听器」→ 应该返 `ok:true, msg: 'tg-monitor-foo 已重启 (建议手动 docker compose up -d --build 重命名为 tg-monitor-bar)'`
3. **完全找不到 tg-monitor-* 场景**: 删所有 tg-monitor-* 容器, 点重启 → 应该返 `ok:false, error: '找不到 tg-monitor 容器,请手动 docker compose up -d --build'`
4. **Sheets 诊断错配**: 同 #2 场景下, Sheet 卡死时驾驶舱诊断卡片应该能扫到 fallback 容器的 log, 正常诊断 OAuth/429/etc

### 回归
- v3.0.8 + v3.0.8.1 + v3.0.8.2 全部功能不变
- _start_tg_monitor 现有 caller (账号增删 auto-restart) 行为不变, 只是 /api/restart 也复用它
- 设置页 COMPANY_NAME 字段 UI 不变 (仍锁定)
