# ADR-0030 · v3.0.11 · WORK_SCHEDULE 默认时段调整

## 背景

2026-04-30 客户反馈:工作时段太长,业务员晚上 22 点后还在被推未回复告警;
周六 11 点前就被打扰,业务员还没开工。

要求调整默认 WORK_SCHEDULE:
- 周一~周五 晚段:从 20:00-23:00 → 20:00-22:00(下班早 1 小时)
- 周六:从「11-13 / 15-20」(2 段) → 只「15-19」一段(简化)
- 周日:仍全休(不变)

## 决策

直接改 `config.py::WORK_SCHEDULE` hardcoded 默认值。所有 7 部门客户都用同一套
工作时段,无需 per-customer 配置。

升级后下一条客户消息进来,`config.is_work_time()` / `work_elapsed_minutes()`
就用新时段算未回复累计分钟。

未回复告警触发 `_no_reply_loop` 已经按工作时段累计,自动适配,无代码逻辑变化。

## 实现

`config.py:260-268`:

```python
WORK_SCHEDULE = {
    0-4: [(11,0,13,0), (15,0,19,0), (20,0,22,0)],  # 周一~五,晚段缩 1h
    5:   [(15,0,19,0)],                             # 周六只 15-19
    6:   [],                                         # 周日休
}
```

## 原因

1. **客户主动反馈** — 不是猜测的运维需求
2. **统一调整** — 7 个客户都同一套,不需要 per-VPS 配置增加复杂度
3. **0 改动业务逻辑** — `is_work_time` / `work_elapsed_minutes` 函数完全
   不动,只改数据
4. **零向后兼容风险** — 升级后行为变化是预期内的(新时段更短),已经在范围内
   没有 active 的未回复告警时窗

## 来源

2026-04-30 客户口头反馈:「周一到周五晚间改成 20:00~22:00,周六改成
15:00~19:00」

## 后果

- 升级后未回复推送量预估**进一步减少**(在 v3.0.10 略过无意义消息基础上,
  晚 22 点后 + 周六上午都不再被打扰)
- **副作用**:如果客户在新时段外发紧急消息(比如周六上午),业务员不会立刻
  收到未回复告警提醒。客户须自行权衡。
- 想恢复老时段:目前只能改代码 + 重发版本(WORK_SCHEDULE 不在 .env 暴露)。
  下个版本可加 `.env::WORK_SCHEDULE_JSON` 让客户自定义。

## 测试

- `is_work_time(2026-05-01 周五 22:30)` → False(原 True)
- `is_work_time(2026-05-01 周五 21:30)` → True
- `is_work_time(2026-05-02 周六 12:00)` → False(原 True)
- `is_work_time(2026-05-02 周六 16:00)` → True
- `is_work_time(2026-05-02 周六 19:30)` → False(原 True,被周六晚段砍掉)
- `is_work_time(2026-05-03 周日 任意)` → False(不变)
