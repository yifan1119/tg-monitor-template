# ADR-0054 · v3.1.8 档案群统一中央台路由(按 account.company 分发)

## 背景

5 类预警(关键词 / 未回复 stage1/2 / 删除 / Session 吊销)v3.0.16 后都走中央台 `_try_central_route`,按 `account.company` 路由到对应公司预警群。但**档案群(媒体存储)**仍按 dept 本地 `.env MEDIA_ARCHIVE_GROUP_ID` 写死,**没统一**:

- 同一公司多个 dept(例如鼎丰 5 个运营 VPS)各自写自己 archive 群 → 客户审查时要看 5 个群
- 改 archive 群只能 SSH 改每个 dept .env + restart
- 客户多公司部署时,跨公司媒体可能写错群(.env 配错)

客户拍板:档案群也统一,跟预警一样按 `account.company` 路由。

## 决策

**配置中心化,执行仍本地**(中央台 bot 没在 dept 私聊里,媒体执行只能 dept 本地):

### 中央台(`tg-central-dashboard v0.40`)

加 endpoint `GET /api/v1/archive_route?company=<>`:
- 鉴权:Bearer `<dept metrics_token>`(跟 SA 拉取同款)
- 查 `alert_routes.archive_chat_id`(已存在字段,admin UI 一直可填)
- 返回 `{"ok": true, "archive_chat_id": "<chat_id>", "found": true/false}`

### tg-monitor(`v3.1.8`)

`media_uploader.py`:
1. 加 `_lookup_central_archive_chat_id(company)` helper:
   - 调中央台 `/api/v1/archive_route?company=<account.company>`
   - 60s TTL cache(避免每条媒体打中央台)
   - 网络挂 / 中央台没配 / 路由没找到 → 返 `None`
2. `forward_to_tg_archive` 改:
   - 先查中央台 chat_id(优先)
   - 没拿到 → fallback `config.MEDIA_ARCHIVE_GROUP_ID`(向后兼容)
3. `_archive_deep_link(archive_msg_id, chat_id=None)` 加参数:
   - 用实际写入的 chat_id(可能是中央台路由的,跟本地不同)生成 t.me/c 深链
   - 没传 chat_id → 老路径(读 config,兼容)

### Bot 选择

dept 本地 `_archive_bot`(老 BOT_TOKEN)发媒体到中央台返的 chat_id。
**运维要求**:dept 的 archive bot 必须被加进对应公司 archive 群(一次性运维步骤)。

## 原因

1. **中央台 bot 不能跨群 forward 媒体**:中央台 bot 没在 dept 私聊里,Telethon 收到的 message 只有 dept 自己 client 能 forward。所以执行必须 dept 本地。
2. **chat_id 路由可以中央台**:跟 alert_chat_id / unreplied_chat_id 一样,都按 company → 中央台 alert_routes 查。
3. **60s cache**:dept 媒体高频,每条调中央台浪费。同公司 chat_id 60s 内不会变,cache 合理。
4. **fallback 兼容**:中央台没配某 company 的 archive_chat_id → 退本地 .env(向后兼容,老 dept 不破)。
5. **deep_link chat_id 传参**:中央台路由的 chat_id 可能跟本地 .env 不同,t.me/c 深链必须用实际写入的群 ID 才能跳对。

## 来源

- 客户反馈(2026-05-19)需要档案群跟预警一样按公司维度路由
- 复用 v0.16 alert_routes.archive_chat_id 字段(原本预留,push_alert 没读)

## 后果

**改动 2 仓**:
- 中央台 `app.py` 加 `/api/v1/archive_route` endpoint(~30 行)
- tg-monitor `media_uploader.py` 加 lookup helper + 60s cache + `forward_to_tg_archive` 优先中央台 + `_archive_deep_link` 加 chat_id 参数(~80 行)

**升级路径**:
- 中央台 push v0.40 → AWS deploy
- tg-monitor 全网升 v3.1.8 → 媒体自动走中央台路由
- 客户在 `/settings/alert_routes` 给每个公司填 `archive_chat_id`(已有 UI,只是没读)
- 把每 dept 的 archive bot 加进对应公司 archive 群

**风险**:
- 中央台 down → fallback 本地 .env,业务不中断(已 try/except)
- 60s cache stale → 改 chat_id 后 60s 内 dept 还用旧值,可接受
- 客户没把 dept bot 加进新公司 archive 群 → bot.send_photo 报错 → 跟之前 .env 配错群同款症状

**回滚**:tg-monitor `.env` 不需要改;客户 / 运维想退回老行为 → 中央台 `alert_routes.archive_chat_id` 清空 → dept 自动 fallback 本地 .env。
