# ADR-0010: 预警分页整行缺失自动 writeback(无限重试保零丢失)

**版本**:v2.10.24.3(hotfix 续,承接 v2.10.24.2)
**日期**:2026-04-22
**状态**:Accepted

## 背景

v2.10.24.2 修好了「预警分页 A/B 栏空白(商务人员/所属公司同步不到)」的历史脏数据,但没有解决另一个更严重的零丢失问题:**整行预警数据缺失**。

### 问题链路(代码层面)

三个预警分页的写入路径(`bot.py`):

| 类型 | 写入时机 | 代码位置 | 失败策略(v2.10.24.2 之前) |
|---|---|---|---|
| `keyword` | 命中即写 | `bot.py:185-198`(3 retry × 2s) | 3 次都失败 → 静默 → **整行永久缺失** |
| `no_reply` | 审核通过时写 | `bot.py:_write_alert_to_sheet`(3 retry × 2s) | 同上 |
| `deleted` | 审核通过时写 | 同上 | 同上 |

所有三个路径都是 **3 次 retry,每次间隔 2 秒**,即**共 6 秒**的容错窗口。一旦 Google Sheets API 出现任一情况超过 6 秒:

- **429 配额爆**(`sync_headers` 旧 60s 间隔时代客户 150+ 账号 30 秒就撞配额)
- **连接短暂抖动 / DNS 闪断**
- **`worksheet()` 内部的 `fetch_sheet_metadata` RPC 暂时失败**

整行数据就会永久消失 —— 客户视角「预警推送有,但表格没这一行」。客户原话:「**不能丢失啊**」。

### 代码层面确认

`bot.py:142/147/190` 原三处都是 `try/except ... log error + sleep 2 + retry;3 次后 return`,没有**任何**持久化失败标记,意味着第 3 次失败后这条预警与分页的联系**彻底断开**,之后即便系统完全恢复也没有任何 loop 会回来补这一行 —— 因为没有任何数据指针「这行该写但没写」。

相比之下,**消息分页**在 v2.10.23(ADR-0001)就已经有 `messages.sheet_written` 标记位 + `_sheets_flush_loop` 无限重试模式,从未出现丢消息。**预警分页是同一个机制的盲区**,必须按 messages 同样思路补齐。

## 决策

**采用「`alerts.sheet_written` 标记位 + `_alert_writeback_loop` 无限重试」方案**(类比 `messages.sheet_written` + `_sheets_flush_loop`)。

### 1. DB 层(`database.py` + migration `_migrate_to_2`)

```sql
-- alerts 表新增:
ALTER TABLE alerts ADD COLUMN sheet_written INTEGER DEFAULT 0;
ALTER TABLE alerts ADD COLUMN keyword TEXT DEFAULT '';
ALTER TABLE alerts ADD COLUMN last_write_error TEXT DEFAULT '';
CREATE INDEX idx_alerts_sheet_written ON alerts(sheet_written);

-- v2.10.24.3 前历史 alerts 全部标 1(不追补,仅保障升级后的零丢失):
UPDATE alerts SET sheet_written = 1 WHERE sheet_written = 0;
```

`user_version` 从 1 升到 2。幂等(`_safe_add_column` + `CREATE INDEX IF NOT EXISTS`)。

**新 helper**(经 Codex round2/round3 修正后最终形态 —— claim-first 三态):
- `insert_alert(..., keyword='')` — 新增 `keyword` 参数,keyword 类型的关键词独立存
- `claim_alert_for_sheet_write(alert_id)` — **append 前**原子抢写权
  `UPDATE sheet_written=1, claimed_at=now WHERE id=? AND (sheet_written=0 OR claimed_at < stale_threshold)`,
  rowcount=1 = 本调用写;=0 = 别路径正在写 / 已写完,跳过
- `mark_alert_sheet_done(alert_id)` — append 成功后标 final done(清 `claimed_at`)
- `rollback_alert_sheet_claim(alert_id, err)` — append 失败,把 claim 回滚(sheet_written=0 + claimed_at=NULL)
- `record_alert_write_error(alert_id, error)` — 记录失败原因(截 500 字)
- `get_unwritten_alerts(limit=100)` — 捡 `sheet_written=0` + stale claim(`sheet_written=1 AND claimed_at < stale`)
- `count_unwritten_alerts()` — 积压量统计(含 stale claim)

**三态**:
- `sheet_written=0` → 未写(可 claim)
- `sheet_written=1 + claimed_at != NULL` → 进行中(超过 stale 阈值可重 claim)
- `sheet_written=1 + claimed_at IS NULL` → 已写完(writeback 不再触碰)

历史 alerts 在 `_migrate_to_2` 一刀标 sheet_written=1,`_migrate_to_3` 新加的 `claimed_at` 默认 NULL → 天然视为"已写完",不被误拾。

兼容别名:`mark_alert_sheet_written(alert_id)` = `mark_alert_sheet_done(alert_id)`,维持旧 API。

### 2. 写入路径层(`bot.py` 三处)

**所有**三个路径统一用 claim-first 三段式:

```python
if not db.claim_alert_for_sheet_write(alert_id):
    return  # 别路径已 claim 或已写完,跳过
append_ok = False
for attempt in range(3):
    try:
        ws.append_row(...)
        append_ok = True
        break
    except Exception as e:
        last_err = e
        ...
if append_ok:
    db.mark_alert_sheet_done(alert_id)          # 清 claimed_at = final done
else:
    db.rollback_alert_sheet_claim(alert_id, e)  # 回 sheet_written=0 让 writeback 接力
```

**`send_keyword_alert` 顺序反转**(v2.10.24.3 关键改动):

```python
# v2.10.24.3 前:写 Sheet → 成功/失败 → 最后才 insert_alert(Sheet 写失败 alert_id 都没,永久丢)
# v2.10.24.3 后:insert_alert(sheet_written=0) → claim → 写 Sheet → done / rollback
alert_id = db.insert_alert("keyword", ..., message_text=text, keyword=keyword)
# TG 推送(失败只 log,不阻断 Sheet 写入流程)
try: await bot.send_message(...)
except Exception as e: logger.error(...)
# Sheet 写入(claim-first,失败走 writeback loop)
if db.claim_alert_for_sheet_write(alert_id):
    try:
        ws.append_row(...)
        db.mark_alert_sheet_done(alert_id)
    except Exception as e:
        db.rollback_alert_sheet_claim(alert_id, e)
```

`_write_alert_to_sheet`(no_reply / deleted approved 路径)同结构 —— claim → append → done / rollback。

### 3. Loop 层(`sheets.py` + `tasks.py`)

**`SheetsWriter.writeback_pending_alerts(limit=50)`**:

```python
rows = db.get_unwritten_alerts(limit=50)  # 含 stale claim
for row in rows:
    if not db.claim_alert_for_sheet_write(row.id):
        continue  # 别路径抢到 / 已完成
    try:
        ws = self.spreadsheet.worksheet(tab_name_for_type[row.type])  # 同一轮缓存
        self._rate_limit()
        ws.append_row([...按预警类型拼列...])
        db.mark_alert_sheet_done(row.id)
    except Exception as e:
        db.rollback_alert_sheet_claim(row.id, e)
        if "429" in str(e): break  # 本轮截断,下轮 loop 再来
```

**`_alert_writeback_loop`** 加到 `tasks.py` 的 `asyncio.gather`(第 11 个 loop):

```python
await asyncio.sleep(30)  # 启动后等 30 秒,让 sheets/DB 初始化
while self._running:
    if not config.ALERT_WRITEBACK_DISABLED and self.sheets:
        self.sheets.writeback_pending_alerts()
    await asyncio.sleep(config.ALERT_WRITEBACK_INTERVAL_SEC)  # 默认 60s
```

### 4. 配置(`config.py`)

```python
ALERT_WRITEBACK_DISABLED = os.environ.get("ALERT_WRITEBACK_DISABLED", "false").lower() == "true"
ALERT_WRITEBACK_INTERVAL_SEC = int(os.environ.get("ALERT_WRITEBACK_INTERVAL_SEC", "60"))
# Codex round3 P0:claim crash stale 回收阈值(默认 300s = 5 min)
ALERT_WRITEBACK_CLAIM_STALE_SEC = int(os.environ.get("ALERT_WRITEBACK_CLAIM_STALE_SEC", "300"))
```

- 默认**开**(客户零操作升级即生效)
- 可关 —— 异常场景回滚方便

## 原因

### 为什么跟 `messages.sheet_written` 走同一个模式

v2.10.23(ADR-0001)已经在 messages 层用 `sheet_written` 标记位 + `_sheets_flush_loop` 无限重试跑了 3 个多月,**实测零丢消息**。这个模式:
- 简单(一个 int 栏位 + 一个 loop)
- 可观测(SELECT 一下就知道积压)
- 自愈(429 退了 loop 会补)
- 幂等(mark=1 后不会再被拾起)

预警分页是完全同构的问题,照搬现成模式最稳。

### 为什么不直接用 `messages.sheet_written` 扩展

`messages` 表的 sheet 指的是「账号分页」,用 `messages.sheet_row` 精确定位列号。**预警分页**是三张独立的汇总分页(`信息未回复预警<部门>` / `关键词监听<部门>` / `信息删除预警<部门>`),行是**追加**的,没有精确的 row 概念 —— 每次 `append_row` 写尾部。语义彻底不同,必须用 `alerts` 表的独立标记。

### 为什么新加 `keyword` 栏位而不是复用 `message_text`

原 `keyword` 类型的 `message_text` 存 `f"[{keyword}] {text}"`(`bot.py:200` 历史代码)。writeback 要把「关键词」「消息内容」拆成两列写入分页:

| 选项 | 缺点 |
|---|---|
| 从 `[kw] text` 字符串里拆 `]` | 如果 `text` 本身含 `]` → 拆错 → 脆弱 |
| 新加 `keyword` 栏位 | 干净可靠,历史 `message_text` 不动 |

选后者。历史 alert 的 `message_text` 保持 `[kw] text` 格式(已 `sheet_written=1` 不会被 writeback 重写),新插入的拆开存。

### 为什么 `send_keyword_alert` 要反转顺序

v2.10.24.3 之前:
```
写 Sheet → 成功/失败 → 最后 insert_alert
```

如果写 Sheet 失败 → 没 `alert_id`,后面 `insert_alert` 即使成功,那行也 **没关联到** Sheet 写入失败事件 → writeback 层无从拾起。

v2.10.24.3 后反转为 「先 insert_alert 拿 id → 再写 Sheet」,这样:
- 写 Sheet 失败 → `sheet_written=0` → loop 接力
- 即使 `insert_alert` 后 process crash(极端)→ 下次重启 loop 也会拾起(因为 `sheet_written=0` 持久化在 DB)

### 为什么历史 alerts 一刀标 sheet_written=1(不追补)

如果标 0 → writeback 会把**升级前一年**的所有预警重新 `append_row` 到分页尾部 → 客户看到表格多出几千行重复记录 → 灾难。

v2.10.24.2 的 `backfill_alert_history` 只补 **A/B 栏**(空的公司/商务)—— 那是往**已有行**里填空,幂等安全。但 writeback 是 `append_row` **新增行**,不能对历史运行。

客户的具体场景是 「升级后新产生的预警零丢失」,不要求「把历史漏写的都翻出来补」(历史到底有多少漏写不好追溯,盲补风险太大)。一刀标 1 = 从升级那一刻起零丢失。

### 为什么改用 claim-first + stale 回收(Codex round 2 + round 3 修复)

**round 1 原决策**:「简单优先,不做 claim」——「写完再 mark」

**round 2 驳回**(Codex Major A):「append 成功 → mark 失败」会让 `sheet_written` 停留在 0,下轮 loop 再写一次 Sheet → **重复行**;mark 的 SQLite OperationalError 并非小概率(高并发 / VACUUM / checkpoint 时有)。

**round 2 新决策**:claim-first —— 把 `UPDATE sheet_written=1` 前移到 append 前。

**round 3 驳回**(Codex P0):简单的 claim-first「claim commit 后 append 前 crash/OOM/SIGTERM」→ sheet_written=1 卡死,writeback 永不再拾 → **永久漏一行**。不满足「零丢失」承诺。

**round 3 最终决策**:**claim-first + stale 回收**(三态)

1. `alerts.claimed_at TEXT DEFAULT NULL` 新加列(`_migrate_to_3`)
2. claim SQL:
   ```sql
   UPDATE sheet_written=1, claimed_at=now
    WHERE id=? AND (sheet_written=0 OR (sheet_written=1 AND claimed_at < stale_threshold))
   ```
3. append 成功 → `mark_alert_sheet_done` → `UPDATE claimed_at=NULL`(final done)
4. append 失败 → `rollback_alert_sheet_claim` → `UPDATE sheet_written=0, claimed_at=NULL`(立刻可重试)
5. `get_unwritten_alerts` 补 stale claim:`sheet_written=0 OR (sheet_written=1 AND claimed_at IS NOT NULL AND claimed_at < stale_threshold)`

**crash 场景行为**:
| 时机 | DB 状态 | 后果 |
|---|---|---|
| claim 后、append 前 crash | sheet_written=1 + claimed_at=<crash时> | stale(默认 5 min)后 writeback 重 claim → 重试写一行 |
| append 成功、mark 前 crash | 同上 | stale 后 writeback 再写一次 → **重复一行**(at-least-once 代价) |
| append 失败、rollback 前 crash | 同上 | stale 后 writeback 重试,成功则恢复(若 Sheet 已部分成功则重复一行) |

**权衡**:选「at-least-once 极端场景偶尔多写一行 vs 永久漏行」→ 选前者。零丢失承诺兑现。

**stale 阈值**:`ALERT_WRITEBACK_CLAIM_STALE_SEC` 默认 300s(5 min),远大于 bot 路径的 6s(3 retry × 2s) + asyncio.to_thread 开销,不会误拾正常进行中的 claim。

**保留兼容别名**:`mark_alert_sheet_written(alert_id)` → 等价于 `mark_alert_sheet_done(alert_id)`,维持旧 API。

### 为什么一轮限 50 条(不是 100 / 不限)

- 50 条 × 1.5s rate limit = 75 秒 API 消耗
- 默认 loop 间隔 60 秒 → 一轮 75 秒略超但不冲突(asyncio.sleep 是 loop 结束后才计)
- 配额:Sheets 写 60 writes/min/user → 50 条 < 60,安全
- 积压多 → 下一轮继续撈,4-5 轮清空 200 条

### 为什么 429 就整轮截断 loop 退避,不是单条退避

单条 429 退避会在本轮内连串多次 429 → 浪费。整轮截断 + 下个 loop 周期(60s)再试 → 天然退避,跟 v2.10.24.1(ADR-0008)的思路一致。

## 来源

- **客户需求**:「预警分页整行缺失(429 > 6 秒)... 这个能解决吗 不能丢失啊」(2026-04-22)
- **代码调查**:Grep `append_row.*预警|关键词监听` 找到三个写入路径,确认都是 3 retry × 2s 的 6 秒容错窗口
- **开发者分析**:对比 messages 的 `sheet_written` + `_sheets_flush_loop` 模式(v2.10.23 ADR-0001 已验证稳定 3 个月),发现 alerts 层是同样问题的盲区

## 后果

### 正面

- **零丢失保障**:429 > 6 秒 / 连接短暂不可达不再导致整行缺失,loop 无限重试直到成功
- **零操作升级**:客户 `./update.sh` 重启 → migration 自动跑 → 新预警自动走新路径
- **可观测**:`SELECT COUNT(*) FROM alerts WHERE sheet_written=0` 一下就知道积压;`last_write_error` 有错误原因
- **向后兼容**:
  - `.env` 零配置(两个新开关默认值 = 期望行为)
  - 历史 alerts 不动(一刀标 1)
  - DB migration 幂等(`_safe_add_column`)
- **代码架构一致性**:复用 `messages.sheet_written` + flush loop 的现成模式,容易理解维护

### 中性

- `alerts` 表多 3 列(`sheet_written` / `keyword` / `last_write_error`)
- 多一个 asyncio loop(第 11 个)
- 启动日志多一行可能:`writeback_pending_alerts: 完成 — 写入 N 行`(只在有积压时才 log)

### 需要后续观察

- **claim-first 极端重复**:claim commit 后 append 真正成功但 mark_done 前 crash → stale 阈值(5 min)后 writeback 重试 → 多写一行。理论罕见(SQLite WAL autocommit 级 UPDATE 基本毫秒级,且两次 UPDATE 之间只有一次 gspread append_row HTTP)。生产 1-2 周后对比 DB `sheet_written=1 且 claimed_at=NULL` 条数 vs Sheet 实际行数评估
- **keyword message_text 格式不一致**:历史是 `[kw] text`,新的是纯 `text`。不影响 Sheet 分页写入(writeback 是从 `keyword` 字段拿关键词,不是从 `message_text` 拆),但若未来有报表按 `message_text` 聚合得注意格式差异。
- **keyword `has_alert_today` 放宽**:v2.10.24.3 改成 keyword 类型只要 DB 里有今日行就算已处理(不像 no_reply/deleted 要求 `bot_message_id IS NOT NULL`)。因为 keyword 顺序反转后 TG 推送是「记 log 不阻断」,如果还要求 `bot_message_id IS NOT NULL` → TG 推送失败但 Sheet 写入成功 → 下次触发重插 + 重写 Sheet 重复行。keyword 无审批流,insert 即已处理。
- **keyword `has_alert_today` + `insert_alert` 非原子**(Codex round3 P1 遗留):并发两条同 peer keyword 命中可能各插一条各自 append → 多写一行。实际极低(Telethon 单 peer 秒内两次匹配极罕见),若生产报重复再加 `(type,peer_id,date)` 部分唯一索引。

### 不在本版做(后续可能)

- Web UI 加「立即 writeback」按钮(手动触发一轮)
- 积压超过阈值 → 推预警群(类似 `_sheets_backlog_loop` 但针对 alerts)
- Codex 审阅后若发现 P0 / P1 再加补丁
