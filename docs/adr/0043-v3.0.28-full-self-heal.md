# ADR-0043: v3.0.28 — 升级链一次根治 + 4 个新 agent action

## 背景

v3.0.16 ~ v3.0.27 一连串版本试图解决「客户升级后没生效」/「中央台没接通」问题,但每个版本只修一个症状,**根因复合**,客户始终卡在老版本。扫描 sheet 1W_L8DY 实际 44 台 VPS 版本分布:

| 版本 | 台数 |
|---|---|
| v2.10.22(2026-04-20 前) | 18 |
| v3.0.13(2026-04-30) | 14 |
| v3.0.14(2026-05-08) | 5 |
| v3.0.15(2026-05-09) | 2 |
| 不可达(SSL/网络) | 5 |

**核心问题 5 个已知根因**(每一个都阻塞升级):

1. **dept 容器 Dockerfile 没装 git** → agent.upgrade `subprocess.run(["git", ...])` 直接 ENOENT,远程升级永远失败
2. **update.sh `CENTRAL_PUSH_URL` 自动补逻辑 bug**:`! grep -q "^CENTRAL_PUSH_URL="` 只要这行存在(哪怕空值 `=`)就不覆盖 → 部分 VPS 永远接不上中央台
3. **客户 git working tree 卡在某个 feature 分支** → `git pull` 拉的不是 main → update.sh / README 永远不更新
4. **agent 缺关键运维 action**:没 `set_env` / `tail_logs` / `reload_oauth` / `fix_sheets` → 远程能看不能修,客户问题最终还是要 SSH
5. **agent.upgrade 单一路径**:只走容器内 git,无 fallback,根因 #1 + #3 任一发作就完全无法升级

## 决策

v3.0.28 一次性修这 5 个根因 + 加 4 个 action,**客户跑一次 update.sh 之后,所有已知问题永远不再出现**:

### dept 端改动

| 文件 | 改动 | 修哪个根因 |
|---|---|---|
| `Dockerfile` | `apt-get install` 加 `git` | #1 |
| `update.sh` | 强制 `git checkout main`(branch != main 时 force checkout) | #3 |
| `update.sh` | `CENTRAL_PUSH_URL` / `CENTRAL_PUSH_TOKEN` 自动补改用 `grep -qE "^...=.+$"`,空值也覆盖 | #2 |
| `agent.action_upgrade` | 加 fallback 路径:容器无 git → 起临时 `alpine:3.20` 容器,挂 docker socket + host_repo,apk add git docker-cli,跑完整 update.sh | #5(+ 让 v3.0.26/27 也能升 v3.0.28)|
| `agent.action_set_env` | 新增:改 `.env` 白名单 14 个 key + restart tg-web | #4 |
| `agent.action_tail_logs` | 新增:docker logs tg-monitor / tg-web / tg-caddy 最近 N 行 | #4 |
| `agent.action_reload_oauth` | 新增:走 v3.0.7 SheetsWriter.reload_credentials 路径,失败 fallback restart 双容器 | #4 |
| `agent.action_fix_sheets` | 新增:调 tg-web 容器内已有的 `/api/diag/sheets-fix-stuck` | #4 |

### 中央台 v0.20 配套

`/admin/fleet` UI 加 4 个按钮(单台 + 批量):查看 log / 改 .env / OAuth 重载 / 修 Sheets。`fleet.py` 后端透传到对应 dept agent action。

## 鉴权 / 安全

跟 v3.0.26 / v3.0.27 一致:Bearer token + HMAC + nonce + timestamp + rate limit + audit_log。

**`set_env` 永禁 14 个白名单之外的 key**:BOT_TOKEN / API_ID / API_HASH / WEB_PASSWORD / METRICS_TOKEN / SHEET_ID / GOOGLE_OAUTH_* 等,即使 token 泄露也改不了关键凭据。

**`upgrade` regex 校验 target_tag**:`^(main|v\d+\.\d+(?:\.\d+){0,2})$` 防 shell injection。

**alpine 临时容器**:`remove=True` 自动清理 + 仅挂 docker socket 和 host_repo,不挂别的路径。

## 原因

为啥这次不再「打地鼠」:
- 之前每版只修 1 个症状(self-check / docker cp 兜底 / strict mode 等),根因 #1-#5 同时存在时仍失败
- v3.0.28 一次修 5 个根因,**任何已知失败路径都被堵死**

为啥 alpine 临时容器路径必要:
- v3.0.26 / v3.0.27 客户 VPS 没装 git → 调 agent.upgrade 直接 ENOENT 死循环
- alpine 容器 + apk add git + 挂 docker socket + host_repo = 完全独立执行环境,**不依赖现有 tg-web 容器有什么**
- 之后所有升级路径都走这条,即使 image 又出 bug 也能远程修

为啥 4 个 action 是 MVP 必须:
- 看 log 是诊断必备(现在出问题完全 black box)
- 改 .env 是修配置必备(关键词/工作时间/中央台 URL 等)
- OAuth reload 是 Sheets 故障最高频痛点
- Sheets fix 包装现有 endpoint,远程一键即可

## 来源

- 用户反馈:「我已经推了好几个版本,一直叫他们更新,你是在耍我吗?」
- sheet 1W_L8DY 实际版本扫描数据(38/44 不在最新版,跨 7 个版本差距)
- 鼎丰公司02 实测 case:agent v3.0.26 调 upgrade → `[Errno 2] No such file or directory: 'git'`

## 后果

✅ 利:
- **客户配合最后一次**:跑 v3.0.28 update.sh 之后,所有后续运维(升级 / 改配置 / 看 log / 修 OAuth / 修 Sheets)全部远程一键
- **永远修了 5 个已知根因**:Dockerfile / update.sh / agent 三层都修
- **alpine fallback 路径**:即使未来 image 出问题,也能远程修复(不再死循环)

⚠ 风险:
- **客户必须真跑这次 update.sh**:已知 sheet 上 38/44 没升级,这次群发命令要严格说明
- **alpine 路径首次跑慢**:apk add git docker-cli 约 30-60 秒(image rebuild 也要时间)
- **set_env 改 .env 后 tg-monitor 不自动 reload**:只 restart tg-web,如果改的 key 影响 listener(如 KEYWORDS)需要 tg-monitor 也 reload — `config.reload_if_env_changed` 自动热加载,5-10 秒生效
- **alpine 临时容器有 docker socket**:任何能调 set_env / upgrade 的人也能创建容器(理论上),但白名单 + audit 兜底

## 验证

agent.py 单测全过:
- ✓ set_env 白名单防御(BOT_TOKEN / API_HASH / WEB_PASSWORD 被拒)
- ✓ upgrade regex 防 shell injection(`main; rm -rf /` 被拒)
- ✓ upgrade dry_run 双路径(container_git / alpine_container 都识别)
- ✓ tail_logs service 校验(只允许 tg-monitor / tg-web / tg-caddy)
