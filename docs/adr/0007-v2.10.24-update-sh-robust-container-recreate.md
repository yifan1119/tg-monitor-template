# ADR-0007: `update.sh` orphan cleanup 放宽 + 容器缺失检测

**版本**:v2.10.24
**日期**:2026-04-21

## 背景

**问题 1:容器名冲突(客户反馈)**

某客户升级时遇到:
```
Error response from daemon: Conflict. The container name "/tg-monitor-<部门>" is already
in use by container "ced4d19beb29165...". You have to remove (or rename) that container.
```

查 `update.sh` v2.10.22 的 orphan cleanup 逻辑:
```bash
for c in $ORPHANS; do
    proj=$(docker inspect -f '{{ index .Config.Labels "com.docker.compose.project" }}' "$c" ...)
    if [ "$proj" != "tg-${COMPANY_NAME}" ]; then    # ← 只清 label 不一致的
        docker rm -f "$c"
    fi
done
```

**只清 label 跟当前 project 不一致的容器**。但如果容器 label **正好匹配**(`tg-<部门>`),但容器处于某种异常状态(上次升级中断残留、compose 状态文件跟实际不一致、Removed 但占用着名字),update.sh 跳过清理 → `docker compose up --build` 走到创建步骤撞「already in use」→ 升级失败。

`install.sh` v2.10.20 已经放宽了这个判断(同名即清),但 `update.sh` 漏同步。

**问题 2:容器缺失被「已是最新版」误导**

同一个客户升级失败后,尝试手动 `docker rm -f tg-monitor-<部门> tg-web-<部门>` 清理,然后再跑 `./update.sh`:

```bash
$ ./update.sh
📥 检查远端版本...
ℹ 当前已是最新版 (5e0718a),无需升级
  如需强制重建容器: docker compose -p tg-<部门> up -d --build
```

`update.sh` 看到本地 SHA == 远端 SHA,直接 exit 0 返回,**没检查容器是否存在**。客户的两个容器已经被手动删掉了,但 update.sh 不管,让客户自己跑底下那条 `docker compose up -d --build` 才能救回来。

**客户不是技术背景,容易漏跑那条命令 → 监控服务长时间宕机**。

## 决策

### 改动 1:orphan cleanup 无条件清(跟 install.sh 对齐)

```bash
# v2.10.24: 不再检查 compose project label
ORPHANS=$(docker ps -a --format '{{.Names}}' \
    | grep -E "^tg-(monitor|web)-${COMPANY_NAME}$" || true)
for c in $ORPHANS; do
    proj=$(docker inspect ... 2>/dev/null || echo "")
    echo "  🧹 清理同名容器: $c (compose project=\"$proj\")"
    docker rm -f "$c" >/dev/null 2>&1 || true
done
```

**关键点**:
- 不检查 `proj` 值(任何 label 都清)
- 只清 `tg-monitor` 和 `tg-web` 两个名字 — **不碰 `tg-caddy`**
  - Caddy 是 profile 服务,v2.10.22 末端的 HTTPS 保护块依赖 `tg-caddy-<部门>` 容器**存在**
  - 清了 Caddy 容器 = 破坏 HTTPS 自恢复机制,v2.10.22 保护块检测不到容器,不会拉起
  - 风险 > 收益,不清

### 改动 2:容器缺失检测

```bash
if [ "$OLD_SHA" = "$REMOTE_SHA" ]; then
    MISSING=""
    for c in "tg-monitor-${COMPANY_NAME}" "tg-web-${COMPANY_NAME}"; do
        if ! docker ps -a --format '{{.Names}}' | grep -q "^${c}$"; then
            MISSING="${MISSING} ${c}"
        fi
    done

    if [ -z "$MISSING" ]; then
        # 代码最新 + 容器都在 → 正常退出
        echo "ℹ 当前已是最新版 (${OLD_SHORT}),所有容器存在,无需升级"
        exit 0
    else
        # 代码最新 + 容器缺失 → 跳过 pull 但继续重建
        echo "ℹ 当前代码已是最新 (${OLD_SHORT}),但检测到容器缺失:${MISSING}"
        echo "  跳过 git pull,继续重建容器..."
        NEW_SHA="$OLD_SHA"
        NEW_SHORT="$OLD_SHORT"
        SKIP_CODE_PULL=1
    fi
fi

if [ -z "$SKIP_CODE_PULL" ]; then
    # ... 原有 stash + git reset 流程 ...
fi
```

`SKIP_CODE_PULL` 让后续的 `.env migrate` + orphan cleanup + `docker compose up --build` + 健康检查 + HTTPS 保护块都**正常执行**,只跳过代码拉取(因为代码本来就是最新)。

### 改动 3:不碰 tg-caddy(安全保障)

如前所述,不清 caddy 容器,让 v2.10.22 的 HTTPS 保护块照旧工作:
- caddy 在跑 → 不碰
- caddy 挂了 → 保护块拉起
- caddy 不存在 → 保护块检测不到,不会重建(此时客户需手动跑 `compose --profile https up -d caddy`)

## 原因

### 为什么不 force 把 caddy 也纳入 orphan cleanup?

**风险**:
1. 升级过程中清掉 caddy → 保护块检测不到「容器存在」→ 不会自动重建
2. 客户 HTTPS 直接挂,不知道怎么救

**替代方案**:在 orphan cleanup 同时 `--profile https up -d caddy` 保证重建。但这会:
- 对没装 HTTPS 的客户创建不必要的 caddy 容器(污染)
- 违反「只在需要时启用 profile」原则

**结论**:不清 caddy 更安全。客户遇到 caddy 撞冲突的概率极低(caddy 独占 80/443 端口,名字冲突场景罕见)。

### 为什么 MISSING 只检查 monitor + web,不检查 caddy?

- caddy 是 profile 服务,不是必须存在 — 没装 HTTPS 的客户本来就没这容器
- 检查 caddy 会误判「缺失」,触发不必要的 rebuild

### 为什么不直接 `docker compose up -d --build` 跳过所有检查?

太粗暴:
- 忽略 orphan cleanup → 还是会撞 already in use
- 忽略健康检查 → 失败不会自动回滚
- 忽略 Caddy 保护 → HTTPS 可能不恢复

维持 update.sh 全流程,只是加一个「容器缺失也触发重建」的入口。

## 来源

1. 客户升级时遇到 container name already in use(2026-04-21)
2. 客户手动 `docker rm -f` 后被 update.sh「已是最新版」误导
3. 通过 grep + Read + 实际截图分析定位到 update.sh 两个问题
4. 按新全局规则「回答用户问题必须先查代码」(ADR 生成依据)

## 后果

### 立即效果
- 客户升级再也不会撞 container name already in use(同名即清)
- 客户误删容器后跑 update.sh 能自动修复(容器缺失检测)
- Caddy 和 HTTPS 保护逻辑不受影响

### 可能的副作用
- orphan cleanup 现在更激进 — 如果客户手动跑着其他 project 叫 tg-monitor-<部门> 的容器(极罕见),会被误清。但 `tg-monitor-<部门>` 这个命名本来就是本系统独占,不应该被其他 project 占用
- MISSING 检测增加一次 `docker ps -a` 调用(~50ms),可忽略

### 相关代码
- `update.sh:59-93`(容器缺失检测 + SKIP_CODE_PULL 逻辑)
- `update.sh:95-123`(stash + git reset 包在 `if [ -z "$SKIP_CODE_PULL" ]` 里)
- `update.sh:148-162`(orphan cleanup 放宽 + 不碰 caddy)

### 回滚
`bash rollback.sh` 回 v2.10.23 — 但 rollback.sh 也走 git reset + compose up --build 流程,同样会遇到容器冲突。建议回滚前先手动清老容器。
