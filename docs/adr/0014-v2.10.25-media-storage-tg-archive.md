# ADR-0014: 媒体存储切换 — 从 Google Drive 改为 TG 档案群(feature flag)

## 背景

线上客户反馈:外事号会偶尔收到违规内容(图片 / 文件)。原架构下:
1. Telethon 监听到媒体
2. `media_uploader.upload_media` 下载到内存 → 上传到客户的 Google Drive
3. Sheet 单元格写 `=HYPERLINK(drive_url, "📎 filename")`

违规内容上传后 Google 侧会扫描到 → 客户的 Google 账号被冻结 → 整套系统连带挂掉(Sheets 写不进、Drive 403、OAuth 失效)。

**客户说法**:「有些用户发的档案或是图片 有不当内容 导致有人的谷歌帐户被冻结」。

原方案的核心问题:**客户个人 Google 账号承担了内容审查风险**,一旦某条消息被判违规,整个业务系统瘫痪。

## 决策

v2.10.25 引入 `MEDIA_STORAGE_MODE` feature flag,取值:
- `drive`(默认,向后兼容) — 保持 v2.10.24 原逻辑,上传到 Drive
- `tg_archive` — Bot 转发到 `MEDIA_ARCHIVE_GROUP_ID` 指定的 TG 群,Sheet 写 `=HYPERLINK("t.me/c/...", "图片 #N")` 跳转到档案群对应消息
- `off` — 不处理媒体,Sheet 只显示文字占位(`[图片]` / `[文件]` 等)

### tg_archive 模式行为

**转发范围(v2.10.25 测试期迭代后)**:
- photo → `send_photo`(10MB 以内);超过降级 `send_document`
- file(document)→ `send_document`(50MB 硬限)
- voice → `send_voice`(发送失败降级 `send_document`,仍保留 .ogg 文件可播放)
- video / sticker → **不转发**,Sheet 仅显示 `[视频]` / `[贴纸]` 占位

**决策演进**:
- 首版设计(按用户首次要求):只转 photo + file,voice / video / sticker 全部保留文字占位
- 测试期(同一次 PR 内):用户补充「语音也转吧」→ 语音加进转发范围
- video / sticker 仍保留文字占位 — 视频单条常 > 50MB(Bot API 硬限)触发失败率高、且客户场景审查图 + 文 + 语音足够;贴纸转档案群噪声大且无审查价值

**Sheet 单元格格式**:
- `=HYPERLINK("https://t.me/c/{gid}/{archive_msg_id}", "图片 #42")`
- `=HYPERLINK("https://t.me/c/{gid}/{archive_msg_id}", "文件 #42")`
- `=HYPERLINK("https://t.me/c/{gid}/{archive_msg_id}", "语音 #42")`
- 不带文件名、不带方向箭头(用户明确要求「不带名」「不需要标方向」)

**档案群消息 caption 格式**:
```
【文件提醒】
中心/部门:{company}
{OPERATOR_LABEL}:{operator}
外事号:{account.name}
{PEER_ROLE_LABEL}:{peer_name}
文件编号:#{media_seq}
文件内容:图片              (photo)
文件内容:文件:xxx.pdf     (file with name)
文件内容:文件              (file without name)
文件内容:语音              (voice)
```

**编号规则**:
- `media_seq` 单账号内单调递增
- 每账号独立序号,跨账号不互相影响
- **原子分配**(Codex P1 round1 修复):用 `account_seq` 专用计数表 + `BEGIN IMMEDIATE` 事务,
  避免 `MAX+1` 方式在 `_handle_message` / `_backfill_peer` / `pull_history` 三路并发 coroutine
  对同账号交错时读到同样最大值 → 派出重复编号
- 转发失败时 seq 已分配但不回收(允许跳号,`#41` → `#43`),接受代价换简化逻辑

### DB schema 变动(migration 4)

```sql
ALTER TABLE messages ADD COLUMN media_seq INTEGER DEFAULT 0;
ALTER TABLE messages ADD COLUMN archive_msg_id INTEGER DEFAULT 0;
CREATE TABLE account_seq (
    account_id INTEGER PRIMARY KEY,
    media_seq INTEGER NOT NULL DEFAULT 0
);
```
- 历史行 `media_seq=0 / archive_msg_id=0`
- `drive` / `off` 模式下不写这两列,保持 0
- `tg_archive` 模式下 `insert_message(media_seq=..., archive_msg_id=...)` 传入
- `account_seq` 表用 `INSERT OR IGNORE ... VALUES (account_id, 0)` + `UPDATE media_seq = media_seq + 1 RETURNING ...`
  原子递增,跨协程安全

### Bot 注入策略

`media_uploader._archive_bot` 模块级变量,`main.py` 在创建 `AlertBot` 后调用 `media_uploader.set_archive_bot(alert_bot.bot)` 注入 aiogram `Bot` 实例。

`drive` / `off` 模式下 `_archive_bot` 不被使用,注入无副作用。

### Bot API 限制处理

- Telegram Bot API:`send_photo` 10MB / `send_document` 50MB
- Photo > 10MB → 降级用 `send_document`(档案群里显示为文件不是缩图,但不丢失)
- 文件 > 50MB → 跳过转发,Sheet 回落到 `[文件]` 占位
- 客户可配的 `MEDIA_MAX_MB`(原有)仍生效,叠加保护

### `MEDIA_ARCHIVE_GROUP_ID` 格式校验(Codex P1 round1 修复)

`is_tg_archive_enabled()` 除了检查模式 / bot 实例,还强制要求 `MEDIA_ARCHIVE_GROUP_ID` 是 supergroup/channel 格式:负数且 `abs(gid)` 以 `100` 开头(对应 `-100xxxxxxxxxx`)。

未通过格式校验时:
1. 首次 warn 一次(不 spam),后续静默
2. `is_tg_archive_enabled()` 返回 False → 媒体 fallback 到文字占位

防的是:
- 客户误填 `+1234567890`(正数 user ID)→ bot 把媒体 DM 发到那个用户(敏感数据泄漏)
- 客户误填普通 group ID(负数但不以 -100 开头)→ t.me/c 深链 404
- 环境变量里填字符串 / 乱码 → Python 层失败回 0 → 同第一项拦截

### `tasks.py` 清理循环 gate

`_media_cleanup_loop` 仅在 `MEDIA_STORAGE_MODE=drive` 时跑 `cleanup_old_media`。非 drive 模式下 Drive 根本没存新东西,清理会空跑浪费 API 额度。

### 三个 listener 路径统一分发

`_handle_message` / `_backfill_peer` / `pull_history` 三个媒体处理点原本都直调 `media_uploader.upload_media()`。v2.10.25 抽成 `Listener._resolve_media_display(msg, account_id, media_type, peer_name, text_fallback)` 统一分发,返回 `(text, media_seq, archive_msg_id)`。

### 本版范围 — backend + flag + UI(UI 在测试过程中拉进 v2.10.25)

v2.10.25:
- ✅ backend 改造 + feature flag + DB migration
- ✅ `MEDIA_STORAGE_MODE=drive` 老客户升级无感
- ✅ `templates/setup.html` 加模式下拉选择器(Drive / TG 档案群 / 关闭)
- ✅ `MEDIA_ARCHIVE_GROUP_ID` 输入框条件显示(仅 tg_archive 模式下可见)
- ✅ 模式切换时 Drive 相关字段(folder ID、retention days)自动隐藏
- ✅ web.py 保存时对 mode 值做枚举校验(非法值回退 drive)

v2.10.26(后续版本):
- 🔜 档案群 ID 自动识别(让客户把 bot 加群 + 发 /bind_archive 自动回填)
- 🔜 切换后的迁移向导
- 🔜 档案群 preflight 检查(bot 在群里 + 有发送权限)

## 原因

**为什么不继续用 Drive?**
- 风险集中在单个 Google 账号,一条违规消息可连带击穿整个业务系统(Sheets + Drive 一锅端)
- 客户不掌控 Google 风控,申诉成本高、恢复时间不可控

**为什么选 TG 档案群而不是其他云?**
- Bot 天然在生态里,零新依赖
- 档案群不怕违规内容冻结(TG 对私群档案容忍度远高于 Google Drive)
- 部署成本为零(建群 → bot 加群 → 填 ID)
- 即便极端情况下档案群被封,业务系统(Sheets)仍能正常写入 — 风险不再合流

**为什么用 bot 而不是 user client 转发?**
- User client 已经在监听消息,再用它转发要给每个账号的 session 加额外动作 → 200+ 账号全量放大
- Bot 单独一份,简洁、可观测、额度隔离
- 用户明确要求:「不能透过 BOT 转发吗」

**为什么默认保持 drive?**
- 老客户升级无感是硬约束(CLAUDE.md 明文规定)
- 强制迁移会引发信任问题,让客户自己选切换时机

**为什么 video/sticker 不转发(voice 已改成转发)?**
- voice 首版不转 — 用户起初说「只需要转发图片跟文件而已」
- **测试期用户补充「语音也转吧」** → voice 加进转发
- video 仍不转 — 常超过 Bot API 50MB 硬限,失败率高
- sticker 仍不转 — 噪声大、审查价值低
- 业务场景核心是客户审查广告主沟通(图 + 文 + 语音),video / sticker 用得少

**为什么 Sheet 单元格不带文件名?**
- 用户明确要求:「需要超连接 不带名」
- 文件名在档案群 caption 里有(`文件内容:文件:xxx.pdf`),Sheet 保持清爽
- 点链接跳过去就能看到全文件名 + 预览

## 来源

- **客户线上反馈**(2026-04-22):「有些用户发的档案或是图片 有不当内容 导致有人的谷歌帐户被冻结了 想把这区块改成 透过TG转发 而不是继续存在谷歌云端 表单变成原本一开始的只显示文字 图片 文件 但后续接收到的图片文件要给编号」
- **设计约束对话**(2026-04-22):
  - 「不能透过BOT转发吗」→ 确定用 bot(不用 user client)
  - 「只需要转发图片跟文件而已」→ 确定不转语音/视频/贴纸
  - 「不需要标方向 语音影片不需要 SHEET 写入 图片 文件 带编号就好」→ 确定 Sheet label 格式
  - 「需要超连接 不带名」→ 确定 Sheet 超链接不带文件名
  - 「做完 先别直接推」→ 保留给用户审阅后再决定推不推

## 后果

### 正面
- **根治 Google 账号冻结风险**:违规内容不再落 Drive,客户 Google 账号安全
- **风险隔离**:即便档案群出问题,Sheets 业务仍能正常写入(方向箭头 / 文字消息全程不经过档案群)
- **可回滚**:切回 `MEDIA_STORAGE_MODE=drive` 立刻回到旧行为(老消息的 Drive 链接不动)
- **向后兼容**:老客户升级默认 `drive`,0 感知,符合 CLAUDE.md 硬性约束
- **Sheet 更清爽**:编号 + 超链接比长文件名友好,主管一眼扫完

### 副作用 / 需要后续跟进
- **档案群消息 ID 稳定性**:Bot 转发后档案群删除或清理会让链接失效。需要文档提示「档案群不要清消息」
- **Bot API 10/50MB 限制**:超限文件 `send_document` 会失败 → fallback 占位。客户可能遇到大视频/大压缩包,需要在 v2.10.26 UI 上提示
- **档案群成员管理**:bot 必须在群里且有发送权限,切换前要有 preflight 检查(v2.10.26 UI)
- **媒体 seq 不保序**:转发失败时 seq 跳号(客户看「图片 #42」之后可能是「图片 #44」)。接受 — 不追求严格连续,只要单调递增
- **drive → tg_archive 切换时**:老消息 Sheet 里还是 Drive 超链接,不回填(代价大、收益小),新消息才走新路径
- **supergroup ID 格式**:用户填 `MEDIA_ARCHIVE_GROUP_ID=-1001234567890`,代码自动剥离 `-100` 前缀拼 t.me/c 链接。普通群(无 -100 前缀)也支持但主流客户都用 supergroup

- [ ] 写客户 FAQ:档案群不要删消息 / bot 必须在群里

## 教训

**「安全」是风险集中度的函数,不是单点概率**。原架构下,一条违规消息 = 整套系统挂。改成 TG 档案群后,档案群出问题也只影响档案群本身,Sheets 照常写入。**降低风险不是降低某次触发概率,是缩小触发后的爆炸半径**。

Feature flag 默认保守(drive)+ 新选项 opt-in(tg_archive)是 CLAUDE.md 第 6 条「Feature flag 默认关」的标准落地 — 200+ 账号客户宁可错过优化,不能升级就挂。
