# ADR-0023: v3.0.8.1 docker cp 漏同步根治 + 驾驶舱 admin 按钮前端隐藏

## 背景

v3.0.8 发布后客户反馈:**升级了驾驶舱看不到「立刻深度诊断」按钮**。

按规矩查代码 — 不是新代码 bug, 是 CLAUDE.md 硬规定 #8 的老坑:

`docker-compose.yml` 的 `tg-web` 容器启动 command 写的是:
```yaml
command: sh -c "cp -rf /app/repo/*.py /app/repo/templates /app/ 2>/dev/null; python web.py"
```

`cp -rf /app/repo/*.py /app/repo/templates /app/` 这条命令对 templates 是**目录复制**:
- 镜像里 `/app/templates` 目录已经存在 (Dockerfile COPY 进去的)
- `cp -rf` 行为: 目标目录已存在 → 把整个 `/app/repo/templates` 目录复制成嵌套 `/app/templates/templates/dashboard.html`
- Flask 还是读 `/app/templates/dashboard.html` (镜像旧版),**新文件去了 /app/templates/templates/ 没人读**
- 结果: 客户跑 `./update.sh` 镜像 rebuild 了, 镜像里 templates 是新的, 但 cp 把它放到错位置, 客户看到的还是镜像 COPY 时的版本

CLAUDE.md 硬规定 #8 早就写下来 v3.0.x 系列要根治, **每次升级靠人工 docker exec cp 同步太脆**, 经常忘。

---

第二个问题:**v3.0.8 引入的「立刻深度诊断」/「一键修复」/「立刻重启监听器」按钮 — 普通成员能看到但点了 403**。

后端 endpoint 已经收紧 (`@admin_required`), 但**前端按钮无条件渲染**, 任何登录用户都看到。点了:
- 「立刻深度诊断」 → GET `/api/diag/sheets-stuck-detail` 403 → modal 显示加载失败
- 「一键修复」 → POST `/api/diag/sheets-fix-stuck` 403 → 按钮变红
- 「立刻重启监听器」 → POST `/api/restart` 实际是 `@login_required`, 普通成员能成功 (但 v3.0.8.1 起 UI 收紧, 只让 admin 看见)

UX 烂 — 客户看到按钮不点会想 "这是干嘛的", 点了报错会想 "我没权限那为什么给我看"。

## 决策

两件事一次发, **v3.0.8.1**:

### 1. docker-compose.yml command 显式 cp 列表

#### tg-web 改动

```yaml
# 旧 (出 bug 的)
command: sh -c "cp -rf /app/repo/*.py /app/repo/templates /app/ 2>/dev/null; python web.py"

# 新 (v3.0.8.1)
command: sh -c "
  cp -rf /app/repo/*.py /app/ 2>/dev/null;
  cp -rf /app/repo/templates/*.html /app/templates/ 2>/dev/null;
  cp -rf /app/repo/README.md /app/README.md 2>/dev/null;
  cp -rf /app/repo/release_notes.json /app/release_notes.json 2>/dev/null;
  python web.py
"
```

**关键变化**: `templates/*.html` (文件级 glob, 不是目录复制) → 不再嵌套, 文件直接覆盖到 `/app/templates/`。

加 `README.md` (Flask `_app_version_string` 读这个显示登入页版本号) + `release_notes.json` (升级通知文案)。

#### tg-monitor 改动

```yaml
# 旧
command: sh -c "cp -rf /app/repo/*.py /app/ 2>/dev/null; python main.py"

# 新 (v3.0.8.1) — 加 README/release_notes 给 update_checker 用
command: sh -c "
  cp -rf /app/repo/*.py /app/ 2>/dev/null;
  cp -rf /app/repo/release_notes.json /app/release_notes.json 2>/dev/null;
  cp -rf /app/repo/README.md /app/README.md 2>/dev/null;
  python main.py
"
```

tg-monitor 不渲染 UI 不需要 templates。但 `update_checker.py` 推送通知时如果走 `/app/release_notes.json` 路径会读老镜像版本。`update_checker:38` 的 `REPO_ROOT / "release_notes.json"` = `/app/repo/release_notes.json` 走 host bind mount 是 OK 的, 但保险起见同步一份到 `/app/`。

### 2. 驾驶舱 admin 按钮前端隐藏

`web.py::dashboard_page` 加 `is_admin` 上下文:

```python
me = flask_session.get("username", "")
return render_template("dashboard.html", ..., is_admin=is_admin(me))
```

`templates/dashboard.html` 顶部加全局 JS 标志:

```javascript
const IS_ADMIN = {{ 'true' if is_admin else 'false' }};
```

3 处按钮渲染逻辑加 `IS_ADMIN` 检查:
- `sh-deep-diag-wrap` → `display = IS_ADMIN ? 'block' : 'none'`
- `wait_or_restart` 路径 → admin 看「立刻重启监听器」按钮, 普通成员看一行说明文字 "请联系管理员重启监听器"
- `restart_monitor` 路径 → 同上

**重要不动**:
- `reauth` / `check_sheet` 路径(去设置页链接)所有登录用户都能看(自己跳走的事, 设置页本身有权限拦截)
- 后端 `/api/restart` 仍 `@login_required` 不收紧 — 账号管理页历史按钮还在用, 改后端会破坏 v2.x 兼容
- `/api/diag/sheets-stuck-detail` + `/api/diag/sheets-fix-stuck` 维持 v3.0.8 的 `@admin_required`

## 原因

1. **客户反馈优先**:升级了看不到按钮 = 客户体感升级失败, 把这个根治后续所有 templates / docs 改动客户升级自动生效

2. **CLAUDE.md 硬规定 #8 应该执行**:写了好久没动, 这次借势统一改 compose command, 以后客户升级靠 docker compose up -d --build 一条命令搞定, 不用再 docker exec cp。同时把 docs/ 不同步留 next version (本版只 cp templates + README + release_notes 这 3 类客户能看到的 UI 文件)

3. **前端 admin 按钮隐藏比后端拦截重要**:后端拦截是底线, 但前端 UX 才是客户看到的。点了 403 比看不到按钮糟糕 — 看不到按钮普通成员根本不知道这个能力存在 (按设计)。

4. **降级模式**:普通成员看到的诊断卡片仍有「请联系管理员」文字提示, 不是空白, 让他知道发生了什么 + 该找谁

## 来源

- 客户反馈: 「我没看到你说的按钮 我更新了」(2026-04-25)
- 现场诊断: docker-compose.yml `cp -rf templates 目录复制` 形成嵌套, Flask 读旧版
- 用户提问: 「正常的用户看不到按钮是吗」 → 实际现状是普通用户能看到按钮但点了 403, UX 烂
- CLAUDE.md 硬规定 #8 (2026-04 写的): "长期修法 (还没做): 改 docker-compose.yml 的 command 覆盖所有非 state 文件" → 这次根治
- Codex 审阅 P1: `cp ... 2>/dev/null` 静默吞错 → cp 真失败时容器还是启动用旧文件, 复现"升级了但看不到新内容"的同源风险 → 改 `cp ... || echo CP_*_FAIL` 失败 marker 输出到 docker logs, 操作员能立刻看到; 容器仍照常启动 (不 abort) 避免 host bind mount 短暂 race 时无法启动

## 后果

### 正面
- 客户升级 v3.0.8.1 之后, **所有后续 templates / README / release_notes 改动自动生效**, 不用人工 docker exec cp。CLAUDE.md 硬规定 #8 长期修法落地
- 普通成员不再看到 admin 按钮, 不再点了 403 — UX 收紧到 "看不到的能力 = 不存在的能力"
- 普通成员看到诊断卡片 + "请联系管理员" 引导, 知道该找谁

### 潜在问题 / trade-off
- **`docs/` 还没 cp**: ADR 文档暂不在容器里 (web.py 没读), 但下次有需要时也加进去. 留个记号
- **scripts/** 同理: caddy-doctor.sh 等运维脚本都在 host /root/tg-monitor-<部门>/scripts/ 下, 容器里不需要
- **既有 v3.0.8 容器升级到 v3.0.8.1 必经过 `docker compose up -d --build`**: rebuild image + recreate container, 这是 update.sh 标准流程不会破。但要确保 `--build` 真生效, 镜像 hash 变 → 容器 recreate → 新 command 跑新 cp 列表
- **是否破坏其他 cp -rf 用法**: 全文 `cp -rf` 在 .py 上跟之前一样, 只改了 templates 一处行为。grep 验证只 docker-compose.yml 这两处用 cp -rf

### 未涉及 (v3.0.9+)
- 把 `cp` 列表抽成独立 `entrypoint.sh` 脚本 — 命令长可读性差, 留 next version 抽出去
- `docs/` 同步 — ADR 进容器 (Flask 直接 serve `/docs/<n>` 给客户看 ADR 历史) 留 v3.0.9
- `scripts/` 同步 — 仅 host 用不进容器, 不动
- `/api/restart` 后端收紧到 `@admin_required` — 跟 v3.0.8 的 admin 按钮统一 — 但破坏账号管理页老按钮的 v2.x 兼容性, 留 v4.0 主版本号大动时再改

## 升级指引

```bash
cd /root/tg-monitor-<部门> && ./update.sh
```

升级后:
- docker compose up -d --build → image rebuild → command 跑新 cp 列表 → templates / README / release_notes 都同步到 /app/
- 客户打开驾驶舱:
  - **管理员账号** 看到「立刻深度诊断」按钮 (Sheet 卡死时显示)
  - **普通成员** 看不到这些按钮, 看到 "请联系管理员重启监听器" 文字提示
  - reauth / check_sheet 链接所有登录用户都能看 (无变化)

## 验证

### 静态自查 (push 前)
- `python3 -c "import yaml; yaml.safe_load(open('docker-compose.yml'))"` — YAML OK
- `python3 -c "import ast; ast.parse(open('web.py').read())"` — Python syntax OK
- `grep "templates/\\*\\.html" docker-compose.yml` — 必须有 1 处 (web 容器), 0 处 `cp .* templates ` 不带 `*.html` (旧目录复制语法)
- `grep "IS_ADMIN" templates/dashboard.html` — 至少 4 处 (1 定义 + 3 按钮 gate)
- `grep "is_admin=is_admin" web.py` — 1 处 (dashboard_page render_template)
- DIV balance: dashboard.html `<div>:</div>` 平衡

### 集成 (dev 容器)
1. **目录嵌套测试**: 在 dev 容器跑 `ls /app/templates/`, **不能** 出现 `templates/templates/` 嵌套
2. **新模板生效**: 改 dashboard.html 加一行 marker, restart 容器, 浏览器刷新看到 marker
3. **admin 隐藏**: 用普通成员账号登录驾驶舱, **不应** 看到「立刻深度诊断」按钮; 用 admin 账号登录, 应该看到
4. **后端不破**: 普通成员手动 curl POST `/api/diag/sheets-fix-stuck` 仍 403 (后端收紧不动)

### 回归
- v3.0.8 全部功能不变(append API / 令牌桶 / 深度诊断后端 / 一键修复后端)
- v3.0.7 OAuth 自愈不变
- v3.0.6 日志面板不变
- 账号管理页「重启监听」按钮不变(/api/restart `@login_required` 不动)
