# ADR-0006: `sync_headers` 单账号异常隔离(同 flush_pending 逻辑)

**版本**:v2.10.23(追加补丁)
**日期**:2026-04-21

## 背景

客户反馈:在账号分页 A2:B3 填了「渠道人员:林知言」+「中心/部门:渠道中心-鼎丰公司」,
但 TG 预警群推送显示这两栏是空值。

查代码(`sheets.py:sync_headers` v2.10.22 实现):

```python
def sync_headers(self):
    accounts = db.get_conn().execute("SELECT * FROM accounts").fetchall()
    for account in accounts:
        ws = self.get_or_create_sheet(account)
        if not ws:
            continue
        self._rate_limit()
        header_data = ws.get("A2:B3")   # ← 任一账号 API 错就 raise
        ...
```

**外层 `_patrol_loop`** 只有整体 try/except,不能隔离单账号:
```python
try:
    self.sheets.sync_headers()
    ...
except Exception as e:
    logger.error(...)
```

**结果**:如果 accounts 表里某个账号的 Sheet 分页异常(tab 被手改名 / 被删 /
单账号撞 Google 429 等),`ws.get("A2:B3")` 抛异常 → `sync_headers` 中断 →
**排在后面的账号永远不被同步** → 这些账号 DB 里 operator/company 长期为空 → 预警推送空白。

跟 ADR-0001(`flush_pending` 全局 LIMIT 500 单账号失败中断整批)是**完全相同的架构问题**。
ADR-0001 只修了 `flush_pending`,`sync_headers` 没同步修,属于遗漏。

## 决策

跟 ADR-0001 同样的处理手法:**单账号 try/except 隔离**。

拆成两个函数:
- `sync_headers`:外层 for loop,每账号 try/except,失败只打 warning + continue
- `_sync_one_account_headers(account)`:单账号逻辑(原 for 内的代码)

```python
def sync_headers(self):
    accounts = db.get_conn().execute("SELECT * FROM accounts").fetchall()
    for account in accounts:
        try:
            self._sync_one_account_headers(account)
        except Exception as e:
            logger.warning(
                "[sync_headers] 账号 id=%s phone=%s 同步失败,跳过: %s",
                account["id"], account["phone"], e,
            )
            continue
```

单账号失败 → 只跳过该账号 → 后面账号继续同步 → DB 最终保持跟 Sheet B2/B3 一致。

## 原因

### 为什么和 ADR-0001 同时修?
按吴献忠方法论「分批发布 > 大改版」,理想情况下应该单独出 v2.10.24 hotfix。
但这次 ADR-0001 刚推,客户还没升级。为避免客户「升级两次」的体验,把
sync_headers 修复一起并入 v2.10.23 发布。作为追加补丁,不独立 bump 版本。

### 为什么 v2.10.23 原版漏了?
ADR-0001 重点在 `flush_pending` 的消息写入路径(客户直接看到的「表格空白」)。
`sync_headers` 的同账号中断问题,只在「Problem 2」(推送空值)被客户反馈后
才暴露出来 — 这类 bug 需要客户特定使用方式(手动填 B2/B3)才触发。

### 为什么不直接 refactor `_rate_limit` / 退避也加到 sync_headers?
`sync_headers` 调用 Google API 频率较低(每 60s 一次,读取操作),不像 flush 频繁写。
目前不加 429 per-account 退避,只做 try/except 隔离,控制改动范围。
如果未来 sync_headers 也撞限额成为痛点,再按 flush 模式加退避(P2)。

## 来源

客户(渠道中心-鼎丰公司,具体匿名)2026-04-21 反馈:
- 问题 1:关键词分页有记录但账号分页搜不到(已经 ADR-0001 覆盖)
- **问题 2:B2/B3 填了但推送空白** ← 本 ADR 触发

基于新的「回答前必须先查代码」规则(全局 CLAUDE.md),
Grep + Read 发现 `sync_headers` 存在同 ADR-0001 的隔离问题。

## 后果

### 立即效果
- v2.10.23 升级后,即使 accounts 表里有「异常账号」(分页被删/改名/API 错),
  其他账号的 B2/B3 → DB 同步照常工作
- 「Problem 2」场景:60 秒内,"宋锦瑟" 等正常账号的 operator/company 会从 Sheet 同步到 DB
- 下一次预警推送就带上正确值,不再空白

### 没改的
- `sync_headers` 仍然不加 429 per-account 退避(后续迭代)
- 单个对话槽 row6 同步的 try/except 之前就有(l.605),不影响

### 相关代码
- `sheets.py:sync_headers`(改成外层 for + 单账号 try/except)
- `sheets.py:_sync_one_account_headers`(新增,封装原 for 内逻辑)
- `tasks.py:_patrol_loop`(不改,外层 try/except 保留作兜底)
