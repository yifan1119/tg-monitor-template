# ADR-0012: `release_notes.json` 里 `v2.10.24` key 改为四个 hotfix 累计说明(补旧客户展示路径)

## 背景

v2.10.24.4(ADR-0011)修了 `update_checker.py` 的版本号 regex 只匹配三段的 bug。但这是**客户端代码**,修完之后**还没升级的旧客户**仍然跑旧 regex:

```
客户本地 = 72439e0 (v2.10.24.1 或更早)
GitHub HEAD = 1aabfe3 (v2.10.24.4)
客户端 update_checker.check_once():
  fetch remote subject = "v2.10.24.4: ..."
  OLD regex 抓 → "v2.10.24"(截断)
  去 release_notes.json 找 "v2.10.24" key → 旧的「容器冲突修补」说明
```

所以旧客户看后台「版本升级」弹窗时,会一直看到容器冲突修补的说明,而不是真正的 v2.10.24.1~4 新内容。

用户反馈「推完还是一样」— 客户端截图依然显示容器冲突说明,跟预期一致(先有鸡先有蛋,修的是客户端 regex 所以得等客户升上来)。

## 决策

**workaround:改 `release_notes.json` 里 `v2.10.24` key 的 value**,从单一 v2.10.24 容器冲突说明改成 **v2.10.24 + .1 + .2 + .3 + .4 五版累计说明**。

这样:
- **旧客户**(regex 只三段):`v2.10.24.*` 被截成 `v2.10.24` → fetch 最新 release_notes.json → 拿到累计说明 ✓
- **新客户**(已升 v2.10.24.4,regex 扩到任意段):`v2.10.24.5` 精确匹配 → 拿到 `v2.10.24.5` 独立 key 的说明 ✓

同时新增 `v2.10.24.5` key 作为本次补丁本身的说明。

bump 到 v2.10.24.5 + tag,因为:
- commit subject 必须带 `v2.10.24.5` 前缀,旧 regex 才能抓到 `v2.10.24` → 命中新覆盖的 key
- 如果用 `docs:` prefix,旧 regex 抓不到版本号 → 进入「详情见 GitHub 提交记录」兜底文案,还是不对

## 原因

**为什么覆盖历史 key 而不是其他方案?**

其他方案权衡过:
1. **加 short_sha override key**:需要先知道 commit sha,再加到同一 commit → 鸡蛋问题;amend + force push 会违反「Never force push」规则
2. **把 v2.10.24 key 改名 / 加 alias**:`_notes_for` 没有 alias 机制,改代码又回到鸡蛋问题
3. **等客户升上来再说**:客户本次升级看错说明成为永久历史,体验差
4. **覆盖 v2.10.24 key**(采用):零代码改动,纯 JSON 文案,风险最低,覆盖全部旧客户场景

**覆盖 v2.10.24 key 的代价 vs 收益**:
- **代价**:`v2.10.24` release note 不再是单一版本的原始说明
- **收益**:所有旧客户(还没升到 v2.10.24.4)都能看到完整累计说明 — 比单一容器冲突说明信息量大得多

**代价可接受,因为**:
- `release_notes.json` 是**客户端 UI 文案**,不是档案
- 真实历史在 `ADR-0007`、`commit d087c9d`、`README.md` 版本列表 — 这三处都保留不动
- `v2.10.24` key 在客户升到 v2.10.24.4 之后就再也不会被读取(新 regex 抓 `v2.10.24.5` 精确匹配)

## 来源

- **用户反馈**:2026-04-22 晚,用户看到截图 `72439e0 → 1aabfe3` 但弹窗仍是 v2.10.24 的容器冲突说明。提问「推完还没更新的也能看到正确资讯吗?」,确认了这条 workaround 路径有效。

## 后果

**正面**:
- 旧客户下次 `update_checker` 巡检(6h 自动 / 手动点「检查更新」)会 fetch 最新 `release_notes.json` → 看到累计说明
- 新客户精确匹配 `v2.10.24.5` 拿到本 patch 说明
- 客户现在打开后台就能看到正确内容,不需要等下一版

**副作用**:
- `v2.10.24` release note 不再是单一版本说明 — 如果将来有人从 release_notes.json 查 v2.10.24 单独的白话说明会拿到累计版。但 `release_notes.json` 本身就是 UI 文案,不是档案,不应该用来查单一版本历史
- 未来发新的四段版本 hotfix(比如 v2.10.25.1)同样会被旧客户 regex 截成 `v2.10.25` → 需要届时同步更新 `v2.10.25` key 内容(如果还有在跑 v2.10.24.3 以下的旧客户)
- 一旦所有客户都升到 v2.10.24.4+,这个 workaround 就不需要维护了

**需要后续**:
- 下次发新版本前记得考虑:是否还有客户跑 < v2.10.24.4 的?有的话 `release_notes.json` 的「兜底」key(被三段 regex 命中的版本号)要同步更新
- 统计客户升级到 v2.10.24.4 的比例,等 100% 升上后可以把 `v2.10.24` key 还原到原单一说明(如果需要)

## 教训

客户端代码里的 display-layer bug 一修要考虑:**修了之后还没升级的客户仍然受旧 bug 影响**。对于这种情况:
1. 修 bug 本身(v2.10.24.4)
2. 同时想办法让数据层 / 配置层能绕过 bug(v2.10.24.5 覆盖 key)
3. 两者同时发,客户体验最佳

这次走了两步(先 .4 修 regex 后 .5 补文案),未来可以一步到位。
